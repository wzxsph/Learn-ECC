# 05 - Rules 编写

学习如何编写定制化的开发规则。

## 概述

ECC [Rules](../../参考文档/rules/代码风格.md) 是一套强制性开发规则，涵盖代码风格、测试、安全、Git 工作流、代理编排和性能优化等方面。这些规则确保代码质量和一致性，通过自动化检查和审查流程实施。

---

## 1. Rules 格式

### 1.1 基本格式

Rules 使用 Markdown 格式，包含 YAML frontmatter：

```yaml
---
name: rule-name
description: 规则描述
category: 分类
---
```

### 1.2 规则层级

| 层级 | 说明 | 优先级 |
|------|------|--------|
| **全局规则** | `~/.claude/rules/` | 最高 |
| **项目规则** | 项目 `.claude/rules/` | 中 |
| **本地规则** | 当前会话 | 最低 |

### 1.3 优先级

全局规则 > 项目规则 > 本地规则

## 2. 规则类型详解

### 2.1 代码风格规则

定义代码组织的核心原则和最佳实践。

#### 核心要求

**不可变性（关键原则）**

```typescript
// 错误示例 - 修改原对象
function modify(user, name) {
  user.name = name;  // 改变了原对象
  return user;
}

// 正确示例 - 返回新对象
function update(user, name) {
  return { ...user, name };  // 创建新副本
}
```

**核心设计原则**

| 原则 | 说明 | 实践 |
|------|------|------|
| **KISS** | Keep It Simple | 首选最简单的可行方案 |
| **DRY** | Don't Repeat Yourself | 避免复制粘贴导致的实现漂移 |
| **YAGNI** | You Aren't Gonna Need It | 在真正需要时重构 |

**命名约定**

| 类型 | 约定 | 示例 |
|------|------|------|
| 变量和函数 | camelCase | `calculateTotal` |
| 布尔值 | is/has/should/can 前缀 | `isActive` |
| 接口、类型、组件 | PascalCase | `UserProfile` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |

### 2.2 测试规则

确保代码质量的强制性测试标准。

#### 最低测试覆盖率：80%

| 测试类型 | 说明 |
|----------|------|
| 单元测试 | 独立函数、工具类、组件 |
| 集成测试 | API 端点、数据库操作 |
| E2E 测试 | 关键用户流程 |

#### TDD 工作流

```
1. 编写测试 (RED)   - 先写一个会失败的测试
2. 运行测试          - 验证测试失败
3. 编写最小实现 (GREEN) - 编写通过测试的最小代码
4. 运行测试          - 验证测试通过
5. 重构 (IMPROVE)    - 改进代码结构
6. 验证覆盖率        - 确保达到 80%+
```

### 2.3 安全规则

强制性安全检查清单。

#### 提交前必须检查

- [ ] 无硬编码凭证（API keys、密码、tokens）
- [ ] 所有用户输入验证
- [ ] SQL 注入防护（参数化查询）
- [ ] XSS 防护（输出转义）
- [ ] CSRF 保护启用
- [ ] 认证/授权验证
- [ ] 速率限制
- [ ] 错误消息不泄露敏感数据

#### 密钥管理

- **绝不** 在源代码中硬编码密钥
- **始终** 使用环境变量或密钥管理器
- 启动时验证必需的密钥
- 轮换任何可能暴露的密钥

### 2.4 Git 工作流规则

Git 提交规范和 PR 工作流。

#### 提交信息格式

```
<type>: <description>

<optional body>
```

**Type 类型**：
- `feat`: 新功能
- `fix`: 错误修复
- `refactor`: 重构
- `docs`: 文档
- `test`: 测试
- `chore`: 维护
- `perf`: 性能
- `ci`: CI/CD

#### PR 工作流

1. 分析完整提交历史
2. 使用 `git diff [base-branch]...HEAD` 查看所有变更
3. 起草综合性 PR 总结
4. 包含测试计划和 TODOs
5. 如是新分支，使用 `-u` 标志推送

### 2.5 代理编排规则

多 Agent 协作模式。

#### 立即使用原则

| 场景 | Agent |
|------|-------|
| 复杂功能请求 | **planner** |
| 代码刚编写/修改 | **code-reviewer** |
| Bug 修复或新功能 | **tdd-guide** |
| 架构决策 | **architect** |

#### 并行任务执行

```markdown
# 好：并行执行
启动 3 个 Agent 并行：
1. Agent 1: 认证模块安全分析
2. Agent 2: 缓存系统性能审查
3. Agent 3: 工具类型检查

# 坏：不必要的顺序执行
先 agent 1，然后 agent 2，然后 agent 3
```

### 2.6 性能优化规则

模型选择和上下文管理。

#### 模型选择策略

| 模型 | 用途 | 成本 |
|------|------|------|
| **Haiku 4.5** | 轻量 Agent、频繁调用、配对编程 | 低 |
| **Sonnet 4.6** | 主要开发工作、多 Agent 编排 | 中 |
| **Opus 4.5** | 复杂架构决策、深度推理 | 高 |

#### 上下文窗口管理

避免在最后 20% 上下文窗口进行：
- 大规模重构
- 跨多文件的特性实现
- 复杂交互调试

## 3. 规则编写

### 3.1 触发条件

```yaml
---
name: no-console-log
description: 阻止提交包含 console.log 的代码
trigger:
  - pre-commit
  - pre-push
---
```

### 3.2 检查逻辑

```markdown
## 检查逻辑

1. 扫描修改的文件
2. 查找 `console.log`、`console.error` 等
3. 如果找到，阻止提交并显示警告
```

### 3.3 修复建议

```markdown
## 修复建议

- 使用结构化日志库（如 winston、pino）
- 配置日志级别（debug、info、warn、error）
- 确保生产环境日志级别合适
```

## 4. 违规处理

### 4.1 严重程度分级

| 级别 | 含义 | 操作 |
|------|------|------|
| CRITICAL | 安全漏洞或数据丢失风险 | **阻止** - 合并前必须修复 |
| HIGH | Bug 或重大质量问题 | **警告** - 合并前应修复 |
| MEDIUM | 可维护性问题 | **提示** - 建议修复 |
| LOW | 样式或小建议 | **备注** - 可选 |

### 4.2 修复流程

```
1. 分析违规原因
2. 确定修复方案
3. 执行修复
4. 验证修复
5. 重新提交审查
```

## 5. 代码质量检查清单

完成工作前必须检查：

- [ ] 代码可读且命名良好
- [ ] 函数短小（<50 行）
- [ ] 文件专注（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 适当的错误处理
- [ ] 无硬编码值（使用常量或配置）
- [ ] 使用不可变模式（无 mutation）
- [ ] 测试覆盖率 ≥80%
- [ ] 所有安全检查通过

## 6. 创建项目特定规则

### 6.1 项目规则位置

```
项目/
└── .claude/
    └── rules/
        ├── README.md
        └── 自定义规则文件
```

### 6.2 继承全局规则

项目规则继承全局规则，可在此基础上扩展：

```markdown
# 项目特定规则

继承自全局规则，增加以下要求：

## 额外要求

- [ ] 项目特定的编码规范
- [ ] 特定的测试要求
- [ ] 特定的文档要求
```
