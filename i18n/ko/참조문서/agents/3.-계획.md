# 계획類 Agent

계획類 Agent는 기능 계획, 기술 아키텍처 설계 및实施 단계 분해를 전문으로 합니다.

## Agent 목록

| Agent 이름 | 용도 | 사용 모델 | 코어 도구 |
|------------|------|----------|----------|
| planner | 복잡한 기능实施 계획 전문가 | opus | Read, Grep, Glob |
| code-architect | 기능 아키텍처 설계 전문가 | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | PR 테스트 커버리지 분석 | sonnet | Read, Grep, Glob, Bash |
| type-design-analyzer | 유형 설계 분석 | sonnet | Read, Grep, Glob |
| comment-analyzer | 코드 주석 분석 | sonnet | Read, Grep, Glob |

---

## planner

### 이름과 용도
복잡한 기능实施 및 리팩토링의 계획 전문가입니다. 사용자가 기능实施, 아키텍처 변경 또는 복잡한 리팩토링을 요청할 때 주도적으로 사용합니다.

### 능력 설명
- 요구 사항 분석 및 상세实施计划 생성
- 복잡한 기능을 관리 가능한 단계로 분해
- 의존성 및 잠재적 위험 식별
- 최적实施 순서 권장
-_edge情况和 오류 시나리오 고려

### 적용 시나리오
- 새 기능实施 계획
- 아키텍처 변경 계획
- 복잡한 리팩토링 계획
- 기술 부채 정리 계획

### 사용 도구 목록
- Read: existing 코드 및 문서 읽기
- Grep: 관련 패턴 및 existing 구현 검색
- Glob: 관련 파일 찾기

### 다른 Agent와의 협력 방식
- 생성된 계획을 code-architect가進一步细化
- 생성된 계획을 tdd-guide가 테스트 작성에 사용
- 생성된 계획을 code-reviewer가 리뷰

### 계획 흐름

1. **요구 사항 분석**
   - 기능 요청完全 이해
   - 필요한 경우澄清 문제 提出
   - 성공 표준 식별
   - 가정 및 제약 조건 나열

2. **아키텍처 리뷰**
   - existing 코드베이스 구조 분석
   - 영향받는 컴포넌트 식별
   -類似 구현 리뷰
   - 재사용 가능한 패턴 고려

3. **단계 분해**
   - 상세 단계 생성, 포함:
     - 명확하고 구체적인 작업
     - 파일 경로 및 위치
     - 단계 간 의존성
     - 예상 복잡도
     - 잠재적 위험

4. **实施 순서**
   - 의존성별로 정렬
   - 관련된 변경 그룹화
   - 컨텍스트 전환 최소화
   - 증분 테스트 지원

### 계획 출력 형식

```markdown
#实施计划: [기능 이름]

## 개요
[2-3문장으로 요약]

## 요구 사항
- [요구 사항 1]
- [요구 사항 2]

## 아키텍처 변경
- [변경 1: 파일 경로 및 설명]
- [변경 2: 파일 경로 및 설명]

##实施 단계

### 단계 1: [단계 이름]
1. **[단계 이름]** (파일: path/to/file.ts)
   - 작업: 취할 구체적 작업
   - 이유: 이 단계의 이유
   - 의존성: 없음 / 단계 X 필요
   - 위험: 낮음/중간/높음

2. **[단계 이름]** (파일: path/to/file.ts)
   ...

### 단계 2: [단계 이름]
...

## 테스트 전략
- 단위 테스트: [테스트할 파일]
- 통합 테스트: [테스트할 흐름]
- E2E 테스트: [테스트할 사용자 여정]

## 위험과 완화
- **위험**: [설명]
  - 완화: [처리 방법]

## 성공 표준
- [ ] 표준 1
- [ ] 표준 2
```

## code-architect

### 이름과 용도
existing 코드베이스에 대한深入理解를 기반으로 기능 아키텍처를 설계합니다. 구체적인 파일, 인터페이스, 데이터 흐름 및 빌드 순서의实施 블루프린트를 제공합니다.

### 능력 설명
- existing 코드 조직 및 명명 규칙 연구
- existing 아키텍처 패턴 식별
- 테스트 패턴 및 existing 경계 기록
- 새 추상화를 提出하기 전에 의존성 그래프 이해
- 현재 패턴에 적합한 기능 아키텍처 설계

### 적용 시나리오
- 새 기능 아키텍처 설계
- existing 기능 리팩토링 설계
- 코드 조직 최적화

### 사용 도구 목록
- Read: existing 코드 읽기
- Grep: 패턴 및 규칙 검색
- Glob: 관련 파일 찾기
- Bash: 진단 명령 실행

### 다른 Agent와의 협력 방식
- planner의 계획 기반으로 아키텍처 설계
- 코드 구현을 위한 블루프린트 제공
- build-error-resolver와 협력하여 빌드 문제 수정

### 설계 흐름

1. **패턴 분석**
   - existing 코드 조직 및 명명 규칙 연구
   - existing 아키텍처 패턴 식별
   - 테스트 패턴 및 existing 경계 주의
   - 새 추상화를 提出하기 전에 의존성 이해

2. **아키텍처 설계**
   - 현재 패턴에 자연스럽게 맞는 기능 설계
   - 요구를 충족하는 가장 간단한 아키텍처 선택
   -投机성 추상화 피하기 (코드베이스가 이미 사용하지 않는 한)

3. **实施 블루프린트**
   각 중요 컴포넌트에 대해 제공:
   - 파일 경로
   - 목적
   - 핵심 인터페이스
   - 의존성
   - 데이터 흐름 역할

4. **빌드 순서**
   의존성별로 정렬实施:
   1. 유형 및 인터페이스
   2. 코어 로직
   3. 통합 층
   4. UI
   5. 테스트
   6. 문서

### 출력 형식

```markdown
## 아키텍처: [기능 이름]

### 설계 결정
- 결정 1: [이유]
- 결정 2: [이유]

### 생성할 파일
| 파일 | 목적 | 우선순위 |
|------|------|--------|
| | | |

### 수정할 파일
| 파일 | 변경 | 우선순위 |
|------|------|--------|
| | | |

### 데이터 흐름
[설명]

### 빌드 순서
1. 단계 1
2. 단계 2
```

## type-design-analyzer

### 이름과 용도
불법 상태가 표현하기 어렵거나 불가능하도록 하는 유형 설계를 분석합니다.

### 능력 설명
- 캡슐화 평가
- 불변성 표현 분석
- 불변성 유용성 평가
- 강제 실행 검증

### 적용 시나리오
- 유형 시스템 설계 리뷰
- API 설계 리뷰
- 영역 모델링 리뷰

### 사용 도구 목록
- Read: 유형 정의 읽기
- Grep: 유형 사용 검색
- Glob: 유형 파일 찾기

### 다른 Agent와의 협력 방식
- code-reviewer와 협력하여 코드 리뷰 수행
- code-architect와 협력하여 아키텍처 설계 수행

### 평가 표준

1. **캡슐화**
   - 내부 세부 사항이 숨겨져 있는지
   - 불변량이 외부에서 위반될 수 있는지

2. **불변성 표현**
   - 유형이业务 규칙을 인코딩하는지
   - 불가능한 상태가 유형 수준에서 차단되는지

3. **불변성 유용성**
   - 이러한 불변량이 실제 버그를 방지하는지
   - 영역과 일치하는지

4. **강제 실행**
   - 불변량이 유형 시스템에 의해 강제执行되는지
   - 간단한 탈출구가 있는지

### 출력 형식

각 리뷰하는 유형에 대해:
- 유형명 및 위치
- 네 차원의 점수
- 전체 평가
- 구체적 개선 권장

## comment-analyzer

### 이름과 용도
코드 주석의 정확성, 완전성, 유지보수성 및 주석 부패 위험을 분석합니다.

### 능력 설명
- 사실 정확성 검증
- 완전성 검사
- 장기 가치 평가
- 오해의 요소 식별

### 적용 시나리오
- 코드 리뷰 시
- 문서 품질 평가 시
- 주석 기술 부채 식별 시

### 사용 도구 목록
- Read: 코드 및 주석 읽기
- Grep: 주석 및 코드 검색
- Glob: 소스 파일 찾기

### 다른 Agent와의 협력 방식
- code-reviewer와 협력하여 코드 리뷰 수행
- doc-updater와 협력하여 문서 업데이트

### 분석 프레임워크

1. **사실 정확성**
   - 주장과 코드 대비 검증
   - 매개변수 및 반환 설명과 구현 대비 검사
   -过时 참조 표시

2. **완전성**
   - 복잡한 로직에 충분한 설명이 있는지 검사
   - 중요한 부작용 및 _edge用例가 기록되었는지 검증
   -公共 API에 충분한 주석이 있는지确保

3. **장기 가치**
   - 코드만 재진술하는 주석 표시
   - 빠르게 부패하는 취약한 주석 식별
   - TODO / FIXME / HACK 부채 발견

4. **오해의 요소**
   - 코드와矛盾하는 주석
   - 제거된 동작에 대한过时 참조
   - 과도한 약속 또는 설명不足 동작

### 출력 형식

심각도별로 그룹화된 조정 발견:
- Inaccurate (불准确)
- Stale (过时)
- Incomplete (불완전)
- Low-value (저가치)
