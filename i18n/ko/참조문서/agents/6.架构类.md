# 아키텍처類 Agent

아키텍처類 Agent 专门负责시스템아키텍처설계、网络아키텍처계획和성능 최적화。

## Agent 목록

| Agent 이름 | 용도 | 사용모델 | 코어도구 |
|------------|------|----------|----------|
| architect | 软件아키텍처专家 | opus | Read, Grep, Glob |
| network-architect | 企业网络아키텍처설계 | sonnet | Read, Grep |
| homelab-architect | 家庭/实验室网络계획 | sonnet | Read, Grep |
| code-explorer | 코드库깊이분석 | sonnet | Read, Grep, Glob |
| performance-optimizer | 성능분석和최적화 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| harness-optimizer | Agent harness 설정최적화 | sonnet | Read, Grep, Glob, Bash, Edit |

## architect

### 이름과 용도
软件아키텍처专家，专注于시스템설계、可확장性和기술决策。在계획新기능、리팩토링大型시스템或做出아키텍처决策时主动사용。

### 능력 설명
- 为新기능설계시스템아키텍처
- 评估기술权衡
- 권장패턴和모범 사례
- 识别可확장性瓶颈
- 为未来增长계획
- 确保코드库一致性

### 적용 시나리오
- 新기능아키텍처설계
- 大型시스템리팩토링
- 아키텍처决策制定
- 기술选型决策
- 확장性계획

### 사용 도구 목록
- Read: 읽기现있음아키텍처
- Grep: 搜索패턴
- Glob: 찾기파일

### 다른 Agent와의 협력 방식
- planner 基于아키텍처설계생성实施계획
- code-architect 为特定기능설계아키텍처
- build-error-resolver 处理빌드문제
- code-reviewer 리뷰코드 품질

### 아키텍처决策기록 (ADRs)

对于重要아키텍처决策，생성 ADR：

```markdown
# ADR-001: Use Redis for Semantic Search Vector Storage

## Context
필요存储和查询 1536 维 embeddings 用于语义市场搜索。

## Decision
사용带있음向量搜索기능的 Redis Stack。

## Consequences

### Positive
- 快速向量相似性搜索 (<10ms)
- 内置 KNN 算法
- 简单배포
- 良好성능高达 100K 向量

### Negative
- 메모리存储（大데이터集昂贵）
- 없음클러스터时单点故障
- 仅限于余弦相似度

### Alternatives Considered
- **PostgreSQL pgvector**: 较慢，但持久存储
- **Pinecone**: 托管서비스，成本较高
- **Weaviate**: 更多기능，更复杂설정

## Status
Accepted

## Date
2025-01-15
```

## network-architect

### 이름과 용도
从需求설계企业或多站点网络아키텍처。사용现있음网络技能进行专注的라우팅、검증、자동化和故障排除。

### 능력 설명
- 校园、브랜치、WAN、데이터中心、云邻近和混合网络계획
- IP 寻址、分段、라우팅域、管理平面访问
- 冗余、모니터링和마이그레이션排序
- 仅설계和리뷰，不应用설정

### 적용 시나리오
- 企业网络설계
- 多站点网络아키텍처
- 데이터中心网络계획
- 云网络통합

### 사용 도구 목록
- Read: 읽기需求和现있음설정
- Grep: 搜索网络패턴

### 다른 Agent와의 협력 방식
- network-config-validation 검증预变更설정
- network-bgp-diagnostics 进行 BGP 진단
- network-interface-health 进行인터페이스健康분석
- cisco-ios-patterns 提供 IOS/IOS-XE 语法
- netmiko-ssh-automation 进行网络자동化

### 워크플로우

1. 重述目标、约束和非目标
2. 识别显著改变아키텍처的缺失需求
3. 选择拓扑并解释原因
4. 在讨论硬件之前설계라우팅和分段
5. 定义管理平面、로그、모니터링、백업和롤백모델
6. 生成보유검증门和롤백点的分단계实施계획
7. 列出残余风险和仍需从运营商가져오기的证据

### 설계기본值

- 优先라우팅边界而非확장 layer-2 설계
- 优先明确分段用于管理、서버、用户、访客、IoT/OT 和受监管环境
- 不假设 BGP、OSPF、EVPN、SD-WAN 或微分段是必需的
- 将보안控制作为아키텍처的一일부，而非事后想法

## homelab-architect

### 이름과 용도
从硬件清单、目标和操作员经验水平설계家庭和小型实验室网络계획。보유보안分단계变更和롤백指导。

### 능력 설명
- 家庭和小型实验室게이트웨이、交换机、AP、NAS、서버
- 本地 DNS、DHCP、访客网络、IoT 隔离
- 远程访问계획
- 仅계획和리뷰，不应用설정

### 적용 시나리오
- 家庭网络설계
- 小型实验室网络계획
- homelab 网络설정

### 사용 도구 목록
- Read: 읽기硬件规格和需求
- Grep: 搜索网络패턴

### 다른 Agent와의 협력 방식
- homelab-network-readiness 变更前评估
- homelab-network-setup 进行 IP 范围、DHCP 预留、电缆
- network-config-validation 검증게이트웨이或交换机설정
- network-interface-health 进行链路、端口、电缆분석

### 워크플로우

1. 盘点硬件：게이트웨이/라우팅器、交换机、AP、서버、NAS、DNS 파싱器、ISP 交接和远程访问路径
2. 확인目标：隔离、访客 Wi-Fi、广告屏蔽、本地서비스、远程访问、백업、모니터링、학습实验室或家庭可靠性
3. 将目标与硬件能力匹配。如果硬件불가지원 VLAN、本地 DNS 或보안远程访问，指出并提出分단계업그레이드路径
4. 先설계最小的있음用拓扑，然后是선택的后续단계
5. 在임의破坏性变更之前定义롤백和访问보안
6. 生成实施순서，在각단계保持互联网、DNS 和管理访问可복구

### 보안기본值

- 不권장将管理인터페이스暴露给互联网
- 不권장비활성화防火墙규칙、인증、DNS 过滤或分段作为故障排除快捷方式
- 在파싱器보유静态地址、健康검사和回退路径之前，避免将 DHCP DNS 更改为本地파싱器
- 除非操作员가능在变更后到达게이트웨이、交换机和 AP，否则避免 VLAN 마이그레이션
- 优先사용通俗易懂的解释和小规模可逆단계

## performance-optimizer

### 능력 설명
- 성능분석 - 识别慢코드路径、메모리泄漏和瓶颈
- Bundle 최적화 - 减少 JavaScript 패키지크기、지연加载、코드分割
- 실행时최적화 - 개선算法效率、减少필요 없음要的计算
- React/渲染최적화 - 防止필요 없음要的重新渲染、최적화컴포넌트树
- 데이터베이스和网络 - 최적화查询、减少 API 调用、구현캐시
- 메모리 관리 - 检测泄漏、최적화메모리사용、정리资源

### 적용 시나리오

### 핵심성능지표

| Metric | Target | Action if Exceeded |
|--------|--------|-------------------|
| First Contentful Paint | < 1.8s | 최적화핵심路径、内联핵심 CSS |
| Largest Contentful Paint | < 2.5s | 지연加载图片、최적화서버응답 |
| Time to Interactive | < 3.8s | 코드分割、减少 JavaScript |
| Cumulative Layout Shift | < 0.1 | 为图片保留공간、避免布局跳动 |
| Total Blocking Time | < 200ms | 分解长태스크、사용 web workers |
| Bundle Size (gzip) | < 200KB | Tree shaking、지연加载、코드分割 |

### 算法분석

검사低效算法：

| Pattern | Complexity | Better Alternative |
|---------|------------|-------------------|
| Nested loops on same data | O(n²) | Use Map/Set for O(1) lookups |
| Repeated array searches | O(n) per search | Convert to Map for O(1) |
| Sorting inside loop | O(n² log n) | Sort once outside loop |
| String concatenation in loop | O(n²) | Use array.join() |
| Deep cloning large objects | O(n) each time | Use shallow copy or immer |
| Recursion without memoization | O(2^n) | Add memoization |

### React 성능 최적화

常见 React 反패턴：

```tsx
// BAD: 渲染中内联함수생성
<Button onClick={() => handleClick(id)}>Submit</Button>

// GOOD: 사용 useCallback 稳定回调
const handleButtonClick = useCallback(() => handleClick(id), [handleClick, id]);
<Button onClick={handleButtonClick}>Submit</Button>

// BAD: 渲染中对象생성
<Child style={{ color: 'red' }} />

// GOOD: 稳定对象引用
const style = useMemo(() => ({ color: 'red' }), []);
<Child style={style} />

// BAD: 每次渲染的昂贵计算
const sortedItems = items.sort((a, b) => a.name.localeCompare(b.name));

// GOOD: 记忆化昂贵计算
const sortedItems = useMemo(
  () => [...items].sort((a, b) => a.name.localeCompare(b.name)),
  [items]
);
```

### 메모리泄漏检测

常见메모리泄漏패턴：

```typescript
// BAD: 없음정리的事件监听器
useEffect(() => {
  window.addEventListener('resize', handleResize);
  // Missing cleanup!
}, []);

// GOOD: 정리事件监听器
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);

// BAD: 없음정리的定时器
useEffect(() => {
  setInterval(() => pollData(), 1000);
  // Missing cleanup!
}, []);

// GOOD: 정리定时器
useEffect(() => {
  const interval = setInterval(() => pollData(), 1000);
  return () => clearInterval(interval);
}, []);
```

### 성공지표

- Lighthouse 성능分数 > 90
- 모든 Core Web Vitals 在"良好"范围内
- Bundle 크기在预算内
- 없음检测到메모리泄漏
- 테스트套件仍然통과
- 없음성능回归
