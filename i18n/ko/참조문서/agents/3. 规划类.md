# 계획類 Agent

계획類 Agent 专门负责기능계획、기술아키텍처설계和实施단계分解。

## Agent 목록

| Agent 이름 | 용도 | 사용모델 | 코어도구 |
|------------|------|----------|----------|
| planner | 复杂기능实施계획专家 | opus | Read, Grep, Glob |
| code-architect | 기능아키텍처설계专家 | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | PR 테스트覆盖분석 | sonnet | Read, Grep, Glob, Bash |
| type-design-analyzer | 유형설계분석 | sonnet | Read, Grep, Glob |
| comment-analyzer | 코드주석분석 | sonnet | Read, Grep, Glob |

---

## planner

### 이름과 용도
复杂기능实施和리팩토링的계획专家。在用户요청기능实施、아키텍처变更或复杂리팩토링时主动사용。

### 능력 설명
- 需求분석和详细实施계획생성
- 将复杂기능分解为可管理的단계
- 识别依赖关系和潜在风险
- 권장最优实施순서
- 고려边缘情况和오류场景

### 적용 시나리오
- 新기능实施계획
- 아키텍처变更계획
- 复杂리팩토링계획
- 기술债务정리계획

### 사용 도구 목록
- Read: 읽기现있음코드和문서
- Grep: 搜索相关패턴和现있음구현
- Glob: 찾기相关파일

### 다른 Agent와의 협력 방식
- 生成的계획供 code-architect 进一步细化
- 生成的계획供 tdd-guide 编写테스트
- 生成的계획供 code-reviewer 리뷰

### 계획流程

1. **需求분석**
   - 完全理解기능요청
   - 如있음필요提出澄清문제
   - 识别성공표준
   - 列出假设和约束

2. **아키텍처리뷰**
   - 분석现있음코드库구조
   - 识别受影响的컴포넌트
   - 리뷰클래스似구현
   - 고려可复用패턴

3. **단계分解**
   - 생성详细단계，패키지含：
     - 清晰、具体的操作
     - 파일路径和位置
     - 단계之间的依赖关系
     - 预估复杂度
     - 潜在风险

4. **实施순서**
   - 按依赖关系排序
   - 将相关变更分组
   - 最小化上下文切换
   - 지원增量테스트

### 계획输出형식

```markdown
# 实施계획: [기능名称]

## 개요
[2-3 句话总结]

## 需求
- [需求 1]
- [需求 2]

## 아키텍처变更
- [变更 1: 파일路径和描述]
- [变更 2: 파일路径和描述]

## 实施단계

### 단계 1: [단계名称]
1. **[단계名称]** (파일: path/to/file.ts)
   - 操作: 要采取的具体操作
   - 原因: 此단계的原因
   - 依赖: 없음 / 필요단계 X
   - 风险: 低/中/高

2. **[단계名称]** (파일: path/to/file.ts)
   ...

### 단계 2: [단계名称]
...

## 테스트전략
- 단위 테스트: [要테스트的파일]
- 통합 테스트: [要테스트的流程]
- E2E 테스트: [用户旅程要테스트]

## 风险与缓解
- **风险**: [描述]
  - 缓解: [如何处理]

## 성공표준
- [ ] 표준 1
- [ ] 표준 2
```

## code-architect

### 이름과 용도
基于对现있음코드库的深入理解설계기능아키텍처。提供具体파일、인터페이스、데이터流和빌드순서的实施蓝图。

### 능력 설명
- 研究现있음코드组织和命名约定
- 识别已있음的아키텍처패턴
- 기록테스트패턴和现있음边界
- 理解依赖关系图后再提出新抽象
- 설계适合当前패턴的기능아키텍처

### 적용 시나리오
- 新기능아키텍처설계
- 现있음기능리팩토링설계
- 코드组织최적화

### 사용 도구 목록
- Read: 읽기现있음코드
- Grep: 搜索패턴和约定
- Glob: 찾기相关파일
- Bash: 실행진단명령

### 다른 Agent와의 협력 방식
- 基于 planner 的계획进行아키텍처설계
- 为코드구현提供蓝图
- 与 build-error-resolver 协作수정빌드문제

### 설계流程

1. **패턴분석**
   - 研究现있음코드组织和命名约定
   - 识别已있음的아키텍처패턴
   - 注意테스트패턴和现있음边界
   - 在提出新抽象前理解依赖关系

2. **아키텍처설계**
   - 설계기능以自然契合当前패턴
   - 选择满足需求的最简单아키텍처
   - 避免投机性抽象（除非코드库已사용）

3. **实施蓝图**
   为각重要컴포넌트提供：
   - 파일路径
   - 目的
   - 핵심인터페이스
   - 依赖关系
   - 데이터流角色

4. **빌드순서**
   按依赖关系排序实施：
   1. 유형和인터페이스
   2. 코어逻辑
   3. 통합层
   4. UI
   5. 테스트
   6. 문서

### 输出형식

```markdown
## 아키텍처: [기능名称]

### 설계决策
- 决策 1: [理由]
- 决策 2: [理由]

### 要생성的파일
| 파일 | 目的 | 优先级 |
|------|------|--------|
| | | |

### 要修改的파일
| 파일 | 变更 | 优先级 |
|------|------|--------|
| | | |

### 데이터流
[描述]

### 빌드순서
1. 단계 1
2. 단계 2
```

## type-design-analyzer

### 이름과 용도
분석유형설계是否使非法状态难以或不可能表示。

### 능력 설명
- 封装评估
- 不变性表达분석
- 不变性있음用性评估
- 强制실행검증

### 적용 시나리오
- 유형시스템설계리뷰
- API 설계리뷰
- 领域建模리뷰

### 사용 도구 목록
- Read: 읽기유형定义
- Grep: 搜索유형사용
- Glob: 찾기유형파일

### 다른 Agent와의 협력 방식
- 与 code-reviewer 协作进行코드 리뷰
- 与 code-architect 协作进行아키텍처설계

### 评估표준

1. **封装**
   - 内部细节是否숨김
   - 不변수是否가능从外部违反

2. **不变性表达**
   - 유형是否인코딩业务규칙
   - 不可能状态是否在유형级别被阻止

3. **不变性있음用性**
   - 这些不변수是否防止真实 bug
   - 是否与领域对齐

4. **强制실행**
   - 不변수是否由유형시스템强制실행
   - 是否있음简单的逃逸舱

### 输出형식

对각리뷰的유형：
- 유형名和位置
- 四个维度的评分
- 整体评估
- 具体개선권장

## comment-analyzer

### 이름과 용도
분석코드주석的准确性、完整性、유지보수성和주석腐坏风险。

### 능력 설명
- 事实准确性검증
- 完整性검사
- 长期价值评估
- 误导性元素识别

### 적용 시나리오
- 코드 리뷰时
- 문서품질评估时
- 识别주석기술债务时

### 사용 도구 목록
- Read: 읽기코드和주석
- Grep: 搜索주석和코드
- Glob: 찾기源파일

### 다른 Agent와의 협력 방식
- 与 code-reviewer 协作进行코드 리뷰
- 与 doc-updater 协作업데이트문서

### 분석프레임워크

1. **事实准确性**
   - 검증声明与코드对比
   - 검사参数和돌아가기描述与구현对比
   - 标记过时引用

2. **完整性**
   - 검사复杂逻辑是否있음足够解释
   - 검증重要副作用和边缘用例是否已기록
   - 确保公共 API 있음足够주석

3. **长期价值**
   - 标记只重述코드的주석
   - 识别会快速腐坏的脆弱주석
   - 发现 TODO / FIXME / HACK 债务

4. **误导性元素**
   - 与코드矛盾的주석
   - 对已제거行为的过时引用
   - 过度承诺或描述不足的行为

### 输出형식

按심각性分组的咨询发现：
- Inaccurate（不准确）
- Stale（过时）
- Incomplete（不完整）
- Low-value（低价值）
