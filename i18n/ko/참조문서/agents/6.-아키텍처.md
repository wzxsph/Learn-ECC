# 아키텍처類 Agent

아키텍처類 Agent 전담시스템아키텍처설계、네트워크네트워크아키텍처계획및성능 최적화。

## Agent 목록

| Agent 이름 | 용도 | 사용모델 | 코어도구 |
|------------|------|----------|----------|
| architect | 软件아키텍처专家 | opus | Read, Grep, Glob |
| network-architect | 企业네트워크네트워크아키텍처설계 | sonnet | Read, Grep |
| homelab-architect | 家庭/实验室네트워크네트워크계획 | sonnet | Read, Grep |
| code-explorer | 코드库깊이분석 | sonnet | Read, Grep, Glob |
| performance-optimizer | 성능분석및최적화 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| harness-optimizer | Agent harness 설정최적화 | sonnet | Read, Grep, Glob, Bash, Edit |

## architect

### 이름과 용도
软件아키텍처专家，专注于시스템설계、可확장성및기술决策。在계획新기능、리팩토링大型시스템또는做出아키텍처决策시主动사용。

### 능력 설명
- 위新기능설계시스템아키텍처
- 评估기술权衡
- 권장패턴및모범 사례
- 识别可확장성瓶颈
- 위未来增长계획
- 确保코드库一致성

### 적용 시나리오
- 新기능아키텍처설계
- 大型시스템리팩토링
- 아키텍처决策制정
- 기술选型决策
- 확장성계획

### 사용 도구 목록
- Read: 읽기现있음아키텍처
- Grep: 검색패턴
- Glob: 찾기파일

### 다른 Agent와의 협력 방식
- planner 기반아키텍처설계생성实施계획
- code-architect 위特정기능설계아키텍처
- build-error-resolver 处처리빌드문제
- code-reviewer 리뷰코드 품질

### 아키텍처리뷰流程

#### 1. 当前상태분석
- 리뷰现있음아키텍처
- 识别패턴및约정
- 기록기술债务
- 评估可확장성제한

#### 2. 요구구收集
- 기능요구구
- 非기능요구구（성능、보안、可확장성）
- 통합点
- 데이터流요구구

#### 3. 설계提案
- 高层아키텍처그래프
- 컴포넌트职责
- 데이터모델
- API 契约
- 통합패턴

#### 4. 权衡분석
각설계决策기록：
- **优点**: 收益및优势
- **결여点**: 결여点및제한
- **替代方案**: 고려의기타옵션
- **决策**: 最终选择및처리由

#### 1. 모듈化및관련注点분离

- 高效데이터베이스쿼리
- 캐시전략
- 로드 밸런싱고려

#### 3. 유지보수성
- 清晰의코드组织
- 一致의패턴
- 全面의문서
- 易于테스트
- 易于처리解

#### 4. 보안
- 纵深防御
- 最小권한원칙
- 경계입력검증
- 기본보안

#### 5. 성능

- 最小化네트워크네트워크요청
- 최적화데이터베이스쿼리
- 适当캐시
- 지연불러오기

### 아키텍처决策기록 (ADRs)

대한重요아키텍처决策，생성 ADR：

```markdown
# ADR-001: Use Redis for Semantic Search Vector Storage

## Context
필요저장및쿼리 1536 维 embeddings 사용于语义市场검색。

## Decision
사용带있음向量검색기능의 Redis Stack。

## Consequences

### Positive
- 快速向量관련似성검색 (<10ms)
- 内置 KNN 算法
- 简单배포
- 良好성능高达 100K 向量

### Negative
- 메모리저장（大데이터集昂贵）
- 없음클러스터시单点故障
- 仅限于余弦관련似度

### Alternatives Considered
- **PostgreSQL pgvector**: 较慢，但持久저장
- **Pinecone**: 托管서비스，成本较高
- **Weaviate**: 更多기능，更复杂설정

## Status
Accepted

## Date
2025-01-15
```

## network-architect

### 이름과 용도
从요구구설계企业또는多站点네트워크네트워크아키텍처。사용现있음네트워크네트워크技가능진행专注의라우팅、검증、자동化및장애排除。

### 능력 설명
- 校园、브랜치、WAN、데이터中心、云邻近및混合네트워크네트워크계획
- IP 寻址、분段、라우팅域、管처리平面액세스
- 冗余、모니터링및마이그레이션정렬
- 仅설계및리뷰，불应사용설정

### 적용 시나리오
- 企业네트워크네트워크설계
- 多站点네트워크네트워크아키텍처
- 데이터中心네트워크네트워크계획
- 云네트워크네트워크통합

### 사용 도구 목록
- Read: 읽기요구구및现있음설정
- Grep: 검색네트워크네트워크패턴

### 다른 Agent와의 협력 방식
- network-config-validation 검증预变更설정
- network-bgp-diagnostics 진행 BGP 진단
- network-interface-health 진행인터페이스健康분석
- cisco-ios-patterns 제공 IOS/IOS-XE 语法
- netmiko-ssh-automation 진행네트워크네트워크자동化

### 워크플로우

1. 重述目标、约束및非目标
2. 识别显著改变아키텍처의결여失요구구
3. 选择拓扑그리고解释이유
4. 在讨论硬件之前설계라우팅및분段
5. 정义管처리平面、로그、모니터링、백업및롤백모델
6. 生成보유검증门및롤백点의분단계实施계획
7. 列出残余风险및仍요구从运营商가져오기의证据

### 설계기본값

- 우선라우팅경계而非확장 layer-2 설계
- 우선明确분段사용于管처리、서버、사용户、게스트、IoT/OT 및受监管环境
- 불假设 BGP、OSPF、EVPN、SD-WAN 또는微분段이必요구의
- 将보안控制作위아키텍처의一일부，而非事后想法

## homelab-architect

### 이름과 용도
从硬件清单、目标및작업员经验水平설계家庭및小型实验室네트워크네트워크계획。보유보안분단계变更및롤백指导。

### 능력 설명
- 家庭및小型实验室게이트웨이、交换机、AP、NAS、서버
- 本地 DNS、DHCP、게스트네트워크네트워크、IoT 隔离
- 远程액세스계획
- 仅계획및리뷰，불应사용설정

### 적용 시나리오
- 家庭네트워크네트워크설계
- 小型实验室네트워크네트워크계획
- homelab 네트워크네트워크설정

### 사용 도구 목록
- Read: 읽기硬件规格및요구구
- Grep: 검색네트워크네트워크패턴

### 다른 Agent와의 협력 방식
- homelab-network-readiness 变更前评估
- homelab-network-setup 진행 IP 范围、DHCP 预留、电缆
- network-config-validation 검증게이트웨이또는交换机설정
- network-interface-health 진행链路、포트、电缆분석

### 워크플로우

1. 盘点硬件：게이트웨이/라우팅器、交换机、AP、서버、NAS、DNS 파싱器、ISP 交接및远程액세스경로
2. 확인目标：隔离、게스트 Wi-Fi、广告屏蔽、本地서비스、远程액세스、백업、모니터링、학습实验室또는家庭可靠성
3. 将目标与硬件가능力매칭。만약硬件불가지원 VLAN、本地 DNS 또는보안远程액세스，指出그리고提出분단계업그레이드경로
4. 先설계最小의있음사용拓扑，然后이선택의后续단계
5. 在임의破坏성变更之前정义롤백및액세스보안
6. 生成实施순서，在각단계保持互联네트워크、DNS 및管처리액세스可복구

### 보안기본값

- 불권장将管처리인터페이스暴露给互联네트워크
- 불권장비활성화防火墙규칙、인증、DNS 필터링또는분段作위장애排除快捷방식
- 在파싱器보유静态地址、健康검사및回退경로之前，避免将 DHCP DNS 更改위本地파싱器
- 除非작업员가능在变更后到达게이트웨이、交换机및 AP，아니오则避免 VLAN 마이그레이션
- 우선사용通俗易懂의解释및小规模可逆단계

## code-explorer

### 이름과 용도
통과추적실행경로、맵아키텍처层및기록의존의존관련系来깊이분석现있음코드库기능，위新개발제공정보。

### 능력 설명
- 入口点发现
- 실행경로추적
- 아키텍처层맵
- 패턴识别
- 의존의존문서화

### 적용 시나리오
- 처리解现있음기능시
- 新개발시작前
- 코드리팩토링前
- 디버깅문제시

### 사용 도구 목록
- Read: 읽기코드
- Grep: 검색코드패턴
- Glob: 찾기파일

### 다른 Agent와의 협력 방식
- code-architect 기반探索설계新기능
- code-reviewer 리뷰코드 품질
- planner 기반처리解생성계획

### 분석流程

#### 1. 入口点发现
- 找到기능또는区域의主요入口点
- 从사용户작업또는外部触发器추적整个스택

#### 2. 실행경로추적
- 跟随调사용链从入口到완료
- 注意브랜치逻辑및비동기경계
- 맵데이터转换및오류경로

#### 3. 아키텍처层맵
- 识别코드触及의层
- 처리解这些层如何通信
- 注意可复사용경계및反패턴

#### 4. 패턴识别
- 识别이미在사용의패턴및추상
- 기록命名约정및코드组织원칙

#### 5. 의존의존문서화
- 맵外部库및서비스
- 맵内部모듈의존의존
- 识别값得复사용의共享도구

## performance-optimizer

### 이름과 용도
성능분석및최적화专家。主动사용于识别瓶颈、최적화慢코드、减少패키지크기및提高실행시성능。

### 능력 설명
- 성능분석 - 识别慢코드경로、메모리泄漏및瓶颈
- Bundle 최적화 - 减少 JavaScript 패키지크기、지연불러오기、코드분割
- 실행시최적화 - 개선算法效率、减少필요 없음요의计算
- React/渲染최적화 - 防止필요 없음요의重新渲染、최적화컴포넌트트리
- 데이터베이스및네트워크네트워크 - 최적화쿼리、减少 API 调사용、구현캐시
- 메모리 관리 - 감지泄漏、최적화메모리사용、정리资源

### 적용 시나리오
- 성능문제진단시
- 성능 최적화前
- Lighthouse 분数下降시
- Bundle 크기增加시
- 메모리사용增长시
- 页面불러오기变慢시

### 사용 도구 목록
- Read: 읽기코드
- Write: 쓰기최적화코드

### 算法분석

검사低效算法：

| Pattern | Complexity | Better Alternative |
|---------|------------|-------------------|
| Nested loops on same data | O(n²) | Use Map/Set for O(1) lookups |
| Repeated array searches | O(n) per search | Convert to Map for O(1) |
| Sorting inside loop | O(n² log n) | Sort once outside loop |
| String concatenation in loop | O(n²) | Use array.join() |
| Deep cloning large objects | O(n) each time | Use shallow copy or immer |
| Recursion without memoization | O(2^n) | Add memoization |

### React 성능 최적화

常见 React 反패턴：

```tsx
// BAD: 渲染中内联함수생성
<Button onClick={() => handleClick(id)}>Submit</Button>

// GOOD: 사용 useCallback 稳정回调
const handleButtonClick = useCallback(() => handleClick(id), [handleClick, id]);
<Button onClick={handleButtonClick}>Submit</Button>

// BAD: 渲染中객체생성
<Child style={{ color: 'red' }} />

// GOOD: 稳정객체引사용
const style = useMemo(() => ({ color: 'red' }), []);
<Child style={style} />

// BAD: 每次渲染의昂贵计算
const sortedItems = items.sort((a, b) => a.name.localeCompare(b.name));

// GOOD: 记忆化昂贵计算
const sortedItems = useMemo(
  () => [...items].sort((a, b) => a.name.localeCompare(b.name)),
  [items]
);
```

### 메모리泄漏감지

常见메모리泄漏패턴：

```typescript
// BAD: 없음정리의이벤트监听器
useEffect(() => {
  window.addEventListener('resize', handleResize);
  // Missing cleanup!
}, []);

// GOOD: 정리이벤트监听器
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);

// BAD: 없음정리의정시器
useEffect(() => {
  setInterval(() => pollData(), 1000);
  // Missing cleanup!
}, []);

// GOOD: 정리정시器
useEffect(() => {
  const interval = setInterval(() => pollData(), 1000);
  return () => clearInterval(interval);
}, []);
```
