# 테스트類 Agent

테스트類 Agent 专门负责테스트驱动개발、E2E 테스트、테스트커버리지분석和품질保证。

## Agent 목록

| Agent 이름 | 용도 | 사용모델 | 코어도구 |
|------------|------|----------|----------|
| tdd-guide | TDD 테스트驱动개발专家 | sonnet | Read, Write, Edit, Bash, Grep |
| e2e-runner | E2E 테스트专家 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| mle-reviewer | ML 工程코드 리뷰（훈련/추론/모니터링） | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | PR 테스트覆盖분석 | sonnet | Read, Grep, Glob, Bash |

---

## tdd-guide

### 이름과 용도
테스트驱动개발专家，强制실행写테스트优先的메서드论。主动用于编写新기능、수정 bug 或리팩토링코드。确保 80%+ 的테스트커버리지。

### 능력 설명
- 强制실행테스트优先的메서드论
- 指导 Red-Green-Refactor 周期
- 确保 80%+ 테스트커버리지
- 编写全面的테스트套件（单元、통합、E2E）
- 在实施前捕获边缘用例

### 적용 시나리오
- 编写新기능时
- 수정 bug 时
- 리팩토링코드时
- 필요 TDD 워크플로우时

### 사용 도구 목록
- Read: 읽기现있음코드
- Write: 编写테스트
- Edit: 修改테스트和구현
- Bash: 실행테스트명령
- Grep: 搜索테스트패턴

### 다른 Agent와의 협력 방식
- code-reviewer 在 TDD 周期后리뷰코드
- build-error-resolver 수정테스트빌드오류
- e2e-runner 进行핵심用户流的 E2E 테스트

### TDD 워크플로우

#### 1. 先写테스트 (RED)
编写描述预期行为的실패테스트。

#### 2. 실행테스트 - 검증它 FAIL
```bash
npm test
```

#### 3. 写最小구현 (GREEN)
只写足够的코드让테스트통과。

#### 4. 실행테스트 - 검증它 PASS

#### 5. 리팩토링 (IMPROVE)
제거重复，개선名称，최적화 - 테스트필수保持绿色。

#### 6. 검증커버리지
```bash
npm run test:coverage
# 요구: 80%+ 브랜치、함수、行、语句커버리지
```

### 필수테스트的边缘用例

1. **Null/Undefined** 输入
2. **空** 数组/字符串
3. **없음效유형** 传递
4. **边界值** (最小/最大)
5. **오류路径** (网络실패、데이터베이스오류)
6. **竞态条件** (동시성操作)
7. **大데이터** (10k+ 项的성능)
8. **特殊字符** (Unicode、表情符号、SQL 字符)

### 反패턴 - 필수避免

- 테스트구현细节（内部状态）而非行为
- 테스트相互依赖（共享状态）
- 断言太少（통과但不검증임의东西）
- 外部依赖未 mock (Supabase、Redis、OpenAI 等)

### 품질검사清单

- [ ] 모든公共함수都있음단위 테스트
- [ ] 모든 API 端点都있음통합 테스트
- [ ] 핵심用户流있음 E2E 테스트
- [ ] 边缘用例已覆盖 (null、empty、invalid)
- [ ] 오류路径已테스트（不只是 happy path）
- [ ] 外部依赖已 mock
- [ ] 테스트相互独立（없음共享状态）
- [ ] 断言具体且있음意义
- [ ] 커버리지 80%+

### v1.8 Eval-Driven TDD 附录

将 eval-driven 개발통합到 TDD 流程：

1. 在实施前定义能力 + 回归 evals
2. 실행基线并捕获실패特征
3. 구현最小통과变更
4. 重新실행테스트和 evals；보고서 pass@1 和 pass@3
5. 게시핵심路径应在머지前达到 pass^3 稳定性

## e2e-runner

### 이름과 용도
E2E 테스트专家，사용 Vercel Agent Browser（首选）和 Playwright 后备。主动用于生成、维护和실행 E2E 테스트。

### 능력 설명
- 테스트旅程생성 - 为用户流编写테스트
- 테스트维护 - 保持테스트与 UI 变化동기화
- 不稳定테스트管理 - 识别和隔离不稳定테스트
- 테스트보고서 - 生成 HTML 보고서和 JUnit XML
- CI/CD 통합 - 确保테스트在管道中可靠실행
- 截图/视频/trace 管理

### 적용 시나리오
- 핵심用户旅程필요검증时
- 필요端到端검증时
- CI/CD 管道中실행 E2E 테스트时
- 发现통합문제时

### 사용 도구 목록
- Read: 읽기页面内容
- Write: 编写테스트
- Edit: 修改테스트
- Bash: 실행 Playwright/Agent Browser 명령
- Grep: 搜索테스트
- Glob: 찾기테스트파일

### 다른 Agent와의 협력 방식
- tdd-guide 编写单元/통합 테스트
- code-reviewer 리뷰테스트품질
- mle-reviewer 리뷰 ML 相关테스트

### 主要도구: Agent Browser

**优先于原始 Playwright 사용 Agent Browser** - 语义选择器、AI 최적화、자동等待、基于 Playwright。

```bash
# 설정
npm install -g agent-browser && agent-browser install

# 핵심 워크플로우
agent-browser open https://example.com
agent-browser snapshot -i          # 가져오기带 refs 的元素 [ref=e1]
agent-browser click @e1            # 按 ref 点击
agent-browser fill @e2 "text"     # 按 ref 填充输入
agent-browser wait visible @e5     # 等待元素可见
agent-browser screenshot result.png
```

### 后备: Playwright

当 Agent Browser 不可用时，直接사용 Playwright。

```bash
npx playwright test                        # 실행모든 E2E 테스트
npx playwright test tests/auth.spec.ts     # 실행特定파일
npx playwright test --headed               # 可见浏览器
npx playwright test --debug                # 사용검사器디버깅
npx playwright test --trace on             # 带 trace 실행
npx playwright show-report                 # 查看 HTML 보고서
```

### 워크플로우

#### 1. 계획
- 识别핵심用户旅程（인증、코어기능、支付、CRUD）
- 定义场景: happy path、边缘用例、오류用例
- 按风险排序: HIGH（金融、인증）、MEDIUM（搜索、导航）、LOW（UI 최적화）

#### 2. 생성
- 사용页面对象모델 (POM) 패턴
- 优先사용 `data-testid` 选择器
- 在핵심단계추가断言
- 在핵심时刻捕获截图
- 사용适当的等待（从不 `waitForTimeout`）

#### 3. 실행
- 本地실행 3-5 次검사不稳定性
- 사용 `test.fixme()` 或 `test.skip()` 隔离不稳定테스트
- 上传 artifact 到 CI

### 핵심원칙

- **사용语义选择器**: `[data-testid="..."]` > CSS 选择器 > XPath
- **等待条件而非时间**: `waitForResponse()` > `waitForTimeout()`
- **자동等待内置**: `page.locator().click()` 자동等待；原始 `page.click()` 不等待
- **隔离테스트**: 각테스트해야 함独立；없음共享状态
- **快速실패**: 在각핵심단계사용 `expect()` 断言
- **재시도时 trace**: 설정 `trace: 'on-first-retry'` 用于디버깅실패

### 不稳定테스트处理

```typescript
// 隔离
test('flaky: market search', async ({ page }) => {
  test.fixme(true, 'Flaky - Issue #123')
})

// 识别不稳定性
// npx playwright test --repeat-each=10
```

常见原因: 竞态条件（사용자동等待选择器）、网络计时（等待응답）、动画计时（等待 `networkidle`）。

### 성공지표

- 모든핵심旅程통과 (100%)
- 总体통과率 > 95%
- 不稳定率 < 5%
- 테스트持续时间 < 10 分钟
- Artifact 上传并可访问

## mle-reviewer

### 이름과 용도
生产머신러닝工程리뷰专家，专注于데이터契约、特征管道、훈련可复现性、离线/在线评估、모델서비스、모니터링和롤백。

### 능력 설명
- 문제定义和决策품질리뷰
- 지표、임계값和오류분석리뷰
- 데이터契约和泄漏검사
- 훈련可复现性검증
- 评估和推广流程리뷰
- 서비스和배포보안리뷰
- 모니터링和事件응답계획

### 적용 시나리오
- ML、MLOps、모델훈련코드变更时
- 추론、特征存储、评估코드变更时
- 모델推广决策时

### 사용 도구 목록
- Read: 읽기 ML 코드和설정
- Grep: 搜索패턴
- Glob: 찾기파일
- Bash: 실행 pytest, ruff, mypy

### 다른 Agent와의 협력 방식
- python-reviewer 处理 Python 风格、유형、오류 처리
- pytorch-build-resolver 处理 tensor/CUDA/훈련실패
- database-reviewer 处理特征表、태그存储
- security-reviewer 处理 secrets、PII、pickle 보안
- performance-optimizer 处理지연、메모리、GPU 利用率
- build-error-resolver 处理 CI、依赖、原生확장실패
- pr-test-analyzer 분석테스트覆盖
- silent-failure-hunter 发现管道静默실패
- e2e-runner 进行产品流 E2E 테스트
- a11y-architect 리뷰预测可访问性
- doc-updater 업데이트문서

### 핵심리뷰领域

#### 문제定义和决策품질
- 变更是否从用户或시스템决策시작，而非모델아키텍처偏好
- 利益相关者和실패成本是否明确
- 지표选择是否遵循오류预算

#### 지표、임계값和오류분석
- 基线和当前生产行为是否比较
- 임계값和설정是否作为产品决策
- 假阳性和假阴性是否直接검사

#### 데이터契约和泄漏
- 实体粒度、主键、时间戳是否明确
- 分割是否遵守时间、用户/实体分组
- 特征연결是否时间点正确
- 敏感属性是否排除或正当处理

#### 훈련可复现性
- 훈련是否可从코드、설정、데이터集버전、种子실행
- 超参数、预处理、依赖버전是否기록
- 随机性和 GPU 非确定性是否있음意处理

#### 评估和推广
- 지표是否与基线和当前生产모델比较
- 推广门是否在选择前声明
- 切片지표是否覆盖重要큐

#### 서비스和배포
- 훈련和서비스转换是否共享或等价테스트
- 输入패턴是否거부过时、缺失、없음效特征
- 추론路径是否있음타임아웃、资源限制、回退逻辑

#### 모니터링和事件응답
- 모니터링是否覆盖서비스健康、特征漂移、预测漂移
- 로그是否패키지含足够标识符
- 롤백是否命名前一 artifact、설정、데이터依赖

### 常见阻塞器

- 在时间相关或用户相关데이터上随机 train/test 分割
- 特征生成사용预测时不可用的字段
- 离线지표개선而핵심切片回归
- 훈련预处理被手动복사到서비스코드
- 모델버전不在预测로그中

### 승인표준

- **APPROVE**: 없음핵심/高 MLE 风险，相关테스트或评估门통과
- **APPROVE WITH WARNINGS**: 只있음中等문제，있음明确的后续行动
- **BLOCK**: 임의可能的泄漏、不可复现的推广、不보안的서비스行为、缺少生产배포롤백、敏感데이터暴露、或핵심评估差距

## pr-test-analyzer

### 이름과 용도
리뷰 Pull Request 테스트覆盖품질和完整性，强调行为覆盖和真实 bug 预防。

### 능력 설명
- 识别变更코드映射
- 검증行为覆盖
- 评估테스트품질
- 发现覆盖差距

### 적용 시나리오
- PR 리뷰时
- 评估테스트커버리지时
- 검증테스트充分性时

### 사용 도구 목록
- Read: 읽기变更코드和테스트
- Grep: 搜索테스트
- Glob: 찾기테스트파일
- Bash: 실행테스트명령

### 다른 Agent와의 협력 방식
- tdd-guide 개선테스트
- e2e-runner 进行端到端覆盖
- code-reviewer 进行코드 품질리뷰

### 분석流程

#### 1. 识别变更코드
- 映射变更的함수、클래스和모듈
- 定位对应테스트
- 识别新的未테스트코드路径

#### 2. 行为覆盖
- 검사각기능是否있음테스트
- 검증边缘用例和오류路径
- 确保重要통합被覆盖

#### 3. 테스트품질
- 优先选择있음意义的断言而非없음抛出검사
- 标记不稳定的패턴
- 검사隔离性和테스트名称清晰度

#### 4. 覆盖差距
按影响评级：
- critical（핵심）
- important（重要）
- nice-to-have（锦上添花）

### 输出형식

1. 覆盖总结
2. 핵심差距
3. 개선권장
4. 积极观察
