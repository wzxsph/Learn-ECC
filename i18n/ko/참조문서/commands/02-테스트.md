## /go-test

**용도 설명**: Go 언어의 TDD 워크플로우. 표驱动 테스트 사용, 먼저 테스트 작성 후 구현, 80%+ 커버리지 검증.

**사용 메서드**:
```
/go-test [기능 설명]
```

**사용 시나리오**:
- 새 Go 함수 구현
-现有 Go 코드에 테스트 커버리지 추가
- 버그 수정 (먼저 실패하는 테스트 작성)
- Go TDD 방법론 학습

**예제 워크플로우**:
```go
// Step 1: 인터페이스 정의
func ValidateEmail(email string) error {
    panic("not implemented")
}

// Step 2: 표驱动 테스트 작성 (RED)
tests := []struct {
    name    string
    email   string
    wantErr bool
}{
    {"simple email", "user@example.com", false},
    {"empty string", "", true},
    {"no at sign", "userexample.com", true},
}

// Step 3: 테스트 실행 - FAIL 확인

// Step 4: 최소 코드 구현 (GREEN)
func ValidateEmail(email string) error {
    if email == "" {
        return ErrEmailEmpty
    }
    if !emailRegex.MatchString(email) {
        return ErrEmailInvalid
    }
    return nil
}

// Step 5: PASS 확인

// Step 6: 커버리지 검사
go test -cover ./...
// coverage: 100.0%
```

**커버리지 명령**:
```bash
go test -cover ./...                    # 기본 커버리지
go test -coverprofile=coverage.out      # 커버리지 파일 생성
go tool cover -html=coverage.out        # 브라우저에서 보기
go test -race -cover ./...              # 경합 감지
```

**사용 메서드**:
```
/kotlin-test [기능 설명]
```

**사용 시나리오**:
- 새 Kotlin 함수 또는 클래스 구현
- Android/KMP 프로젝트에 테스트 커버리지 추가
- 버그 수정 (먼저 실패하는 테스트 작성)
- Kotlin TDD 방법론 학습

**Kotest 테스트 스타일**:
```kotlin
// StringSpec (가장 간단)
class CalculatorTest : StringSpec({
    "add two positive numbers" {
        Calculator.add(2, 3) shouldBe 5
    }
})

// FunSpec (표준 단위 테스트)
class RegistrationValidatorTest : FunSpec({
    test("valid registration returns Valid") {
        val request = RegistrationRequest(
            name = "Alice",
            email = "alice@example.com",
            password = "SecureP@ss1"
        )
        val result = validateRegistration(request)
        result.shouldBeInstanceOf<ValidationResult.Valid>()
    }
})

// BehaviorSpec (BDD 스타일)
class OrderServiceTest : BehaviorSpec({
    Given("a valid order") {
        When("placed") {
            Then("should be confirmed") { /* ... */ }
        }
    }
})
```

**코루틴 테스트**:
```kotlin
test("concurrent fetch completes") {
    runTest {
        val result = service.fetchAll()
        result.shouldNotBeEmpty()
    }
}
```

**커버리지 명령**:
```bash
./gradlew koverHtmlReport     # HTML 보고서
./gradlew koverVerify         # 커버리지 임계값 검증
./gradlew koverXmlReport      # CI용 XML 보고서
./gradlew test                # 테스트 실행
```

## /rust-test

**용도 설명**: Rust 언어의 TDD 워크플로우. `#[test]`, rstest, proptest 및 mockall 사용, 먼저 테스트 작성 후 구현, 80%+ 커버리지 검증 (cargo-llvm-cov).

**사용 메서드**:
```
/rust-test [기능 설명]
```

**사용 시나리오**:
- 새 Rust 함수, 메서드 또는 trait 구현
- Rust 프로젝트에 테스트 커버리지 추가
- 버그 수정 (먼저 실패하는 테스트 작성)
- Rust TDD 방법론 학습

**테스트 패턴**:
```rust
// 단위 테스트
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn adds_two_numbers() {
        assert_eq!(add(2, 3), 5);
    }

    #[test]
    fn handles_error() -> Result<(), Box<dyn std::error::Error>> {
        let result = parse_config(r#"port = 8080"#)?;
        assert_eq!(result.port, 8080);
        Ok(())
    }
}

// 매개변수화된 테스트 (rstest)
#[rstest]
#[case("hello", 5)]
#[case("", 0)]
#[case("rust", 4)]
fn test_string_length(#[case] input: &str, #[case] expected: usize) {
    assert_eq!(input.len(), expected);
}

// 비동기 테스트
#[tokio::test]
async fn fetches_data_successfully() {
    let client = TestClient::new().await;
    let result = client.get("/data").await;
    assert!(result.is_ok());
}

// 속성 테스트 (proptest)
proptest! {
    #[test]
    fn encode_decode_roundtrip(input in ".*") {
        let encoded = encode(&input);
        let decoded = decode(&encoded).unwrap();
        assert_eq!(input, decoded);
    }
}
```

**커버리지 명령**:
```bash
cargo llvm-cov                    # 커버리지 요약
cargo llvm-cov --html             # HTML 보고서
cargo llvm-cov --fail-under-lines 80  # 임계값 미만이면 실패
cargo test                        # 테스트 실행
```

## /cpp-test

**용도 설명**: C++ 언어의 TDD 워크플로우. GoogleTest/GoogleMock 및 CMake/CTest 사용, 먼저 테스트 작성 후 구현, 커버리지 검증 (gcov/lcov).

**사용 메서드**:
```
/cpp-test [기능 설명]
```

**사용 시나리오**:
- 새 C++ 클래스 또는 함수 구현
- 테스트 커버리지 추가
- 버그 수정 (먼저 실패하는 테스트 작성)
- C++ TDD 방법론 학습

**GoogleTest 패턴**:
```cpp
// 기본 테스트
TEST(SuiteName, TestName) {
    EXPECT_EQ(add(2, 3), 5);
    EXPECT_NE(result, nullptr);
    EXPECT_TRUE(is_valid);
    EXPECT_THROW(func(), std::invalid_argument);
}

// Fixture
class DatabaseTest : public ::testing::Test {
protected:
    void SetUp() override { db_ = create_test_db(); }
    void TearDown() override { db_.reset(); }
    std::unique_ptr<Database> db_;
};

TEST_F(DatabaseTest, InsertsRecord) {
    db_->insert("key", "value");
    EXPECT_EQ(db_->get("key"), "value");
}

// 매개변수화된 테스트
class PrimeTest : public ::testing::TestWithParam<std::pair<int, bool>> {};

TEST_P(PrimeTest, ChecksPrimality) {
    auto [input, expected] = GetParam();
    EXPECT_EQ(is_prime(input), expected);
}

INSTANTIATE_TEST_SUITE_P(Primes, PrimeTest,
    ::testing::Values(
        std::make_pair(2, true),
        std::make_pair(4, false),
        std::make_pair(7, true)
    ));
```

**커버리지 명령**:
```bash
cmake -DCMAKE_CXX_FLAGS="--coverage" -B build
cmake --build build
ctest --test-dir build
lcov --capture --directory build --output-file coverage.info
genhtml coverage.info --output-directory coverage_html
```

## /flutter-test

**용도 설명**: Flutter/Dart 테스트. 테스트 문제를 보고하고 점진적으로 수정하며, 단위, 컴포넌트, 골든 및 통합 테스트를 다릅니다.

**사용 메서드**:
```
/flutter-test                # 모든 테스트 실행
/flutter-test --coverage     # 커버리지 포함
/flutter-test test/file.dart # 특정 테스트 파일 실행
```

**사용 시나리오**:
- 기능 구현 후破坏 없음 확인
- 새 코드의 테스트 커버리지 검사
- 특정 테스트 파일 수정
- PR 커밋 전

**일반적인 테스트 실패 및 수정**:

| 실패 유형 | 일반 수정 |
|----------|----------|
| `Expected: <X> Actual: <Y>` | assertion 업데이트 또는 구현 수정 |
| `Widget not found` | finder 선택기 수정 또는 widget 이름 업데이트 |
| `Golden file not found` | `flutter test --update-goldens` 실행하여 생성 |
| `MissingPluginException` | 테스트 설정에서 플랫폼 채널 mock |
| `pumpAndSettle timed out` | 명시적 `pump(Duration)`로 교체 |

## /test-coverage

**용도 설명**: 테스트 커버리지 분석, 격차 식별 및 목표 임계값 (기본 80%)에 도달하기 위한缺失 테스트 생성.

**사용 메서드**:
```
/test-coverage
```

**워크플로우**:
1. **테스트 프레임워크 감지** - 프로젝트 유형에 따라 커버리지 명령 선택
2. **커버리지 보고서 분석** - 80% 미만 파일 나열
3. **缺失 테스트 생성** - 우선순위별로 테스트 생성
4. **검증** - 커버리지 개선 확인 위해 다시 실행

**테스트 프레임워크 감지**:

| 지시자 | 커버리지 명령 |
|--------|----------|
| `jest.config.*` | `npx jest --coverage` |
| `vitest.config.*` | `npx vitest run --coverage` |
| `pytest.ini` | `pytest --cov=src --cov-report=json` |
| `Cargo.toml` | `cargo llvm-cov --json` |
| `go.mod` | `go test -coverprofile=coverage.out` |
