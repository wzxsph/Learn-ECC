# 핵심 워크플로우명령

本문서介绍 ECC 中用于코어개발워크플로우的명령。

---

## /plan

**용도설명**: 重述需求、评估风险，并생성分步实施계획。在触碰임의코드之前**等待用户확인**。

**사용메서드**:
```
/plan [기능描述 | PRD파일路径]
```

**사용场景**:
- 시작新기능개발
- 进行重大아키텍처变更
- 复杂的리팩토링작업
- 需求不明确或模糊时

**상세 설명**:
该명령会：
1. **重述需求** - 清晰表达필요빌드的内容
2. **识别风险** - 揭示潜在문제和阻碍
3. **생성分步계획** - 将实施分解为단계
4. **等待확인** - 필수获得用户明确승인后才能계속

지원多种输入패턴：
| 输入 | 패턴 | 行为 |
|------|------|------|
| `path/to/name.prd.md` | PRD 패턴 | 읽기 PRD，选择下一个보류的交付里程碑 |
| 기타 markdown 路径 | 引用패턴 | 읽기파일作为上下文并生成内联계획 |
| 自由형식文本 | 对话패턴 | 生成内联계획 |
| 空输入 | 澄清패턴 | 询问해야 함계획什么 |

## /code-review

**용도설명**: 本地未커밋变更或 GitHub PR 的全面코드 리뷰。

**사용메서드**:
```
/code-review                 # 本地리뷰패턴
/code-review [PR号 | PR链接]  # PR 리뷰패턴
```

**사용场景**:
- 커밋코드之前进行全面리뷰
- PR 리뷰以发现潜在문제
- 학습코드库中的모범 사례

**本地리뷰패턴**:
1. 收集变更파일（`git diff --name-only HEAD`）
2. 검사보안문제（평문 자격 증명、SQL注入、XSS等）
3. 검사코드 품질문제（함수>50行、파일>800行等）
4. 生成带심각级别标注的보고서

**PR 리뷰패턴**:
1. 가져오기 PR 元데이터和差异
2. 검사相关프로젝트사양和계획문서
3. 읽기完整变更파일
4. 应用 7 个리뷰클래스别（正确性、타입 안전성、패턴合规、보안、성능、完整性、유지보수성）
5. 실행검증명령（유형검사、lint、테스트、빌드）
6. 게시리뷰意见到 GitHub

## /build-fix

**용도설명**: 检测프로젝트빌드시스템并逐步수정빌드/유형오류。

**사용메서드**:
```
/build-fix
```

**사용场景**:
- 빌드실패时
- TypeScript/JavaScript 유형오류
- 컴파일오류
- 依赖문제

**워크플로우**:
1. **检测빌드시스템** - 根据파일유형选择合适的빌드명령
2. **파싱오류** - 按파일和依赖순서分组
3. **逐步수정** - 一次수정一个오류
4. **검증** - 每次수정后重新실행빌드

지원的빌드시스템：

| 指示器 | 빌드명령 |
|--------|----------|
| `package.json` + `build` 스크립트 | `npm run build` |
| `tsconfig.json` (TypeScript) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall` 或 `mypy` |

## /verify

**용도설명**: 검증코드变更是否按预期작업。用于검증 PR、확인수정或테스트变更。

**사용메서드**:
```
/verify [quick]           # 快速검증（권장）
/verify full             # 完整검증
```

**사용场景**:
- 검증 PR 更改
- 확인수정있음效
- 手动테스트变更
- 本地검증后再푸시

**검증단계**:
1. 실행相关테스트
2. 검사타입 안전성性
3. 검증빌드성공
4. 검사 linting

---

## /quality-gate

**용도설명**: 为파일或프로젝트范围실행 ECC 품질管道并보고서수정단계。

**사용메서드**:
```
/quality-gate [path|.] [--fix] [--strict]
```

**사용场景**:
- 按需실행품질검사
- 在 CI 管道中통합
- 确保코드符合프로젝트표준

**管道단계**:
1. 检测目标언어/도구
2. 실행형식化程序검사
3. 실행 lint/유형검사
4. 生成简洁的수정列表

## /tdd

**용도설명**: 테스트驱动개발워크플로우。遵循红-绿-리팩토링루프。

**사용메서드**:
```
/tdd                      # 시작 TDD 워크플로우
```

**사용场景**:
- 구현新기능
- 수정 bug（先写실패的테스트）
- 추가테스트覆盖
- 학습 TDD 메서드论

**TDD 루프**:
```
RED    → 写一个실패的테스트
GREEN  → 编写最少的코드使테스트통과
REFACTOR → 개선코드，테스트保持绿色
REPEAT → 下一个테스트用例
```

**모범 사례**:
- **先写테스트** - 在임의구현之前
- **每次修改后실행테스트** - 검증进度
- **테스트行为，而非구현** - 关注인터페이스而非细节
- **패키지含边界情况** - 空值、null、最大值等

## 명령통합关系

```
需求不明 → /plan-prd (생성 PRD)
     ↓
需求明确 → /plan (생성实施계획)
     ↓
/tdd (或 /feature-dev) 구현
     ↓
/build-fix 수정빌드오류
     ↓
/code-review 코드 리뷰
     ↓
/quality-gate 품질门禁
     ↓
/verify 검증
     ↓
/pr 생성 PR
```
