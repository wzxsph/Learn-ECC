# 핵심 워크플로우 명령

본 문서는 ECC에서 코어 개발 워크플로우용 명령을 소개합니다.

---

## /plan

**용도 설명**: 요구 사항 재진술, 위험 평가 및 단계별 구현 계획 생성. 임의 코드 작성 전에 **사용자 확인을 기다립니다**.

**사용 메서드**:
```
/plan [기능 설명 | PRD 파일 경로]
```

**사용 시나리오**:
- 새 기능 개발 시작
- 중대한 아키텍처 변경
- 복잡한 리팩토링 작업
- 요구 사항이 불분명하거나 모호할 때

**상세 설명**:
이 명령은:
1. **요구 사항 재진술** - 구축할 내용을 명확히 표현
2. **위험 식별** - 잠재적 문제 및 장애물 드러내기
3. **단계별 계획 생성** - 구현을 단계로 분해
4. **확인 대기** - 계속하려면 사용자의 명시적 승인을 받아야 함

여러 입력 패턴 지원:
| 입력 | 패턴 | 동작 |
|------|------|------|
| `path/to/name.prd.md` | PRD 패턴 | PRD를 읽고 다음 보류 중인 전달 마일스톤 선택 |
| 기타 markdown 경로 | 인용 패턴 | 파일을 컨텍스트로 읽고 인라인 계획 생성 |
| 자유 형식 텍스트 | 대화 패턴 | 인라인 계획 생성 |
| 빈 입력 |澄清 패턴 | 무엇을 계획해야 하는지 질문 |

## /code-review

**용도 설명**: 로컬 미커밋 변경 또는 GitHub PR의 포괄적인 코드 리뷰.

**사용 메서드**:
```
/code-review                 # 로컬 리뷰 패턴
/code-review [PR 번호 | PR 링크]  # PR 리뷰 패턴
```

**사용 시나리오**:
- 커밋하기 전에 포괄적인 리뷰
- 잠재적 문제를 발견하기 위한 PR 리뷰
- 코드베이스의 모범 사례 학습

**로컬 리뷰 패턴**:
1. 변경된 파일 수집 (`git diff --name-only HEAD`)
2. 보안 문제 검사 (평문 자격 증명, SQL 인젝션, XSS 등)
3. 코드 품질 문제 검사 (함수 >50행, 파일 >800행 등)
4. 심각도 수준이 표시된 보고서 생성

**PR 리뷰 패턴**:
1. PR 메타데이터 및 차이점 가져오기
2. 관련 프로젝트 사양 및 계획 문서 검사
3. 완전한 변경된 파일 읽기
4. 7개 리뷰 범주 적용 (정확성, 타입 안전성, 패턴 준수, 보안, 성능, 완전성, 유지보수성)
5. 검증 명령 실행 (유형 검사, lint, 테스트, 빌드)
6. GitHub에 리뷰 의견 게시

## /build-fix

**용도 설명**: 프로젝트 빌드 시스템 감지 및 점진적 빌드/유형 오류 수정.

**사용 메서드**:
```
/build-fix
```

**사용 시나리오**:
- 빌드 실패 시
- TypeScript/JavaScript 유형 오류
- 컴파일러 오류
- 의존성 문제

**워크플로우**:
1. **빌드 시스템 감지** - 파일 유형에 따라 적합한 빌드 명령 선택
2. **오류 파싱** - 파일 및 의존성 순서별로 그룹화
3. **점진적 수정** - 한 번에 하나의 오류 수정
4. **검증** - 각 수정 후 빌드 다시 실행

지원하는 빌드 시스템:

| 지시자 | 빌드 명령 |
|--------|----------|
| `package.json` + `build` 스크립트 | `npm run build` |
| `tsconfig.json` (TypeScript) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall` 또는 `mypy` |

## /verify

**용도 설명**: 코드 변경이 예상대로 작동하는지 검증. PR 검증, 수정 확인 또는 테스트 변경 검증에 사용.

**사용 메서드**:
```
/verify [quick]           #快速 검증 (권장)
/verify full             # 완전한 검증
```

**사용 시나리오**:
- PR 변경 검증
- 수정的有效性 확인
- 수동 테스트 변경
- 푸시 전 로컬 검증

**검증 단계**:
1. 관련 테스트 실행
2. 타입 안전성 검사
3. 빌드 성공 검증
4. linting 검사

## /quality-gate

**용도 설명**: 파일 또는 프로젝트 범위에 ECC 품질 파이프라인 실행 및 수정 단계 보고.

**사용 메서드**:
```
/quality-gate [path|.] [--fix] [--strict]
```

**사용 시나리오**:
- 요청 시 품질 검사 실행
- CI 파이프라인에 통합
- 코드가 프로젝트 표준을 준수하는지 확인

**파이프라인 단계**:
1. 대상 언어/도구 감지
2. 포맷터 검사 실행
3. lint/유형 검사 실행
4.简洁한 수정 목록 생성

## /tdd

**용도 설명**: 테스트驱动 개발 워크플로우.紅-绿-리팩토링 루프를 따릅니다.

**사용 메서드**:
```
/tdd                      # TDD 워크플로우 시작
```

**사용 시나리오**:
- 새 기능 구현
- 버그 수정 (먼저 실패하는 테스트 작성)
- 테스트 커버리지 추가
- TDD 방법론 학습

**TDD 루프**:
```
RED    → 실패하는 테스트 작성
GREEN  → 테스트를 통과하는 최소 코드 작성
REFACTOR → 테스트가 green 상태를 유지하면서 코드 개선
REPEAT → 다음 테스트 케이스로
```

**모범 사례**:
- **먼저 테스트 작성** - 임의 구현 이전에
- **수정 후마다 테스트 실행** - 진행 상황 검증
- **테스트 동작이지实现，而非** - 세부사항而非 인터페이스에 집중
- **경계 상황 포함** - null, 최대값 등

## 명령 통합 관계

```
요구 사항 불분명 → /plan-prd (PRD 생성)
     ↓
요구 사항 명확 → /plan (구현 계획 생성)
     ↓
/tdd (또는 /feature-dev) 구현
     ↓
/build-fix 빌드 오류 수정
     ↓
/code-review 코드 리뷰
     ↓
/quality-gate 품질 gate
     ↓
/verify 검증
     ↓
/pr PR 생성
```
