# 사용자 정의 개발

## 개요

본 가이드는 ECC Hooks 시스템용 사용자 정의 훅 개발 방법, 기본 구조, 모범 사례 및 exit 0 요구 사항을 소개합니다.

## 기본 구조

### 최소 훅 템플릿

```javascript
// my-custom-hook.js
let data = '';
process.stdin.on('data', chunk => data += chunk);
process.stdin.on('end', () => {
  const input = JSON.parse(data);

  // 도구 정보 접근
  const toolName = input.tool_name;        // "Edit", "Bash", "Write" 등
  const toolInput = input.tool_input;      // 도구 특정 매개변수
  const toolOutput = input.tool_output;    // PostToolUse에서만 사용 가능

  // 경고 (비블로킹): stderr에 쓰기
  console.error('[Hook] 경고 정보');

  //阻止 (PreToolUse만): exit code 2
  // process.exit(2);

  // 항상 원본 데이터를 stdout에 출력
  console.log(data);
});
```

## 종료码 요구 사항

### 핵심 규칙: exit 0 on 非핵심 오류

** 중요**: 모든 훅은非핵심 오류 시 exit 0해야 하며, 의도치 않게 도구 실행을 차단하지 않도록 해야 합니다.

| 종료码 | 의미 | 사용 시나리오 |
|--------|------|----------|
| 0 | 성공 | 계속 실행, 또는非핵심 경고 |
| 2 | 차단 | PreToolUse의 핵심 차단을 위해서만 |
| 기타 非零 | 오류 | 로그만 기록, **사용 금지** |

### 왜 非零 종료码가 나쁜가?

훅이 非零 종료码로退出하면 (2 제외), Claude Code가 전체 도구 호출을 실패로 표시하며, 이는 다음을 유발할 수 있습니다:
- 도구 실행이 의도치 않게 중단됨
- 사용자 경험受损
- 디버깅困难的 문제

### 올바른 오류 처리

```javascript
// 오류 예제 - 이렇게 하지 마세요
process.stdin.on('end', () => {
  try {
    const input = JSON.parse(data);
    // 처리 로직
  } catch (e) {
    console.error('[Hook] Error:', e.message);
    process.exit(1);  // 오류: 도구를 차단함
  }
  console.log(data);
});

// 올바른 예제 - 이렇게 하세요
process.stdin.on('end', () => {
  try {
    const input = JSON.parse(data);
    // 처리 로직
  } catch (e) {
    console.error('[Hook] Error:', e.message);
    // 非핵심 오류, exit 0으로 계속 실행
    console.log(data);
    return;
  }
  console.log(data);
});
```

## 훅 입력 패턴

### HookInput 인터페이스

```typescript
interface HookInput {
  tool_name: string;          // 도구 이름
  tool_input: {               // 도구 입력 매개변수
    command?: string;         // Bash: 명령
    file_path?: string;        // Edit/Write/Read: 파일 경로
    old_string?: string;       // Edit: 교체될 텍스트
    new_string?: string;       // Edit: 교체 텍스트
    content?: string;          // Write: 파일 내용
  };
  tool_output?: {             // PostToolUse만
    output?: string;          // 명령/도구 출력
  };
}
```

### 도구 정보 접근

```javascript
process.stdin.on('end', () => {
  const input = JSON.parse(data);

  // 도구 이름 가져오기
  console.log('Tool:', input.tool_name);

  // 도구 유형에 따라 처리
  if (input.tool_name === 'Bash') {
    console.log('Command:', input.tool_input.command);
  }

  if (input.tool_name === 'Edit' || input.tool_name === 'Write') {
    console.log('File:', input.tool_input.file_path);
  }

  // PostToolUse에서 출력 접근 가능
  if (input.tool_output) {
    console.log('Output:', input.tool_output.output);
  }
});
```

## 비동기 훅

주 흐름을 차단하지 않아야 하는 훅 (백그라운드 분석 등)의 경우:

```json
{
  "matcher": "Edit|Write|MultiEdit",
  "hooks": [{
    "type": "command",
    "command": "node my-slow-hook.js",
    "async": true,
    "timeout": 30
  }]
}
```

### 비동기 훅 주의사항

- 비동기 훅은 백그라운드에서 실행됨
- 도구 실행을 차단할 수 없음
- 30초 이내에 완료되어야 함
- 적합한 경우: 로그, 분석, 텔레메트리

## 블로킹 훅

도구 실행을 반드시 차단해야 하는 경우 (PreToolUse만):

```javascript
// PreToolUse에서 차단
process.exit(2);  // 종료码 2는 차단 표시
```

### 언제 차단 사용するか

- 보안 검사 실패 시 (키 감지 등)
- 하드 규칙 위반 시
- 데이터 손실 가능 작업 시

### 언제 차단 사용하지 않는か

- 코딩 스타일 문제 (경고 사용)
- 非핵심 검사
- 권장 사항 검사

## 성능 모범 사례

### 빠르게 유지

- PreToolUse 훅: <200ms
- PostToolUse 동기화 훅: <1초
- 비동기 훅: <30초

### 블로킹 작업 피하기

```javascript
// 나쁜 것: 동기화 파일 읽기
const content = fs.readFileSync('large-file.txt', 'utf8');

// 좋은 것: 비동기 또는 지연 로딩
fs.readFile('large-file.txt', 'utf8', (err, content) => {
  // 처리
});
```

### 결과 캐싱

```javascript
// 비용이 큰 작업 캐싱
let cachedResult = null;
let cacheTime = 0;
const CACHE_TTL = 60000; // 1분

function getCachedResult() {
  const now = Date.now();
  if (!cachedResult || now - cacheTime > CACHE_TTL) {
    cachedResult = expensiveOperation();
    cacheTime = now;
  }
```

## 테스트 훅

### 수동 테스트

```bash
# PreToolUse 훅 테스트
echo '{"tool_name":"Bash","tool_input":{"command":"echo test"}}' | node scripts/hooks/my-hook.js

# PostToolUse 훅 테스트
echo '{"tool_name":"Bash","tool_input":{"command":"echo test"},"tool_output":{"output":"test\n"}}' | node scripts/hooks/my-hook.js
```

### 테스트 출력 형식

```javascript
// 올바른 stdout 출력 (JSON)
console.log(data);  // 원본 입력 데이터

// 올바른 stderr 출력 (경고)
console.error('[Hook] Warning message');

// 올바르지 않은 방법 - 기타 내용 stdout에 출력하지 마세요
console.log('Some message');  // 이것은 데이터 흐름을破坏
```

### 일반적인 문제

| 문제 | 원인 | 해결책 |
|------|------|----------|
| 도구가 의도치 않게 차단됨 | 훅이 非零로退出 (2 제외) | exit 0 및 stderr 경고 사용으로 전환 |
| 걸림 | 훅이 종료되지 않음 | 항상 stdout에 출력해야 함 |
| 데이터 손상 | stdout에 非JSON 출력 | 원본 data만 출력 |
| 성능 문제 | 훅이 너무 느림 | 최적화 또는 비동기로 변경 |

## 모범 사례 요약

1. **항상 非핵심 오류 시 exit 0** - 非零 종료码 (2 제외) 사용 금지
2. **경고는 console.error 사용** - console.log 사용 금지
3. **빠르게 유지** - PreToolUse <200ms
4. **항상 원본 data를 stdout에 출력** - 데이터 흐름 변경 금지
5. **오래 걸리는 작업은 비동기 처리** - async: true 설정
6. **크로스 플랫폼 경로 처리** - path.join 사용
7. **출력 형식 테스트** - stdout이 원본 JSON인지 확인
8. **훅 문서화** - 명확한 description 추가
