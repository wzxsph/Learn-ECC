# 테스트 규칙

## 규칙 개요

ECC Rules 테스트 규칙은 코드 품질을 보장하는强制性 테스트 표준을 정의합니다. 이 규칙은 테스트驱动 개발(TDD),最低 커버리지 요구 및 테스트 구조 양식을 다루며, 모든 코드가 충분히 검증되도록 보장합니다.

## 코드 요구사항

### 최저 테스트 커버리지: 80%

세 가지 테스트 유형을 반드시 포함해야 합니다:

| 테스트 유형 | 설명 | 커버리지 범위 |
|----------|------|----------|
| 단위 테스트 | 독립 함수, 도구 클래스, 컴포넌트 | 개별 유닛 |
| 통합 테스트 | API 엔드포인트, 데이터베이스 작업 | 통합 지점 |
| E2E 테스트 | 핵심 사용자 플로우 | 중요 사용자 플로우 |

### 테스트驱动 개발 (TDD)

**强制性 워크플로우**:

```
1. 테스트 작성 (RED)   - 실패할 테스트를 먼저 작성
2. 테스트 실행          - 테스트 실패 확인
3. 최소 구현 작성 (GREEN) - 테스트를 통과하는 최소 코드 작성
4. 테스트 실행          - 테스트 통과 확인
5. 리팩토링 (IMPROVE)    - 코드 구조 개선
6. 커버리지 검증        - 80%+ 달성 확인
```

## 실시 세부사항

### AAA 패턴 (Arrange-Act-Assert)

모든 테스트는 AAA 구조를 따라야 합니다:

```typescript
test('코사인 유사도를 정확 계산', () => {
  // Arrange - 테스트 데이터 준비
  const vector1 = [1, 0, 0];
  const vector2 = [0, 1, 0];

  // Act - 테스트할 작업 실행
  const similarity = calculateCosineSimilarity(vector1, vector2);

  // Assert - 결과 검증
  expect(similarity).toBe(0);
});
```

### 테스트 명명 사양

테스트되는 동작을 설명하는 설명적 이름 사용:

```typescript
// 권장 - 예상 동작 설명
test('시장 일치 항목이 없을 때 빈 배열 반환', () => {});
test('API 키 누락 시 오류 발생', () => {});
test('Redis 사용 불가 시 하위 문자열 검색으로 대체', () => {});

// 피하기 - 모호한 명명
test('test1', () => {});
test('edge case', () => {});
```

### 테스트 문제 해결

테스트 실패 시:

| 단계 | 작업 |
|------|------|
| 1 | **tdd-guide** agent 사용 |
| 2 | 테스트 격리 확인 |
| 3 | Mock 정확성 검증 |
| 4 | 테스트가 아닌 구현 수정 (테스트 자체가 잘못된 경우만 해당) |

### Agent 지원

| Agent | 용도 | 사용 타이밍 |
|-------|------|----------|
| **tdd-guide** | 테스트驱动 개발 안내 | 새 기능 개발, 오류 수정 시 강제 사용 |
| **code-reviewer** | 코드 리뷰 | 코드 작성 완료 후 |

## 위반 처리

### 커버리지 부족

- 80% 미만 커버리지의 코드는 병합 불가
- CI/CD 파이프라인이 낮은 커버리지 코드 커밋을 차단해야 함
- `c8` 또는 유사 도구를 사용하여 커버리지 보고서 생성

### 테스트 실패 처리

```
1. 실패 원인 분석
2. 구현 문제인지 테스트 문제인지 진단
3. 구현 문제인 경우 - 구현 코드 수정
4. 테스트 문제인 경우 - 테스트 코드 수정
5. 모든 테스트 통과 후 커밋 확인
```

### 일반적인 문제

| 문제 | 원인 | 해결책 |
|------|------|----------|
| 불안정 테스트 | 비동기 작업, 경합 조건 | 대기 시간 증가, Mock 사용 |
| 테스트 격리 실패 | 공유 상태 오염 | 각 테스트 전 상태 재설정 |
| Mock不正确 | 기대치와 실제 불일치 | mock 설정이 정확한지 검증 |
