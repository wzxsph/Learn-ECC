# 코딩 스타일 규칙

## 규칙 개요

ECC Rules 코딩 스타일 규칙은 코드 구성의 핵심 원칙과 모범 사례를 정의합니다. 이러한 규칙은 코드가 가독성, 유지보수성 및 일관성을 유지하도록 보장하며, 불변성부터 명명 규칙에 이르기까지 모든 측면을 다룹니다.

## 코드 요구사항

### 불변성 (핵심 원칙)

**필수**: 항상 새 객체를 생성하고, 기존 객체는 절대 수정하지 마세요.

```typescript
// 오류 예제 - 원본 객체 수정
function modify(user, name) {
  user.name = name;  // 원본 객체를 변경함
  return user;
}

// 올바른 예제 - 새 객체 반환
function update(user, name) {
  return { ...user, name };  // 새副本 생성
}
```

**理由**: 불변 데이터는 숨은 부작용을防止하고, 디버깅을更容易하며, 보안 동시성을支持합니다.

### 핵심 설계 원칙

| 원칙 | 설명 |実践 |
|------|------|------|
| **KISS** | Keep It Simple | 가장 간단한可行方案 우선, 과도한 최적화避免,明晰优于巧妙 |
| **DRY** | Don't Repeat Yourself | 반복 로직을 공유 함수 또는 도구 클래스로 추출, 복사粘贴导致的实现漂移 避免 |
| **YAGNI** | You Aren't Gonna Need It | 미래에 필요할 특징이나 추상화を構築하지 마세요, 간단히 시작, 진짜 필요할 때 리팩토링 |

### 파일 구성

| 요구 | 설명 |
|------|------|
| 여러 소규모 파일 > 소수 대규모 파일 | 높은 응집도, 낮은 결합도 |
| 일반적인 줄 수 | 200-400행, 최대 800행 |
| 대규모 모듈 분리 | 대규모 모듈에서 도구 함수 추출 |
| 구성 방식 | 기능/영역별 구성, 유형별 아닌 |

### 오류 처리

| 요구 | 설명 |
|------|------|
| 명시적 처리 | 각 레이어에서 명시적으로 오류 처리 |
| UI 코드 | 사용자 친화적인 오류 메시지 제공 |
| 서버 측 |詳細な 오류 컨텍스트 기록 |
| 금지 행위 | 오류를 자동吞噬하지 마세요 |

### 입력 검증

| 요구 | 설명 |
|------|------|
| 경계 검증 | 시스템 경계에서 모든 입력 검증 |
| 스키마 검증 | 가능한 경우 스키마 기반 검증 사용 |
|快速실패 | 명확한 오류 메시지로快速실패 |
| 외부 데이터 | 외부 데이터를 절대 신뢰하지 마세요 (API 응답, 사용자 입력, 파일 내용) |

## 실시 세부사항

### 명명 규칙

| 유형 | 규칙 | 예제 |
|------|------|------|
| 변수 및 함수 | camelCase + 설명적 이름 | `calculateTotal`, `userName` |
| 부울 값 | is/has/should/can 접두사 | `isActive`, `hasPermission` |
| 인터페이스, 유형, 컴포넌트 | PascalCase | `UserProfile`, `ButtonComponent` |
| 상수 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT`, `API_TIMEOUT` |
| 커스텀 Hooks | camelCase + use 접두사 | `useUserData`, `useFetch` |

### 코드 스멜

#### 깊은 중첩

4层 이상 중첩은早期 리턴으로 리팩토링해야 합니다:

```typescript
// 피하기
function processOrder(order) {
  if (order) {
    if (order.items) {
      if (order.items.length > 0) {
        // 처리 로직
      }
    }
  }
}

// 권장
function processOrder(order) {
  if (!order || !order.items || order.items.length === 0) {
    return;
  }
  // 처리 로직
}
```

#### 매직 넘버

매직 넘버 대신 명명된 상수 사용:

```typescript
// 피하기
setTimeout(callback, 300000);

// 권장
const SESSION_TIMEOUT_MS = 5 * 60 * 1000; // 5분
setTimeout(callback, SESSION_TIMEOUT_MS);
```

#### 긴 함수

대규모 함수를 책임이 명확한 소규모 함수로 분리:

| 함수 길이 | 권장 |
|----------|------|
| <50행 | 이상적 |
| 50-100행 |容許, 분할 고려 |
| >100행 | 필수 분리 |

## 위반 처리

### 코드 품질 검사 목록

작업 완료 전에 반드시 확인:

- [ ] 코드가 가독성 있고 이름이良好
- [ ] 함수가 짧음 (<50행)
- [ ] 파일이 집중됨 (<800행)
- [ ] 깊은 중첩 없음 (>4层)
- [ ] 적절한 오류 처리
- [ ] 하드코딩된 값 없음 (상수 또는 설정 사용)
- [ ] 불변 패턴 사용 (변형 없음)

### 심각도 수준

| 수준 | 의미 | 작업 |
|------|------|------|
| CRITICAL | 보안 취약점 또는 데이터 손실 위험 | **차단** - 병합 전에 반드시 수정 |
| HIGH | 버그 또는重大的品質問題 | **경고** - 병합 전에 수정해야 함 |
| MEDIUM | 유지보수성 문제 | **정보** - 권장 수정 |
| LOW | 스타일 또는 작은 권장 | **참고** - 선택 |
