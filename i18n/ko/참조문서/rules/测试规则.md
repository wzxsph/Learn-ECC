# 테스트 규칙

## 규칙개요

ECC Rules 테스트 규칙定义了确保코드 품질的强制性테스트표준。该규칙涵盖테스트驱动개발(TDD)、最低커버리지요구和테스트구조사양，确保모든코드都经过充分검증。

## 코어요구

### 最低테스트커버리지：80%

필수패키지含以下三种테스트類型：

| 테스트類型 | 설명 | 覆盖范围 |
|----------|------|----------|
| 단위 테스트 | 独立함수、도구클래스、컴포넌트 | Individual units |
| 통합 테스트 | API端点、데이터베이스操作 | Integration points |
| E2E 테스트 | 핵심사용자 플로우 | Critical user flows |

### 테스트驱动개발（TDD）

**强制워크플로우**：

```
1. 编写테스트 (RED)   - 先写一个会실패的테스트
2. 실행테스트          - 검증테스트실패
3. 编写最小구현 (GREEN) - 编写통과테스트的最小코드
4. 실행테스트          - 검증테스트통과
5. 리팩토링 (IMPROVE)    - 개선코드구조
6. 검증커버리지        - 确保达到80%+
```

## 实施细节

### AAA패턴（Arrange-Act-Assert）

모든테스트필수遵循AAA구조：

```typescript
test('正确计算余弦相似度', () => {
  // Arrange - 准备테스트데이터
  const vector1 = [1, 0, 0];
  const vector2 = [0, 1, 0];

  // Act - 실행被테스트的操作
  const similarity = calculateCosineSimilarity(vector1, vector2);

  // Assert - 검증结果
  expect(similarity).toBe(0);
});
```

### 테스트命名사양

사용描述性名称설명被테스트的行为：

```typescript
// 권장 - 描述预期行为
test('当없음市场匹配查询时돌아가기空数组', () => {});
test('当API密钥缺失时抛出오류', () => {});
test('当Redis不可用时回退到子字符串搜索', () => {});

// 避免 - 模糊命名
test('test1', () => {});
test('edge case', () => {});
```

### 테스트故障排除

当테스트실패时：

| 단계 | 操作 |
|------|------|
| 1 | 사용 **tdd-guide** agent |
| 2 | 검사테스트隔离 |
| 3 | 검증Mock正确性 |
| 4 | 수정구현，而非테스트（除非테스트本身있음误） |

### Agent지원

| Agent | 용도 | 사용 타이밍 |
|-------|------|----------|
| **tdd-guide** | 테스트驱动개발指导 | 新기능개발、오류수정时强制사용 |
| **code-reviewer** | 코드 리뷰 | 코드编写완료后 |

## 违规处理

### 커버리지不足

- 커버리지低于80%的코드不得머지
- CI/CD流水线应阻止低커버리지코드커밋
- 사용 `c8` 或클래스似도구生成커버리지보고서

### 테스트실패处理

```
1. 분석실패原因
2. 진단是구현문제还是테스트문제
3. 如果是구현문제 - 수정구현코드
4. 如果是테스트문제 - 수정테스트코드
5. 确保모든테스트통과后再커밋
```

### 常见문제

| 문제 | 原因 | 해결方案 |
|------|------|----------|
| Flaky테스트 | 비동기操作、竞态条件 | 增加等待时间、사용mock |
| 테스트隔离실패 | 共享状态污染 | 각테스트前재설정状态 |
| Mock不正确 | 期望值与实际不符 | 검증mock설정是否正确 |
