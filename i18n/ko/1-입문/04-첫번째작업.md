# 04-첫 번째 임무

## 학습 목표

- 완전한 개발 플로 이해
- /ecc:plan → /ecc:code-review → /ecc:build-fix → 제출 의 완전한 플로 마스터
- 독립적으로 단순한 기능의 개발을 완료할 수 있다

---

## 완전한 개발 플로

ECC는 다음과 같은 개발 플로를 권장：

```
Step 1: /ecc:plan        → 임무 계획
Step 2: /ecc:code-review → 코드 리뷰
Step 3: /ecc:build-fix   → 빌드 수리
Step 4: 코드 제출     → Git 제출
```

---

## Step 1: /ecc:plan 사용 계획

### 입력

계산기 기능을 구현한다고 가정：

```
/ecc:plan 사칙연산(덧셈, 뺄셈, 곱셈, 나눗셈)을 지원하는 계산기를 구현
```

### 예상 출력

/ecc:plan 명령은：

1. **요구조件 재진술** - 「덧셈, 뺄셈, 곱셈, 나눗셈이라는 4가지 기본 연산을 지원하는 계산기를 구현...」
2. **위험 평가** - 「주의 필요: 나눗셈에서 0으로 나누는 경우...」
3. **분단 계획** -
   - Phase 1: 계산기 코어 함수 생성
   - Phase 2: 4가지 연산 구현
   - Phase 3: 에러 처리 추가
   - Phase 4: 유닛 테스트 작성
4. **확인 대기** - 「상기 계획 시작 가능 여부 확인하세요」

### 확인 후

사용자가 확인하면, ECC는 각 단계 시작을 기다린다.

### 팁

- PRD 파일 모드를 사용하면 더 상세한 계획을 얻을 수 있다
- 복잡한 기능은 먼저 PRD를 작성하고 계획

---

## Step 2: /ecc:code-review 사용 리뷰

### 입력

코드 작성 완료 후, 실행：

```
/ecc:code-review
```

### 예상 출력

/ecc:code-review 은：

1. 변경 파일 수집
2. 7개 카테고리로 리뷰：
   - 정확성
   - 타입 안전
   - 패턴 준수
   - 보안
   - 성능
   - 완전성
   - 유지보수성
3. CRITICAL/HIGH/MEDIUM/LOW 레벨로 리뷰 보고서 생성

### 리뷰 보고서 예

```
## 코드 리뷰 보고서

### CRITICAL
- [ ] 하드코딩된 자격 증명: src/auth.js 23행

### HIGH
- [ ] 함수가 너무 김: src/calculator.js 45-78행（34행）
- [ ] 에러 처리가 누락됨: src/calculator.js 나눗셈 함수

### MEDIUM
- [ ] 명명 규칙이 없음: 변수 `tmp`는 `temporary`로 변경해야 함

### 조작 제안
1. 하드코딩된 API Key 제거
2. calculator.js의 나눗셈 함수 리팩토링, 0으로 나누기 체크 추가
3. 너무 긴 함수 분리
```

### 팁

- 제출 전에 반드시 /ecc:code-review 실행
- CRITICAL와 HIGH 문제는 반드시 수정

---

## Step 3: /ecc:build-fix 사용 수리

### 입력

빌드가 실패한 경우, 실행：

```
/ecc:build-fix
```

### 예상 출력

/ecc:build-fix 은：

1. **빌드 시스템 감지** - 「TypeScript 프로젝트 감지」
2. **빌드 실행** - 「tsc --noEmit 실행...」
3. **에러 분석** - 파일별로 그룹화
4. **점진적 수리** - 한 번에 하나의 에러 수리
5. **검증** - 각 수리 후 재빌드

### 출력 예

```
[build-fix] TypeScript 프로젝트 감지
[build-fix] tsc --noEmit 실행...
[build-fix] 3개의 에러 발견：

에러 1: src/calculator.ts:15 - 타입 'string'은(는) 타입 'number'에 할당 불가
에러 2: src/calculator.ts:28 - 프로퍼티 'result'이(가) 존재하지 않음
에러 3: src/calculator.ts:42 - 파라미터 'b'이(가) 사용되지 않음

[build-fix] 에러 1 수리 시작...
[build-fix] 수리 완료, 검증 중...
[build-fix] 빌드 성공！
```

### 팁

- 빌드 수리는 증분적, 한 번에 하나의 에러 수리
- 언제든지 Ctrl+C로 중지 가능

---

## Step 4: 코드 제출

### 제출 플로

1. 모든 테스트가 통과ことを確認
2. 빌드가 성공ことを確認
3. 코드 리뷰 문제가 수정되었음을 확인

### Git 제출 규범

ECC는 conventional commits 포맷을 사용：

```
<type>: <description>

[optional body]
```

**Type 타입**:
- `feat` - 새 기능
- `fix` - 버그 수리
- `refactor` - 리팩토링
- `docs` - 문서
- `test` - 테스트
- `chore` - 일상 유지보수
- `perf` - 성능 최적화
- `ci` - CI 설정

### 제출 예

```
feat: 계산기 4가지 기본 연산 구현

- 덧셈 함수 추가
- 뺄셈 함수 추가
- 곱셈 함수 추가
- 나눗셈 함수 추가（0으로 나누기 체크 포함）

해결: Issue #123
```

### 팁

- 제출 전에 `/ecc:verify quick`로 빠른 검증
- 큰 기능은 여러 번에 나눠 제출, 작은 기능은 한 번에 제출

---

## 완전한 예

### 시나리오

단순한 문자열 유틸리티 함수 구현: `truncate(str, maxLength)`, 문자열을 지정된 길이로截断。

### Step 1: 계획

```
/ecc:plan truncate 함수 구현, 문자열을 지정된 길이로截断
```

출력：
```
## 요구사항 재진술
truncate(str, maxLength) 함수 구현, maxLength를 넘는 부분은 ...로 치환

## 위험 평가
- maxLength가 3보다 작은 경우의 처리
- str이 빈 문자열이거나 null인 경우의 처리
- Unicode 문자의 처리

## 분단 계획
1. src/string-utils.ts 생성
2. truncate 함수 구현
3. 경계 상황 테스트 추가
4. JSDoc 문서 추가

시작 가능 여부 확인하세요.
```

### Step 2: 구현

（사용자 확인 후, ECC가 코드 구현을協力）

### Step 3: 리뷰

```
/ecc:code-review
```

출력：
```
## 코드 리뷰 보고서
상태：APPROVED

### 리뷰 결과
- 정확성：통과
- 타입 안전：통과
- 패턴 준수：통과
- 보안：통과
- 성능：통과
- 완전성：더 많은 경계 테스트 추가 제안
- 유지보수성：통과

### 제안
- 빈 문자열 테스트 추가
- null 값 테스트 추가
```

### Step 4: 빌드 검증

```
/ecc:build-fix
```

```
[build-fix] TypeScript 프로젝트 감지
[build-fix] tsc --noEmit 실행...
[build-fix] 빌드 성공！
```

### Step 5: 제출

```
feat: truncate 문자열截断 함수 구현

- 지정된 최대 길이 지원
- 초과 부분은 ...로 치환
- 빈 문자열과 null 값 처리
```

---


##小结

1. **완전한 플로**: /ecc:plan → 구현 → /ecc:code-review → /ecc:build-fix → 제출
2. **/ecc:plan** 은 계획에 사용, 방향의 정확성確保
3. **/ecc:code-review** 은 리뷰에 사용, 문제 발견
4. **/ecc:build-fix** 은 빌드 에러 수리에 사용
5. **제출** 은 conventional commits 포맷을 사용
