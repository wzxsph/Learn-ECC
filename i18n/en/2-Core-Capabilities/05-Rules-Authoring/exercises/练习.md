# Exercises: Rules Authoring

## Exercise 1: Create Basic Rules

### Objective
Create basic rules for a project.

### Scenario
You need to create rules for a JavaScript project.

### Steps

1. **Define rule requirements**

   ```markdown
   # Project Rules Requirements

   ## Must-have rules
   1. No hardcoded credentials
   2. Minimum 80% test coverage
   3. Functions no more than 50 lines

   ## Should-have rules
   1. Consistent naming conventions
   2. No console.log in production code
   ```

2. **Create rule files**

   Create the following files:
   - `security/no-hardcoded-credentials.md`
   - `testing/minimum-coverage.md`
   - `coding-style/max-function-length.md`

3. **Add YAML frontmatter**

   ```yaml
   ---
   name: no-hardcoded-credentials
   severity: critical
   trigger: Bash
   ---
   ```

4. **Define check logic and fix suggestions**

   For each rule, define:
   - What to check
   - How to fix violations

### Acceptance Criteria
- [ ] Created at least 3 rule files
- [ ] Each rule has YAML frontmatter
- [ ] Each rule has check logic and fix suggestions
- [ ] Rules follow consistent format

## Exercise 2: Create Multi-level Rules

### Objective
Understand and implement rule priority and inheritance.

### Steps

1. **Understand rule levels**

   | Level | Location | Priority |
   |-------|----------|----------|
   | Global | `~/.claude/rules/` | Highest |
   | Project | `.claude/rules/` | Medium |
   | Local | Current session | Lowest |

2. **Design rule hierarchy**

   ```markdown
   # Rule Hierarchy Design

   ## Global Rules (Highest priority)
   - Security baseline
   - Basic code style

   ## Project Rules (Medium priority)
   - Project-specific requirements
   - Enhanced standards

   ## Local Rules (Lowest priority)
   - Session-specific overrides
   ```

3. **Create project rules that extend global rules**

   ```markdown
   # Project Security Rules

   Inherited from global security rules.

   ## Additional Requirements

   - API rate limiting required
   - Input validation for all endpoints
   ```

4. **Test priority override**

   Create a local rule that overrides a project rule:
   ```markdown
   ---
   name: local-override
   override: project:specific-rule
   ---
   ```

### Acceptance Criteria
- [ ] Understood rule priority system
- [ ] Created rules at different levels
- [ ] Tested rule inheritance
- [ ] Documented rule hierarchy

## Exercise 3: Implement Rule Enforcement

### Objective
Make rules enforceable through automation.

### Steps

1. **Choose enforcement mechanism**

   | Mechanism | Use case |
   |-----------|----------|
   | PreToolUse Hook | Block dangerous operations |
   | PostToolUse Hook | Log and analyze |
   | CI/CD pipeline | Enforce at build time |
   | Git hooks | Enforce at commit time |

2. **Create enforcing Hook**

   ```json
   {
     "name": "security-rules-hook",
     "matcher": {
       "tool": "Bash"
     },
     "condition": "command.includes('git commit')",
     "hooks": [
       {
         "type": "command",
         "when": "before",
         "command": "node scripts/enforce-rules.js",
         "continue": false
       }
     ]
   }
   ```

3. **Create enforcement script**

   `scripts/enforce-rules.js`:
   ```javascript
   const { execSync } = require('child_process');
   
   function checkRules() {
     const checks = [
       { name: 'Security', command: 'npm run check:security' },
       { name: 'Coverage', command: 'npm run check:coverage' },
       { name: 'Lint', command: 'npm run lint' }
     ];
   
     for (const check of checks) {
       try {
         execSync(check.command, { stdio: 'inherit' });
       } catch (e) {
         console.error(`✗ ${check.name} check failed`);
         return false;
       }
     }
     return true;
   }
   
   process.exit(checkRules() ? 0 : 1);
   ```

4. **Configure CI integration**

   ```yaml
   # .github/workflows/rules.yml
   name: Rules Enforcement
   on: [push, pull_request]
   jobs:
     enforce:
       runs-on: ubuntu-latest
       steps:
         - uses: actions/checkout@v2
         - name: Run Rule Checks
           run: node scripts/enforce-rules.js
   ```

### Acceptance Criteria
- [ ] Created enforcement mechanism
- [ ] Implemented Hook or script
- [ ] Integrated with CI/CD
- [ ] Documented enforcement process
