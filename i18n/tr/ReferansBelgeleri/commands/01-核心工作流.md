# Temel İş Akışı命令

本文档介绍 ECC 中Kullanılan核心开发工作流的命令。

---

## /plan

**KullanımAçıklama**: 重述需求、评估风险，并创建分步实施计划。在触碰任何代码之前**等待用户确认**。

**Kullanım Yöntemi**:
```
/plan [功能Tanım | PRD文件路径]
```

**Kullanım Senaryosu**:
- Başla新功能开发
- 进行重大架构变更
- 复杂的重构工作
- 需求不明确或模糊时

**详细Açıklama**:
该命令会：
1. **重述需求** - 清晰表达Gerekli构建的内容
2. **识别风险** - 揭示潜在问题和阻碍
3. **创建分步计划** - 将实施分解为阶段
4. **等待确认** - Zorunlu获得用户明确批准后才能继续

Destekle多种输入模式：
| 输入 | 模式 | 行为 |
|------|------|------|
| `path/to/name.prd.md` | PRD 模式 | 读取 PRD，选择下一个待处理的交付里程碑 |
| 其他 markdown 路径 | 引用模式 | 读取文件作为上下文并生成内联计划 |
| 自由格式文本 | 对话模式 | 生成内联计划 |
| 空输入 | 澄清模式 | 询问Olmalı规划什么 |

## /code-review

**KullanımAçıklama**: 本地未提交变更或 GitHub PR 的全面Kod İnceleme。

**Kullanım Yöntemi**:
```
/code-review                 # 本地审查模式
/code-review [PR号 | PR链接]  # PR 审查模式
```

**Kullanım Senaryosu**:
- 提交代码之前进行全面审查
- PR 审查以发现潜在问题
- 学习代码库中的En İyi Uygulamalar

**本地审查模式**:
1. 收集变更文件（`git diff --name-only HEAD`）
2. 检查Güvenlik问题（Sabit Kodlu Kimlik Bilgileri、SQL注入、XSS等）
3. 检查Kod Kalitesi问题（函数>50行、文件>800行等）
4. 生成带严重级别标注的报告

**PR 审查模式**:
1. 获取 PR 元数据和差异
2. 检查相关项目规范和规划文档
3. 读取完整变更文件
4. 应用 7 个审查Kategori（正确性、TürGüvenlik、模式合规、Güvenlik、性能、完整性、可维护性）
5. 运行Doğrulama命令（Tür检查、lint、Test、构建）
6. 发布审查意见到 GitHub

## /build-fix

**KullanımAçıklama**: 检测项目构建系统并逐步修复构建/Tür错误。

**Kullanım Yöntemi**:
```
/build-fix
```

**Kullanım Senaryosu**:
- 构建失败时
- TypeScript/JavaScript Tür错误
- 编译错误
- 依赖问题

**工作流程**:
1. **检测构建系统** - 根据文件Tür选择合适的构建命令
2. **解析错误** - 按文件和依赖顺序分组
3. **逐步修复** - 一次修复一个错误
4. **Doğrulama** - 每次修复后重新Yapılandırmayı Çalıştır

Destekle的构建系统：

| 指示器 | 构建命令 |
|--------|----------|
| `package.json` + `build` 脚本 | `npm run build` |
| `tsconfig.json` (TypeScript) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall` 或 `mypy` |

## /verify

**KullanımAçıklama**: Doğrulama代码变更是否按预期工作。KullanılanDoğrulama PR、确认修复或Test变更。

**Kullanım Yöntemi**:
```
/verify [quick]           # 快速Doğrulama（推荐）
/verify full             # 完整Doğrulama
```

**Kullanım Senaryosu**:
- Doğrulama PR 更改
- 确认修复有效
- 手动Test变更
- 本地Doğrulama后再推送

**DoğrulamaAdımlar**:
1. 运行相关Test
2. 检查TürGüvenlik性
3. Doğrulama构建成功
4. 检查 linting

## /quality-gate

**KullanımAçıklama**: 为文件或项目范围运行 ECC 质量管道并报告修复Adımlar。

**Kullanım Yöntemi**:
```
/quality-gate [path|.] [--fix] [--strict]
```

**Kullanım Senaryosu**:
- 按需运行质量检查
- 在 CI 管道中集成
- 确保代码符合项Hedef准

**管道Adımlar**:
1. 检测Hedef语言/工具
2. 运行格式化程序检查
3. 运行 lint/Tür检查
4. 生成简洁的修复列表

---

## /tdd

**KullanımAçıklama**: Test Güdümlü Geliştirme工作流。遵循红-绿-重构循环。

**Kullanım Yöntemi**:
```
/tdd                      # 启动 TDD 工作流
```

**Kullanım Senaryosu**:
- Gerçekleştir新功能
- 修复 bug（先写失败的Test）
- 添加Test覆盖
- 学习 TDD 方法论

**TDD 循环**:
```
RED    → 写一个失败的Test
GREEN  → 编写最少的代码使TestVasıtasıyla
REFACTOR → 改进代码，Test保持绿色
REPEAT → 下一个Test用例
```

**En İyi Uygulamalar**:
- **先写Test** - 在任何Gerçekleştir之前
- **每次修改后运行Test** - Doğrulama进度
- **Test行为，而非Gerçekleştir** - 关注接口而非细节
- **包含边界情况** - 空值、null、最大值等

## 命令集成关系

```
需求不明 → /plan-prd (创建 PRD)
     ↓
需求明确 → /plan (创建实施计划)
     ↓
/tdd (或 /feature-dev) Gerçekleştir
     ↓
/build-fix Yapı Hatalarını Onarma
     ↓
/code-review Kod İnceleme
     ↓
/quality-gate 质量门禁
     ↓
/verify Doğrulama
     ↓
/pr 创建 PR
```
