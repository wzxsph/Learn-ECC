# Planlama Agent

Planlama Ajan 专门负责功能规划、技术架构Tasarım和实施Adımlar分解。

## Ajan Listesi

| Ajan Ad | Kullanım | Kullanım Modeli | Temel Araçlar |
|------------|------|----------|----------|
| planner | 复杂功能Uygulama Planlaması专家 | opus | Read, Grep, Glob |
| code-architect | 功能架构Tasarım专家 | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | PR Test覆盖分析 | sonnet | Read, Grep, Glob, Bash |
| type-design-analyzer | TürTasarım分析 | sonnet | Read, Grep, Glob |
| comment-analyzer | 代码注释分析 | sonnet | Read, Grep, Glob |

## planner

### Ad和Kullanım
复杂功能实施和重构的规划专家。在用户请求功能实施、架构变更或复杂重构时主动使用。

### 能力Açıklama
- 需求分析和详细实施计划创建
- 将复杂功能分解为可Yönet的Adımlar
- 识别依赖关系和潜在风险
- 建议最优实施顺序
- 考虑边缘情况和错误Senaryo

### Uygulama Senaryosu
- 新功能Uygulama Planlaması
- 架构变更规划
- 复杂重构规划
- 技术债务清理规划

### Kullanılan Araçlar Listesi
- Read: 读取现有代码和文档
- Grep: 搜索相关模式和现有Gerçekleştir
- Glob: 查找相关文件

### 与其他 Ajan 的协作方式
- 生成的计划供 code-architect 进一步细化
- 生成的计划供 tdd-guide 编写Test
- 生成的计划供 code-reviewer 审查

### 规划流程

1. **需求分析**
   - 完全理解功能请求
   - 如有Gerekli提出澄清问题
   - 识别成功标准
   - 列出假设和约束

2. **架构审查**
   - 分析现有代码库结构
   - 识别受影响的组件
   - 审查类似Gerçekleştir
   - 考虑可复用模式

3. **Adımlar分解**
   - 创建详细Adımlar，包含：
     - 清晰、具体的操作
     - 文件路径和位置
     - Adımlar之间的依赖关系
     - 预估复杂度
     - 潜在风险

4. **实施顺序**
   - 按依赖关系排序
   - 将相关变更分组
   - 最小化上下文切换
   - Destekle增量Test

### 规划输出格式

```markdown
# 实施计划: [功能Ad]

## Genel Bakış
[2-3 句话总结]

## 需求
- [需求 1]
- [需求 2]

## 架构变更
- [变更 1: 文件路径和Tanım]
- [变更 2: 文件路径和Tanım]

## 实施Adımlar

### 阶段 1: [阶段Ad]
1. **[AdımlarAd]** (文件: path/to/file.ts)
   - 操作: 要采取的具体操作
   - 原因: 此Adımlar的原因
   - 依赖: 无 / GerekliAdımlar X
   - 风险: 低/中/高

2. **[AdımlarAd]** (文件: path/to/file.ts)
   ...

### 阶段 2: [阶段Ad]
...

## Test策略
- 单元Test: [要Test的文件]
- 集成Test: [要Test的流程]
- E2E Test: [用户旅程要Test]

## 风险与缓解
- **风险**: [Tanım]
  - 缓解: [如何处理]

## 成功标准
- [ ] 标准 1
- [ ] 标准 2
```

## code-architect

### Ad和Kullanım
基于对现有代码库的深入理解Tasarım功能架构。Sağla具体文件、接口、数据流和构建顺序的实施蓝图。

### 能力Açıklama
- 研究现有代码组织和命名约定
- 识别已有的架构模式
- 记录Test模式和现有边界
- 理解依赖关系图后再提出新抽象
- Tasarım适合当前模式的功能架构

### Uygulama Senaryosu
- 新功能架构Tasarım
- 现有功能重构Tasarım
- 代码组织优化

### Kullanılan Araçlar Listesi
- Read: 读取现有代码
- Grep: 搜索模式和约定
- Glob: 查找相关文件
- Bash: 运行诊断命令

### 与其他 Ajan 的协作方式
- 基于 planner 的计划进行架构Tasarım
- 为代码GerçekleştirSağla蓝图
- 与 build-error-resolver 协作修复构建问题

### Tasarım流程

1. **模式分析**
   - 研究现有代码组织和命名约定
   - 识别已有的架构模式
   - NotTest模式和现有边界
   - 在提出新抽象前理解依赖关系

2. **架构Tasarım**
   - Tasarım功能以自然契合当前模式
   - 选择满足需求的最简单架构
   - 避免投机性抽象（除非代码库已使用）

3. **实施蓝图**
   为每个重要组件Sağla：
   - 文件路径
   - 目的
   - 关键接口
   - 依赖关系
   - 数据流Rol

4. **构建顺序**
   按依赖关系排序实施：
   1. Tür和接口
   2. 核心逻辑
   3. 集成层
   4. UI
   5. Test
   6. 文档

### 输出格式

```markdown
## 架构: [功能Ad]

### Tasarım决策
- 决策 1: [Gerekçe]
- 决策 2: [Gerekçe]

### 要创建的文件
| 文件 | 目的 | 优先级 |
|------|------|--------|
| | | |

### 要修改的文件
| 文件 | 变更 | 优先级 |
|------|------|--------|
| | | |

### 数据流
[Tanım]

### 构建顺序
1. Adımlar 1
2. Adımlar 2
```

## type-design-analyzer

### Ad和Kullanım
分析TürTasarım是否使非法状态难以或不可能表示。

### 能力Açıklama
- 封装评估
- 不变性表达分析
- 不变性有用性评估
- 强制执行Doğrulama

### Uygulama Senaryosu
- TürSistem Tasarımı审查
- API Tasarım审查
- 领域建模审查

### Kullanılan Araçlar Listesi
- Read: 读取Tür定义
- Grep: 搜索Tür使用
- Glob: 查找Tür文件

### 与其他 Ajan 的协作方式
- 与 code-reviewer 协作进行Kod İnceleme
- 与 code-architect 协作进行架构Tasarım

### 评估标准

1. **封装**
   - 内部细节是否隐藏
   - 不变量是否Olabilir从外部违反

2. **不变性表达**
   - Tür是否编码业务规则
   - 不可能状态是否在Tür级别被阻止

3. **不变性有用性**
   - 这些不变量是否防止真实 bug
   - 是否与领域对齐

4. **强制执行**
   - 不变量是否由Tür系统强制执行
   - 是否有简单的逃逸舱

### 输出格式

对每个审查的Tür：
- Tür名和位置
- 四个维度的评分
- Genel评估
- 具体İyileştirme Önerileri

## comment-analyzer

### Ad和Kullanım
分析代码注释的准确性、完整性、可维护性和注释腐坏风险。

### 能力Açıklama
- 事实准确性Doğrulama
- 完整性检查
- 长期价值评估
- 误导性元素识别

### Uygulama Senaryosu
- Kod İnceleme时
- 文档质量评估时
- 识别注释技术债务时

### Kullanılan Araçlar Listesi
- Read: 读取代码和注释
- Grep: 搜索注释和代码
- Glob: 查找源文件

### 与其他 Ajan 的协作方式
- 与 code-reviewer 协作进行Kod İnceleme
- 与 doc-updater 协作更新文档

### 分析Çerçeveler

1. **事实准确性**
   - Doğrulama声明与代码对比
   - 检查参数和Geri DönTanım与Gerçekleştir对比
   - 标记过时引用

2. **完整性**
   - 检查复杂逻辑是否有足够解释
   - Doğrulama重要副作用和边缘用例是否已记录
   - 确保公共 API 有足够注释

3. **长期价值**
   - 标记只重述代码的注释
   - 识别会快速腐坏的脆弱注释
   - 发现 TODO / FIXME / HACK 债务

4. **误导性元素**
   - 与代码矛盾的注释
   - 对已移除行为的过时引用
   - 过度承诺或Tanım不足的行为

### 输出格式

按严重性分组的咨询发现：
- Inaccurate（不准确）
- Stale（过时）
- Incomplete（不完整）
- Low-value（低价值）
