## tdd-guide

### Ad和Kullanım
Test Güdümlü Geliştirme专家，强制执行写Test优先的方法论。主动Kullanılan编写新功能、修复 bug 或重构代码。确保 80%+ 的Test覆盖率。

### 能力Açıklama
- 强制执行Test优先的方法论
- 指导 Red-Green-Refactor 周期
- 确保 80%+ Test覆盖率
- 编写全面的Test套件（单元、集成、E2E）
- 在实施前捕获边缘用例

### Uygulama Senaryosu
- 编写新功能时
- 修复 bug 时
- 重构代码时
- Gerekli TDD 工作流程时

### Kullanılan Araçlar Listesi
- Read: 读取现有代码
- Write: 编写Test
- Edit: 修改Test和Gerçekleştir
- Bash: 运行Test命令
- Grep: 搜索Test模式

### 与其他 Ajan 的协作方式
- code-reviewer 在 TDD 周期后Kodu İncele
- build-error-resolver 修复Test构建错误
- e2e-runner 进行关键用户流的 E2E Test

### TDD 工作流

#### 1. 先写Test (RED)
编写Tanım预期行为的失败Test。

#### 2. 运行Test - Doğrulama它 FAIL
```bash
npm test
```

#### 3. 写最小Gerçekleştir (GREEN)
只写足够的代码让TestVasıtasıyla。

#### 4. 运行Test - Doğrulama它 PASS

#### 5. 重构 (IMPROVE)
移除重复，改进Ad，优化 - TestZorunlu保持绿色。

#### 6. Doğrulama覆盖率
```bash
npm run test:coverage
# 要求: 80%+ 分支、函数、行、语句覆盖率
```

### ZorunluTest的边缘用例

1. **Null/Undefined** 输入
2. **空** 数组/字符串
3. **无效Tür** 传递
4. **边界值** (最小/最大)
5. **错误路径** (网络失败、数据库错误)
6. **竞态条件** (并发操作)
7. **大数据** (10k+ 项的性能)
8. **特殊字符** (Unicode、表情符号、SQL 字符)

### 反模式 - Zorunlu避免

- TestGerçekleştir细节（内部状态）而非行为
- Test相互依赖（共享状态）
- 断言太少（Vasıtasıyla但不Doğrulama任何东西）
- 外部依赖未 mock (Supabase、Redis、OpenAI 等)

### 质量检查清单

- [ ] 所有公共函数都有单元Test
- [ ] 所有 API 端点都有集成Test
- [ ] 关键用户流有 E2E Test
- [ ] 边缘用例已覆盖 (null、empty、invalid)
- [ ] 错误路径已Test（不只是 happy path）
- [ ] 外部依赖已 mock
- [ ] Test相互独立（无共享状态）
- [ ] 断言具体且有意义
- [ ] 覆盖率 80%+

### v1.8 Eval-Driven TDD 附录

将 eval-driven 开发集成到 TDD 流程：

1. 在实施前定义能力 + 回归 evals
2. 运行基线并捕获失败特征
3. Gerçekleştir最小Vasıtasıyla变更
4. 重新运行Test和 evals；报告 pass@1 和 pass@3
5. 发布关键路径应在合并前达到 pass^3 稳定性

### 主要工具: Ajan Browser

**优先于原始 Playwright 使用 Ajan Browser** - 语义选择器、AI 优化、自动等待、基于 Playwright。

```bash
# 设置
npm install -g agent-browser && agent-browser install

# Temel İş Akışı
agent-browser open https://example.com
agent-browser snapshot -i          # 获取带 refs 的元素 [ref=e1]
agent-browser click @e1            # 按 ref 点击
agent-browser fill @e2 "text"     # 按 ref 填充输入
agent-browser wait visible @e5     # 等待元素可见
agent-browser screenshot result.png
```

### 后备: Playwright

当 Ajan Browser 不可用时，直接使用 Playwright。

```bash
npx playwright test                        # 运行所有 E2E Test
npx playwright test tests/auth.spec.ts     # 运行特定文件
npx playwright test --headed               # 可见浏览器
npx playwright test --debug                # 使用检查器调试
npx playwright test --trace on             # 带 trace 运行
npx playwright show-report                 # 查看 HTML 报告
```

### 工作流

#### 1. 计划
- 识别关键用户旅程（认证、核心功能、支付、CRUD）
- 定义Senaryo: happy path、边缘用例、错误用例
- 按风险排序: HIGH（金融、认证）、MEDIUM（搜索、导航）、LOW（UI 优化）

#### 2. 创建
- 使用页面对象模型 (POM) 模式
- 优先使用 `data-testid` 选择器
- 在关键Adımlar添加断言
- 在关键时刻捕获截图
- 使用适当的等待（从不 `waitForTimeout`）

#### 3. 执行
- 本地运行 3-5 次检查不稳定性
- 使用 `test.fixme()` 或 `test.skip()` 隔离不稳定Test
- 上传 artifact 到 CI

### 关键原则

- **使用语义选择器**: `[data-testid="..."]` > CSS 选择器 > XPath
- **等待条件而非时间**: `waitForResponse()` > `waitForTimeout()`
- **自动等待内置**: `page.locator().click()` 自动等待；原始 `page.click()` 不等待
- **隔离Test**: 每个TestOlmalı独立；无共享状态
- **快速失败**: 在每个关键Adımlar使用 `expect()` 断言
- **重试时 trace**: 配置 `trace: 'on-first-retry'` Kullanılan调试失败

### 不稳定Test处理

```typescript
// 隔离
test('flaky: market search', async ({ page }) => {
  test.fixme(true, 'Flaky - Issue #123')
})

// 识别不稳定性
// npx playwright test --repeat-each=10
```

常见原因: 竞态条件（使用自动等待选择器）、网络计时（等待响应）、动画计时（等待 `networkidle`）。

### 成功指标

- 所有关键旅程Vasıtasıyla (100%)
- 总体Vasıtasıyla率 > 95%
- 不稳定率 < 5%
- Test持续时间 < 10 分钟
- Artifact 上传并可访问

## mle-reviewer

### Ad和Kullanım
生产机器学习工程审查专家，专注于数据契约、特征管道、训练可复现性、离线/在线评估、模型服务、监控和回滚。

- 服务和部署Güvenlik审查
- 监控和事件响应规划

### Uygulama Senaryosu

- 推理、特征存储、评估代码变更时

### 与其他 Ajan 的协作方式
- python-reviewer 处理 Python 风格、Tür、Hata Yönetimi
- pytorch-build-resolver 处理 tensor/CUDA/训练失败
- database-reviewer 处理特征表、标签存储
- security-reviewer 处理 secrets、PII、pickle Güvenlik
- performance-optimizer 处理延迟、内存、GPU 利用率
- build-error-resolver 处理 CI、依赖、原生扩展失败
- pr-test-analyzer 分析Test覆盖
- silent-failure-hunter 发现管道静默失败
- e2e-runner 进行产品流 E2E Test
- a11y-architect 审查预测可访问性
- doc-updater 更新文档

### 关键审查领域

#### 问题定义和决策质量
- 变更是否从用户或系统决策Başla，而非模型架构偏好
- 利益相关者和失败成本是否明确
- 指标选择是否遵循错误预算

#### 指标、阈值和错误分析
- 基线和当前生产行为是否比较
- 阈值和配置是否作为产品决策
- 假阳性和假阴性是否直接检查

#### 数据契约和泄漏
- 实体粒度、主键、时间戳是否明确
- 分割是否遵守时间、用户/实体分组
- 特征连接是否时间点正确
- 敏感属性是否排除或正当处理

#### 训练可复现性
- 训练是否可从代码、配置、数据集版本、种子运行
- 超参数、预处理、依赖版本是否记录
- 随机性和 GPU 非确定性是否有意处理

#### 评估和推广
- 指标是否与基线和当前生产模型比较
- 推广门是否在选择前声明
- 切片指标是否覆盖重要队列

#### 服务和部署
- 训练和服务转换是否共享或等价Test
- 输入模式是否拒绝过时、缺失、无效特征
- 推理路径是否有超时、资源限制、回退逻辑

#### 监控和事件响应
- 监控是否覆盖服务健康、特征漂移、预测漂移
- 日志是否包含足够标识符
- 回滚是否命名前一 artifact、配置、数据依赖

### 常见阻塞器

- 在时间相关或用户相关数据上随机 train/test 分割
- 特征生成使用预测时不可用的字段
- 离线指标改进而关键切片回归
- 训练预处理被手动复制到服务代码
- 模型版本不在预测日志中

### 批准标准

- **APPROVE**: 无关键/高 MLE 风险，相关Test或评估门Vasıtasıyla
- **APPROVE WITH WARNINGS**: 只有中等问题，有明确的后续行动
- **BLOCK**: 任何可能的泄漏、不可复现的推广、不Güvenlik的服务行为、缺少生产部署回滚、敏感数据暴露、或关键评估差距

## pr-test-analyzer

### Ad和Kullanım
审查 Pull Request Test覆盖质量和完整性，强调行为覆盖和真实 bug 预防。

### 能力Açıklama
- 识别变更代码映射
- Doğrulama行为覆盖
- 评估Test质量
- 发现覆盖差距

### Uygulama Senaryosu
- PR 审查时
- 评估Test覆盖率时
- DoğrulamaTest充分性时

### Kullanılan Araçlar Listesi
- Read: 读取变更代码和Test
- Grep: 搜索Test
- Glob: 查找Test文件
- Bash: 运行Test命令

### 与其他 Ajan 的协作方式
- tdd-guide 改进Test
- e2e-runner 进行端到端覆盖
- code-reviewer 进行Kod Kalitesi审查

### 分析流程

#### 1. 识别变更代码
- 映射变更的函数、类和模块
- 定位对应Test
- 识别新的未Test代码路径

#### 2. 行为覆盖
- 检查每个功能是否有Test
- Doğrulama边缘用例和错误路径
- 确保重要集成被覆盖

#### 3. Test质量
- 优先选择有意义的断言而非无抛出检查
- 标记不稳定的模式
- 检查隔离性和TestAd清晰度

#### 4. 覆盖差距
按影响评级：
- critical（关键）
- important（重要）
- nice-to-have（锦上添花）

### 输出格式

1. 覆盖总结
2. 关键差距
3. İyileştirme Önerileri
4. 积极观察
