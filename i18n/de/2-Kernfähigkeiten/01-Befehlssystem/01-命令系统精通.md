# 01 - Befehlssystem meistern

Tiefgreifend das ECC-Befehlssystem lernen, alle Befehlstypen und Nutzungsszenarien beherrschen.

## Übersicht

ECC bietet **75 [Befehle](../../Referenzdokumente/commands/01-核心工作流.md)**, unterteilt in drei Hauptkategorien: Schrägstrich-Befehle (Slash Commands), Skript-Befehle und Agent-Befehle. Diese Befehle verbessern die Entwicklungseffizienz erheblich durch standardisierte Workflows.

## Befehlsdateiformat

Jeder Befehl ist eine `.md`-Datei, enthält YAML Frontmatter + Markdown-Haupttext:

```yaml
---
description: "Kurze Beschreibung des Befehls"
argument-hint: "[Optionaler Parameterhinweis]"
name: befehl-name          # Optional, Standard verwendet Dateiname
command: true              # Optional, CLI Passthrough-Befehl
allowed_tools: ["Bash"]    # Optional, verfügbare Tools einschränken
---
```

### Frontmatter-Felder

| Feld | Erforderlich | Beschreibung |
|------|------|------|
| description | Ja | Angezeigt in `/` Befehlsliste |
| argument-hint | Nein | Parameterhinweis |
| name | Nein | Dateinamen als Befehlsnamen überschreiben |
| command | Nein | Als CLI Passthrough-Befehl markieren |
| allowed_tools | Nein | Verfügbare Tools einschränken |

## 2. Befehlsketten und Kombinationen

Mehrere Befehle können kombiniert werden um komplexe Workflows zu bilden:

```markdown
# Typischer Entwicklungsworkflow
/plan → /tdd → /code-review → /verify → /pr

# Problembehebungsworkflow
/build-fix → /code-review → /test-coverage

# Multi-Modell-Kollaboration
/multi-plan → /multi-execute → /pr
```

## 3. Benutzerdefinierte Befehlserstellung

### Befehlsspeicherorte

| Typ | Pfad |
|------|------|
| Befehlsdateien | `commands/` |
| Registrierungsmechanismus | `.claude-plugin/plugin.json` |

### Erstellungsschritte

1. `.md`-Datei im `commands/`-Verzeichnis erstellen
2. YAML Frontmatter hinzufügen
3. Befehlsinhalt (Markdown-Format) schreiben
4. Befehl automatisch bei ECC registrieren

## 4. Best Practices

### Befehlsauswahl-Leitfaden

| Szenario | Empfohlene Befehle |
|------|----------|
| Neue Funktionsentwicklung | `/plan` → `/tdd` |
| Code-Review | `/code-review` |
| Bug-Behebung | `/build-fix` → `/verify` |
| Testgetrieben | `/tdd` → `/test-coverage` |
| PR erstellen | `/pr` |
| Muster lernen | `/learn` → `/evolve` |

### Befehlsausführungsreihenfolge

Workflow-Prinzip folgen:
1. Planung: `/plan` oder `/prp-plan`
2. Implementierung: `/tdd` oder Feature-Befehle
3. Review: `/code-review`
4. Verifizierung: `/verify` oder `/quality-gate`
5. Commit: `/pr`
