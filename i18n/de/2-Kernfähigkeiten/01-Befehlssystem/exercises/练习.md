# Übung: Befehlssystem meistern

## Übung 1: Alle Befehlskategorien erkunden

### Ziel
Alle Befehlskategorien von Claude Code kennenlernen und jeden Kategorie-Schlüsselbefehl identifizieren.

### Schritte

1. **Alle verfügbaren Befehle auflisten**
   - In Claude Code `/` eingeben um alle Befehle zu sehen
   - Auf Befehlskategorie-Organisationsweise achten

2. **Jede Kategorie erkunden**
   - Für jede Kategorie `/help` oder `?` verwenden um Hilfe zu erhalten
   - Schlüsselbefehle und Verwendung jeder Kategorie notieren

3. **Befehlsliste aufzeichnen**
   Eigene Befehls-Übersicht erstellen:

   | Kategorie | Befehl | Verwendung | Häufigkeit |
   |------|------|------|----------|
   | Entwicklungsworkflow | /plan | Implementierungsplan erstellen | ★★★ |
   | ... | ... | ... | ... |

### Abnahmekriterien
- [ ] Mindestens 5 Kategorien mit Befehlen auflisten können
- [ ] 20+ häufig verwendete Befehle notiert
- [ ] Befehlskategorie-Logik verstanden

## Übung 2: Mehrere Befehle kombiniert verwenden

### Ziel
Lernen wie man mehrere Befehle kombiniert um kohärente Entwicklungsworkflows zu erstellen.

### Szenario
Einen angenommenen Build-Fehler beheben und Code-Review durchführen.

### Schritte

1. **Szenario simulieren**
   Angenommen, du entwickelst ein Go-Projekt und hast Build-Fehler:

   ```
   # Szenario-Setup (nicht tatsächlich ausführen)
   Angenommen folgende Build-Fehler müssen behoben werden:
   - import cycle detected
   - undefined: someFunction
   ```

2. **Befehlskombinations-Übung**
   Folgende Befehlskombination in Reihenfolge denken und notieren:

   a. Zuerst `/plan` verwenden um Reparatur-Schritte zu planen
   b. `/go-build` verwenden um Build zu versuchen
   c. `/build-fix` verwenden um Fehler zu beheben
   d. `/code-review` verwenden um zu reviewen
   e. `/verify` verwenden um Reparatur zu verifizieren

3. **Workflow aufzeichnen**
   Dein Reparatur-Workflow-Dokument erstellen:

   ```markdown
   ## Build-Reparatur-Workflow

   1. /plan - Problem analysieren, Reparaturplan erstellen
   2. /build-fix - Reparatur ausführen
   3. /code-review - Reparaturqualität prüfen
   4. /verify - Funktionalität verifizieren
   ```

### Abnahmekriterien
- [ ] Befehlskombinations-Logik verstanden
- [ ] Mindestens 3 benutzerdefinierte Workflows erstellt
- [ ] Best Practices für Befehlskombination notiert

## Übung 3: Benutzerdefinierte Workflows erstellen

### Ziel
Basierend auf Projektanforderungen automatisierte Workflow-Befehlskombinationen erstellen.

### Szenario
Entwicklungsworkflow für ein angenommenes Flutter-Projekt erstellen.

### Schritte

1. **Workflow-Ziele definieren**
   Dein Flutter-Projekt braucht folgende Workflows:

   - Tägliches Development: Coding → Testen → Bauen
   - Code-Review: Selbstprüfung → Review → Reparatur
   - Release-Vorbereitung: Testen → Bauen → Verifizieren

2. **Befehlskombinationen entwerfen**

   **Täglicher Development-Workflow**:
   ```
   /flutter-build → /flutter-test → /verify
   ```

   **Code-Review-Workflow**:
   ```
   /flutter-review → /build-fix → /code-review
   ```

3. **Workflow-Dokument erstellen**
   `CLAUDE_WORKFLOWS.md` im Projekt erstellen:

   ```markdown
   # Claude Code Workflows

   ## Flutter Development Workflow

   ### Tägliches Development
   1. `/plan` verwenden um Aufgabe zu planen
   2. `/flutter-build` verwenden zum Bauen
   3. `/flutter-test` verwenden zum Testen
   4. `/verify` verwenden zum Verifizieren

   ### Code-Review
   1. `/flutter-review` verwenden zur Selbstprüfung
   2. `/build-fix` verwenden um Probleme zu beheben
   3. `/code-review` verwenden um Review anzufordern

   ### Release-Vorbereitung
   1. Vollständigen Test-Suite ausführen
   2. Build-Verifizierung ausführen
   3. `/verify` für finale Bestätigung verwenden
   ```

4. **Praxis-Verifizierung**
   Wenn reales Projekt vorhanden, diesen Workflow ausführen.

### Abnahmekriterien
- [ ] 3 vollständige Workflows entworfen
- [ ] Workflow-Dokument erstellt
- [ ] Mindestens 1 Workflow in echtem Projekt getestet

## Übungsantworten-Referenz

### Übung 1 Befehlsliste-Beispiel

| Kategorie | Befehl | Verwendung |
|------|------|------|
| Entwicklungsworkflow | /plan | Implementierungsplanung |
| Entwicklungsworkflow | /tdd | Testgetriebene Entwicklung |
| Entwicklungsworkflow | /build-fix | Build-Reparatur |
| Entwicklungsworkflow | /verify | Änderungsverifizierung |
| Code-Review | /code-review | Generisches Code-Review |
| Code-Review | /security-review | Sicherheitsreview |
| Build | /go-build | Go-Build |
| Build | /rust-build | Rust-Build |
| Projekt | /project-init | Projektinitialisierung |
| Projekt | /projects | Projektliste |
| Skill | /skill-create | Skill erstellen |
| Skill | /learn | Lernen-Extraktion |
| Lernen | /deep-research | Tiefgehende Recherche |

### Übung 3 Workflow-Vorlage

```markdown
# [Projektname] Claude Code Workflows

## Entwicklungsworkflow
- Täglich: /plan → [Coding] → /verify
- Reparatur: /build-fix → /code-review → /verify
- Release: /test → /build → /verify

## Nutzungshinweise
1. `/plan` verwenden um jede Aufgabe zu starten
2. Nach Coding `/verify` ausführen
3. Für wichtige Änderungen `/code-review` verwenden
```
