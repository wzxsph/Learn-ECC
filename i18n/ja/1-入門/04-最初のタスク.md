# 04-最初任務

## 学習目標

- 完全な開発フローを理解する
- /ecc:plan → /ecc:code-review → /ecc:build-fix → 提交の完全フローを掌握
- 独立して単純な機能の开发を完了できる

---

## 完全な開発フロー

ECCは 다음과 같은開発フローを推奨：

```
Step 1: /ecc:plan        → 任務を計画
Step 2: /ecc:code-review → コードをレビュー
Step 3: /ecc:build-fix   → ビルドを修復
Step 4: コードを提交     → Git提交
```

---

## Step 1: /ecc:plan を使用計画

### 入力

計算機機能を実装すると仮定します：

```
/ecc:plan 加減乗除の4種類の演算をサポートする計算機を実装
```

### 予想出力

/ecc:plan コマンドは：

1. **要件を再述** - 「加法、減法、乘法、除法という4種類の基本演算をサポートする計算機を実装...」
2. **リスク評価** - 「注意が必要：除法における零での除算情况...」
3. **分段計画** -
   - Phase 1: 計算機コア関数を作成
   - Phase 2: 4種類の演算を実装
   - Phase 3: エラー処理を追加
   - Phase 4: ユニットテストを作成
4. **確認を待つ** - 「上記計画是否可以開始するか確認してください」

### 確認後

ユーザーが確認すると、ECCは各段階を開始するのを待ちます。

### 小技

- PRDファイルモードを使用するとより詳細な計画を取得
- 複雑な機能は先にPRDを作成し、それから計画

---

## Step 2: /ecc:code-review を使用レビュー

### 入力

コード作成完了後、実行：

```
/ecc:code-review
```

### 予想出力

/ecc:code-review は：

1. 変更ファイルを収集
2. 7つのカテゴリでレビュー：
   - 正しさ
   - 型安全
   - パターン準拠
   - セキュリティ
   - パフォーマンス
   - 完全性
   - メンテナンス性
3. CRITICAL/HIGH/MEDIUM/LOWレベルでレビュー報告を生成

### レビュー報告例

```
## コードレビュー報告

### CRITICAL
- [ ] ハードコードされた凭证：src/auth.js 第23行

### HIGH
- [ ] 関数が長すぎる：src/calculator.js 第45-78行（34行）
- [ ] エラー処理が欠落：src/calculator.js 除法関数

### MEDIUM
- [ ] 命名が不規則：変数 `tmp` は `temporary` に変更すべき

### 操作提案
1. ハードコードされた API Key を削除
2. calculator.js の除法関数をリファクタリングし、零で割るチェックを追加
3. 長すぎる関数を分割
```

### 小技

- 提交前に必ず /ecc:code-review を実行
- CRITICAL と HIGH の問題は必ず修正

---

## Step 3: /ecc:build-fix を使用修復

### 入力

ビルドが失敗した場合、実行：

```
/ecc:build-fix
```

### 予想出力

/ecc:build-fix は：

1. **ビルドシステムを検出** - 「TypeScriptプロジェクトを検出」
2. **ビルドを実行** - 「tsc --noEmit を実行...」
3. **エラーを解析** - ファイルごとにグループ化
4. **逐步的に修復** - 一度に1つのエラーを修復
5. **検証** - 各修复後に再ビルド

### 出力例

```
[build-fix] TypeScriptプロジェクトを検出
[build-fix] tsc --noEmit を実行...
[build-fix] 3個のエラーを発見：

エラー 1: src/calculator.ts:15 - 型 'string' は型 'number' に割り当て不可
エラー 2: src/calculator.ts:28 - プロパティ 'result' が存在しない
エラー 3: src/calculator.ts:42 - パラメータ 'b' が使用されていない

[build-fix] エラー1の修復を開始...
[build-fix] 修復完了、検証中...
[build-fix] ビルド成功！
```

### 小技

- ビルド修復は増分的で、一度に1つのエラーを修復
- いつでも Ctrl+C で停止可能

---

## Step 4: コードを提出

### 提出フロー

1. すべてのテストが通过ことを確認
2. ビルドが成功することを確認
3. コードレビュー問題が修正されたことを確認

### Git 提出規範

ECCは conventional commits フォーマットを使用：

```
<type>: <description>

[optional body]
```

**Type タイプ**:
- `feat` - 新機能
- `fix` - bug修復
- `refactor` - リファクタリング
- `docs` - ドキュメント
- `test` - テスト
- `chore` - 日常メンテナンス
- `perf` - パフォーマンス最適化
- `ci` - CI設定

### 提出例

```
feat: 計算機の4種類の基本演算を実装

- 加法関数を追加
- 減法関数を追加
- 乗法関数を追加
- 除法関数を追加（零で割るチェックを含む）

解决：Issue #123
```

### 小技

- 提出前に `/ecc:verify quick` で快速検証
- 大きな機能は複数回で提出、小さい機能は一度で提出

## 完全例

### シナリオ

単純な文字列ユーティリティ関数を実装：`truncate(str, maxLength)`、文字列を指定された長さに截断。

### Step 1: 計画

```
/ecc:plan truncate関数を実装、文字列を指定された長さに截断
```

出力：
```
## 要件再述
truncate(str, maxLength) 関数を実装、maxLength 超える部分は ... で置換

## リスク評価
- maxLength が3より小さい場合の処理
- str が空または null の場合の処理
- Unicode文字の処理

## 分段計画
1. src/string-utils.ts を作成
2. truncate関数を実装
3. 境界情况テストを追加
4. JSDoc ドキュメントを追加

開始是否可以確認してください。
```

### Step 2: 実装

（ユーザーが確認後、ECCがコード実装を协力）

### Step 3: レビュー

```
/ecc:code-review
```

出力：
```
## コードレビュー報告
状態：APPROVED

### レビュー結果
- 正しさ：通過
- 型安全：通過
- パターン準拠：通過
- セキュリティ：通過
- パフォーマンス：通過
- 完全性：より多くの境界テストの追加を提案
- メンテナンス性：通過

### 提案
- 空文字列テストを追加
- null値テストを追加
```

### Step 4: ビルド検証

```
/ecc:build-fix
```

```
[build-fix] TypeScriptプロジェクトを検出
[build-fix] tsc --noEmit を実行...
[build-fix] ビルド成功！
```

### Step 5: 提出

```
feat: truncate 文字列截断関数を実装

- 指定された最大長さをサポート
- 超える部分は ... で置換
- 空文字列とnull値を処理
```

---


## 小結

1. **完全フロー**: /ecc:plan → 実装 → /ecc:code-review → /ecc:build-fix → 提出
2. **/ecc:plan** は計画に使用し、方向の正确性を確保
3. **/ecc:code-review** はレビューに使用し、問題を発見
4. **/ecc:build-fix** はビルドエラーの修復に使用
5. **提出** は conventional commits フォーマットを使用
