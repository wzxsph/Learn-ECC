## エンコード標準

### 用途説明
跨プロジェクト适用的基线エンコード约定，含む括命名、可读性、不可变性和コード品質レビュー。

### 使用タイミング
- 起動新プロジェクト或モジュール
- レビューコード品質和保守性
- リファクタリング既存のコード遵循约定
- 强制命名、フォーマット或構造一致性
- 設定linting、フォーマット化或タイプ検査ルール
- 纳入新贡献者エンコード约定

### コア原則

**1. 可读性优先**
- コード阅读次数多於编写次数
- 清晰的変数和関数名前
- 优先使用自文書化コード而非コメント
- 一致的フォーマット化

**2. KISS原則**
- 保持簡單，不要过度設計
- 避免过早最適化
- 容易理解 > 聪明的コード

**3. DRY原則**
- 提取公共逻辑到関数
- 作成可复用コンポーネント
- 跨モジュール共享ツール
- 避免コピー粘贴编程

**4. YAGNI原則**
- 不ビルド尚未必要的特性
- 避免 speculative generality
- 仅在必要时追加複雜性
- 先簡單，必要时リファクタリング

export function useDebounce<T>(value: T, delay: number): T {
  const [debouncedValue, setDebouncedValue] = useState<T>(value)

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value)
    }, delay)
    return () => clearTimeout(handler)
  }, [value, delay])

  return debouncedValue
}

## エラー処理

### 用途説明
跨TypeScript、Python和Go的健壮エラー処理パターン，含む括タイプ化エラー、エラー边界、再試行、断路器和ユーザー友好エラーメッセージ。

### 使用タイミング
- 為新モジュール或サービス設計エラータイプ或例外层次構造
- 為不可靠的外部依賴追加再試行逻辑或断路器
- レビューAPI端點は否缺少エラー処理
- 実装ユーザー友好エラーメッセージ和反馈
- デバッグ级联失敗或静默エラー吞並

### コア原則

1. **快速失敗並大声** - 在边界处暴露エラー
2. **タイプ化エラー优於字符串消息** - エラーは一等公民，あり構造
3. **ユーザー消息 ≠ 開発者消息** - 向ユーザー表示友好文本，サーバー端記録完整上下文
4. **永远不静默吞並エラー** - 各`catch`块must處理、重新抛出或記録
5. **エラーはAPI合约的一一部** - 記録クライアント可能收到的各エラー码

### TypeScriptエラー処理

```typescript
// タイプ化エラークラス
export class AppError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly statusCode: number = 500,
    public readonly details?: unknown,
  ) {
    super(message)
    this.name = this.constructor.name
    Object.setPrototypeOf(this, new.target.prototype)
  }
}

export class NotFoundError extends AppError {
  constructor(resource: string, id: string) {
    super(`${resource} not found: ${id}`, 'NOT_FOUND', 404)
  }
}

export class ValidationError extends AppError {
  constructor(message: string, details: { field: string; message: string }[]) {
    super(message, 'VALIDATION_ERROR', 422, details)
  }
}

// Resultパターン
type Result<T, E = AppError> =
  | { ok: true; value: T }
  | { ok: false; error: E }

async function fetchUser(id: string): Promise<Result<User>> {
  try {
    const user = await db.users.findUnique({ where: { id } })
    if (!user) return { ok: false, error: new NotFoundError('User', id) }
    return { ok: true, value: user }
  } catch (e) {
    return { ok: false, error: new AppError('Database error', 'DB_ERROR') }
  }
}

// APIエラー処理器
function handleApiError(error: unknown): NextResponse {
  if (error instanceof AppError) {
    return NextResponse.json(
      { error: { code: error.code, message: error.message, ...(error.details && { details: error.details }) } },
      { status: error.statusCode }
    )
  }
  console.error('Unexpected error:', error)
  return NextResponse.json(
    { error: { code: 'INTERNAL_ERROR', message: 'An unexpected error occurred' } },
    { status: 500 }
  )
}
```

class AppError(Exception):
    def __init__(self, message: str, code: str, status_code: int = 500):
        super().__init__(message)
        self.code = code
        self.status_code = status_code

class NotFoundError(AppError):
    def __init__(self, resource: str, id: str):
        super().__init__(f"{resource} not found: {id}", "NOT_FOUND", 404)

class ValidationError(AppError):
    def __init__(self, message: str, details: list[dict] | None = None):
        super().__init__(message, "VALIDATION_ERROR", 422)
        self.details = details or []

# FastAPI全局例外處理器
@app.exception_handler(AppError)
async def app_error_handler(request: Request, exc: AppError) -> JSONResponse:
    return JSONResponse(
        status_code=exc.status_code,
        content={"error": {"code": exc.code, "message": str(exc)}},

async def generic_error_handler(request: Request, exc: Exception) -> JSONResponse:

### 指数退避再試行

```typescript
interface RetryOptions {
  maxAttempts?: number
  baseDelayMs?: number
  maxDelayMs?: number
  retryIf?: (error: unknown) => boolean
}

async function withRetry<T>(fn: () => Promise<T>, options: RetryOptions = {}): Promise<T> {
  const { maxAttempts = 3, baseDelayMs = 500, maxDelayMs = 10_000, retryIf = () => true } = options

  let lastError: unknown

  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await fn()
    } catch (error) {
      lastError = error
      if (attempt === maxAttempts || !retryIf(error)) throw error

      const jitter = Math.random() * baseDelayMs
      const delay = Math.min(baseDelayMs * 2 ** (attempt - 1) + jitter, maxDelayMs)
      await new Promise(resolve => setTimeout(resolve, delay))
    }
  }

  throw lastError
}
```

### エラー処理検査清单

- [ ] 各`catch`块處理、重新抛出或記録 - なし静默吞並
- [ ] APIエラー遵循標準信封`{ error: { code, message } }`
- [ ] ユーザー可见消息不含む含スタックトレース或内部细节
- [ ] 完整エラー上下文在サーバー端記録
- [ ] 自定義エラークラス拡張带あり`code`欄位的基`AppError`
- [ ] 非同期関数將エラー传递給調用者 - なしfire-and-forget
- [ ] 再試行逻辑仅再試行可再試行エラー（非4xxクライアントエラー）
- [ ] Reactコンポーネント含む装在`ErrorBoundary`中處理渲染エラー

### autonomous-loops

**用途**: 自主ループパターン和アーキテクチャ - 從簡單順序管道到 RFC 驱动的多プロキシ DAG システム

**ループパターン谱系**:
| パターン | 複雜度 | 適用シナリオ |
|------|--------|----------|
| Sequential Pipeline | 低 | 日常開発ステップ、スクリプト化ワークフロー |
| NanoClaw REPL | 低 | 交互式持久セッション |
| Infinite Agentic Loop | 中 | 並列內容生成、仕様驱动作業 |
| Continuous Claude PR Loop | 中 | 多日迭代プロジェクト，CI 门禁 |
| De-Sloppify Pattern | 附加 | 任意 Implementer ステップ后的品質クリーンアップ |
| Ralphinho / RFC-Driven DAG | 高 | 大型機能、多单元並列作業與マージキュー |

**使用タイミング**:
- 設定自主開発ワークフロー
- 选择合适的ループアーキテクチャ
- ビルド CI/CD 风格的継続開発管道
- 実行带マージ協調的並列プロキシ

**重要パターン**:

1. **Sequential Pipeline** - 簡單的 `claude -p` 管道
2. **NanoClaw REPL** - 内置持久化ループ，session-aware
3. **Infinite Agentic Loop** - 仕様驱动的並列生成
4. **Continuous Claude PR Loop** - 本番対応 shell スクリプト，自動作成 PR、等待 CI、自動マージ
5. **De-Sloppify Pattern** - 追加專門的クリーンアップ/リファクタリングステップ
6. **Ralphinho** - RFC 驱动的多プロキシ DAG 编排
