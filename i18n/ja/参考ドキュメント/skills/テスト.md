# テスト

本一部涵盖各プログラミング言語的テスト戦略、TDDワークフロー和E2Eテストパターン。

## TDDワークフロー

### 用途説明
TDDワークフロー確保すべてコード開発遵循テスト驱动開発原則，を持つ全面的テストカバレッジ。

### 使用タイミング
- 编写新機能或機能
- 修正bug或問題
- リファクタリング既存のコード
- 追加API端點
- 作成新コンポーネント

### コア概念

**1. テスト先行**
总は先写テスト，然後実装コード使テスト通過。

**2. カバレッジ要件**
最低80%カバレッジ（ユニットテスト + 統合テスト + E2Eテスト）

**3. テスト類型**
- ユニットテスト：独立関数和コンポーネント
- 統合テスト：API端點和データベース操作
- E2Eテスト：重要ユーザー流和浏览器自動化

**4. RED-GREEN-REFACTORループ**
- RED: 先写失敗的テスト
- GREEN: 写最简コード使テスト通過
- REFACTOR: 保持テスト绿色的情况下改善コード

### TDD流程ステップ

```bash
# 1. 写テスト（RED）
npm test  # テストすべき失敗

# 2. 実装コード（GREEN）
# 写最简コード使テスト通過

# 3. リファクタリング（REFACTOR）
# 改善コード品質，保持テスト绿色

# 4. 検証カバレッジ
npm run test:coverage  # 検証80%+カバレッジ
```

### テスト隔离原則

```typescript
// エラー：テスト相互依賴
test('creates user', () => { /* ... */ })
test('updates same user', () => { /* 依賴前1テスト */ })

// 正确：各テスト独立設定
test('creates user', () => {
  const user = createTestUser()
  // テスト逻辑
})

test('updates user', () => {
  const user = createTestUser()
  // 更新逻辑
})
```

## Pythonテスト

### 用途説明
Pythonテスト戦略使用pytest、TDDメソッド论、fixtures、mocking、參數化和カバレッジ要件。

### 使用タイミング
- 遵循TDDメソッド论编写新Pythonコード
- 設計Pythonプロジェクトテスト套件
- レビューPythonテストカバレッジ
- 設定テストインフラ

### コア概念

**1. pytest Fixtures**
使用`@pytest.fixture`作成可复用テストデータ。

**2. 參數化テスト**
使用`@pytest.mark.parametrize`用異なる输入実行テスト。

**3. Mocking**
使用`unittest.mock`的`@patch`模拟外部依賴。

**4. カバレッジ**
使用`--cov`测量カバレッジ，目標80%+。

### 使用サンプル

```python
import pytest
from unittest.mock import patch

# Fixtures
@pytest.fixture
def database():
    db = Database(":memory:")
    db.create_tables()
    yield db
    db.close()

# 參數化テスト
@pytest.mark.parametrize("input,expected", [
    ("hello", "HELLO"),
    ("world", "WORLD"),
    ("PyThOn", "PYTHON"),
])
def test_uppercase(input, expected):
    assert input.upper() == expected

# Mocking
@patch("mypackage.external_api_call")
def test_with_mock(api_call_mock):
    api_call_mock.return_value = {"status": "success"}
    result = my_function()
    assert result["status"] == "success"

# 例外テスト
with pytest.raises(ValueError, match="invalid input"):
    raise ValueError("invalid input provided")

# 実行コマンド
pytest --cov=mypackage --cov-report=html
```

## Goテスト

### 用途説明
Goテストパターン含む括表驱动テスト、子テスト、基准テスト、模糊テスト和テストカバレッジ，遵循TDDメソッド和惯用Go实践。

### 使用タイミング
- 编写新的Go関数或メソッド
- 為既存のコード追加テストカバレッジ
- 為パフォーマンス重要コード作成基准テスト
- 実装模糊テスト进行输入検証
- 在Goプロジェクト中遵循TDDワークフロー

### コア概念

**1. 表驱动テスト**
使用表驱动パターン実装全面覆寫。

**2. 子テスト**
使用`t.Run`組織相關テスト。

**3. テスト辅助関数**
使用`t.Helper()`标记辅助関数。

**4. インターフェースMocking**
使用インターフェース进行依賴解耦和テスト。

**5. 基准テスト和模糊テスト**
パフォーマンステスト和随机输入テスト。

### 使用サンプル

```go
// 表驱动テスト
func TestAdd(t *testing.T) {
    tests := []struct {
        name     string
        a, b     int
        expected int
    }{
        {"positive numbers", 2, 3, 5},
        {"negative numbers", -1, -2, -3},
        {"zero values", 0, 0, 0},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := Add(tt.a, tt.b)
            if got != tt.expected {
                t.Errorf("Add(%d, %d) = %d; want %d", tt.a, tt.b, got, tt.expected)
            }
        })
    }
}

// テスト辅助関数
func setupTestDB(t *testing.T) *sql.DB {
    t.Helper()
    db, err := sql.Open("sqlite3", ":memory:")
    if err != nil {
        t.Fatalf("failed to open database: %v", err)
    }
    t.Cleanup(func() { db.Close() })
    return db
}

// インターフェースMocking
type UserRepository interface {
    GetUser(id string) (*User, error)
    SaveUser(user *User) error
}

type MockUserRepository struct {
    GetUserFunc  func(id string) (*User, error)
    SaveUserFunc func(user *User) error
}

func (m *MockUserRepository) GetUser(id string) (*User, error) {
    return m.GetUserFunc(id)
}

// 基准テスト
func BenchmarkProcess(b *testing.B) {
    data := generateTestData(1000)
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        Process(data)
    }
}

// 実行コマンド
go test -cover ./...
go test -bench=. -benchmem
go test -fuzz=FuzzParse -fuzztime=30s
```

## Rustテスト

### 用途説明
Rustテストパターン含む括ユニットテスト、統合テスト、非同期テスト、屬性テスト、mocking和カバレッジ，遵循TDDメソッド论。

### 使用タイミング
- 编写新的Rust関数、メソッド或trait
- 為既存のコード追加テストカバレッジ
- 為パフォーマンス重要コード作成基准テスト
- 実装屬性テスト进行输入検証
- 在Rustプロジェクト中遵循TDDワークフロー

### コア概念

**1. モジュール级テスト**
在`#[cfg(test)]`モジュール中编写ユニットテスト。

**2. 統合テスト**
在`tests/`ディレクトリ中编写統合テスト。

**3. 非同期テスト**
使用`#[tokio::test]`进行非同期テスト。

**4. 屬性テスト**
使用`proptest`进行基づいて屬性的テスト。

**5. Mocking**
使用`mockall`模拟trait。

### 使用サンプル

```rust
// モジュール级テスト
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn creates_user_with_valid_email() {
        let user = User::new("Alice", "alice@example.com").unwrap();
        assert_eq!(user.display_name(), "Alice");
    }

    #[test]
    fn rejects_invalid_email() {
        let result = User::new("Bob", "not-an-email");
        assert!(result.is_err());
    }
}

// 非同期テスト
#[tokio::test]
async fn fetches_data_successfully() {
    let client = TestClient::new().await;
    let result = client.get("/data").await;
    assert!(result.is_ok());
}

// 屬性テスト
proptest! {
    #[test]
    fn encode_decode_roundtrip(input in ".*") {
        let encoded = encode(&input);
        let decoded = decode(&encoded).unwrap();
        prop_assert_eq!(input, decoded);
    }
}

// Mocking with mockall
use mockall::{automock, predicate::eq};

#[automock]
trait UserRepository {
    fn find_by_id(&self, id: u64) -> Option<User>;
    fn save(&self, user: &User) -> Result<(), StorageError>;
}

#[test]
fn service_returns_user_when_found() {
    let mut mock = MockUserRepository::new();
    mock.expect_find_by_id()
        .with(eq(42))
        .times(1)
        .returning(|_| Some(User { id: 42, name: "Alice".into() }));

    let service = UserService::new(Box::new(mock));
    let user = service.get_user(42).unwrap();
    assert_eq!(user.name, "Alice");
}

// 実行コマンド
cargo test
cargo llvm-cov --fail-under-lines 80
cargo bench
```

## E2Eテスト

### 用途説明
Playwright E2Eテストパターン提供します了ビルド穩定、快速、可維護E2Eテスト套件的全面パターン，含む括页面物件モデル、設定、CI/CD統合和防抖テスト戦略。

### 使用タイミング
- 作成E2Eテスト検証完整ユーザー流
- 使用Playwright进行浏览器自動化
- 設定CI/CD中的E2Eテスト
- 處理不穩定テスト

### コア概念

**1. 页面物件モデル(POM)**
將页面交互封裝在页面物件クラス中。

**2. テスト隔离**
各テスト独立設定データ。

**3. 定位器戦略**
使用语义定位器而非CSSクラス名。

**4. 防抖テスト**
识别和處理不穩定テスト。

**5. Artifacts管理**
截圖、视频、トレース記録使用デバッグ。

### 使用サンプル

```typescript
import { test, expect } from '@playwright/test'

// 页面物件モデル
export class ItemsPage {
  readonly page: Page
  readonly searchInput: Locator
  readonly itemCards: Locator

  constructor(page: Page) {
    this.page = page
    this.searchInput = page.locator('[data-testid="search-input"]')
    this.itemCards = page.locator('[data-testid="item-card"]')
  }

  async goto() {
    await this.page.goto('/items')
    await this.page.waitForLoadState('networkidle')
  }

  async search(query: string) {
    await this.searchInput.fill(query)
    await this.page.waitForResponse(resp => resp.url().includes('/api/search'))
  }
}

// テスト構造
test.describe('Item Search', () => {
  let itemsPage: ItemsPage

  test.beforeEach(async ({ page }) => {
    itemsPage = new ItemsPage(page)
    await itemsPage.goto()
  })

  test('should search by keyword', async ({ page }) => {
    await itemsPage.search('test')
    const count = await itemsPage.getItemCount()
    expect(count).toBeGreaterThan(0)
    await expect(itemsPage.itemCards.first()).toContainText(/test/i)
  })
})

// Playwright設定
export default defineConfig({
  testDir: './tests/e2e',
  retries: process.env.CI ? 2 : 0,
  use: {
    baseURL: process.env.BASE_URL || 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
  },
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox', use: { ...devices['Desktop Firefox'] } },
  ],
})

// 防抖テスト
test('conditional skip', async ({ page }) => {
  test.skip(process.env.CI, 'Flaky in CI - Issue #123')
})

test('flaky: complex search', async ({ page }) => {
  test.fixme(true, 'Known flaky test - Issue #123')
})
```
