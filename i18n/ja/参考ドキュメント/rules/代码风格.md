# コーディングスタイルルール

## ルール概要

ECC Rules コーディングスタイルルール定义了コード组织的コア原則和ベストプラクティス。这些ルール确保コード保持可读性、保守性和一致性，涵盖从不可变性到命名约定的すべて方面。

## コア要件

### 不可变性（重要原則）

**must**：始终作成新对象，绝不修改现あり对象。

```typescript
// エラーサンプル - 修改原对象
function modify(user, name) {
  user.name = name;  // 改变了原对象
  return user;
}

// 正确サンプル - 戻る新对象
function update(user, name) {
  return { ...user, name };  // 作成新副本
}
```

**理由**：不可变データ防止非表示副作用，使デバッグ更容易，并サポートセキュリティ並行処理。

### コア設計原則

| 原則 | 説明 | 实践 |
|------|------|------|
| **KISS** | Keep It Simple | 首选最简单的可行方案，避免过早最適化，优先検討清晰而非巧妙 |
| **DRY** | Don't Repeat Yourself | 将重复逻辑提取到共享関数或ツールクラス，避免コピー粘贴导致的実装漂移 |
| **YAGNI** | You Aren't Gonna Need It | 不ビルド未来可能必要的特性或抽象，简单開始，在真正必要时リファクタリング |
### ファイル组织

| 要件 | 説明 |
|------|------|
| 多小ファイル > 少大ファイル | 高内聚、低耦合 |
| 典型行数 | 200-400行，最多800行 |
| 大モジュール拆分 | 从大モジュール中提取ツール関数 |
| 组织方式 | 按機能/领域组织，而非按タイプ |

### エラー処理

| 要件 | 説明 |
|------|------|
| 显式处理 | 各レイヤー都要显式处理エラー |
| UIコード | 提供用户友好的エラーメッセージ |
| サービス端 | 記録详细的エラー上下文 |
| 禁止行为 | 不得静默吞掉エラー |

### 输入検証

| 要件 | 説明 |
|------|------|
| 边界検証 | 在システム边界検証すべて输入 |
| Schema検証 | 尽可能使用基于Schema的検証 |
| 快速失敗 | 用清晰的エラーメッセージ快速失敗 |
| 外部データ | 永远不信任外部データ（API応答、用户输入、ファイル内容） |
## 实施细节

### 命名约定

| タイプ | 约定 | サンプル |
|------|------|------|
| 変数和関数 | camelCase + 描述性名称 | `calculateTotal`, `userName` |
| 布尔值 | is/has/should/can 前缀 | `isActive`, `hasPermission` |
| インターフェース、タイプ、コンポーネント | PascalCase | `UserProfile`, `ButtonComponent` |
| 定数 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT`, `API_TIMEOUT` |
| 自定义Hooks | camelCase + use前缀 | `useUserData`, `useFetch` |

### コード异味

#### 深层嵌套

超过4层嵌套应使用早期戻るリファクタリング：

```typescript
// 避免
function processOrder(order) {
  if (order) {
    if (order.items) {
      if (order.items.length > 0) {
        // 处理逻辑
      }
    }
  }
}

// 推奨
function processOrder(order) {
  if (!order || !order.items || order.items.length === 0) {
    return;
  }
  // 处理逻辑
}
```

#### 魔法数字

使用命名定数代替魔法数字：

```typescript
// 避免
setTimeout(callback, 300000);

// 推奨
const SESSION_TIMEOUT_MS = 5 * 60 * 1000; // 5分钟
setTimeout(callback, SESSION_TIMEOUT_MS);
```

#### 长関数

拆分大型関数为を持つ清晰职责的小型関数：

| 関数長さ | 推奨 |
|----------|------|
| <50行 | 理想 |
| 50-100行 | 可受け入れ，検討拆分 |
| >100行 | must拆分 |

## 违规处理

### コード品質チェック清单

完了作業前must検査：

- [ ] コード可读且命名良好
- [ ] 関数短小（<50行）
- [ ] ファイル专注（<800行）
- [ ] なし深层嵌套（>4层）
- [ ] 适当的エラー処理
- [ ] なし硬エンコード值（使用定数或設定）
- [ ] 使用不可变パターン（なしmutation）

### 重大程度分级

| 级别 | 含义 | 操作 |
|------|------|------|
| CRITICAL | セキュリティ漏洞或データ丢失风险 | **阻止** - マージ前must修正 |
| HIGH | Bug或重大品質問題 | **警告** - マージ前应修正 |
| MEDIUM | 保守性問題 | **提示** - 推奨修正 |
| LOW | 样式或小推奨 | **备注** - オプション |
