# テストルール

## ルール概要

ECC Rules テストルール定義了確保コード品質的强制性テスト標準。该ルール涵盖テスト驱动開発(TDD)、最低カバレッジ要件和テスト構造仕様，確保すべてコード都經過充分検証。

## コア要件

### 最低テストカバレッジ：80%

must含む含以下三种テスト類型：

| テスト類型 | 説明 | 覆寫范围 |
|----------|------|----------|
| ユニットテスト | 独立関数、ツールクラス、コンポーネント | Individual units |
| 統合テスト | API端點、データベース操作 | Integration points |
| E2Eテスト | 重要ユーザーフロー | Critical user flows |

### テスト驱动開発（TDD）

**强制ワークフロー**：

```
1. 编写テスト (RED)   - 先写1会失敗的テスト
2. 実行テスト          - 検証テスト失敗
3. 编写最小実装 (GREEN) - 编写通過テスト的最小コード
4. 実行テスト          - 検証テスト通過
5. リファクタリング (IMPROVE)    - 改善コード構造
6. 検証カバレッジ        - 確保达到80%+
```

## 實施细节

### AAAパターン（Arrange-Act-Assert）

すべてテストmust遵循AAA構造：

```typescript
test('正确计算余弦相似度', () => {
  // Arrange - 准备テストデータ
  const vector1 = [1, 0, 0];
  const vector2 = [0, 1, 0];

  // Act - 実行被テスト的操作
  const similarity = calculateCosineSimilarity(vector1, vector2);

  // Assert - 検証結果
  expect(similarity).toBe(0);
});
```

### テスト命名仕様

使用記述性名前説明被テスト的行為：

```typescript
// 推奨 - 記述预期行為
test('當ない市場匹配查詢时戻る空陣列', () => {});
test('當API密鑰缺失时抛出エラー', () => {});
test('當Redis不可用时回退到子字符串搜索', () => {});

// 避免 - 模糊命名
test('test1', () => {});
test('edge case', () => {});
```

### テスト故障排除

當テスト失敗时：

| ステップ | 操作 |
|------|------|
| 1 | 使用 **tdd-guide** agent |
| 2 | 検査テスト隔离 |
| 3 | 検証Mock正确性 |
| 4 | 修正実装，而非テスト（除非テスト本身あり误） |

### Agentサポート

| Agent | 用途 | 使用タイミング |
|-------|------|----------|
| **tdd-guide** | テスト驱动開発指导 | 新機能開発、エラー修正时强制使用 |
| **code-reviewer** | コードレビュー | コード编写完了后 |

## 违规處理

### カバレッジ不足

- カバレッジ低於80%的コード不得マージ
- CI/CD流水线应阻止低カバレッジコードコミット
- 使用 `c8` 或クラス似ツール生成カバレッジレポート

### テスト失敗處理

```
1. 分析失敗原因
2. 診断は実装問題还はテスト問題
3. 如果は実装問題 - 修正実装コード
4. 如果はテスト問題 - 修正テストコード
5. 確保すべてテスト通過后再コミット
```

### 常见問題

| 問題 | 原因 | 解決方案 |
|------|------|----------|
| Flakyテスト | 非同期操作、竞态條件 | 增加等待時間、使用mock |
| テスト隔离失敗 | 共享狀態污染 | 各テスト前リセット狀態 |
| Mock不正确 | 期望值與实际不符 | 検証mock設定は否正确 |
