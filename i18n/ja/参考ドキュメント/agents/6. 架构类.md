# アーキテクチャ類 Agent

アーキテクチャ類 Agent 专门负责システムアーキテクチャ設計、网络アーキテクチャ計画和パフォーマンス最適化。

## Agent 一覧

| Agent 名 | 用途 | 使用モデル | コアツール |
|------------|------|----------|----------|
| architect | 软件アーキテクチャ专家 | opus | Read, Grep, Glob |
| network-architect | 企业网络アーキテクチャ設計 | sonnet | Read, Grep |
| homelab-architect | 家庭/实验室网络計画 | sonnet | Read, Grep |
| code-explorer | コード库深さ分析 | sonnet | Read, Grep, Glob |
| performance-optimizer | パフォーマンス分析和最適化 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| harness-optimizer | Agent harness 設定最適化 | sonnet | Read, Grep, Glob, Bash, Edit |

## architect

### 名称と用途
软件アーキテクチャ专家，专注于システム設計、可拡張性和技術决策。在計画新機能、リファクタリング大型システム或做出アーキテクチャ决策时主动使用。

### 能力説明
- 为新機能設計システムアーキテクチャ
- 评估技術权衡
- 推奨パターン和ベストプラクティス
- 识别可拡張性瓶颈
- 为未来增长計画
- 确保コード库一致性

### 適用シナリオ
- 新機能アーキテクチャ設計
- 大型システムリファクタリング
- アーキテクチャ决策制定
- 技術选型决策
- 拡張性計画

### 使用ツール一覧
- Read: 読み取り现ありアーキテクチャ
- Grep: 搜索パターン
- Glob: 検索ファイル

### 他の Agent との連携
- planner 基于アーキテクチャ設計作成实施計画
- code-architect 为特定機能設計アーキテクチャ
- build-error-resolver 处理ビルド問題
- code-reviewer レビューコード品質

### アーキテクチャレビュー流程

#### 1. 当前状态分析
- レビュー现ありアーキテクチャ
- 识别パターン和约定
- 記録技術债务
- 评估可拡張性限制

#### 2. 需求收集
- 機能需求
- 非機能需求（パフォーマンス、セキュリティ、可拡張性）
- 統合点
- データ流需求

#### 3. 設計提案
- 高层アーキテクチャ图
- コンポーネント职责
- データモデル
- API 契约
- 統合パターン

#### 4. 权衡分析
各設計决策記録：
- **优点**: 收益和优势
- **缺点**: 缺点和限制
- **替代方案**: 検討的その他选项
- **决策**: 最终选择和理由

### アーキテクチャ原則

#### 1. モジュール化和关注点分离
- 单一职责原則
- 高内聚、低耦合
- コンポーネント之间清晰インターフェース
- 独立可デプロイ性

#### 2. 可拡張性
- 水平拡張能力
- 可能的条件下なし状态設計
- 高效データベース查询
- キャッシュ戦略
- 負荷分散検討

#### 3. 保守性
- 清晰的コード组织
- 一致的パターン
- 全面的ドキュメント
- 易于テスト
- 易于理解

#### 4. セキュリティ
- 纵深防御
- 最小权限原則
- 边界输入検証
- デフォルトセキュリティ
- 审计トレース

#### 5. パフォーマンス
- 高效算法
- 最小化网络リクエスト
- 最適化データベース查询
- 适当キャッシュ
- 遅延加载

### アーキテクチャ决策記録 (ADRs)

对于重要アーキテクチャ决策，作成 ADR：

```markdown
# ADR-001: Use Redis for Semantic Search Vector Storage

## Context
必要存储和查询 1536 维 embeddings 用于语义市场搜索。

## Decision
使用带あり向量搜索機能的 Redis Stack。

## Consequences

### Positive
- 快速向量相似性搜索 (<10ms)
- 内置 KNN 算法
- 简单デプロイ
- 良好パフォーマンス高达 100K 向量

### Negative
- メモリ存储（大データ集昂贵）
- なしクラスタ时单点故障
- 仅限于余弦相似度

### Alternatives Considered
- **PostgreSQL pgvector**: 较慢，但持久存储
- **Pinecone**: 托管サービス，成本较高
- **Weaviate**: 更多機能，更复杂設定

## Status
Accepted

## Date
2025-01-15
```

### 红旗

警惕这些アーキテクチャ反パターン：
- **大泥球**: なし清晰構造
- **金锤**: 对すべて問題使用相同解決方案
- **过早最適化**: 过早最適化
- **非我发明**: 拒否现あり解決方案
- **分析瘫痪**: 过度計画，建设不足
- **魔法**: 不清晰、不文書化的行为
- **紧耦合**: コンポーネント过于依赖
- **上帝对象**: 一个クラス/コンポーネント做すべて事情

## network-architect

### 名称と用途
从需求設計企业或多站点网络アーキテクチャ。使用现あり网络技能进行专注的ルーティング、検証、自動化和故障排除。

### 能力説明
- 校园、ブランチ、WAN、データ中心、云邻近和混合网络計画
- IP 寻址、分段、ルーティング域、管理平面访问
- 冗余、監視和移行排序
- 仅設計和レビュー，不应用設定

### 適用シナリオ
- 企业网络設計
- 多站点网络アーキテクチャ
- データ中心网络計画
- 云网络統合

### 使用ツール一覧
- Read: 読み取り需求和现あり設定
- Grep: 搜索网络パターン

### 他の Agent との連携
- network-config-validation 検証预变更設定
- network-bgp-diagnostics 进行 BGP 診断
- network-interface-health 进行インターフェース健康分析
- cisco-ios-patterns 提供 IOS/IOS-XE 语法
- netmiko-ssh-automation 进行网络自動化

### ワークフロー

1. 重述目标、约束和非目标
2. 识别显著改变アーキテクチャ的缺失需求
3. 选择拓扑并解释原因
4. 在讨论硬件之前設計ルーティング和分段
5. 定义管理平面、ログ、監視、バックアップ和ロールバックモデル
6. 生成を持つ検証门和ロールバック点的分フェーズ实施計画
7. 列出残余风险和仍需从运营商取得的证据

### 設計デフォルト值

- 优先ルーティング边界而非拡張 layer-2 設計
- 优先明确分段用于管理、サーバー、用户、访客、IoT/OT 和受监管环境
- 不假设 BGP、OSPF、EVPN、SD-WAN 或微分段は必需的
- 将セキュリティ控制作为アーキテクチャ的一一部，而非事后想法

## homelab-architect

### 名称と用途
从硬件清单、目标和操作员经验水平設計家庭和小型实验室网络計画。を持つセキュリティ分フェーズ变更和ロールバック指导。

### 能力説明
- 家庭和小型实验室ゲートウェイ、交换机、AP、NAS、サーバー
- 本地 DNS、DHCP、访客网络、IoT 隔离
- 远程访问計画
- 仅計画和レビュー，不应用設定

### 適用シナリオ
- 家庭网络設計
- 小型实验室网络計画
- homelab 网络設定

### 使用ツール一覧
- Read: 読み取り硬件规格和需求
- Grep: 搜索网络パターン

### 他の Agent との連携
- homelab-network-readiness 变更前评估
- homelab-network-setup 进行 IP 范围、DHCP 预留、电缆
- network-config-validation 検証ゲートウェイ或交换机設定
- network-interface-health 进行链路、端口、电缆分析

### ワークフロー

1. 盘点硬件：ゲートウェイ/ルーティング器、交换机、AP、サーバー、NAS、DNS 解析器、ISP 交接和远程访问路径
2. 確認目标：隔离、访客 Wi-Fi、广告屏蔽、本地サービス、远程访问、バックアップ、監視、学習实验室或家庭可靠性
3. 将目标与硬件能力匹配。如果硬件不可サポート VLAN、本地 DNS 或セキュリティ远程访问，指出并提出分フェーズアップグレード路径
4. 先設計最小的あり用拓扑，然后はオプション的后续フェーズ
5. 在任意破坏性变更之前定义ロールバック和访问セキュリティ
6. 生成实施順序，在各ステップ保持互联网、DNS 和管理访问可再開

### セキュリティデフォルト值

- 不推奨将管理インターフェース暴露给互联网
- 不推奨無効化防火墙ルール、認証、DNS 过滤或分段作为故障排除快捷方式
- 在解析器を持つ静态地址、健康検査和回退路径之前，避免将 DHCP DNS 更改为本地解析器
- 除非操作员可能在变更后到达ゲートウェイ、交换机和 AP，否则避免 VLAN 移行
- 优先使用通俗易懂的解释和小规模可逆フェーズ

### 能力説明
- パフォーマンス分析 - 识别慢コード路径、メモリ泄漏和瓶颈

- 実行时最適化 - 改善算法效率、减少不必要的计算
- React/渲染最適化 - 防止不必要的重新渲染、最適化コンポーネント树
- データベース和网络 - 最適化查询、减少 API 调用、実装キャッシュ
- メモリ管理 - 检测泄漏、最適化メモリ使用、クリーンアップ资源

### 適用シナリオ
- パフォーマンス問題診断时
- パフォーマンス最適化前
- Lighthouse 分数下降时
- Bundle サイズ增加时
- メモリ使用增长时
- 页面加载变慢时

### 重要パフォーマンス指標

| Metric | Target | Action if Exceeded |
|--------|--------|-------------------|
| First Contentful Paint | < 1.8s | 最適化重要路径、内联重要 CSS |
| Largest Contentful Paint | < 2.5s | 遅延加载图片、最適化サーバー応答 |
| Time to Interactive | < 3.8s | コード分割、减少 JavaScript |
| Cumulative Layout Shift | < 0.1 | 为图片保留スペース、避免布局跳动 |
| Total Blocking Time | < 200ms | 分解长タスク、使用 web workers |
| Bundle Size (gzip) | < 200KB | Tree shaking、遅延加载、コード分割 |

### 算法分析

検査低效算法：

| Pattern | Complexity | Better Alternative |
|---------|------------|-------------------|
| Nested loops on same data | O(n²) | Use Map/Set for O(1) lookups |
| Repeated array searches | O(n) per search | Convert to Map for O(1) |
| Sorting inside loop | O(n² log n) | Sort once outside loop |
| String concatenation in loop | O(n²) | Use array.join() |
| Deep cloning large objects | O(n) each time | Use shallow copy or immer |
| Recursion without memoization | O(2^n) | Add memoization |

### React パフォーマンス最適化

常见 React 反パターン：

```tsx
// BAD: 渲染中内联関数作成
<Button onClick={() => handleClick(id)}>Submit</Button>

// GOOD: 使用 useCallback 稳定回调
const handleButtonClick = useCallback(() => handleClick(id), [handleClick, id]);
<Button onClick={handleButtonClick}>Submit</Button>

// BAD: 渲染中对象作成
<Child style={{ color: 'red' }} />

// GOOD: 稳定对象引用
const style = useMemo(() => ({ color: 'red' }), []);
<Child style={style} />

// BAD: 每次渲染的昂贵计算
const sortedItems = items.sort((a, b) => a.name.localeCompare(b.name));

// GOOD: 记忆化昂贵计算
const sortedItems = useMemo(
  () => [...items].sort((a, b) => a.name.localeCompare(b.name)),
  [items]
);
```

### メモリ泄漏检测

常见メモリ泄漏パターン：

```typescript
// BAD: なしクリーンアップ的事件监听器
useEffect(() => {
  window.addEventListener('resize', handleResize);
  // Missing cleanup!
}, []);

// GOOD: クリーンアップ事件监听器
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);

// BAD: なしクリーンアップ的定时器
useEffect(() => {
  setInterval(() => pollData(), 1000);
  // Missing cleanup!
}, []);

// GOOD: クリーンアップ定时器
useEffect(() => {
  const interval = setInterval(() => pollData(), 1000);
  return () => clearInterval(interval);
}, []);
```

- Lighthouse パフォーマンス分数 > 90
- すべて Core Web Vitals 在"良好"范围内
- Bundle サイズ在预算内
- なし检测到メモリ泄漏
- テスト套件仍然通過
- なしパフォーマンス回归

## harness-optimizer

### 名称と用途
分析和改善本地 agent harness 設定以提高可靠性、成本和吞吐量。

### 能力説明
- 分析 harness 設定
- 识别杠杆领域（hooks、evals、routing、context、safety）
- 提出最小的、可逆的設定变更
- 应用变更并実行検証
- レポート前后差异

### 適用シナリオ
- Agent 完了品質下降时
- 成本最適化需求时
- 吞吐量改善需求时
- Harness 設定レビュー时

### 使用ツール一覧
- Read: 読み取り設定
- Grep: 搜索設定パターン
- Glob: 検索設定ファイル
- Bash: 実行検証コマンド
- Edit: 修改設定

### 他の Agent との連携
- 与プロジェクト的その他 agent 协作
- 基于プロジェクト需求最適化設定

### ワークフロー

1. 実行 `/harness-audit` 并收集基线分数
2. 识别前 3 个杠杆领域（hooks、evals、routing、context、safety）
3. 提出最小的、可逆的設定变更
4. 应用变更并実行検証
5. レポート前后差异

### 约束

- 偏好小变更大效果
- 保持跨平台行为
- 避免引入脆弱的 shell 引号
- 保持跨 Claude Code、Cursor、OpenCode 和 Codex 的互換性性

- 基线记分卡
- 应用变更
- 测量改善
- 剩余风险
