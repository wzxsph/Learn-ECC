# アーキテクチャ類 Agent

アーキテクチャ類 Agent は、システムアーキテクチャ設計、ネットワークアーキテクチャ計画、パフォーマンス最適化を担当します。

## Agent 一覧

| Agent 名 | 用途 | 使用モデル | コアツール |
|------------|------|----------|----------|
| architect | ソフトウェアアーキテクチャ専門家 | opus | Read, Grep, Glob |
| network-architect | 企業ネットワークアーキテクチャ設計 | sonnet | Read, Grep |
| homelab-architect | 家庭/ラボネットワーク計画 | sonnet | Read, Grep |
| code-explorer | コードベース深さ分析 | sonnet | Read, Grep, Glob |
| performance-optimizer | パフォーマンス分析と最適化 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| harness-optimizer | Agent harness 設定最適化 | sonnet | Read, Grep, Glob, Bash, Edit |

## architect

### 名前と用途
ソフトウェアアーキテクチャ専門家、システム設計、拡張性と技術的意思決定を専門とする。新機能計画、大きなシステムのリファクタリング、アーキテクチャ意思決定を行う際に積極的に使用する。

### 能力説明
- 新機能のためのシステムアーキテクチャ設計
- 技術的トレードオフの評価
- 推奨パターンとベストプラクティス
- 拡張性のボトルネックを特定
- 将来の成長を計画
- コードベースの整合性を確保

### 適用シナリオ
- 新機能アーキテクチャ設計
- 大規模システムリファクタリング
- アーキテクチャ意思決定
- 技術選択の意思決定
- 拡張性計画

### 使用ツール一覧
- Read: 既存のアーキテクチャを読み取り
- Grep: パターンを検索
- Glob: ファイルを検索

### 他の Agent との連携
- planner に基づいてアーキテクチャ設計を実施計画を作成
- code-architect 特定の機能のためにアーキテクチャを設計
- build-error-resolver ビルド問題を処理
- code-reviewer コード品質をレビュー

### アーキテクチャレビュー流れ

#### 1. 現在の状態分析
- 既存のアーキテクチャをレビュー
- パターンと約束事を特定
- 技術的負債を記録
- 拡張性の制約を評価

#### 2. 要件収集
- 機能要件
- 非機能要件（パフォーマンス、セキュリティ、拡張性）
- 統合点
- データフロー要件

#### 3. 設計提案
- 高レベルアーキテクチャ図
- コンポーネント責務
- データモデル
- API 契約
- 統合パターン

#### 4. トレードオフ分析
各設計意思決定を記録：
- **長所**: メリットと利点
- **短所**: 缺点と制約
- **代替案**: 検討中のその他オプション
- **意思決定**: 最終選択と理由

### アーキテクチャ原則

#### 1. モジュール化と関心の分離
- 単一責任原則
- 高凝縮、低結合
- コンポーネント間の明確なインターフェース
- 独立デプロイ可能性

#### 2. 拡張性
- 水平拡張能力
- 可能な条件下でステートレス設計
- 効率的なデータベースクエリ
- キャッシュ戦略
- 負荷分散の検討

#### 3. 保守性
- 明確なコード編成
- 一貫したパターン
- 包括的なドキュメント
- テストの容易さ
- 理解の容易さ

#### 4. セキュリティ
- 多層防御
- 最小権限原則
- 境界入力検証
- デフォルトセキュリティ
- 監査トレース

#### 5. パフォーマンス
- 効率的なアルゴリズム
- ネットワークリクエストの最小化
- データベースクエリの最適化
- 適切なキャッシュ
- 遅延読み込み

### アーキテクチャ意思決定記録 (ADRs)

重要なアーキテクチャ意思決定に対しては、ADR を作成：

```markdown
# ADR-001: Use Redis for Semantic Search Vector Storage

## Context
セマンティック市場検索に 1536 次元_embeddings を保存・検索する必要がある。

## Decision
ベクトル検索機能付きの Redis Stack を使用。

## Consequences

### Positive
- 高速ベクトル類似性検索 (<10ms)
- 内蔵 KNN アルゴリズム
- シンプルなデプロイ
- 最大 100K ベクターまで良好パフォーマンス

### Negative
- メモリストレージ（大数据集は高昂）
- クラスタなしの場合、単一障害点
- コサイン類似度に限定

### Alternatives Considered
- **PostgreSQL pgvector**: より遅いが、永続ストレージ
- **Pinecone**: マネジメントサービス、より高コスト
- **Weaviate**:  更多機能、より複雑な設定

## Status
Accepted

## Date
2025-01-15
```

### システム設計チェックリスト

#### 機能要件
- [ ] ユーザーストーリーが記録されている
- [ ] API 契約が定義されている
- [ ] データモデルが指定されている
- [ ] UI/UX フローがマッピングされている

#### 非機能要件
- [ ] パフォーマンス目標が定義されている（遅延、スループット）
- [ ] 拡張性要件が指定されている
- [ ] セキュリティ要件が特定されている
- [ ] 可用性目標が設定されている（稼働時間 %）

#### 技術設計
- [ ] アーキテクチャ図が作成されている
- [ ] コンポーネント責務が定義されている
- [ ] データフローが記録されている
- [ ] 統合点が特定されている
- [ ] エラー処理戦略が定義されている
- [ ] テスト戦略が計画されている

#### 運用
- [ ] デプロイ戦略が定義されている
- [ ] 監視とアラートが計画されている
- [ ] バックアップと再開戦略
- [ ] ロールバック計画が記録されている

### 警告旗

これらのアーキテクチャ反パターンを警戒：
- **大きな泥だまり**: 明確な構造がない
- **金のハンマー**: すべての問題に同じ解決策を使用
- **過早最適化**: 早すぎる最適化
- **notinvented here**: 既存の解決策を拒否
- **分析麻痺**: 過度な計画、構築不足
- **魔法**: 不明確で文書化されていない動作
- **密結合**: コンポーネントが相互に過度に依存
- **上帝オブジェクト**: 1 クラス/コンポーネントがすべてを行う

## network-architect

### 名前と用途
要件から企業或多站点ネットワークアーキテクチャを設計。既存のネットワークスキルを使用して、ルーティング、検証、自動化、トラブルシューティングに専念。

### 能力説明
- キャンパス、ブランチ、WAN、データセンター、クラウド近接、ハイブリッドネットワーク計画
- IP アドレッシング、セグメンテーション、ルーティングドメイン、管理プレーンアクセス
- 冗長性、監視、移行序列
- 設計とレビューのみアプリケーション設定ではない

### 適用シナリオ
- 企業ネットワーク設計
- 多站点ネットワークアーキテクチャ
- データセンターネットワーク計画
- クラウドネットワーク統合

### 使用ツール一覧
- Read: 要件と既存の設定を読み取り
- Grep: ネットワークパターンを検索

### 他の Agent との連携
- network-config-validation 変更前設定を検証
- network-bgp-diagnostics BGP 診断を実行
- network-interface-health インターフェース健全性分析を実行
- cisco-ios-patterns IOS/IOS-XE 構文を提供
- netmiko-ssh-automation ネットワーク自動化を実行

### ワークフロー

1. 目標、制約、非目標を要約
2. アーキテクチャを著しく変更する欠落要件を特定
3.  топология選擇し理由を説明
4. ハードウェア議論の前にルーティングとセグメンテーションを設計
5. 管理プレーン、ログ、監視、バックアップ、ロールバックモデルを定義
6. 検証ゲートとロールバックポイントを持つ段階的実施計画を生成
7. 残留リスクと.operator からまだ必要とする証拠をリスト

### 設計デフォルト値

- レイヤー 2 設計よりルーティング境界を優先
- 管理、サーバー、ユーザー、ゲスト、IoT/OT、規制環境の明示的セグメンテーションを優先
- BGP、OSPF、EVPN、SD-WAN、微分段は必須ではないと想定しない
- セキュリティコントロールをアーキテクチャの一部として、後付け想法ではなく扱う

## homelab-architect

### 名前と用途
ハードウェア清单、目標、演算子経験レベルから家庭と小規模ラボネットワーク計画を設計。セキュリティと段階的変更およびロールバック指導を持つ。

### 能力説明
- 家庭と小規模ラボゲートウェイ、スイッチ、AP、NAS、サーバー
- ローカル DNS、DHCP、ゲストネットワーク、IoT 分離
- リモートアクセス計画
- 計画とレビューのみアプリケーション設定ではない

### 適用シナリオ
- 家庭ネットワーク設計
- 小規模ラボネットワーク計画
- homelab ネットワーク設定

### 使用ツール一覧
- Read: ハードウェア仕様と要件を読み取り
- Grep: ネットワークパターンを検索

### 他の Agent との連携
- homelab-network-readiness 変更前に評価
- homelab-network-setup IP 範囲、DHCP 予約、ケーブルを実行
- network-config-validation ゲートウェイまたはスイッチ設定を検証
- network-interface-health リンク、ポート、ケーブルを分析

### ワークフロー

1. ハードウェアを года盘点: ゲートウェイ/ルーター、スイッチ、AP、サーバー、NAS、DNS リゾルバー、ISP ハンドオフ、リモートアクセスパス
2. 目標を確認: 分離、ゲスト Wi-Fi、广告屏蔽、ローカルサービス、リモートアクセス、バックアップ、監視、学習ラボ、家族信頼性
3. 目標をハードウェア能力に一致させる。ハードウェアが VLAN、ローカル DNS、セキュリティリモートアクセスをサポートしない場合は指摘し、段階的アップグレードパスを提案
4. 最小限の動作可能トポロジを先に設計し、その後オプションの後続フェーズ
5. 破壊的変更の前定义了ロールバックとアクセスセキュリティ
6. 實施順序を生成し、各ステップでインターネット、DNS、管理アクセスを再開可能に保つ

### セキュリティデフォルト値

- 管理インターフェースをインターネットに露出することを推奨しない
- 防火壁ルール、認証、DNS フィルタリング、セグメンテーションを故障排除のショートカットとして無効化を推奨しない
- リゾルバーを持つ静的アドレス、健康检查、フォールバックパスの前に、DHCP DNS をローカルリゾルバーに変更することを避ける
- オペレーターが変更後にゲートウェイ、スイッチ、AP に到達できる可能性がない限り、VLAN 移行を避ける
- わかりやすい説明と小規模可逆フェーズを優先

## code-explorer

### 名前と用途
実行パスのトレース、アーキテクチャ層のマップ、依存関係，记录を通じて既存のコードベースの機能を深く分析し、新開発に信息を提供。

### 能力説明
- エントリーポイント発見
- 実行パストレース
- アーキテクチャ層マッピング
- パターン識別
- 依存関係の文書化

### 適用シナリオ
- 既存の機能を理解するとき
- 新開発開始前
- コードリファクタリング前
-  デバッグ问题时

### 使用ツール一覧
- Read: コードを読み取り
- Grep: コードパターンを検索
- Glob: ファイルを検索

### 他の Agent との連携
- code-architect に基づいて新機能の探索を設計
- code-reviewer コード品質をレビュー
- planner に基づいて理解を作成し計画を立案

### 分析流れ

#### 1. エントリーポイント発見
- 機能または領域の主要なエントリーポイントを見つける
- ユーザー操作または外部トリガーから entireスタックをトレース

#### 2. 実行パストレース
- 入口から完了まで呼び出し鎖に従う
-  分岐論理と非同期境界に注意
- データ変換とエラーパスをマップ

#### 3. アーキテクチャ層マッピング
- コードが触れる層を特定
- これらの層がどのように通信するかを理解
- 再利用可能境界と反パターンをメモ

#### 4. パターン識別
- 既に使用されているパターンと抽象を特定
- 命名約束とコード構成原則を記録

#### 5. 依存関係文書化
- 外部ライブラリхибник とサービスをマップ
- 内部モジュール依存関係をマップ
- 再利用価値のある共有ツールを特定

## performance-optimizer

### 名前と用途
パフォーマンス分析と最適化専門家。ボトルネックの特定、遅いコードの最適化、包含サイズの削減、実行時パフォーマンスの向上積極的に使用する。

### 能力説明
- パフォーマンス分析 - 遅いコードパス、メモリリーク、ボトルネックを特定
- バンドル最適化 - JavaScript 包含サイズの削減、遅延読み込み、コード分割
- 実行時最適化 - アルゴリズム効率の改善、不必要な計算の削減
- React/描画最適化 - 不必要な再レンダーを防止、コンポーネントツリーの最適化
- データベースとネットワーク - クエリの最適化、API 呼出しの削減、キャッシュ実装
- メモリ管理 - リーク 检测、最適化されたメモリ使用、クリーンアップリソース

### 適用シナリオ
- パフォーマンス問題の診断時
- パフォーマンス最適化前
- Lighthouse スコアが低下した時
- バンドルサイズが増加した時
- メモリ使用量が増加した時
- ページ読み込みが低速になった時

### 使用ツール一覧
- Read: コードを読み取り
- Write: 最適化コードを書き込み
- Edit: 最適化を編集
- Bash: パフォーマンス分析ツールを実行
- Grep: パフォーマンスパターンを検索
- Glob: ファイルを検索

### 他の Agent との連携
- code-reviewer コード品質チェックを共有
- mle-reviewer ML パフォーマンス問題を処理
- build-error-resolver ビルド問題を処理

### 主要パフォーマンス指標

| Metric | Target | Action if Exceeded |
|--------|--------|-------------------|
| First Contentful Paint | < 1.8s | 重要パスを最適化、重要 CSS をインライン化 |
| Largest Contentful Paint | < 2.5s | 画像を遅延読み込み、サーバー応答を最適化 |
| Time to Interactive | < 3.8s | コード分割、JavaScript を削減 |
| Cumulative Layout Shift | < 0.1 | 画像にスペースを確保、レイアウトシフトを回避 |
| Total Blocking Time | < 200ms | 長タスクを分解、Web Workers を使用 |
| Bundle Size (gzip) | < 200KB | Tree shaking、遅延読み込み、コード分割 |

### アルゴリズム分析

非効率的なアルゴリズムを検査：

| Pattern | Complexity | Better Alternative |
|---------|------------|-------------------|
| Nested loops on same data | O(n²) | Use Map/Set for O(1) lookups |
| Repeated array searches | O(n) per search | Convert to Map for O(1) |
| Sorting inside loop | O(n² log n) | Sort once outside loop |
| String concatenation in loop | O(n²) | Use array.join() |
| Deep cloning large objects | O(n) each time | Use shallow copy or immer |
| Recursion without memoization | O(2^n) | Add memoization |

### React パフォーマンス最適化

一般的な React 反パターン：

```tsx
// BAD: 描画中にインライン関数を作成
<Button onClick={() => handleClick(id)}>Submit</Button>

// GOOD: useCallback を使用して安定コールバック
const handleButtonClick = useCallback(() => handleClick(id), [handleClick, id]);
<Button onClick={handleButtonClick}>Submit</Button>

// BAD: 描画中にオブジェクトを作成
<Child style={{ color: 'red' }} />

// GOOD: 安定したオブジェクト参照
const style = useMemo(() => ({ color: 'red' }), []);
<Child style={style} />

// BAD: 每次描画の高昂な計算
const sortedItems = items.sort((a, b) => a.name.localeCompare(b.name));

// GOOD: 記憶された高昂な計算
const sortedItems = useMemo(
  () => [...items].sort((a, b) => a.name.localeCompare(b.name)),
  [items]
);
```

### メモリリーク検出

一般的なメモリリークパターン：

```typescript
// BAD: クリーンアップなしのイベントリスナー
useEffect(() => {
  window.addEventListener('resize', handleResize);
  // Missing cleanup!
}, []);

// GOOD: クリーンアップイベントリスナー
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);

// BAD: クリーンアップなしのタイマー
useEffect(() => {
  setInterval(() => pollData(), 1000);
  // Missing cleanup!
}, []);

// GOOD: クリーンアップタイマー
useEffect(() => {
  const interval = setInterval(() => pollData(), 1000);
  return () => clearInterval(interval);
}, []);
```

### 成功指標

- Lighthouse パフォーマンススコア > 90
- すべての Core Web Vitals が「良好」範囲内
- バンドルサイズが予算内
- メモリリークが検出されていない
- テストスイートがまだ合格
- パフォーマンスのの後退がない

## harness-optimizer

### 名前と用途
ローカル agent harness 設定的分析と改善、信頼性、コスト、スループットを向上させる。

### 能力説明
- harness 設定を分析
- レバー領域を特定（hooks、evals、routing、context、safety）
- 最小限の、可逆的な設定変更を提案
- 設定を適用し検証を実行
- 前後の差分をレポート

### 適用シナリオ
- Agent の結果品質が低下した時
- コスト最適化が必要时
- スループット改善が必要时
- Harness 設定レビュー時

### 使用ツール一覧
- Read: 設定を読み取り
- Grep: 設定パターンを検索
- Glob: 設定ファイルを検索
- Bash: 検証コマンドを実行
- Edit: 設定を変更

### 他の Agent との連携
- プロジェクトのその他 agent と協業
- プロジェクトの要件に基づいて設定を最適化

### ワークフロー

1. `/harness-audit` を実行し、ベースラインスコアを収集
2. 上位 3 つのレバー領域を特定（hooks、evals、routing、context、safety）
3. 最小限の、可逆的な設定変更を提案
4. 設定を適用し検証を実行
5. 前後の差分をレポート

### 制約

- 小さな変更で大きな効果を優先
- クロスプラットフォーム動作を維持
- 脆弱なシェル引用符の導入を避ける
- Claude Code、Cursor、OpenCode、Codex 間の互換性を維持

### 出力

- ベースラインスコアカード
- 設定への変更
- 測定された改善
- 残留リスク
