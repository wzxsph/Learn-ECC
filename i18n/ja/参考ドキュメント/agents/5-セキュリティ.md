# セキュリティ類 Agent

セキュリティ類 Agent は、セキュリティ脆弱性検出、準拠性レビューと機密情報保護を担当します。

## Agent 一覧

| Agent 名 | 用途 | 使用モデル | コアツール |
|------------|------|----------|----------|
| security-reviewer | セキュリティ脆弱性検出と修正専門家 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| silent-failure-hunter | 静的失敗検出専門家 | sonnet | Read, Grep, Glob, Bash |
| healthcare-reviewer | 医療アプリケーションコードレビュー（PHI/臨床セキュリティ） | opus | Read, Grep, Glob |
| opensource-sanitizer | オープンソースプロジェクト漏えい検査専門家 | sonnet | Read, Grep, Glob, Bash |

---

## security-reviewer

### 名前と用途
セキュリティ脆弱性検出と修正専門家。ユーザー入力、認証、API エンドポイントまたは機密データを處理するコードを作成後に積極的に使用。secrets、SSRF、注入、安全でない暗号化と OWASP Top 10 脆弱性をマーク。

### 能力説明
- 脆弱性検出 - OWASP Top 10 と常见セキュリティ問題を識別
- Secrets 検出 - ハードコードされた API キー、パスワード、tokens を発見
- 入力検証 - すべてのユーザー入力が適切にクリーンアップされていることを確保
- 認証/認可 - 適切なアクセス制御を検証
- 依存関係セキュリティ - 脆弱性がある npm パッケージを検査
- セキュリティベストプラクティス - セキュリティエンコードパターンを強制実行

### 適用シナリオ
- 新 API エンドポイント
- 認証コードの変更
- ユーザー入力處理
- データベースクエリの変更
- ファイルアップロード
- 支払コード
- 外部 API 統合
- 依存関係の更新

### 使用ツール一覧
- Read: コードを読み取り
- Write: セキュリティ修正を書き込み
- Edit: セキュリティ修正を編集
- Bash: npm audit, eslint を実行
- Grep: 敏感パターンを検索
- Glob: ファイルを検索

### 他の Agent との連携
- code-reviewer がコード品質チェックを共有
- healthcare-reviewer が医療固有のセキュリティ問題を処理
- opensource-sanitizer がオープンソース公開前の検査を処理

### 重要なレビュー領域

#### 認証と認可
- セッション管理が安全であることを確認
- パスワード存储が適切であることを確認
- アクセス制御が正しく実装されていることを確認

#### 入力検証
- すべてのユーザー入力が検証されていることを確認
- SQL 注入を防止するためにパラメータ化クエリを使用
- XSS を防止するために出力エンコーディングを確認

#### データ保護
- 機密データが暗号化されていることを確認
- 安全でない保管を検出して修正
- 会議entropy管理を検証

#### API セキュリティ
- API キーが安全に保管されていることを確認
- 速度制限が実装されていることを確認
- CORS 設定が正しいことを確認

### 検出される脆弱性

#### CRITICAL
- SQL 注入
- 認証バイパス
- 機密データの露出
- 安全でない暗号化

#### HIGH
- XSS
- CSRF
- 弱いセッション管理
- 安全でない直接オブジェクト参照

#### MEDIUM
- Information disclosure
- 安全でない設定
-  транспортlayer security の問題

### 修正の優先順位

1. **CRITICAL** - 即座に修正
2. **HIGH** - 1週間以内に修正
3. **MEDIUM** - 2週間以内に修正
4. **LOW** - 次のスプリントで修正

## silent-failure-hunter

### 名前と用途
静的失敗検出専門家。失敗を静かに Поједина的原因是、プログラムが失敗时应即座に失敗而非適切に错误を報告するパターンを検出。

### 能力説明
- 静默エラーパターンの検出
- エラー処理の欠落を識別
- Promise 拒否の未処理を検出
- 例外飞出の未捕获を検出

### 適用シナリオ
- コードレビュー時
- エラー処理の検証時
- システムの信頼性評価時

### 使用ツール一覧
- Read: コードを読み取り
- Grep: エラーパターンを検索
- Glob: ファイルを選択
- Bash: 静的解析ツールを実行

### 他の Agent との連携
- code-reviewer がコード品質をレビュー
- build-error-resolver がビルド問題を処理
- tdd-guide がテストを追加

### 検出されるパターン

#### 1. 缺失エラー処理
```typescript
// 悪い例
fetch(url).then(response => response.json());

// 良い例
fetch(url)
  .then(response => {
    if (!response.ok) throw new Error(`HTTP ${response.status}`);
    return response.json();
  })
  .catch(error => console.error('Failed:', error));
```

#### 2. 未処理の Promise 拒否
```typescript
// 悪い例
async function fetchData() {
  someAsyncOperation(); // 拒否された場合、誰もcatchしない
}

// 良い例
async function fetchData() {
  try {
    await someAsyncOperation();
  } catch (error) {
    console.error('Operation failed:', error);
  }
}
```

#### 3. 空的 catch ブロック
```typescript
// 悪い例
try {
  riskyOperation();
} catch (e) {
  // 何もしない
}

// 良い例
try {
  riskyOperation();
} catch (error) {
  console.error('Operation failed:', error);
  throw error; // または適切に処理
}
```

#### 4. 回调内のエラー
```typescript
// 悪い例
fs.readFile(path, (err, data) => {
  if (err) console.log('error'); // まだ実行 продолжается
  process(data);
});

// 良い例
fs.readFile(path, (err, data) => {
  if (err) {
    console.error('Failed to read file:', err);
    return;
  }
  process(data);
});
```

### 軽減戦略

1. **全局エラー処理** - 捕获されていない例外を処理
2. **Promise の最後 に .catch()** - Promise 拒否を常に処理
3. **エラー境界** - React でエラー境界を使用
4. **構造化されたエラー処理** - カスタムエラータイプを使用

## healthcare-reviewer

### 名前と用途
医療アプリケーションコードレビュー専門家。PHI（保護医療情報）と臨床セキュリティに焦点。

### 能力説明
- PHI 露出の検出
- 臨床データの保護
- HIPAA 準拠性チェック
- 医療アルゴリズムの妥当性検証

### 適用シナリオ
- 医療アプリケーションのコードレビュー
- PHI を処理するコードの変更
- HIPAA 準拠性の検証

### 使用ツール一覧
- Read: コードを読み取り
- Grep: PHI パターンを検索
- Glob: ファイルを選択

### 他の Agent との連携
- security-reviewer が一般的なセキュリティ問題を処理
- code-reviewer がコード品質をレビュー

### HIPAA 準拠性チェック

#### 1. データ暗号化
- 保存時のデータ暗号化
- 传输中のデータ暗号化
- 鍵管理

#### 2. アクセス制御
- 最小権限原則
- ユーザーの認証
- 定期的なアクセスレビュー

#### 3. 監査証跡
- すべての PHI アクセスをログに記録
- 不正アクセスの検出
- ログの完全性

#### 4. データマスキング
- 開発環境での PHI マスキング
- テストデータからの PHI 除去
- ログの PHI 除外

## opensource-sanitizer

### 名前と用途
オープンソースプロジェクト漏えい検査専門家。オープンソース化する前のコードから機密情報を検出して除去。

### 能力説明
- API キーの検出
- 資格情報の検出
- 機密データの検出
- 排除パターンの定義

### 適用シナリオ
- オープンソースプロジェクトの公開前
- コードの 외부 共有前
- 依赖项の漏えい検査

### 使用ツール一覧
- Read: コードを読み取り
- Grep: 機密パターンを検索
- Glob: ファイルを選択
- Bash: 走査ツールを実行

### 検出される機密情報

#### API キー
- AWS キー
- GCP キー
- Azure キー
- Stripe キー
- GitHub トークン

#### 資格情報
- データベースパスワード
- 認証情報
- シークレットキー

#### 個人データ
- メールアドレス
- 電話番号
- 社会保障番号

### 修復戦略

1. **環境変数** - 機密情報を環境変数に移動
2. **.gitignore** - 機密ファイルを無視
3. **ensive 置換** - 実際の値をプレースホルダーに置換
4. **文書化** - 必要な環境変数を文書化
