# セキュリティ類 Agent

セキュリティ類 Agent 专门负责セキュリティ脆弱性検出、合规性レビュー和敏感情報保护。

## Agent 一覧

| Agent 名 | 用途 | 使用モデル | コアツール |
|------------|------|----------|----------|
| security-reviewer | セキュリティ脆弱性検出和修正专家 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| silent-failure-hunter | 静默失敗检测专家 | sonnet | Read, Grep, Glob, Bash |
| healthcare-reviewer | 医疗应用コードレビュー（PHI/临床セキュリティ） | opus | Read, Grep, Glob |
| opensource-sanitizer | 开源プロジェクト泄露検査专家 | sonnet | Read, Grep, Glob, Bash |

## security-reviewer

### 名称と用途
セキュリティ脆弱性検出和修正专家。在编写处理用户输入、認証、API 端点或敏感データ的コード后主动使用。标记 secrets、SSRF、注入、不セキュリティ暗号化和 OWASP Top 10 漏洞。

### 能力説明
- 漏洞检测 - 识别 OWASP Top 10 和常见セキュリティ問題
- Secrets 检测 - 发现硬エンコード API 密钥、密码、tokens
- 输入検証 - 确保すべて用户输入被正确クリーンアップ
- 認証/認可 - 検証适当的访问控制
- 依赖セキュリティ - 検査あり漏洞的 npm パッケージ
- セキュリティベストプラクティス - 强制実行セキュリティエンコードパターン

### 適用シナリオ
- 新 API 端点
- 認証コード变更
- 用户输入处理
- データベース查询变更
- ファイル上传
- 支付コード
- 外部 API 統合
- 依赖更新

### 使用ツール一覧
- Read: 読み取りコード
- Write: 書き込みセキュリティ修正
- Edit: 編集セキュリティ修正
- Bash: 実行 npm audit, eslint
- Grep: 搜索敏感パターン
- Glob: 検索ファイル

### 他の Agent との連携
- code-reviewer 共享コード品質チェック
- healthcare-reviewer 处理医疗特定セキュリティ問題
- opensource-sanitizer 处理开源发布前検査

### 初始扫描

```bash
npm audit --audit-level=high
npx eslint . --plugin security
```

搜索硬エンコード secrets，レビュー高风险区域：auth、API 端点、DB 查询、ファイル上传、支付、webhooks。

### OWASP Top 10 検査

1. **注入** - 查询は否参数化？用户输入は否クリーンアップ？ORM は否セキュリティ使用？
2. **認証破坏** - 密码は否哈希 (bcrypt/argon2)？JWT は否検証？セッションは否セキュリティ？
3. **敏感データ** - HTTPS は否强制？Secrets は否在 env vars？PII は否暗号化？ログは否クリーンアップ？
4. **XXE** - XML 解析器は否セキュリティ設定？外部实体は否無効化？
5. **访问控制破坏** - 各ルーティングは否検査 auth？CORS は否正确設定？
6. **セキュリティ設定エラー** - デフォルト凭据は否更改？生产环境 debug パターンは否閉じる？セキュリティ头は否設定？
7. **XSS** - 输出は否转义？CSP は否設定？フレームワークは否自動转义？
8. **不セキュリティデシリアライズ** - 用户输入は否セキュリティデシリアライズ？
9. **已知漏洞** - 依赖は否最新？npm audit は否通過？
10. **ログ和監視不足** - セキュリティ事件は否記録？は否設定警报？

### コードパターンレビュー

立即标记这些パターン：

| パターン | 重大性 | 修正 |
|------|--------|------|
| 硬エンコード secrets | CRITICAL | 使用 `process.env` |
| 用户输入的 shell コマンド | CRITICAL | 使用セキュリティ API 或 execFile |
| 字符串拼接 SQL | CRITICAL | 参数化查询 |
| `innerHTML = userInput` | HIGH | 使用 `textContent` 或 DOMPurify |
| `fetch(userProvidedUrl)` | HIGH | 白名单許可的域名 |
| 明文密码比较 | CRITICAL | 使用 `bcrypt.compare()` |
| ルーティングなし auth 検査 | CRITICAL | 追加認証ミドルウェア |
| なし锁的余额検査 | CRITICAL | 在事务中使用 `FOR UPDATE` |
| なし速率限制 | HIGH | 追加 `express-rate-limit` |
| 記録密码/secrets | MEDIUM | クリーンアップログ输出 |

### 重要原則

1. **纵深防御** - 多层セキュリティ
2. **最小权限** - 所需最小权限
3. **セキュリティ失敗** - エラー不应暴露データ
4. **不信任输入** - 検証和クリーンアップ一切
5. **定期更新** - 保持依赖最新

### 常见误报

- `.env.example` 中的环境変数（ではない实际 secrets）
- テストファイル中的テスト凭据（如果明确标记）
- 实际公开的公钥 API 密钥
- 用于校验和的 SHA256/MD5（ではない密码）

**始终在标记前検証上下文。**

### 紧急応答

如果发现 CRITICAL 漏洞：
1. 详细レポート記録
2. 立即通知プロジェクト负责人
3. 提供セキュリティコードサンプル
4. 検証修正あり效
5. 如果凭据暴露则轮换 secrets

## silent-failure-hunter

### 名称と用途
レビューコード中的静默失敗、被吞掉的エラー、エラー的回退和缺失的エラー传播。

### 能力説明
- 空 catch 块检测
- 不充分ログ检测
- 危险回退检测
- エラー传播問題检测
- 缺失エラー処理检测

### 適用シナリオ
- コードレビュー时
- 发现奇怪 bug 时
- 管道看似绿色但跳过データ时
- 异常处理レビュー时

### 使用ツール一覧
- Read: 読み取りコード
- Grep: 搜索エラー処理パターン
- Glob: 検索源ファイル
- Bash: 実行テスト

### 他の Agent との連携
- code-reviewer 共享コード品質チェック
- security-reviewer 处理セキュリティ相关問題
- mle-reviewer 处理 ML 管道問題

### 狩猎目标

#### 1. 空 Catch 块
- `catch {}` 或被無視的异常
- 转换为 `null` / 空数组但なし上下文的エラー

#### 2. 不充分ログ
- ログ上下文不足
- エラー重大性不对
- log-and-forget 处理

#### 3. 危险回退
- 非表示真实失敗的デフォルト值
- `.catch(() => [])`
- 优雅但使下游 bug 更难診断的路径

#### 4. エラー传播問題
- 丢失的スタックトレース
- 泛型 rethrows
- 缺失 async 处理

#### 5. 缺失エラー処理
- 网络/ファイル/データベース路径なしタイムアウト或エラー処理
- 事务作業なしロールバック

### 输出フォーマット

各发现：
- 位置
- 重大性
- 問題
- 影响
- 修正推奨

## healthcare-reviewer

### 名称と用途
レビュー医疗应用コード的临床セキュリティ、CDSS 准确性、PHI 合规性和医疗データ完整性。专为 EMR/EHR、临床决策サポート和健康情報システム而設計。

### 能力説明
- CDSS 准确性 - 検証药物相互作用逻辑、剂量検証ルール和临床评分実装
- PHI/PII 保护 - 扫描患者データ在ログ、エラー、応答、URL 和クライアント存储中的暴露
- 临床データ完整性 - 确保审计トレース、ロック記録和级联保护
- 医疗データ正确性 - 検証 ICD-10/SNOMED 映射、实验室参考范围和药物データベース条目
- 統合合规性 - 検証 HL7/FHIR 消息处理和エラー再開

### 適用シナリオ
- EMR/EHR システム開発
- 临床决策サポートシステム
- 健康情報システム
- 医疗データ处理应用

### 使用ツール一覧
- Read: 読み取り医疗コード
- Grep: 搜索 PHI パターン
- Glob: 検索医疗相关ファイル

### 他の Agent との連携
- security-reviewer 处理通用セキュリティ漏洞
- code-reviewer 处理コード品質問題
- silent-failure-hunter 处理静默失敗問題

### 重要検査

#### CDSS 引擎
- [ ] すべて药物相互作用对产生正确警报（双向）
- [ ] 剂量検証ルール在超出范围值时触发
- [ ] 临床评分匹配发布仕様
- [ ] なし假阴性（漏掉的相互作用 = 患者セキュリティ事件）
- [ ] フォーマットエラー的输入产生エラー，而非静默通過

#### PHI 保护
- [ ] 患者データ不在 `console.log`、`console.error` 或エラーメッセージ中
- [ ] PHI 不在 URL 参数或查询字符串中
- [ ] PHI 不在浏览器 localStorage/sessionStorage 中
- [ ] クライアントコード中なし `service_role` 密钥
- [ ] すべて患者データ表あり効化 RLS
- [ ] 跨设施データ隔离已検証

#### 临床ワークフロー
- [ ] 诊疗ロック防止編集（仅附录）
- [ ] 每次临床データ CRUD 都あり审计トレース条目
- [ ] 重要警报不可無視（ではない toast 通知）
- [ ] 临床医生継続进行重要警报时記録覆盖原因
- [ ] 红旗症状触发可见警报

#### データ完整性
- [ ] 患者記録なし CASCADE DELETE
- [ ] 並行処理編集检测（乐观锁或冲突解決）
- [ ] 临床表なし孤立記録
- [ ] 时间戳使用一致的时区

### 输出フォーマット

```
## Healthcare Review: [module/feature]

### Patient Safety Impact: [CRITICAL / HIGH / MEDIUM / LOW / NONE]

### Clinical Accuracy
- CDSS: [検査通過/失敗]
- Drug DB: [検証/問題]
- Scoring: [匹配仕様/偏离]

### PHI Compliance
- 暴露向量検査: [列表]
- 发现問題: [列表或なし]

### Issues
1. [PATIENT SAFETY / CLINICAL / PHI / TECHNICAL] 描述
   - Impact: [潜在危害或暴露]
   - Fix: [所需变更]

### Verdict: [SAFE TO DEPLOY / NEEDS FIXES / BLOCK — PATIENT SAFETY RISK]
```

### ルール

- 对临床准确性あり疑问时，标记为 NEEDS REVIEW - 绝不承認不确定的临床逻辑
- 一次漏掉的药物相互作用比一百个假警报更糟糕
- PHI 暴露始终は CRITICAL 重大性，なし论泄露多小
- 绝不承認静默捕获 CDSS エラー的コード

## opensource-sanitizer

### 名称と用途
在发布前検証开源 fork 已完全クリーンアップ。扫描泄露的 secrets、PII、内部引用和危险ファイル。使用 20+ 正则表达式パターン。生成 PASS/FAIL/PASS-WITH-WARNINGS レポート。

### 能力説明
- Secrets 扫描 - 扫描 API 密钥、密码、tokens
- PII 扫描 - 扫描邮箱、IP 地址
- 内部引用扫描 - 扫描绝对路径、内部域名
- 危险ファイル検査 - 検査 .env、credentials.json 等
- 設定完整性検証 - 検証 .env.example
- Git 历史审计 - 検査泄露的凭据

### 適用シナリオ
- 开源 fork 发布前
- 第三方コード审计前
- コード发布前的セキュリティ検査

### 使用ツール一覧
- Read: 読み取りファイル
- Grep: 搜索敏感パターン
- Glob: 検索ファイル
- Bash: 実行 git log 等

### 他の Agent との連携
- security-reviewer 处理通用セキュリティ漏洞
- code-reviewer 处理コード品質問題
- doc-updater 更新ドキュメント

### ワークフロー

#### ステップ 1: Secrets 扫描 (CRITICAL)

扫描各文本ファイル（排除 `node_modules`、`.git`、`__pycache__`、`*.min.js`、二进制ファイル）：

```
# API 密钥
pattern: [A-Za-z0-9_]*(api[_-]?key|apikey|api[_-]?secret)[A-Za-z0-9_]*\s*[=:]\s*['"]?[A-Za-z0-9+/=_-]{16,}

# AWS
pattern: AKIA[0-9A-Z]{16}
pattern: (?i)(aws_secret_access_key|aws_secret)\s*[=:]\s*['"]?[A-Za-z0-9+/=]{20,}

# データベース URL（带凭据）
pattern: (postgres|mysql|mongodb|redis)://[^:]+:[^@]+@[^\s'"]+

# JWT tokens (3-segment)
pattern: eyJ[A-Za-z0-9_-]{20,}\.eyJ[A-Za-z0-9_-]{20,}\.[A-Za-z0-9_-]+

# 私钥
pattern: -----BEGIN\s+(RSA\s+|EC\s+|DSA\s+|OPENSSH\s+)?PRIVATE KEY-----

# GitHub tokens
pattern: gh[pousr]_[A-Za-z0-9_]{36,}
pattern: github_pat_[A-Za-z0-9_]{22,}
```

#### ステップ 2: PII 扫描 (CRITICAL)

```
# 个人邮箱（ではない通用邮箱如 noreply@、info@）
pattern: [a-zA-Z0-9._%+-]+@(gmail|yahoo|hotmail|outlook|protonmail|icloud)\.(com|net|org)

# 私あり IP 地址（表示内部インフラ）
pattern: (192\.168\.\d+\.\d+|10\.\d+\.\d+\.\d+|172\.(1[6-9]|2\d|3[01])\.\d+\.\d+)

# SSH 接続字符串
pattern: ssh\s+[a-z]+@[0-9.]+
```

#### ステップ 3: 内部引用扫描 (CRITICAL)

```
# 绝对路径到特定用户主ディレクトリ
pattern: /home/[a-z][a-z0-9_-]*/
pattern: /Users/[A-Za-z][A-Za-z0-9_-]*/
pattern: C:\\Users\\[A-Za-z]

# 内部 secret ファイル引用
pattern: \.secrets/
pattern: source\s+~/\.secrets/
```

#### ステップ 4: 危险ファイル検査 (CRITICAL)

検証这些**不存在**：
```
.env (任意变体: .env.local, .env.production, .env.*.local)
*.pem, *.key, *.p12, *.pfx, *.jks
credentials.json, service-account*.json
.secrets/, secrets/
.claude/settings.json
sessions/
*.map (source maps 暴露原始源構造)
node_modules/, __pycache__/, .venv/, venv/
```

#### ステップ 5: Git 历史审计

```bash
# すべきは单一初始コミット
cd PROJECT_DIR
git log --oneline | wc -l
# 如果 > 1，历史未クリーンアップ — FAIL

# 搜索可能的 secrets
git log -p | grep -iE '(password|secret|api.?key|token)' | head -20
```

### 输出フォーマット

生成プロジェクトディレクトリ中的 `SANITIZATION_REPORT.md`：

```markdown
# Sanitization Report: {project-name}

**Date:** {date}
**Auditor:** opensource-sanitizer v1.0.0
**Verdict:** PASS | FAIL | PASS WITH WARNINGS

## Summary

| Category | Status | Findings |
|----------|--------|----------|
| Secrets | PASS/FAIL | {count} findings |
| PII | PASS/FAIL | {count} findings |
| Internal References | PASS/FAIL | {count} findings |
| Dangerous Files | PASS/FAIL | {count} findings |
| Config Completeness | PASS/WARN | {count} findings |
| Git History | PASS/FAIL | {count} findings |

## Critical Findings (Must Fix Before Release)

1. **[SECRETS]** `src/config.py:42` — Hardcoded database password: `DB_P...` (truncated)

## Warnings (Review Before Release)

1. **[CONFIG]** `src/app.py:8` — Port 8080 hardcoded, should be configurable

## .env.example Audit

- Variables in code but NOT in .env.example: {list}
- Variables in .env.example but NOT in code: {list}

## Recommendation

{If FAIL: "Fix the {N} critical findings and re-run sanitizer."}
{If PASS: "Project is clear for open-source release. Proceed to packager."}
{If WARNINGS: "Project passes critical checks. Review {N} warnings before release."}
```

### ルール

- **绝不**表示完整 secret 值 - 截断为前 4 个字符 + "..."
- **绝不**修改源ファイル - 只生成レポート (SANITIZATION_REPORT.md)
- **始终**扫描各文本ファイル，不只は已知拡張名
- **始终**検査 git 历史，即使は新仓库
- **保持偏执** - 误报可受け入れ，漏报不可
- 任意クラス别的单一 CRITICAL 发现 = 整体 FAIL
- 单独警告 = PASS WITH WARNINGS（用户决定）
