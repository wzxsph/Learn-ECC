# テスト類 Agent

テスト類 Agent 专门负责テスト驱动開発、E2Eテスト、テストカバレッジ分析和品質保证。

## Agent 一覧

| Agent 名 | 用途 | 使用モデル | コアツール |
|------------|------|----------|----------|
| tdd-guide | TDD テスト驱动開発专家 | sonnet | Read, Write, Edit, Bash, Grep |
| e2e-runner | E2Eテスト专家 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| mle-reviewer | ML 工程コードレビュー（訓練/推論/監視） | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | PR テスト覆盖分析 | sonnet | Read, Grep, Glob, Bash |

## tdd-guide

### 名称と用途
テスト驱动開発专家，强制実行写テスト优先的メソッド论。主动用于编写新機能、修正 bug 或リファクタリングコード。确保 80%+ 的テストカバレッジ。

### 能力説明
- 强制実行テスト优先的メソッド论
- 指导 Red-Green-Refactor 周期
- 确保 80%+ テストカバレッジ
- 编写全面的テスト套件（单元、統合、E2E）
- 在实施前捕获边缘用例

### 適用シナリオ
- 编写新機能时
- 修正 bug 时
- リファクタリングコード时
- 必要 TDD ワークフロー时

### 使用ツール一覧
- Read: 読み取り现ありコード
- Write: 编写テスト
- Edit: 修改テスト和実装
- Bash: 実行テストコマンド
- Grep: 搜索テストパターン

### 他の Agent との連携
- code-reviewer 在 TDD 周期后レビューコード
- build-error-resolver 修正テストビルドエラー
- e2e-runner 进行重要用户流的 E2E テスト

### TDD ワークフロー

#### 1. 先写テスト (RED)
编写描述预期行为的失敗テスト。

#### 2. 実行テスト - 検証它 FAIL
```bash
npm test
```

#### 3. 写最小実装 (GREEN)
只写足够的コード让テスト通過。

#### 4. 実行テスト - 検証它 PASS

#### 5. リファクタリング (IMPROVE)
移除重复，改善名称，最適化 - テストmust保持绿色。

#### 6. 検証カバレッジ
```bash
npm run test:coverage
# 要件: 80%+ ブランチ、関数、行、语句カバレッジ
```

### mustテスト的边缘用例

1. **Null/Undefined** 输入
2. **空** 数组/字符串
3. **なし效タイプ** 传递
4. **边界值** (最小/最大)
5. **エラー路径** (网络失敗、データベースエラー)
6. **竞态条件** (並行処理操作)
7. **大データ** (10k+ 项的パフォーマンス)
8. **特殊字符** (Unicode、表情符号、SQL 字符)

### 反パターン - must避免

- テスト実装细节（内部状态）而非行为
- テスト相互依赖（共享状态）
- 断言太少（通過但不検証任意东西）
- 外部依赖未 mock (Supabase、Redis、OpenAI 等)

### 品質検査清单

- [ ] すべて公共関数都ありユニットテスト
- [ ] すべて API 端点都あり統合テスト
- [ ] 重要用户流あり E2E テスト
- [ ] 边缘用例已覆盖 (null、empty、invalid)
- [ ] エラー路径已テスト（不只は happy path）
- [ ] 外部依赖已 mock
- [ ] テスト相互独立（なし共享状态）
- [ ] 断言具体且あり意义
- [ ] カバレッジ 80%+

### v1.8 Eval-Driven TDD 附录

将 eval-driven 開発統合到 TDD 流程：

1. 在实施前定义能力 + 回归 evals
2. 実行基线并捕获失敗特征
3. 実装最小通過变更
4. 重新実行テスト和 evals；レポート pass@1 和 pass@3
5. 发布重要路径应在マージ前达到 pass^3 稳定性

## e2e-runner

### 名称と用途
E2Eテスト专家，使用 Vercel Agent Browser（首选）和 Playwright 后备。主动用于生成、维护和実行 E2E テスト。

### 能力説明
- テスト旅程作成 - 为用户流编写テスト
- テスト维护 - 保持テスト与 UI 变化同期
- 不稳定テスト管理 - 识别和隔离不稳定テスト
- テストレポート - 生成 HTML レポート和 JUnit XML
- CI/CD 統合 - 确保テスト在管道中可靠実行
- 截图/视频/trace 管理

### 適用シナリオ
- 重要用户旅程必要検証时
- 必要端到端検証时
- CI/CD 管道中実行 E2E テスト时
- 发现統合問題时

### 使用ツール一覧
- Read: 読み取り页面内容
- Write: 编写テスト
- Edit: 修改テスト
- Bash: 実行 Playwright/Agent Browser コマンド
- Grep: 搜索テスト
- Glob: 検索テストファイル

### 他の Agent との連携
- tdd-guide 编写单元/統合テスト
- code-reviewer レビューテスト品質
- mle-reviewer レビュー ML 相关テスト

### 主要ツール: Agent Browser

**优先于原始 Playwright 使用 Agent Browser** - 语义选择器、AI 最適化、自動等待、基于 Playwright。

```bash
# 設定
npm install -g agent-browser && agent-browser install

# コアワークフロー
agent-browser open https://example.com
agent-browser snapshot -i          # 取得带 refs 的元素 [ref=e1]
agent-browser click @e1            # 按 ref 点击
agent-browser fill @e2 "text"     # 按 ref 填充输入
agent-browser wait visible @e5     # 等待元素可见
agent-browser screenshot result.png
```

### 后备: Playwright

当 Agent Browser 不可用时，直接使用 Playwright。

```bash
npx playwright test                        # 実行すべて E2E テスト
npx playwright test tests/auth.spec.ts     # 実行特定ファイル
npx playwright test --headed               # 可见浏览器
npx playwright test --debug                # 使用検査器デバッグ
npx playwright test --trace on             # 带 trace 実行
npx playwright show-report                 # 查看 HTML レポート
```

### ワークフロー

#### 1. 計画
- 识别重要用户旅程（認証、コア機能、支付、CRUD）
- 定义场景: happy path、边缘用例、エラー用例
- 按风险排序: HIGH（金融、認証）、MEDIUM（搜索、导航）、LOW（UI 最適化）

#### 2. 作成
- 使用页面对象モデル (POM) パターン
- 优先使用 `data-testid` 选择器
- 在重要ステップ追加断言
- 在重要时刻捕获截图
- 使用适当的等待（从不 `waitForTimeout`）

#### 3. 実行
- 本地実行 3-5 次検査不稳定性
- 使用 `test.fixme()` 或 `test.skip()` 隔离不稳定テスト
- 上传 artifact 到 CI

### 重要原則

- **使用语义选择器**: `[data-testid="..."]` > CSS 选择器 > XPath
- **等待条件而非时间**: `waitForResponse()` > `waitForTimeout()`
- **自動等待内置**: `page.locator().click()` 自動等待；原始 `page.click()` 不等待
- **隔离テスト**: 各テストすべき独立；なし共享状态
- **快速失敗**: 在各重要ステップ使用 `expect()` 断言
- **再試行时 trace**: 設定 `trace: 'on-first-retry'` 用于デバッグ失敗

### 不稳定テスト处理

```typescript
// 隔离
test('flaky: market search', async ({ page }) => {
  test.fixme(true, 'Flaky - Issue #123')
})

// 识别不稳定性
// npx playwright test --repeat-each=10
```

常见原因: 竞态条件（使用自動等待选择器）、网络计时（等待応答）、动画计时（等待 `networkidle`）。

### 成功指標

- すべて重要旅程通過 (100%)
- 总体通過率 > 95%
- 不稳定率 < 5%
- テスト持续时间 < 10 分钟
- Artifact 上传并可访问

## mle-reviewer

### 名称と用途
生产機械学習工程レビュー专家，专注于データ契约、特征管道、訓練可复现性、离线/在线评估、モデルサービス、監視和ロールバック。

### 能力説明
- 問題定义和决策品質レビュー
- 指標、しきい値和エラー分析レビュー
- データ契约和泄漏検査
- 訓練可复现性検証
- 评估和推广流程レビュー
- サービス和デプロイセキュリティレビュー
- 監視和事件応答計画

### 適用シナリオ
- ML、MLOps、モデル訓練コード变更时
- 推論、特征存储、评估コード变更时
- モデル推广决策时

### 使用ツール一覧
- Read: 読み取り ML コード和設定
- Grep: 搜索パターン
- Glob: 検索ファイル
- Bash: 実行 pytest, ruff, mypy

### 他の Agent との連携
- python-reviewer 处理 Python 风格、タイプ、エラー処理
- pytorch-build-resolver 处理 tensor/CUDA/訓練失敗
- database-reviewer 处理特征表、タグ存储
- security-reviewer 处理 secrets、PII、pickle セキュリティ
- performance-optimizer 处理遅延、メモリ、GPU 利用率
- build-error-resolver 处理 CI、依赖、原生拡張失敗
- pr-test-analyzer 分析テスト覆盖
- silent-failure-hunter 发现管道静默失敗
- e2e-runner 进行产品流 E2E テスト
- a11y-architect レビュー预测可访问性
- doc-updater 更新ドキュメント

### 重要レビュー领域

#### 問題定义和决策品質
- 变更は否从用户或システム决策開始，而非モデルアーキテクチャ偏好
- 利益相关者和失敗成本は否明确
- 指標选择は否遵循エラー预算

#### 指標、しきい値和エラー分析
- 基线和当前生产行为は否比较
- しきい値和設定は否作为产品决策
- 假阳性和假阴性は否直接検査

#### データ契约和泄漏
- 实体粒度、主键、时间戳は否明确
- 分割は否遵守时间、用户/实体分组
- 特征接続は否时间点正确
- 敏感属性は否排除或正当处理

#### 訓練可复现性
- 訓練は否可从コード、設定、データ集バージョン、种子実行
- 超参数、预处理、依赖バージョンは否記録
- 随机性和 GPU 非确定性は否あり意处理

#### 评估和推广
- 指標は否与基线和当前生产モデル比较
- 推广门は否在选择前声明
- 切片指標は否覆盖重要キュー

#### サービス和デプロイ
- 訓練和サービス转换は否共享或等价テスト
- 输入パターンは否拒否过时、缺失、なし效特征
- 推論路径は否ありタイムアウト、资源限制、回退逻辑

#### 監視和事件応答
- 監視は否覆盖サービス健康、特征漂移、预测漂移
- ログは否パッケージ含足够标识符
- ロールバックは否命名前一 artifact、設定、データ依赖

### 常见阻塞器

- 在时间相关或用户相关データ上随机 train/test 分割
- 特征生成使用预测时不可用的字段
- 离线指標改善而重要切片回归
- 訓練预处理被手动コピー到サービスコード
- モデルバージョン不在预测ログ中

### 承認標準

- **APPROVE**: なし重要/高 MLE 风险，相关テスト或评估门通過
- **APPROVE WITH WARNINGS**: 只あり中等問題，あり明确的后续行动
- **BLOCK**: 任意可能的泄漏、不可复现的推广、不セキュリティ的サービス行为、缺少生产デプロイロールバック、敏感データ暴露、或重要评估差距

## pr-test-analyzer

### 名称と用途
レビュー Pull Request テスト覆盖品質和完整性，强调行为覆盖和真实 bug 预防。

### 能力説明
- 识别变更コード映射
- 検証行为覆盖
- 评估テスト品質
- 发现覆盖差距

### 適用シナリオ
- PR レビュー时
- 评估テストカバレッジ时
- 検証テスト充分性时

### 使用ツール一覧
- Read: 読み取り变更コード和テスト
- Grep: 搜索テスト
- Glob: 検索テストファイル
- Bash: 実行テストコマンド

### 他の Agent との連携
- tdd-guide 改善テスト
- e2e-runner 进行端到端覆盖
- code-reviewer 进行コード品質レビュー

### 分析流程

#### 1. 识别变更コード
- 映射变更的関数、クラス和モジュール
- 定位对应テスト
- 识别新的未テストコード路径

#### 2. 行为覆盖
- 検査各機能は否ありテスト
- 検証边缘用例和エラー路径
- 确保重要統合被覆盖

#### 3. テスト品質
- 优先选择あり意义的断言而非なし抛出検査
- 标记不稳定的パターン
- 検査隔离性和テスト名称清晰度

#### 4. 覆盖差距
按影响评级：
- critical（重要）
- important（重要）
- nice-to-have（锦上添花）

### 输出フォーマット

1. 覆盖总结
2. 重要差距
3. 改善推奨
4. 积极观察
