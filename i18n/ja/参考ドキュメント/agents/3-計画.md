# 計画類 Agent

計画類 Agent は、機能計画、技術アーキテクチャ設計と実施ステップ分解を担当します。

## Agent 一覧

| Agent 名 | 用途 | 使用モデル | コアツール |
|------------|------|----------|----------|
| planner | 複雑な機能実施計画専門家 | opus | Read, Grep, Glob |
| code-architect | 機能アーキテクチャ設計専門家 | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | PR テスト覆寫分析 | sonnet | Read, Grep, Glob, Bash |
| type-design-analyzer | 型設計分析 | sonnet | Read, Grep, Glob |
| comment-analyzer | コードコメント分析 | sonnet | Read, Grep, Glob |

## planner

### 名前と用途
複雑な機能実施とリファクタリングの計画専門家。ユーザー要求 기능実施、アーキテクチャ変更또는复杂的 리팩토링 시에積極的に 사용。

### 能力説明
- 要件分析と詳細な実施計画の作成
- 複雑な機能を管理可能なステップに分解
- 依存関係と潜在リスクを特定
- 最適な実施順序を推奨
- エッジケースとエラーシーンを検討

### 適用シナリオ
- 新機能実施計画
- アーキテクチャ変更計画
- 複雑なリファクタリング計画
- 技術的負債クリーンアップ計画

### 使用ツール一覧
- Read: 既存のコードとドキュメントを読み取り
- Grep: 関連パターンと既存の実装を検索
- Glob: 関連ファイルを検索

### 他の Agent との連携
- 生成された計画を code-architect がさらに詳細化
- 生成された計画を tdd-guide がテスト作成に使用
- 生成された計画を code-reviewer がレビュー

### 計画流れ

1. **要件分析**
   - 機能要求を完全に理解
   - 必要に応じて澄清問題を提起
   - 成功基準を特定

2. **技術調査**
   - 既存のコードパターンを検索
   - 類似の実装を参照
   - 技術的制約を評価

3. **アーキテクチャ設計**
   - 高レベルの設計を作成
   - コンポーネントを定義
   - データフローをマッピング

4. **ステップ分解**
   - 管理可能なタスクに分解
   - 依存関係を特定
   - 優先順位を付ける

5. **リスク評価**
   - 潜在的なリスクを特定
   - 軽減戦略を計画
   - 検証基準を定義

### 出力フォーマット

```markdown
## 機能計画: [機能名]

### 要件
- [要件 1]
- [要件 2]

### アーキテクチャ
- [高レベル設計]
- [コンポーネント図]

### 実施ステップ
1. [ステップ 1]
2. [ステップ 2]
3. [ステップ 3]

### リスクと軽減
- [リスク 1]: [軽減戦略]
- [リスク 2]: [軽減戦略]

### 検証基準
- [基準 1]
- [基準 2]
```

## code-architect

### 名前と用途
機能のためのアーキテクチャ設計専門家。新機能または既存 기능の拡張のアーキテクチャを設計的时候에 적극使用。

### 能力説明
- 機能需要からアーキテクチャを設計
- コンポーネントとモジュールの境界を定義
- データモデルと API 契約を設計
- 非機能要件（パフォーマンス、セキュリティ、拡張性）を考慮
- アーキテクチャの妥当性を検証

### 適用シナリオ
- 新機能のアーキテクチャ設計
- 既存機能の拡張
- アーキテクチャの意思決定
- コードレビューの前のアーキテクチャ検証

### 使用ツール一覧
- Read: 既存のコードとドキュメントを読み取り
- Grep: パターンと依存関係を検索
- Glob: 関連ファイルを検索
- Bash: 検証スクリプトを実行

### 他の Agent との連携
- planner が生成した計画的基础上で設計
- build-error-resolver がビルド問題を処理
- code-reviewer がコード品質をレビュー

### 設計原則

#### 1. 関心の分離
- 異なる关注点を異なるモジュールに配置
- 密的結合を避ける
- 明確なインターフェースを定義

#### 2. 開放/閉鎖原則
- 拡張のために開放
- 修正のために閉鎖
- 戦略パターンの使用

#### 3. 依存関係逆転
- 具象而非抽象に依存
- 依存関係注入を使用
- モック可能を保持

#### 4. 単一責任
- 各モジュールは1つのことを行う
- 凝集度を高める
- 結合度を低下

### アーキテクチャパターン

#### クリーンアーキテクチャ
```
src/
├── domain/          # ビジネスロジック
├── application/     # ユースケース
├── infrastructure/  # 外部 concern
└── presentation/    # UI 層
```

#### マイクロサービス
- サービス境界を明確に定義
- API を介して通信
- 独立してデプロイ可能

#### イベント駆動
- イベントを使用して分离
- 非同期処理
- スケーラビリティ向上

### 出力フォーマット

```markdown
## アーキテクチャ設計: [機能名]

### 概要
[高レベルの説明]

### コンポーネント
| コンポーネント | 責務 | 公開 API |
|------------|------|----------|
| [名前] | [責務] | [API] |

### データフロー
1. [ステップ 1]
2. [ステップ 2]

### 技術的決定
- [決定 1]: [理由]
- [決定 2]: [理由]

### 検証
- [基準 1]
- [基準 2]
```

## pr-test-analyzer

### 名前と用途
Pull Request のテスト覆寫品質と完全性をレビュー。動作覆寫と実際のバグ予防を強調。

### 能力説明
- 変更コードマッピングの識別
- 動作覆寫の検証
- テスト品質の评估
- 覆寫ギャップの発見

### 適用シナリオ
- PR レビュー時
- テストカバレッジ評価時
- テスト十分性の検証時

### 使用ツール一覧
- Read: 変更コードとテストを読み取り
- Grep: テストを検索
- Glob: テストファイルを検索
- Bash: テストコマンドを実行

### 他の Agent との連携
- tdd-guide がテストを改善
- e2e-runner がエンドツーエンドの覆寫を実行
- code-reviewer がコード品質をレビュー

### 分析流れ

#### 1. 変更コードの識別
- 変更の関数、クラスとモジュールをマッピング
- 対応するテストを配置
- 新しい未テストのコードパスを識別

#### 2. 動作覆寫
- 各機能がテストされているかを検査
- エッジケースとエラーパスを検証
- 重要な統合が覆寫されていることを確認

#### 3. テスト品質
- 意味のある断言を選択肢而非なし抛出検査を優先
- 不安定なパターンをマーク
- 隔離性とテスト名前の清晰度を検査

#### 4. 覆寫ギャップ
影響別に評価：
- critical（重要）
- important（重要）
- nice-to-have（追加の利益）

### 出力フォーマット

1. 覆寫サマリー
2. 重要なギャップ
3. 改善推奨
4. 肯定的観察

## type-design-analyzer

### 名前と用途
型設計分析専門家。TypeScript、Java、Go などの静的型付け言語の型設計をレビュー。

### 能力説明
- 型 безопас性の分析
- ジェネリックスの使用評価
- /null 安全性の检查
- API 契約の設計評価

### 適用シナリオ
- 型設計のレビュー
- API 契約の設計
- リファクタリングの計画

### 使用ツール一覧
- Read: コードを読み取り
- Grep: 型パターンを検索
- Glob: ファイルを選択

### 他の Agent との連携
- typescript-reviewer が TypeScript を処理
- java-reviewer が Java を処理
- go-reviewer が Go を処理

## comment-analyzer

### 名前と用途
コードコメント分析専門家。コメントの品質、完全性、正確性を評価。

### 能力説明
- コメントの覆盖率を分析
- 時代遅れのコメントを検出
- JSDoc/Typedoc の品質を評価
-  コメントの明確さを評価

### 適用シナリオ
- コードコメントのレビュー
- ドキュメントの更新計画
- コメントの改善提案

### 使用ツール一覧
- Read: コードを読み取り
- Grep: コメントパターンを検索
- Glob: ファイルを選択

### 他の Agent との連携
- doc-updater がドキュメントを更新
- code-reviewer がコメントの品質をレビュー

---

### コメントの質_checklist

- [ ] 公共 API に JSDoc/Typedoc がある
- [ ] 複雑な論理にインラインコメントがある
- [ ] 時代遅れのコメントがない
- [ ] コメントは「なぜ」を説明し而非「何」を説明
- [ ] TODO/FIXME に issue 番号がある
