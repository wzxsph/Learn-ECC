# テスト類 Agent

テスト類 Agent は、テスト駆動開発、E2E テスト、テストカバレッジ分析と品質保証を担当します。

## Agent 一覧

| Agent 名 | 用途 | 使用モデル | コアツール |
|------------|------|----------|----------|
| tdd-guide | TDD テスト駆動開発専門家 | sonnet | Read, Write, Edit, Bash, Grep |
| e2e-runner | E2E テスト専門家 | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| mle-reviewer | ML エンジニアリングコードレビュー（訓練/推論/監視） | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | PR テスト覆寫分析 | sonnet | Read, Grep, Glob, Bash |

## tdd-guide

### 名前と用途
テスト駆動開発専門家、テスト優先のメソッド論を強制実行。新機能の作成、バグの修正、コードのリファクタリングに使用。80%+ のテストカバレッジを確保。

### 能力説明
- テスト優先のメソッド論を強制実行
- Red-Green-Refactor サイクルの指導
- 80%+ テストカバレッジを確保
- 包括的なテストスイートを作成（ユニット、統合、E2E）
- 実施前にエッジケースをキャプチャ

### 適用シナリオ
- 新機能の作成時
- バグの修正時
- コードのリファクタリング時
- TDD ワークフローが必要な時

### 使用ツール一覧
- Read: 既存のコードを読み取り
- Write: テストを作成
- Edit: テストと実装を変更
- Bash: テストコマンドを実行
- Grep: テストパターンを検索

### 他の Agent との連携
- code-reviewer が TDD サイクル後にコードをレビュー
- build-error-resolver がテストビルドエラーを修正
- e2e-runner が重要なユーザー流れの E2E テストを実行

### TDD ワークフロー

#### 1. まずテストを作成 (RED)
期待される動作の失敗テストを作成。

#### 2. テストを実行 - それが FAIL することを確認
```bash
npm test
```

#### 3. 最小限の実装を作成 (GREEN)
テストが通る足够的コードのみを作成。

#### 4. テストを実行 - それが PASS することを確認

#### 5. リファクタリング (IMPROVE)
重複を排除し、名前を改善し最適化 - テストは緑を保つなければならない。

#### 6. カバレッジを検証
```bash
npm run test:coverage
# 要件: 80%+ ブランチ、関数、行、statement カバレッジ
```

### テストすべきエッジケース

1. **Null/Undefined** 入力
2. **空** 配列/文字列
3. **無効タイプ** の传递
4. **境界値** (最小/最大)
5. **エラーパス** (ネットワーク失敗、データベースエラー)
6. **竞態條件** (並行処理操作)
7. **大数据** (10k+ 項目のパフォーマンス)
8. **特殊文字** (Unicode、エモジー、SQL 文字)

### 反パターン - 避けるべき

- 実装詳細（内部状態）而非行為をテスト
- テスト間の相互依存（共有状態）
- 断言太少（通るが何も検証しない）
- 外部依存が mock されていない (Supabase、Redis、OpenAI 等)

### 品質検査チェックリスト

- [ ] すべての公共関数にユニットテストがある
- [ ] すべての API エンドポイントに統合テストがある
- [ ] 重要なユーザー流れに E2E テストがある
- [ ] エッジケースが覆寫されている (null、empty、invalid)
- [ ] エラーパスがテストされている（ happy path のみではない）
- [ ] 外部依存が mock されている
- [ ] テストが相互に独立している（共有状態なし）
- [ ] 断言が具体的であり意味がある
- [ ] カバレッジ 80%+

### v1.8 Eval-Driven TDD 付録

eval-driven 開発を TDD プロセスに統合：

1. 実施前に能力 + 回帰 evals を定義
2. 基線を実行し失敗特徴をキャプチャ
3. 最小限の合格変更を実装
4. テストと evals を再実行; pass@1 と pass@3 をレポート
5. 重要なパスはマージ前に pass^3 安定性に達するはず

## e2e-runner

### 名前と用途
E2E テスト専門家、Vercel Agent Browser（優先）と Playwright フォールバックを使用。新機能の生成、維持と実行 E2E テストに使用。

### 能力説明
- テスト旅程の作成 - ユーザー流れのためのテストを作成
- テストの維持 - テストを UI 変化と同期させる
- 不安定テストの管理 - 不安定テストを特定し隔离
- テストレポート - HTML レポートと JUnit XML を生成
- CI/CD 統合 - テストがパイプラインで信頼的に実行されることを確保
- スクリーンショット/動画/trace の管理

### 適用シナリオ
- 重要なユーザー旅程の検証が必要な時
- エンドツーエンドの検証が必要な時
- CI/CD パイプラインで E2E テストを実行する時
- 統合問題を検出した時

### 使用ツール一覧
- Read: ページ内容を読み取り
- Write: テストを作成
- Edit: テストを変更
- Bash: Playwright/Agent Browser コマンドを実行
- Grep: テストを検索
- Glob: テストファイルを検索

### 他の Agent との連携
- tdd-guide がユニット/統合テストを作成
- code-reviewer がテスト品質をレビュー
- mle-reviewer が ML 関連テストをレビュー

### 主要ツール: Agent Browser

**Agent Browser を原始 Playwright より優先使用** - セマンティックセレクター、AI 最適化、自动待機、Playwright に基づく。

```bash
# 設定
npm install -g agent-browser && agent-browser install

# コアワークフロー
agent-browser open https://example.com
agent-browser snapshot -i          # refs を持つ要素を取得 [ref=e1]
agent-browser click @e1             # ref でクリック
agent-browser fill @e2 "text"       # ref で入力を埋める
agent-browser wait visible @e5       # 要素が見えるのを待つ
agent-browser screenshot result.png
```

### フォールバック: Playwright

Agent Browser が利用できない時、Playwright を直接使用。

```bash
npx playwright test                        # すべての E2E テストを実行
npx playwright test tests/auth.spec.ts     # 特定のファイルを実行
npx playwright test --headed               # 可见ブラウザ
npx playwright test --debug                # 検査器を使用デバッグ
npx playwright test --trace on             # trace で実行
npx playwright show-report                 # HTML レポートを表示
```

### ワークフロー

#### 1. 計画
- 重要なユーザー旅程を識別（認証、コア機能、支払、CRUD）
- シーンを定義: happy path、エッジケース、エラーケース
- リスクでソート: HIGH（金融、認証）、MEDIUM（検索、ナビゲーション）、LOW（UI 最適化）

#### 2. 作成
- ページオブジェクトモデル (POM) パターンを使用
- `data-testid` セレクターを優先
- 重要なステップに断言を追加
- 重要な瞬間にスクリーンショットをキャプチャ
- 適切な待機を使用（決して `waitForTimeout` ではない）

#### 3. 実行
- 本地で 3-5 回実行して不安定性を検査
- `test.fixme()` または `test.skip()` を使用して不安定テストを隔离
- artifact を CI にアップロード

### 重要な原則

- **セマンティックセレクターを使用**: `[data-testid="..."]` > CSS セレクター > XPath
- **時間而非条件以待**: `waitForResponse()` > `waitForTimeout()`
- **自動待機の内置**: `page.locator().click()` は自動待機; 原始 `page.click()` は待機しない
- **テストを隔離**: 各テストは独立すべき; 共有状態なし
- **快速失敗**: 各重要なステップで `expect()` 断言を使用
- **再試行時 trace**: 失敗をデバッグするために `trace: 'on-first-retry'` を設定

### 不安定テストの処理

```typescript
// 隔離
test('flaky: market search', async ({ page }) => {
  test.fixme(true, 'Flaky - Issue #123')
})

// 不安定性を識別
// npx playwright test --repeat-each=10
```

よくある原因: 竞態條件（自動待機セレクターを使用）、ネットワークのタイミング（応答を待つ）、アニメーションのタイミング（`networkidle` を待つ）。

### 成功指標

- すべての重要な旅程が通る (100%)
- 全体の通過率 > 95%
- 不安定率 < 5%
- テスト継続時間 < 10 分
- Artifact がアップロードされアクセス可能

## mle-reviewer

### 名前と用途
本番機械学習エンジニアリングレビュー専門家、データ契約、特徴パイプライン、訓練再現性、オフライン/オンライン評価、モデルサービス、監視とロールバックに焦点を当てる。

### 能力説明
- 問題定義と意思決定の品質をレビュー
- 指標、しきい値とエラーを分析してレビュー
- データ契約と漏れの検査
- 訓練再現性を検証
- 評価とプロモートプロセスをレビュー
- サービスとデプロイのセキュリティをレビュー
- 監視とイベント対応計画をレビュー

### 適用シナリオ
- ML、MLOps、モデル訓練コードが変更された時
- 推論、特徴存储、評価コードが変更された時
- モデルプロモートの意思決定時

### 使用ツール一覧
- Read: ML コードと設定を読み取り
- Grep: パターンを検索
- Glob: ファイルを検索
- Bash: pytest, ruff, mypy を実行

### 他の Agent との連携
- python-reviewer が Python スタイル、タイプ、エラー処理を処理
- pytorch-build-resolver が tensor/CUDA/訓練失敗を処理
- database-reviewer が特徴表、タグ存储を処理
- security-reviewer が secrets、PII、pickle セキュリティを処理
- performance-optimizer が遅延、メモリ、GPU 使用率を処理
- build-error-resolver が CI、依存関係、原始拡張失敗を処理
- pr-test-analyzer がテスト覆寫を分析
- silent-failure-hunter がパイプラインのsilent失敗を発見
- e2e-runner が製品流れの E2E テストを実行
- a11y-architect が予測の可アクセス性をレビュー
- doc-updater がドキュメントを更新

### 重要なレビュー領域

#### 問題定義と意思決定の品質
- 変更是否是ユーザーまたはシステムの意思決定から始まり，而非モデルアーキテクチャの好み
- 利害関係者と失敗コスト是否是明確
- 指標選択是否是エラーバジェットに従っている

#### 指標、しきい値とエラー分析
- 基線と現在の本番動作是否是比較されている
- しきい値と設定是否是製品意思決定として扱われている
- 偽陽性と偽陰是否是直接検査されている

#### データ契約と漏れ
- エンティティ粒度、主キー、タイムスタンプ是否是明確
- 分割是否是時間、ユーザー/エンティティのグループ化を遵守
- 特徴の接続是否是時間点で正しい
- 機密属性是否是除外または正当な処理がされている

#### 訓練再現性
- 訓練是否是コード、設定、データセットバージョン、シードから実行可能
- ハイパーパラメータ、前処理、依存関係バージョンは是否是記録されている
- ランダム性と GPU 非決定性は是否是意図的に処理されている

#### 評価とプロモート
- 指標是否是基線と現在の本番モデルとを比較
- プロモートゲート是否是選択前に宣言されている
- スライス指標是否是重要なキューを覆寫

#### サービスとデプロイ
- 訓練とサービスの変換是否是共有または同等のテスト
- 入力パターンは是否是古い、欠落、無効な特徴を拒否
- 推論パス是否是タイムアウト、資源制限、フォールバック論理を含む

#### 監視とイベント対応
- 監視是否是サービス健康、特徴ドリフト、予測ドリフトを覆寫
- ログ是否是十分な識別子を含む
- ロールバック是否是前の artifact、設定、データの依存関係に名前を付けている

### よくある阻塞器

- 時間関連またはユーザー関連のデータでのランダム train/test 分割
- 予測時に利用できないフィールドを使用した特徴生成
- オフライン指標の改善而重要なスライスの回帰
- 訓練の前処理がサービスコードに手動でコピーされた
- モデルバージョンが予測ログにない

### 承認基準

- **APPROVE**: 重要な/高い MLE リスクがなく、関連するテストまたは評価ゲートが通過
- **APPROVE WITH WARNINGS**: 中程度の問題のみがあり、明確な後続行動がある
- **BLOCK**: 可能性のある漏れ、再現不可能なプロモート、安全でないサービス動作、本番デプロイのロールバックの欠落、機密データの露出、または重要な評価のギャップがある場合
