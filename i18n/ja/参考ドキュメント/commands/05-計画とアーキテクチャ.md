# 計画とアーキテクチャコマンド

## 概要

計画とアーキテクチャコマンド使用產品計画、機能設計和システムアーキテクチャ决策。這些コマンド確保在写コード之前あり清晰的計画，减少返工和提高コード品質。

## コマンド一覧

### /plan

**用途**: 通用實施計画 - 复述需求、評估風險、作成分ステップ實施計画

**記述**: 在触碰任意コード之前，等待ユーザー確認后作成全面的實施計画。受け入れ自由形式需求或 PRD markdown ファイル。

**输入パターン**:

| 输入 | パターン | 行為 |
|---|---|---|
| `path/to/name.prd.md` | PRD artifact パターン | 読み取り PRD，选择下1待交付里程碑，生成 `.claude/plans/{name}.plan.md` |
| その他 markdown パス | 引用パターン | 読み取りファイル作為上下文，产生内联計画 |
| 自由形式文本 | 對话パターン | 产生内联計画 |
| 空输入 | 澄清パターン | 询问要計画什么 |

**重要參數**:
- `$ARGUMENTS`: `[feature description | path/to/*.prd.md]` - 機能記述或 PRD ファイルパス

**ワークフロー**:
1. **复述需求** - 澄清必要ビルド什么
2. **识别風險** - 列出潜在問題和阻碍
3. **作成ステップ計画** - 分解為フェーズ性實施
4. **等待確認** - must收到ユーザー承認后才能継続

**输出フォーマット** (PRD artifact パターン):
```markdown
# Plan: {Feature Name}

**Source PRD**: {path}
**Selected Milestone**: {milestone or phase name}
**Complexity**: {Small | Medium | Large}

## Summary
{2-3 sentences}

## Patterns to Mirror
| Category | Source | Pattern |
|---|---|---|
| Naming | `path:line` | {short description} |
| Errors | `path:line` | {short description} |
| Tests | `path:line` | {short description} |

## Files to Change
| File | Action | Why |
|---|---|---|
| `path` | CREATE / UPDATE / DELETE | {reason} |

## Tasks
### Task 1: {name}
- **Action**: {what to do}
- **Mirror**: {pattern to follow}
- **Validate**: {command that proves correctness}

## Validation
```bash
{project-specific validation commands}
```

## Risks
| Risk | Likelihood | Mitigation |
|---|---|---|
```

**ベストプラクティス**:
- 使用前先研究コードベース中的既存のパターン
- 在 PRD artifact パターン中作成 `.claude/plans/` ディレクトリ（如不存在）
- 如果 PRD 含む含 `Delivery Milestones` 表格，只更新所选行狀態

**常见陷阱**:
- 空输入时不询问澄清就尝试計画
- 不等待ユーザー確認就開始写コード
- 跳过 Pattern Grounding ステップ

**與その他のコマンド統合**:
- 計画后使用 `tdd-workflow` skill 进行テスト驱动開発
- 使用 `/build-fix` 修正ビルドエラー
- 使用 `/code-review` レビュー完了実装
- 使用 `/pr` 或 `/prp-pr` 開く Pull Request

### /plan-prd

**用途**: 交互式 PRD 生成器

**記述**: 生成以問題為中心的產品需求ドキュメント，含む含問題定義、ユーザー画像、成功指標和 MVP 范围。

**使用场景**: 
- 確定要ビルド的機能
- 必要明确產品需求
- 從零開始新機能計画

**ワークフロー**:
1. FRAME - 了解ユーザー和問題
2. GROUND - 收集证据
3. DECIDE - 確定范围和假设
4. GENERATE - 生成 PRD ファイル

---

### /prp-plan

**用途**: 全面的機能計画

**記述**: 完整的プロジェクト計画，涵盖コードベース分析、風險評估和分ステップ實施計画。

**使用场景**:
- 複雜機能的詳細計画
- 必要コードベース上下文分析
- 多フェーズ實施プロジェクト

---

### /prp-prd

**用途**: PRP ワークフロー PRD 生成器

**記述**: 使用 PRP（Plan-Record-Produce）ワークフロー生成產品需求ドキュメント。

---

### /prp-implement

**用途**: 実行 PRP 計画+検証ループ

**記述**: 按照 PRP ワークフロー実行計画，含む含継続的検証和検査。

---

### /prp-pr

**用途**: 從 PRP ワークフロー作成 PR

**記述**: 從 PRP ワークフロー的結果作成 GitHub Pull Request。

---

### /prp-commit

**用途**: PRP 検証コミット

**記述**: 在 PRP ワークフロー中実行検証並コミットコード。

---

### /multi-plan

**用途**: 多モデル协作計画

**記述**: 结合 Codex 和 Gemini 的双モデル分析，生成综合實施計画。

**使用场景**:
- 必要多角度分析
- 複雜跨領域プロジェクト
- 必要平衡フロントエンド和バックエンド考量
