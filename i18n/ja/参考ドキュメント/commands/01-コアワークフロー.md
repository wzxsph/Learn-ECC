# コアワークフローコマンド

本ドキュメント介紹 ECC 中使用コア開発ワークフロー的コマンド。

---

## /plan

**用途説明**: 重述需求、評估風險，並作成分步實施計画。在触碰任意コード之前**等待ユーザー確認**。

**使用メソッド**:
```
/plan [機能記述 | PRDファイルパス]
```

**使用场景**:
- 開始新機能開発
- 进行重大アーキテクチャ变更
- 複雜的リファクタリング作業
- 需求不明确或模糊时

**詳細説明**:
该コマンド会：
1. **重述需求** - 清晰表达必要ビルド的內容
2. **识别風險** - 揭示潜在問題和阻碍
3. **作成分步計画** - 將實施分解為フェーズ
4. **等待確認** - must获得ユーザー明确承認后才能継続

サポート多种输入パターン：
| 输入 | パターン | 行為 |
|------|------|------|
| `path/to/name.prd.md` | PRD パターン | 読み取り PRD，选择下1保留中的交付里程碑 |
| その他 markdown パス | 引用パターン | 読み取りファイル作為上下文並生成内联計画 |
| 自由フォーマット文本 | 對话パターン | 生成内联計画 |
| 空输入 | 澄清パターン | 询问すべき計画什么 |

## /code-review

**用途説明**: 本地未コミット变更或 GitHub PR 的全面コードレビュー。

**使用メソッド**:
```
/code-review                 # 本地レビューパターン
/code-review [PR号 | PR链接]  # PR レビューパターン
```

**使用场景**:
- コミットコード之前进行全面レビュー
- PR レビュー以发现潜在問題
- 学習コードベース中的ベストプラクティス

**本地レビューパターン**:
1. 收集变更ファイル（`git diff --name-only HEAD`）
2. 検査セキュリティ問題（平文凭证、SQL注入、XSS等）
3. 検査コード品質問題（関数>50行、ファイル>800行等）
4. 生成带重大级别标注的レポート

**PR レビューパターン**:
1. 取得 PR 元データ和差異
2. 検査相關プロジェクト仕様和計画ドキュメント
3. 読み取り完整变更ファイル
4. アプリケーション 7 个レビュークラス别（正确性、型セキュリティ、パターン合规、セキュリティ、パフォーマンス、完整性、保守性）
5. 実行検証コマンド（タイプ検査、lint、テスト、ビルド）
6. 發布レビュー意见到 GitHub

## /build-fix

**用途説明**: 检测プロジェクトビルドシステム並逐步修正ビルド/タイプエラー。

**使用メソッド**:
```
/build-fix
```

**使用场景**:
- ビルド失敗时
- TypeScript/JavaScript タイプエラー
- コンパイルエラー
- 依賴問題

**ワークフロー**:
1. **检测ビルドシステム** - 根據ファイルタイプ选择合适的ビルドコマンド
2. **解析エラー** - 按ファイル和依賴順序分组
3. **逐步修正** - 一次修正1エラー
4. **検証** - 每次修正后重新実行ビルド

サポート的ビルドシステム：

| 指示器 | ビルドコマンド |
|--------|----------|
| `package.json` + `build` スクリプト | `npm run build` |
| `tsconfig.json` (TypeScript) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall` 或 `mypy` |

## /verify

**用途説明**: 検証コード变更は否按预期作業。使用検証 PR、確認修正或テスト变更。

**使用メソッド**:
```
/verify [quick]           # 快速検証（推奨）
/verify full             # 完整検証
```

**使用场景**:
- 検証 PR 更改
- 確認修正あり效
- 手动テスト变更
- 本地検証后再プッシュ

**検証ステップ**:
1. 実行相關テスト
2. 検査型セキュリティ性
3. 検証ビルド成功
4. 検査 linting

---

## /quality-gate

**用途説明**: 為ファイル或プロジェクト范围実行 ECC 品質管道並レポート修正ステップ。

**使用メソッド**:
```
/quality-gate [path|.] [--fix] [--strict]
```

**使用场景**:
- 按需実行品質検査
- 在 CI 管道中統合
- 確保コード符合プロジェクト標準

**管道ステップ**:
1. 检测目標言語/ツール
2. 実行フォーマット化程序検査
3. 実行 lint/タイプ検査
4. 生成简洁的修正一覧

---

## /tdd

**用途説明**: テスト驱动開発ワークフロー。遵循红-绿-リファクタリングループ。

**使用メソッド**:
```
/tdd                      # 起動 TDD ワークフロー
```

**使用场景**:
- 実装新機能
- 修正 bug（先写失敗的テスト）
- 追加テスト覆寫
- 学習 TDD メソッド论

**TDD ループ**:
```
RED    → 写1失敗的テスト
GREEN  → 编写最少的コード使テスト通過
REFACTOR → 改善コード，テスト保持绿色
REPEAT → 下1テスト用例
```

**ベストプラクティス**:
- **先写テスト** - 在任意実装之前
- **每次修改后実行テスト** - 検証进度
- **テスト行為，而非実装** - 关注インターフェース而非细节
- **含む含边界情况** - 空值、null、最大值等

## コマンド統合關係

```
需求不明 → /plan-prd (作成 PRD)
     ↓
需求明确 → /plan (作成實施計画)
     ↓
/tdd (或 /feature-dev) 実装
     ↓
/build-fix 修正ビルドエラー
     ↓
/code-review コードレビュー
     ↓
/quality-gate 品質门禁
     ↓
/verify 検証
     ↓
/pr 作成 PR
```
