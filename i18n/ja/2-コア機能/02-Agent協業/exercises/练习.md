# 演習：Agent コラボレーション

## 演習 1: コードレビュー類 Agent の使用

### 目標
コードレビュー Agent を呼び出し、そのフィードバックメカニズムを理解する。

### シーン
以下の Python コードを作成したと假设（実際の実行は不要）：

```python
def get_user_data(user_id, include_password=False):
    user = db.query("SELECT * FROM users WHERE id = " + user_id)
    if include_password:
        return user
    return {"name": user.name, "email": user.email}
```

### 手順

1. **このコードを分析**
   潜在的な問題を思考：
   - SQL 注入脆弱性
   - パスワード泄露リスク
   - エラー処理の欠落

2. **適切な Agent を選択**
   このコードには以下を使用すべき：
   - `security-reviewer` - セキュリティ脆弱性をチェック
   - `python-reviewer` - Python ベストプラクティスをチェック

3. **Agent フィードバックをシミュレート**
   Agent が提供すると思うフィードバックを記録：

   ```markdown
   ## コードレビューフィードバック

   ### セキュリティ問題 (CRITICAL)
   - SQL 注入：user_id を SQL 文に直接連結
   - パスワード泄露：include_password オプションがパスワードを露出可能性

   ### 品質問題 (HIGH)
   - エラー処理が欠落
   - 入力検証が欠落

   ### 改善提案
   - パラメータ化クエリを使用
   - パスワードフィールドを返さない
   - 例外処理を追加
   ```

4. **修復計画を制定**
   `/plan` の思路を使用して、修復計画を制定：

   ```markdown
   # 修復計画

   1. SQL 注入を修復：パラメータ化クエリを使用
   2. パスワード返送を移除：決してパスワードを返さない
   3. 入力検証を追加：user_id 形式を検証
   4. エラー処理を追加：ユーザーが存在しない情况的を処理
   ```

### 完了基準
- [ ] コードのセキュリティ問題を識別できる
- [ ] レビュー Agent のフィードバック構造を理解
- [ ] 修復計画を制定できる

## 演習 2: ビルドリペア類 Agent の使用

### 目標
ビルド Agent を使用してビルドエラーを診断・修復する方法を学習。

### シーン
Go プロジェクトに以下のビルドエラーがあると假设：

```
# github.com/myproject/pkg
./main.go:15: undefined: " fmt"
./main.go:23: undefined: calculateTotal
./main.go:31: too many arguments to function call
```

### 手順

1. **エラータイプを分析**
   エラーを分類：

   | エラー | タイプ | 重大度 |
   |------|------|----------|
   | undefined: " fmt" | インポートエラー | HIGH |
   | undefined: calculateTotal | 関数未定義 | HIGH |
   | too many arguments | パラメータエラー | MEDIUM |

2. **修復フローを設計**
   Agent コラボレーションを使用：

   ```markdown
   # ビルド修復フロー

   1. /go-build - ビルドを実行して完全エラーを取得
   2. /build-fix - ビルド修復 Agent を呼び出す
   3. エラープリオリティを分析：高Priorityを先に修復
   4. 逐次修復して検証
   5. /verify - ビルド成功を確認
   ```

3. **各エラーの修復方案を設計**

   **エラー 1: undefined: " fmt"**
   ```go
   // 問題：余分な引用符
   import "fmt"  // 正しい書き方

   // 修復：引用符を移除
   ```

   **エラー 2: undefined: calculateTotal**
   ```go
   // 問題：関数が未定義または名前エラー
   // 修復：関数定義を確認し、追加または名前変更が必要
   ```

   **エラー 3: too many arguments**
   ```go
   // 問題：関数呼び出しパラメータ数が一致しない
   // 修復：関数署名を確認し、呼び出しパラメータを修正
   ```

4. **修復経験を記録**
   エラーパターンサマリーを作成：

   ```markdown
   # Go ビルドエラーパターン

   ## 一般的なエラー
   1. undefined: インポート/関数問題
   2. too many/few arguments: パラメータ不一致
   3. undefined: " xxx": インポート構文エラー

   ## 修復戦略
   - エラー数を確認し、その後エラー位置を確認
   - 高Priorityエラーは低Priorityエラー连锁の原因となる可能性
   - 修復後に再ビルドして検証
   ```

### 完了基準
- [ ] ビルドエラーのタイプとPriorityを分析できる
- [ ] 完全な修復フローを設計
- [ ] 一般的なエラーパターンを記録

## 演習 3: Agent 組合ワークフロー

### 目標
複雑なタスクを設計し、複数の Agent のコラボレーションで完了。

### シーン
新しいユーザー認証モジュールを開発、以下の機能を含む：
- ユーザー登録（パスワード強度検証を含む）
- ユーザーログイン（ブルートフォース攻撃防止を含む）
- JWT token 生成と検証

### 手順

1. **タスク分解**
   タスクをサブタスクに分解：

   ```markdown
   # 認証モジュール開発タスク

   ## サブタスク
   1. データベース設計 (Schema)
   2. ユーザー登録 API
   3. ユーザーログイン API
   4. JWT Token 処理
   5. セキュリティメカニズム（パスワード強度、暴力破解防止）
   ```

2. **Agent 分配**
   各サブタスクに Agent を分配：

   | サブタスク | Agent | 理由 |
   |--------|-------|------|
   | データベース設計 | architect | アーキテクチャ視点が必要 |
   | 登録 API | tdd-guide + security-reviewer | TDD + セキュリティ |
   | ログイン API | tdd-guide + security-reviewer | TDD + セキュリティ |
   | JWT 処理 | security-reviewer | セキュリティ敏感 |
   | セキュリティメカニズム | security-reviewer (優先) | セキュリティ優先 |

3. **ワークフローを設計**

   ```
   [開始]
      ↓
   /plan (全体計画)
      ↓
   ┌─────────────────────────────────────┐
   │  並行実行:                            │
   │  - architect: データベース設計         │
   │  - tdd-guide: テスト戦略を設計        │
   └─────────────────────────────────────┘
      ↓
   [実施段階]
      ↓
   ┌─────────────────────────────────────┐
   │  直列 + 並行混合:                      │
   │  1. tdd-guide: 登録テストを作成       │
   │  2. 登録機能を実装                     │
   │  3. security-reviewer: セキュリティレビュー│
   │  ──────────────────────────────────   │
   │  1. tdd-guide: ログインテストを作成    │
   │  2. ログイン機能を実装                   │
   │  3. security-reviewer: セキュリティレビュー│
   └─────────────────────────────────────┘
      ↓
   [統合段階]
      ↓
   ┌─────────────────────────────────────┐
   │  - architect: JWT アーキテクチャ設計    │
   │  - security-reviewer: 全体セキュリティレビュー│
   │  - code-reviewer: コード品質レビュー    │
   │  - /verify: 全体検証                  │
   └─────────────────────────────────────┘
      ↓
   [完了]
   ```

4. **ワークフロードキュメントを作成**

   ```markdown
   # ユーザー認証モジュール開発ワークフロー

   ## 段階 1: 計画
   - /plan を使用して全体計画を制定
   - architect を使用してデータベースと JWT アーキテクチャを設計

   ## 段階 2: TDD 開発
   - tdd-guide を使用してテスト作成をガイド
   - まずテストを書き、それから機能を実装
   - 各サブ機能を单独で検証

   ## 段階 3: セキュリティレビュー
   - security-reviewer を使用して各 API をレビュー
   - 特に注目：パスワード処理、Token セキュリティ、注入防護

   ## 段階 4: 統合検証
   - code-reviewer を使用して全体をレビュー
   - /verify を使用して機能完全性を検証

   ## Agent コラボレーションまとめ
   - architect: アーキテクチャ設計 (1回)
   - tdd-guide: テストガイド (2回)
   - security-reviewer: セキュリティレビュー (3回)
   - code-reviewer: 品質レビュー (1回)
   ```

### 完了基準
- [ ] タスク分解が合理的で、5+ のサブタスクを含む
- [ ] Agent 分配策略が明確
- [ ] ワークフロー設計が清晰
- [ ] 文書化されたワークフローが再利用しやすい
