# 05 - Rules 作成

カスタマイズされた開発規則の作成方法を学習します。

## 概要

ECC [Rules](../../参考ドキュメント/rules/代码风格.md) は強制的な開発規則セットで、コードスタイル、テスト、セキュリティ、Git ワークフロー、エージェント雰囲、 performance最適化などをカバーします。これらの規則はコード品質と一貫性を確保し、自動化されたチェックとレビュー流程を通じて実施されます。

---

## 1. Rules 形式

### 1.1 基本形式

Rules は Markdown 形式を使用し、YAML frontmatter を含みます：

```yaml
---
name: rule-name
description: 規則の説明
category: 分類
---
```

### 1.2 規則レイヤー

| レイヤー | 説明 | 優先度 |
|------|------|--------|
| **全般規則** | `~/.claude/rules/` | 最高 |
| **プロジェクト規則** | プロジェクト `.claude/rules/` | 中 |
| **ローカル規則** | 現在のセッション | 最低 |

### 1.3 優先度

全般規則 > プロジェクト規則 > ローカル規則

## 2. 規則タイプ详解

### 2.1 コードスタイル規則

コード編成のコア原則とベストプラクティスを定義。

#### コア要件

**不変性（重要原則）**

```typescript
// エラー例 - 元のオブジェクトを変更
function modify(user, name) {
  user.name = name;  // 元のオブジェクトを変更
  return user;
}

// 正しい例 - 新規オブジェクトを返す
function update(user, name) {
  return { ...user, name };  // 新規コピーを作成
}
```

**コア設計原則**

| 原則 | 説明 | 実践 |
|------|------|------|
| **KISS** | Keep It Simple | 最もシンプルな実行可能な方案を選択 |
| **DRY** | Don't Repeat Yourself | コピー＆ペーストによる実装ドリフトを回避 |
| **YAGNI** | You Aren't Gonna Need It | 実際に必要時にリファクタリング |

**命名規則**

| タイプ | 規則 | 例 |
|------|------|------|
| 変数と関数 | camelCase | `calculateTotal` |
| 布尔値 | is/has/should/can 接頭辞 | `isActive` |
| インターフェース、タイプ、コンポーネント | PascalCase | `UserProfile` |
| 定数 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |

### 2.2 テスト規則

コード品質の強制的なテスト標準を確保。

#### 最低テストカバレッジ：80%

| テストタイプ | 説明 |
|----------|------|
| ユニットテスト | 独立関数、ツールクラス、コンポーネント |
| 統合テスト | API エンドポイント、データベース操作 |
| E2E テスト | 重要なユーザーフロー |

#### TDD ワークフロー

```
1. テストを作成 (RED)   - 失敗するテストを先に記述
2. テストを実行          - テストが失敗することを検証
3. 最小実装を作成 (GREEN) - テストに合格する最小コードを記述
4. テストを実行          - テストが合格することを検証
5. リファクタリング (IMPROVE)    - コード構造を改善
6. カバレッジを検証        - 80%+ に到達することを確保
```

### 2.3 セキュリティ規則

強制的なセキュリティチェックリスト。

#### 提出前に必ずチェック

- [ ] ハードコードされた凭证なし（API keys、パスワード、tokens）
- [ ] すべてのユーザー入力検証
- [ ] SQL 注入防護（パラメータ化クエリ）
- [ ] XSS 防護（出力エスケープ）
- [ ] CSRF 保護有効
- [ ] 認証/授權検証
- [ ] レート制限
- [ ] エラーメッセージが機密データを漏洩しない

#### 鍵管理

- **決して** ソースコードに鍵をハードコードしない
- **常に** 環境変数または鍵マネージャーを使用
- 起動時に必要な鍵を検証
- 露出した可能性がある鍵を交換

### 2.4 Git ワークフロー規則

Git 提出規範と PR ワークフロー。

#### 提出情報形式

```
<type>: <description>

<optional body>
```

**Type タイプ**：
- `feat`: 新機能
- `fix`: エラー修復
- `refactor`: リファクタリング
- `docs`: ドキュメント
- `test`: テスト
- `chore`: メンテナンス
- `perf`: 性能
- `ci`: CI/CD

#### PR ワークフロー

1. 完全な提交歴史を分析
2. `git diff [base-branch]...HEAD` を使用してすべての変更を表示
3. 包括的な PR 要約を起草
4. テスト計画と TODOs を含める
5. 新規ブランチの場合、`-u` フラグでプッシュ

### 2.5 エージェント雰囲規則

多 Agent 協力モード。

#### 即時使用原則

| シーン | Agent |
|------|-------|
| 複雑な機能要求 | **planner** |
| コード作成/変更直後 | **code-reviewer** |
| Bug 修復または新機能 | **tdd-guide** |
| アーキテクチャ決定 | **architect** |

#### 並行タスク実行

```markdown
# 良い：並行実行
3つの Agent を並行起動：
1. Agent 1: 認証モジュールセキュリティ分析
2. Agent 2: キャッシュシステム性能レビュー
3. Agent 3: ユーティリティタイプチェック

# 悪い：不必要な順序実行
まず agent 1、その次 agent 2、その次 agent 3
```

### 2.6 性能最適化規則

モデル選択とコンテキスト管理。

#### モデル選択戦略

| モデル | 用途 | コスト |
|------|------|------|
| **Haiku 4.5** | 軽量 Agent、频繁呼び出し、配对プログラミング | 低 |
| **Sonnet 4.6** | 主な開発作業、多 Agent 雰囲 | 中 |
| **Opus 4.5** | 複雑なアーキテクチャ決定、深度推論 | 高 |

#### コンテキストウィンドウ管理

最後の 20% コンテキストウィンドウでの以下を回避：
- 大規模リファクタリング
- 複数ファイルにまたがる特性実装
- 複雑な相互作用デバッグ

## 3. 規則作成

### 3.1 トリガー条件

```yaml
---
name: no-console-log
description: console.log を含むコードの提出を阻止
trigger:
  - pre-commit
  - pre-push
---
```

### 3.2 チェックロジック

```markdown
## チェックロジック

1. 変更されたファイルをスキャン
2. `console.log`、`console.error` などを検索
3. 発見した場合、提出を阻止して警告を表示
```

### 3.3 修復提案

```markdown
## 修復提案

- 構造化ログライブラリを使用（winston、pino など）
- ログレベルを設定（debug、info、warn、error）
- 本番環境ログレベルが適切であることを確保
```

## 4. 違反処理

### 4.1 重大度分级

| レベル | 意味 | 操作 |
|------|------|------|
| CRITICAL | セキュリティ脆弱性またはデータ損失リスク | **阻止** - マージ前に必ず修復 |
| HIGH | Bug または重大な品質問題 | **警告** - マージ前に修復すべき |
| MEDIUM | メンテナンス性问题 | **提示** - 修復を検討 |
| LOW | 样式または小さな提案 | **備考** - オプション |

### 4.2 修復フロー

```
1. 違反原因を分析
2. 修復方案を確定
3. 修復を実行
4. 修復を検証
5. 再提出レビュー
```

## 5. コード品質チェックリスト

作業を完了する前に必ずチェック：

- [ ] コードが読みやすく命名が的良好
- [ ] 関数が短い（<50 行）
- [ ] ファイルが专注（<800 行）
- [ ] 深层嵌套がない（>4 层）
- [ ] 適切なエラー処理
- [ ] ハードコード値がない（定数または設定を使用）
- [ ] 不変パターンを使用（mutation なし）
- [ ] テストカバレッジ ≥80%
- [ ] すべてのセキュリティチェックに合格

## 6. プロジェクト特定規則の作成

### 6.1 プロジェクト規則の位置

```
プロジェクト/
└── .claude/
    └── rules/
        ├── README.md
        └── カスタム規則ファイル
```

### 6.2 全般規則の継承

プロジェクト規則は全般規則を継承し，在此基础上扩展：

```markdown
# プロジェクト特定規則

全般規則から継承し、以下を追加：

## 追加要件

- [ ] プロジェクト固有のコーディング規範
- [ ] 特定のテスト要件
- [ ] 特定のドキュメント要件
```
