# Упражнение：Сотрудничество Agent

## Упражнение 1: 使用Код-ревью Agent

### Цель
学习调用Код-ревью Agent，并理解其反馈机制。

### Сценарий
假设你编写了以下 Python 代码（不Требуется实际执行，只是在脑海中）：

```python
def get_user_data(user_id, include_password=False):
    user = db.query("SELECT * FROM users WHERE id = " + user_id)
    if include_password:
        return user
    return {"name": user.name, "email": user.email}
```

### Шаги

1. **Проанализировать этот код**
   思考Возможные проблемы：
   - SQL инъекция漏洞
   - 密码泄露风险
   - Обработка ошибок缺失

2. **Выбрать подходящего Agent**
   对于这段代码，Следует使用：
   - `security-reviewer` - 检查Уязвимости безопасности
   - `python-reviewer` - 检查 Python Лучшие практики

3. **Симулировать отзыв Agent**
   记录你认为 Agent 会给出的反馈：

   ```markdown
   ## Код-ревью反馈

   ### Безопасность问题 (CRITICAL)
   - SQL инъекция：直接拼接 user_id 到 SQL 语句
   - Утечка пароля: опция include_password может раскрыть пароль

   ### Проблема качества (ВЫСОКАЯ)
   - 缺少Обработка ошибок
   - 缺少输入Проверка

   ### Рекомендации по улучшению
   - Использовать параметризованные запросы
   - 不要Назад密码字段
   - Добавить обработку исключений
   ```

4. **Составить план исправлений**
   使用 `/plan` 的思路，Составить план исправлений：

   ```markdown
   # План исправлений

   1. 修复 SQL инъекция：Использовать параметризованные запросы
   2. 移除密码Назад：永远不要Назад密码
   3. 添加输入Проверка：Проверка user_id 格式
   4. 添加Обработка ошибок：Обработать случай отсутствия пользователя
   ```

### Критерии приёмки
- [ ] 能识别代码中的Безопасность问题
- [ ] Понимание структуры отзывов Agent проверки
- [ ] 能Составить план исправлений

## Упражнение 2: 使用Исправление сборки Agent

### Цель
Учиться как使用构建 Agent 诊断和Исправление ошибок сборки。

### Сценарий
假设 Go 项目有以下构建错误：

```
# github.com/myproject/pkg
./main.go:15: не определён: " fmt"
./main.go:23: не определён: calculateTotal
./main.go:31: too many arguments to function call
```

### Шаги

1. **分析错误Тип**
   将错误分类：

   | 错误 | Тип | 严重程度 |
   |------|------|----------|
   | не определён: " fmt" | Ошибка импорта | HIGH |
   | не определён: calculateTotal | Функция не определена | HIGH |
   | too many arguments | Ошибка параметра | MEDIUM |

2. **Спроектировать修复流程**
   使用 Сотрудничество Agent：

   ```markdown
   # Исправление сборки流程

   1. /go-build - Запустить сборку для получения полных ошибок
   2. /build-fix - 调用Исправление сборки Agent
   3. Проанализировать приоритеты ошибок：Сначала исправлять высокоприоритетные
   4. 逐个修复并Проверка
   5. /verify - Подтвердить успешную сборку
   ```

3. **为每个错误Спроектировать修复方案**

   **错误 1: не определён: " fmt"**
   ```go
   // 问题：Лишние кавычки
   import "fmt"  // Правильное написание
   
   // 修复：Убрать кавычки
   ```

   **错误 2: не определён: calculateTotal**
   ```go
   // 问题：Функция не определена或命名错误
   // 修复：Проверить определение функции，可能Требуется添加或重命名
   ```

   **错误 3: too many arguments**
   ```go
   // 问题：Несовпадение количества параметров при вызове функции
   // 修复：检查Сигнатура функции，Исправить параметры вызова
   ```

4. **记录修复经验**
   创建错误模式总结：

   ```markdown
   # Паттерны ошибок Go сборки

   ## Типичные ошибки
   1. не определён: Проблема импорта/функции
   2. too many/few arguments: Несовпадение параметров
   3. не определён: " xxx": Синтаксическая ошибка импорта

   ## Стратегия исправлений
   - Сначала посмотреть на количество ошибок，Потом на расположение ошибок
   - Высокоприоритетные ошибки могут вызывать низкоприоритетные
   - 修复后重新构建Проверка
   ```

### Критерии приёмки
- [ ] 能分析构建错误Тип和优先级
- [ ] Спроектировать了完整的修复流程
- [ ] Документированы типичные паттерны ошибок

## Упражнение 3: Комбинированный рабочий процесс Agent

### Цель
Спроектировать一个复杂Задача，使用多个 Сотрудничество AgentЗавершить。

### Сценарий
开发一个新的用户认证模块，包含以下功能：
- Регистрация пользователя（含Сложность пароляПроверка）
- Вход пользователя（с защитой от брутфорса）
- JWT token 生成和Проверка

### Шаги

1. **Декомпозиция задачи**
   将Декомпозиция задачи为子Задача：

   ```markdown
   # Задача разработки модуля аутентификации

   ## 子Задача
   1. 数据库Спроектировать (Schema)
   2. Регистрация пользователя API
   3. Вход пользователя API
   4. Обработка JWT Token
   5. Безопасность机制（Сложность пароля、Защита от брутфорса）
   ```

2. **Назначение Agent**
   Назначить Agent для каждой подзадачи：

   | 子Задача | Agent | Обоснование |
   |--------|-------|------|
   | 数据库Спроектировать | architect | Требуется架构视角 |
   | 注册 API | tdd-guide + security-reviewer | TDD + Безопасность |
   | 登录 API | tdd-guide + security-reviewer | TDD + Безопасность |
   | JWT 处理 | security-reviewer | Безопасность敏感 |
   | Безопасность机制 | security-reviewer (优先) | Безопасность优先 |

3. **Спроектировать工作流**

   ```
   [Начать]
      ↓
   /plan (规划Общий)
      ↓
   ┌─────────────────────────────────────┐
   │  Параллельное выполнение:                            │
   │  - architect: 数据库Спроектировать               │
   │  - tdd-guide: СпроектироватьТестирование策略            │
   └─────────────────────────────────────┘
      ↓
   [Этап реализации]
      ↓
   ┌─────────────────────────────────────┐
   │  Смесь серийного и параллельного:                      │
   │  1. tdd-guide: 编写注册Тестирование            │
   │  2. Реализовать注册功能                       │
   │  3. security-reviewer: Безопасность审查        │
   │  ──────────────────────────────────   │
   │  1. tdd-guide: 编写登录Тестирование            │
   │  2. Реализовать登录功能                       │
   │  3. security-reviewer: Безопасность审查        │
   └─────────────────────────────────────┘
      ↓
   [Этап интеграции]
      ↓
   ┌─────────────────────────────────────┐
   │  - architect: JWT 架构Спроектировать            │
   │  - security-reviewer: ОбщийБезопасность审查     │
   │  - code-reviewer: Качество кода审查        │
   │  - /verify: ОбщийПроверка                  │
   └─────────────────────────────────────┘
      ↓
   [Завершить]
   ```

4. **Написать документацию рабочего процесса**

   ```markdown
   # Рабочий процесс разработки модуля аутентификации

   ## Этап 1: Планирование
   - 使用 /plan 制定Общий план
   - 使用 architect СпроектироватьБаза данных и JWT архитектура

   ## Этап 2: TDD разработка
   - 使用 tdd-guide 指导Тестирование编写
   - 先写Тестирование，再Реализовать功能
   - 每个子功能单独Проверка

   ## 阶段 3: Безопасность审查
   - 使用 security-reviewer Проверить каждый API
   - Особое внимание：Обработка паролей、Token Безопасность、Защита от инъекций

   ## 阶段 4: 整合Проверка
   - 使用 code-reviewer Общий审查
   - 使用 /verify Проверка功能完整性

   ## Сотрудничество Agent总结
   - architect: 架构Спроектировать (1次)
   - tdd-guide: Тестирование指导 (2次)
   - security-reviewer: Безопасность审查 (3次)
   - code-reviewer: Проверка качества (1次)
   ```

### Критерии приёмки
- [ ] Декомпозиция задачи合理，包含 5+ 个子Задача
- [ ] Назначение Agent策略明确
- [ ] 工作流Спроектировать清晰
- [ ] Документированный рабочий процесс легче использовать повторно
