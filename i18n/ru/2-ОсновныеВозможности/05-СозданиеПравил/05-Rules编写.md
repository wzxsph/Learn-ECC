# 05 - Написание Rules

Учиться как编写定制化的开发规则。

## Обзор

ECC [Rules](../../СправочныеДокументы/rules/Стиль кода.md) 是一套强制性开发规则，涵盖Стиль кода、Тестирование、Безопасность、Git 工作流、Оркестрация агентов和Оптимизация производительности等方面。这些规则确保Качество кода和一致性，Через自动化检查和审查流程实施。

---

## 1. Rules 格式

### 1.1 基本格式

Rules 使用 Markdown 格式，包含 YAML frontmatter：

```yaml
---
name: rule-name
description: 规则Описание
category: 分类
---
```

### 1.2 规则层级

| 层级 | Описание | 优先级 |
|------|------|--------|
| **全局规则** | `~/.claude/rules/` | 最高 |
| **项目规则** | 项目 `.claude/rules/` | 中 |
| **本地规则** | 当前会话 | 最低 |

### 1.3 优先级

全局规则 > 项目规则 > 本地规则

## 2. 规则Тип详解

### 2.1 Стиль кода规则

定义代码组织的核心原则和Лучшие практики。

#### 核心要求

**不可变性（关键原则）**

```typescript
// 错误Пример - 修改原对象
function modify(user, name) {
  user.name = name;  // 改变了原对象
  return user;
}

// 正确Пример - Назад新对象
function update(user, name) {
  return { ...user, name };  // 创建新副本
}
```

**核心Спроектировать原则**

| 原则 | Описание | 实践 |
|------|------|------|
| **KISS** | Keep It Simple | 首选最简单的可行方案 |
| **DRY** | Don't Repeat Yourself | 避免复制粘贴导致的Реализовать漂移 |
| **YAGNI** | You Aren't Gonna Need It | 在真正Требуется时重构 |

**命名约定**

| Тип | 约定 | Пример |
|------|------|------|
| 变量和函数 | camelCase | `calculateTotal` |
| 布尔值 | is/has/should/can 前缀 | `isActive` |
| 接口、Тип、组件 | PascalCase | `UserProfile` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |

### 2.2 Правила тестирования

确保Качество кода的强制性Тестирование标准。

#### 最低Тестирование覆盖率：80%

| Тестирование型 | Описание |
|----------|------|
| 单元Тестирование | 独立函数、工具类、组件 |
| 集成Тестирование | API 端点、数据库操作 |
| E2E Тестирование | 关键用户流程 |

#### TDD 工作流

```
1. 编写Тестирование (RED)   - 先写一个会失败的Тестирование
2. 运行Тестирование          - ПроверкаТестирование失败
3. 编写最小Реализовать (GREEN) - 编写ЧерезТестирование的最小代码
4. 运行Тестирование          - ПроверкаТестированиеЧерез
5. 重构 (IMPROVE)    - 改进代码结构
6. Проверка覆盖率        - 确保达到 80%+
```

### 2.3 Правила безопасности

强制性Безопасность检查清单。

#### 提交前Обязательно检查

- [ ] 无Жёстко закодированные учётные данные（API keys、密码、tokens）
- [ ] 所有用户输入Проверка
- [ ] SQL инъекция防护（参数化查询）
- [ ] XSS 防护（输出转义）
- [ ] CSRF 保护启用
- [ ] 认证/授权Проверка
- [ ] 速率限制
- [ ] 错误消息不泄露敏感数据

#### 密钥Управлять

- **绝不** 在源代码中硬编码密钥
- **始终** 使用环境变量或密钥Управлять器
- 启动时Проверка必需的密钥
- 轮换任何可能暴露的密钥

### 2.4 Git 工作流规则

Git 提交规范和 PR 工作流。

#### 提交信息格式

```
<type>: <description>

<optional body>
```

**Type Тип**：
- `feat`: 新功能
- `fix`: 错误修复
- `refactor`: 重构
- `docs`: 文档
- `test`: Тестирование
- `chore`: 维护
- `perf`: 性能
- `ci`: CI/CD

#### PR 工作流

1. 分析完整提交历史
2. 使用 `git diff [base-branch]...HEAD` 查看所有变更
3. 起草综合性 PR 总结
4. 包含Тестирование计划和 TODOs
5. 如是新分支，使用 `-u` 标志推送

### 2.5 Оркестрация агентов规则

多 Сотрудничество Agent模式。

#### Принцип немедленного использования

| Сценарий | Agent |
|------|-------|
| Запрос сложной функции | **planner** |
| Код только что написан/изменён | **code-reviewer** |
| Исправление бага или новая функция | **tdd-guide** |
| Архитектурные решения | **architect** |

#### Параллельное выполнение задач

```markdown
# 好：Параллельное выполнение
Запустите 3 Агента параллельно:
1. Agent 1: 认证模块Анализ безопасности
2. Агент 2: Проверка производительности кэш-системы
3. Agent 3: 工具Тип检查

# Плохо: Ненужное последовательное выполнение
Сначала agent 1, потом agent 2, потом agent 3
```

### 2.6 Оптимизация производительности规则

模型选择和上下文Управлять。

#### 模型选择策略

| 模型 | Назначение | 成本 |
|------|------|------|
| **Haiku 4.5** | 轻量 Agent、频繁调用、配对编程 | 低 |
| **Sonnet 4.6** | 主要开发工作、Мульти-Agent 编排 | 中 |
| **Opus 4.5** | 复杂架构决策、深度推理 | 高 |

#### 上下文窗口Управлять

避免在最后 20% 上下文窗口进行：
- 大规模重构
- 跨多文件的特性Реализовать
- 复杂交互调试

## 3. 规则编写

### 3.1 触发条件

```yaml
---
name: no-console-log
description: 阻止提交包含 console.log 的代码
trigger:
  - pre-commit
  - pre-push
---
```

### 3.2 检查逻辑

```markdown
## 检查逻辑

1. 扫描修改的文件
2. 查找 `console.log`、`console.error` 等
3. 如果找到，阻止提交并显示Предупреждение
```

### 3.3 修复建议

```markdown
## 修复建议

- 使用结构化日志库（如 winston、pino）
- 配置日志级别（debug、info、warn、error）
- 确保生产环境日志级别合适
```

## 4. 违规处理

### 4.1 严重程度分级

| 级别 | 含义 | 操作 |
|------|------|------|
| CRITICAL | Уязвимости безопасности或数据丢失风险 | **阻止** - 合并前Обязательно修复 |
| HIGH | Bug 或重大质量问题 | **Предупреждение** - 合并前应修复 |
| MEDIUM | 可维护性问题 | **Подсказка** - 建议修复 |
| LOW | 样式或小建议 | **备注** - 可选 |

### 4.2 修复流程

```
1. 分析违规原因
2. 确定修复方案
3. 执行修复
4. Проверка修复
5. 重新提交审查
```

## 5. Качество кода检查清单

Завершить工作前Обязательно检查：

- [ ] 代码可读且命名良好
- [ ] 函数短小（<50 行）
- [ ] 文件专注（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 适当的Обработка ошибок
- [ ] 无硬编码值（使用常量或配置）
- [ ] 使用不可变模式（无 mutation）
- [ ] Тестирование覆盖率 ≥80%
- [ ] 所有Безопасность检查Через

## 6. 创建项目特定规则

### 6.1 项目规则位置

```
项目/
└── .claude/
    └── rules/
        ├── README.md
        └── 自定义规则文件
```

### 6.2 继承全局规则

项目规则继承全局规则，可在此基础上扩展：

```markdown
# 项目特定规则

继承自全局规则，增加以下要求：

## 额外要求

- [ ] 项目特定的编码规范
- [ ] 特定的Тестирование要求
- [ ] 特定的文档要求
```
