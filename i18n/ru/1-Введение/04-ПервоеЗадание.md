# 04-Первое задание

## Цели обучения

- Понять полный процесс разработки
- Освоить полный процесс /ecc:plan → /ecc:code-review → /ecc:build-fix → commit
- Уметь самостоятельно разрабатывать простую функциональность

---

## Полный процесс разработки

ECC рекомендует следующий процесс разработки:

```
Шаг 1: /ecc:plan        → Спланировать задачу
Шаг 2: /ecc:code-review → Проверить код
Шаг 3: /ecc:build-fix   → Исправить сборку
Шаг 4: Зафиксировать код → Git commit
```

---

## Шаг 1: Планирование с помощью /ecc:plan

### Входные данные

Допустим, мы хотим реализовать простой калькулятор:

```
/ecc:plan Реализовать калькулятор, поддерживающий четыре операции: сложение, вычитание, умножение, деление
```

### Ожидаемый результат

Команда /ecc:plan выполнит:

1. **Повторение требований** - "Реализовать калькулятор, поддерживающий четыре базовые операции..."
2. **Оценка рисков** - "Необходимо учесть: деление на ноль..."
3. **Пошаговый план** -
   - Фаза 1: Создать ядро калькулятора
   - Фаза 2: Реализовать четыре операции
   - Фаза 3: Добавить обработку ошибок
   - Фаза 4: Написать unit-тесты
4. **Ожидание подтверждения** - "Пожалуйста, подтвердите, можно ли начинать"

### После подтверждения

После подтверждения пользователем, ECC ожидает ввод пользователя для запуска каждого этапа.

### Советы

- Используйте шаблон PRD файла для более детального планирования
- Для сложной функциональности рекомендуется сначала написать PRD, затем планировать

## Шаг 2: Проверка с помощью /ecc:code-review

### Входные данные

После написания кода запустите:

```
/ecc:code-review
```

### Ожидаемый результат

/ecc:code-review выполнит:

1. Соберет измененные файлы
2. Проведет проверку по 7 категориям:
   - Правильность
   - Типовая безопасность
   - Соответствие шаблонам
   - Безопасность
   - Производительность
   - Полнота
   - Удобство сопровождения
3. Создаст отчет о проверке с уровнями CRITICAL/HIGH/MEDIUM/LOW

### Пример отчета о проверке

```
## Отчет о проверке кода

### CRITICAL
- [ ] Жестко закодированные учетные данные: src/auth.js строка 23

### HIGH
- [ ] Слишком длинная функция: src/calculator.js строки 45-78 (34 строки)
- [ ] Отсутствует обработка ошибок: функция деления в src/calculator.js

### MEDIUM
- [ ] Неправильное именование: переменная `tmp` должна быть переименована в `temporary`

### Рекомендуемые действия
1. Удалить жестко закодированный API Key
2. Рефакторить функцию деления в calculator.js, добавить проверку деления на ноль
3. Разделить слишком длинную функцию
```

### Советы

- Обязательно запускайте /ecc:code-review перед коммитом
- Проблемы CRITICAL и HIGH должны быть исправлены

## Шаг 3: Исправление с помощью /ecc:build-fix

### Входные данные

Если сборка не удалась, запустите:

```
/ecc:build-fix
```

### Ожидаемый результат

/ecc:build-fix выполнит:

1. **Определит систему сборки** - "Обнаружен проект TypeScript"
2. **Запустит сборку** - "Запускается tsc --noEmit..."
3. **Разберет ошибки** - Сгруппирует по файлам
4. **Будет исправлять пошагово** - Одну ошибку за раз
5. **Проверит** - Пересоберет после каждого исправления

### Пример вывода

```
[build-fix] Обнаружен проект TypeScript
[build-fix] Запускается tsc --noEmit...
[build-fix] Найдено 3 ошибки:

Ошибка 1: src/calculator.ts:15 - Тип 'string' не может быть присвоен типу 'number'
Ошибка 2: src/calculator.ts:28 - Свойство 'result' не существует
Ошибка 3: src/calculator.ts:42 - Параметр 'b' не используется

[build-fix] Начинается исправление ошибки 1...
[build-fix] Исправление завершено, проверка...
[build-fix] Сборка успешна!
```

### Советы

- Исправление сборки инкрементальное, по одной ошибке за раз
- Можно в любой момент остановить с помощью Ctrl+C

## Шаг 4: Фиксация кода

### Процесс фиксации

1. Убедитесь, что все тесты проходят
2. Убедитесь, что сборка успешна
3. Убедитесь, что проблемы проверки кода исправлены

### Стандарты Git коммитов

ECC использует формат conventional commits:

```
<type>: <description>

[optional body]
```

**Типы Type**:
- `feat` - Новая функция
- `fix` - Исправление бага
- `refactor` - Рефакторинг
- `docs` - Документация
- `test` - Тест
- `chore` - Рутинное обслуживание
- `perf` - Оптимизация производительности
- `ci` - Конфигурация CI

### Пример коммита

```
feat: Реализовать четыре базовые операции калькулятора

- Добавлена функция сложения
- Добавлена функция вычитания
- Добавлена функция умножения
- Добавлена функция деления (с проверкой деления на ноль)

Связано с: Issue #123
```

### Советы

- Используйте `/ecc:verify quick` для быстрой проверки перед коммитом
- Большие функции разделяйте на несколько коммитов, маленькие функции объединяйте в один коммит

## Полный пример

### Сценарий

Реализуем простую функцию для работы со строками: `truncate(str, maxLength)` - обрезает строку до указанной длины.

### Шаг 1: Планирование

```
/ecc:plan Реализовать функцию truncate, которая обрезает строку до указанной длины
```

Вывод:
```
## Повторение требований
Реализовать функцию truncate(str, maxLength), которая заменяет часть, превышающую maxLength, на ...

## Оценка рисков
- Обработка случая, когда maxLength меньше 3
- Обработка пустой строки или null
- Обработка символов Unicode

## Пошаговый план
1. Создать src/string-utils.ts
2. Реализовать функцию truncate
3. Добавить тесты для граничных случаев
4. Добавить JSDoc документацию

Можно начинать?
```

### Шаг 2: Реализация

(После подтверждения пользователя, ECC помогает реализовать код)

### Шаг 3: Проверка

```
/ecc:code-review
```

Вывод:
```
## Отчет о проверке кода
Статус: ОДОБРЕНО

### Результаты проверки
- Правильность: Пройдено
- Типовая безопасность: Пройдено
- Соответствие шаблонам: Пройдено
- Безопасность: Пройдено
- Производительность: Пройдено
- Полнота: Рекомендуется добавить больше граничных тестов
- Удобство сопровождения: Пройдено

### Рекомендации
- Добавить тест на пустую строку
- Добавить тест на null значение
```

### Шаг 4: Проверка сборки

```
/ecc:build-fix
```

```
[build-fix] Обнаружен проект TypeScript
[build-fix] Запускается tsc --noEmit...
[build-fix] Сборка успешна!
```

### Шаг 5: Коммит

```
feat: Реализовать функцию усечения строки truncate

- Поддержка указания максимальной длины
- Превышающая часть заменяется на ...
- Обработка пустой строки и null значения
```

## Резюме

1. **Полный процесс**: /ecc:plan → Реализация → /ecc:code-review → /ecc:build-fix → Коммит
2. **/ecc:plan** используется для планирования, обеспечивает правильное направление
3. **/ecc:code-review** используется для проверки, находит проблемы
4. **/ecc:build-fix** используется для исправления ошибок сборки
5. **Коммит** использует формат conventional commits
