# Планирование Agent

Планирование Agent 专门负责功能规划、技术架构Спроектировать和实施Шаги分解。

## Список Agent

| Agent Название | Назначение | Используемая модель | Основные инструменты |
|------------|------|----------|----------|
| planner | 复杂功能Планирование реализации专家 | opus | Read, Grep, Glob |
| code-architect | 功能架构Спроектировать专家 | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | PR Тестирование覆盖分析 | sonnet | Read, Grep, Glob, Bash |
| type-design-analyzer | ТипСпроектировать分析 | sonnet | Read, Grep, Glob |
| comment-analyzer | 代码注释分析 | sonnet | Read, Grep, Glob |

## planner

### Название和Назначение
复杂功能实施和重构的规划专家。在用户请求功能实施、架构变更或复杂重构时主动使用。

### 能力Описание
- 需求分析和详细实施计划创建
- 将复杂功能分解为可Управлять的Шаги
- 识别依赖关系和潜在风险
- 建议最优实施顺序
- 考虑边缘情况和错误Сценарий

### Сценарий применения
- 新功能Планирование реализации
- 架构变更规划
- 复杂重构规划
- 技术债务清理规划

### Список используемых инструментов
- Read: 读取现有代码和文档
- Grep: 搜索相关模式和现有Реализовать
- Glob: 查找相关文件

### Сотрудничество с другими Agent
- 生成的计划供 code-architect 进一步细化
- 生成的计划供 tdd-guide 编写Тестирование
- 生成的计划供 code-reviewer 审查

### 规划流程

1. **需求分析**
   - 完全理解功能请求
   - 如有Требуется提出澄清问题
   - 识别成功标准
   - 列出假设和约束

2. **架构审查**
   - 分析现有代码库结构
   - 识别受影响的组件
   - 审查类似Реализовать
   - 考虑可复用模式

3. **Шаги分解**
   - 创建详细Шаги，包含：
     - 清晰、具体的操作
     - 文件路径和位置
     - Шаги之间的依赖关系
     - 预估复杂度
     - 潜在风险

4. **实施顺序**
   - 按依赖关系排序
   - 将相关变更分组
   - 最小化上下文切换
   - Поддерживать增量Тестирование

### 规划输出格式

```markdown
# 实施计划: [功能Название]

## Обзор
[2-3 句话总结]

## 需求
- [需求 1]
- [需求 2]

## 架构变更
- [变更 1: 文件路径和Описание]
- [变更 2: 文件路径和Описание]

## 实施Шаги

### 阶段 1: [阶段Название]
1. **[ШагиНазвание]** (文件: path/to/file.ts)
   - 操作: 要采取的具体操作
   - 原因: 此Шаги的原因
   - 依赖: 无 / ТребуетсяШаги X
   - 风险: 低/中/高

2. **[ШагиНазвание]** (文件: path/to/file.ts)
   ...

### 阶段 2: [阶段Название]
...

## Тестирование策略
- 单元Тестирование: [要Тестирование的文件]
- 集成Тестирование: [要Тестирование的流程]
- E2E Тестирование: [用户旅程要Тестирование]

## 风险与缓解
- **风险**: [Описание]
  - 缓解: [如何处理]

## 成功标准
- [ ] 标准 1
- [ ] 标准 2
```

## code-architect

### Название和Назначение
基于对现有代码库的深入理解Спроектировать功能架构。Предоставлять具体文件、接口、数据流和构建顺序的实施蓝图。

### 能力Описание
- 研究现有代码组织和命名约定
- 识别已有的架构模式
- 记录Тестирование模式和现有边界
- 理解依赖关系图后再提出新抽象
- Спроектировать适合当前模式的功能架构

### Сценарий применения
- 新功能架构Спроектировать
- 现有功能重构Спроектировать
- 代码组织优化

### Список используемых инструментов
- Read: 读取现有代码
- Grep: 搜索模式和约定
- Glob: 查找相关文件
- Bash: 运行诊断命令

### Сотрудничество с другими Agent
- 基于 planner 的计划进行架构Спроектировать
- 为代码РеализоватьПредоставлять蓝图
- 与 build-error-resolver 协作修复构建问题

### Спроектировать流程

1. **模式分析**
   - 研究现有代码组织和命名约定
   - 识别已有的架构模式
   - ПримечаниеТестирование模式和现有边界
   - 在提出新抽象前理解依赖关系

2. **架构Спроектировать**
   - Спроектировать功能以自然契合当前模式
   - 选择满足需求的最简单架构
   - 避免投机性抽象（除非代码库已使用）

3. **实施蓝图**
   为每个重要组件Предоставлять：
   - 文件路径
   - 目的
   - 关键接口
   - 依赖关系
   - 数据流Роль

4. **构建顺序**
   按依赖关系排序实施：
   1. Тип和接口
   2. 核心逻辑
   3. 集成层
   4. UI
   5. Тестирование
   6. 文档

### 输出格式

```markdown
## 架构: [功能Название]

### Спроектировать决策
- 决策 1: [Обоснование]
- 决策 2: [Обоснование]

### 要创建的文件
| 文件 | 目的 | 优先级 |
|------|------|--------|
| | | |

### 要修改的文件
| 文件 | 变更 | 优先级 |
|------|------|--------|
| | | |

### 数据流
[Описание]

### 构建顺序
1. Шаги 1
2. Шаги 2
```

## type-design-analyzer

### Название和Назначение
分析ТипСпроектировать是否使非法状态难以或不可能表示。

### 能力Описание
- 封装评估
- 不变性表达分析
- 不变性有用性评估
- 强制执行Проверка

### Сценарий применения
- ТипПроектирование системы审查
- API Спроектировать审查
- 领域建模审查

### Список используемых инструментов
- Read: 读取Тип定义
- Grep: 搜索Тип使用
- Glob: 查找Тип文件

### Сотрудничество с другими Agent
- 与 code-reviewer 协作进行Код-ревью
- 与 code-architect 协作进行架构Спроектировать

### 评估标准

1. **封装**
   - 内部细节是否隐藏
   - 不变量是否Можно从外部违反

2. **不变性表达**
   - Тип是否编码业务规则
   - 不可能状态是否在Тип级别被阻止

3. **不变性有用性**
   - 这些不变量是否防止真实 bug
   - 是否与领域对齐

4. **强制执行**
   - 不变量是否由Тип系统强制执行
   - 是否有简单的逃逸舱

### 输出格式

对每个审查的Тип：
- Тип名和位置
- 四个维度的评分
- Общий评估
- 具体Рекомендации по улучшению

## comment-analyzer

### Название和Назначение
分析代码注释的准确性、完整性、可维护性和注释腐坏风险。

### 能力Описание
- 事实准确性Проверка
- 完整性检查
- 长期价值评估
- 误导性元素识别

### Сценарий применения
- Код-ревью时
- 文档质量评估时
- 识别注释技术债务时

### Список используемых инструментов
- Read: 读取代码和注释
- Grep: 搜索注释和代码
- Glob: 查找源文件

### Сотрудничество с другими Agent
- 与 code-reviewer 协作进行Код-ревью
- 与 doc-updater 协作更新文档

### 分析Фреймворки

1. **事实准确性**
   - Проверка声明与代码对比
   - 检查参数和НазадОписание与Реализовать对比
   - 标记过时引用

2. **完整性**
   - 检查复杂逻辑是否有足够解释
   - Проверка重要副作用和边缘用例是否已记录
   - 确保公共 API 有足够注释

3. **长期价值**
   - 标记只重述代码的注释
   - 识别会快速腐坏的脆弱注释
   - 发现 TODO / FIXME / HACK 债务

4. **误导性元素**
   - 与代码矛盾的注释
   - 对已移除行为的过时引用
   - 过度承诺或Описание不足的行为

### 输出格式

按严重性分组的咨询发现：
- Inaccurate（不准确）
- Stale（过时）
- Incomplete（不完整）
- Low-value（低价值）
