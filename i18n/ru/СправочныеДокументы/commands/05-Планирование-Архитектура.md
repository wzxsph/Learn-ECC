# Планирование и архитектура命令

## Обзор

Планирование и архитектура命令Используется для产品规划、功能Спроектировать和系统架构决策。这些命令确保在写代码之前有清晰的计划，减少返工和提高Качество кода。

## 命令列表

### /plan

**Назначение**: 通用Планирование реализации - 复述需求、评估风险、创建分Шаги实施计划

**Описание**: 在触碰任何代码之前，等待用户确认后创建全面的实施计划。接受自由形式需求或 PRD markdown 文件。

**输入模式**:

| 输入 | 模式 | 行为 |
|---|---|---|
| `path/to/name.prd.md` | PRD artifact 模式 | 读取 PRD，选择下一个待交付里程碑，生成 `.claude/plans/{name}.plan.md` |
| 其他 markdown 路径 | 引用模式 | 读取文件作为上下文，产生内联计划 |
| 自由形式文本 | 对话模式 | 产生内联计划 |
| 空输入 | 澄清模式 | 询问要规划什么 |

**关键参数**:
- `$ARGUMENTS`: `[feature description | path/to/*.prd.md]` - 功能Описание或 PRD 文件路径

**工作流**:
1. **复述需求** - 澄清Требуется构建什么
2. **识别风险** - 列出潜在问题和阻碍
3. **创建Шаги计划** - 分解为阶段性实施
4. **等待确认** - Обязательно收到用户批准后才能继续

**输出格式** (PRD artifact 模式):
```markdown
# Plan: {Feature Name}

**Source PRD**: {path}
**Selected Milestone**: {milestone or phase name}
**Complexity**: {Small | Medium | Large}

## Summary
{2-3 sentences}

## Patterns to Mirror
| Category | Source | Pattern |
|---|---|---|
| Naming | `path:line` | {short description} |
| Errors | `path:line` | {short description} |
| Tests | `path:line` | {short description} |

## Files to Change
| File | Action | Why |
|---|---|---|
| `path` | CREATE / UPDATE / DELETE | {reason} |

## Tasks
### Task 1: {name}
- **Action**: {what to do}
- **Mirror**: {pattern to follow}
- **Validate**: {command that proves correctness}

## Validation
```bash
{project-specific validation commands}
```

## Risks
| Risk | Likelihood | Mitigation |
|---|---|---|
```

**Лучшие практики**:
- 使用前先研究代码库中的现有模式
- 在 PRD artifact 模式中创建 `.claude/plans/` Содержание（如不存在）
- 如果 PRD 包含 `Delivery Milestones` 表格，只更新所选行状态

**常见陷阱**:
- 空输入时不询问澄清就尝试规划
- 不等待用户确认就Начать写代码
- 跳过 Pattern Grounding Шаги

**与Другие команды集成**:
- 规划后使用 `tdd-workflow` skill 进行Разработка через тестирование
- 使用 `/build-fix` Исправление ошибок сборки
- 使用 `/code-review` 审查ЗавершитьРеализовать
- 使用 `/pr` 或 `/prp-pr` 打开 Pull Request

### /plan-prd

**Назначение**: 交互式 PRD 生成器

**Описание**: 生成以问题为中心的产品需求文档，包含问题定义、用户画像、成功指标和 MVP 范围。

**Сценарий использования**: 
- 确定要构建的功能
- Требуется明确产品需求
- 从零Начать新功能规划

**工作流**:
1. FRAME - 了解用户和问题
2. GROUND - 收集证据
3. DECIDE - 确定范围和假设
4. GENERATE - 生成 PRD 文件

---

### /prp-plan

**Назначение**: 全面的功能规划

**Описание**: 完整的项目规划，涵盖代码库分析、风险评估和分Шаги实施计划。

**Сценарий использования**:
- 复杂功能的详细规划
- Требуется代码库上下文分析
- 多阶段实施项目

---

### /prp-prd

**Назначение**: PRP 工作流 PRD 生成器

**Описание**: 使用 PRP（Plan-Record-Produce）工作流生成产品需求文档。

---

### /prp-implement

**Назначение**: 执行 PRP 计划+Проверка循环

**Описание**: 按照 PRP 工作流执行计划，包含持续的Проверка和检查。

---

### /prp-pr

**Назначение**: 从 PRP 工作流创建 PR

**Описание**: 从 PRP 工作流的结果创建 GitHub Pull Request。

---

### /prp-commit

**Назначение**: PRP Проверка提交

**Описание**: 在 PRP 工作流中执行Проверка并提交代码。

---

### /multi-plan

**Назначение**: 多模型协作规划

**Описание**: 结合 Codex 和 Gemini 的双模型分析，生成综合实施计划。

**Сценарий использования**:
- Требуется多角度分析
- 复杂跨领域项目
- Требуется平衡前端和后端考量
