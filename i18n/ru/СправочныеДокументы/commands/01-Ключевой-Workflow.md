# Основной рабочий процесс命令

本文档介绍 ECC 中Используется для核心开发工作流的命令。

---

## /plan

**НазначениеОписание**: 重述需求、评估风险，并创建分步实施计划。在触碰任何代码之前**等待用户确认**。

**Способ использования**:
```
/plan [功能Описание | PRD文件路径]
```

**Сценарий использования**:
- Начать新功能开发
- 进行重大架构变更
- 复杂的重构工作
- 需求不明确或模糊时

**详细Описание**:
该命令会：
1. **重述需求** - 清晰表达Требуется构建的内容
2. **识别风险** - 揭示潜在问题和阻碍
3. **创建分步计划** - 将实施分解为阶段
4. **等待确认** - Обязательно获得用户明确批准后才能继续

Поддерживать多种输入模式：
| 输入 | 模式 | 行为 |
|------|------|------|
| `path/to/name.prd.md` | PRD 模式 | 读取 PRD，选择下一个待处理的交付里程碑 |
| 其他 markdown 路径 | 引用模式 | 读取文件作为上下文并生成内联计划 |
| 自由格式文本 | 对话模式 | 生成内联计划 |
| 空输入 | 澄清模式 | 询问Следует规划什么 |

## /code-review

**НазначениеОписание**: 本地未提交变更或 GitHub PR 的全面Код-ревью。

**Способ использования**:
```
/code-review                 # 本地审查模式
/code-review [PR号 | PR链接]  # PR 审查模式
```

**Сценарий использования**:
- 提交代码之前进行全面审查
- PR 审查以发现潜在问题
- 学习代码库中的Лучшие практики

**本地审查模式**:
1. 收集变更文件（`git diff --name-only HEAD`）
2. 检查Безопасность问题（Жёстко закодированные учётные данные、SQL注入、XSS等）
3. 检查Качество кода问题（函数>50行、文件>800行等）
4. 生成带严重级别标注的报告

**PR 审查模式**:
1. 获取 PR 元数据和差异
2. 检查相关项目规范和规划文档
3. 读取完整变更文件
4. 应用 7 个审查Категория（正确性、ТипБезопасность、模式合规、Безопасность、性能、完整性、可维护性）
5. 运行Проверка命令（Тип检查、lint、Тестирование、构建）
6. 发布审查意见到 GitHub

## /build-fix

**НазначениеОписание**: 检测项目构建系统并逐步修复构建/Тип错误。

**Способ использования**:
```
/build-fix
```

**Сценарий использования**:
- 构建失败时
- TypeScript/JavaScript Тип错误
- 编译错误
- 依赖问题

**工作流程**:
1. **检测构建系统** - 根据文件Тип选择合适的构建命令
2. **解析错误** - 按文件和依赖顺序分组
3. **逐步修复** - 一次修复一个错误
4. **Проверка** - 每次修复后重新Запустить сборку

Поддерживать的构建系统：

| 指示器 | 构建命令 |
|--------|----------|
| `package.json` + `build` 脚本 | `npm run build` |
| `tsconfig.json` (TypeScript) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall` 或 `mypy` |

## /verify

**НазначениеОписание**: Проверка代码变更是否按预期工作。Используется дляПроверка PR、确认修复或Тестирование变更。

**Способ использования**:
```
/verify [quick]           # 快速Проверка（推荐）
/verify full             # 完整Проверка
```

**Сценарий использования**:
- Проверка PR 更改
- 确认修复有效
- 手动Тестирование变更
- 本地Проверка后再推送

**ПроверкаШаги**:
1. 运行相关Тестирование
2. 检查ТипБезопасность性
3. Проверка构建成功
4. 检查 linting

## /quality-gate

**НазначениеОписание**: 为文件或项目范围运行 ECC 质量管道并报告修复Шаги。

**Способ использования**:
```
/quality-gate [path|.] [--fix] [--strict]
```

**Сценарий использования**:
- 按需运行质量检查
- 在 CI 管道中集成
- 确保代码符合项Цель准

**管道Шаги**:
1. 检测Цель语言/工具
2. 运行格式化程序检查
3. 运行 lint/Тип检查
4. 生成简洁的修复列表

## /tdd

**НазначениеОписание**: Разработка через тестирование工作流。遵循红-绿-重构循环。

**Способ использования**:
```
/tdd                      # 启动 TDD 工作流
```

**Сценарий использования**:
- Реализовать新功能
- 修复 bug（先写失败的Тестирование）
- 添加Тестирование覆盖
- 学习 TDD 方法论

**TDD 循环**:
```
RED    → 写一个失败的Тестирование
GREEN  → 编写最少的代码使ТестированиеЧерез
REFACTOR → 改进代码，Тестирование保持绿色
REPEAT → 下一个Тестирование用例
```

**Лучшие практики**:
- **先写Тестирование** - 在任何Реализовать之前
- **每次修改后运行Тестирование** - Проверка进度
- **Тестирование行为，而非Реализовать** - 关注接口而非细节
- **包含边界情况** - 空值、null、最大值等

## 命令集成关系

```
需求不明 → /plan-prd (创建 PRD)
     ↓
需求明确 → /plan (创建实施计划)
     ↓
/tdd (或 /feature-dev) Реализовать
     ↓
/build-fix Исправление ошибок сборки
     ↓
/code-review Код-ревью
     ↓
/quality-gate 质量门禁
     ↓
/verify Проверка
     ↓
/pr 创建 PR
```
