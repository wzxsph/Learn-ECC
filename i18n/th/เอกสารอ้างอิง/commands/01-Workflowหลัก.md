# คำสั่งขั้นตอนการทำงานหลัก

เอกสารนี้แนะนำคำสั่งสำหรับขั้นตอนการทำงานหลักในการพัฒนาใน ECC

---

## /plan

**วัตถุประสงค์**: อธิบายความต้องการซ้ำ ประเมินความเสี่ยง และสร้างแผนการดำเนินการทีละขั้นตอน รอการยืนยันจากผู้ใช้**ก่อน**แตะโค้ดใดๆ

**วิธีใช้**:
```
/plan [คำอธิบายฟีเจอร์ | path ของไฟล์ PRD]
```

**สถานการณ์การใช้งาน**:
- เริ่มพัฒนาฟีเจอร์ใหม่
- ดำเนินการเปลี่ยนแปลงสถาปัตยกรรมที่สำคัญ
- งานปรับโครงสร้างที่ซับซ้อน
- ความต้องการไม่ชัดเจนหรือคลุมเครือ

**รายละเอียด**:
คำสั่งนี้จะ:
1. **อธิบายความต้องการซ้ำ** - แสดงอย่างชัดเจนว่าต้องสร้างอะไร
2. **ระบุความเสี่ยง** - เปิดเผยปัญหาและอุปสรรคที่อาจเกิดขึ้น
3. **สร้างแผนทีละขั้นตอน** - แบ่งการดำเนินการออกเป็นขั้นตอน
4. **รอการยืนยัน** - ต้องได้รับอนุมัติจากผู้ใช้อย่างชัดเจนก่อนดำเนินต่อ

รองรับหลายโหมดการป้อนข้อมูล:
| ข้อมูลนำเข้า | โหมด | พฤติกรรม |
|------|------|------|
| `path/to/name.prd.md` | โหมด PRD | อ่าน PRD เลือก milestone ถัดไปที่ต้องส่งมอบ |
| path markdown อื่นๆ | โหมดอ้างอิง | อ่านไฟล์เป็นบริบทและสร้างแผนแบบ inline |
| ข้อความเป็นอิสระ | โหมดการสนทนา | สร้างแผนแบบ inline |
| ข้อมูลว่างเปล่า | โหมดชี้แจง | ถามว่าควรวางแผนอะไร |

## /code-review

**วัตถุประสงค์**: การตรวจสอบโค้ดที่ครอบคลุมสำหรับการเปลี่ยนแปลงที่ยังไม่ได้ commit ในเครื่องหรือ GitHub PR

**วิธีใช้**:
```
/code-review                 # โหมดตรวจสอบในเครื่อง
/code-review [หมายเลข PR | URL ของ PR]  # โหมดตรวจสอบ PR
```

**สถานการณ์การใช้งาน**:
- การตรวจสอบที่ครอบคลุมก่อน commit โค้ด
- การตรวจสอบ PR เพื่อค้นหาปัญหาที่อาจเกิดขึ้น
- เรียนรู้แนวปฏิบัติที่ดีที่สุดใน codebase

**โหมดตรวจสอบในเครื่อง**:
1. รวบรวมไฟล์ที่เปลี่ยนแปลง (`git diff --name-only HEAD`)
2. ตรวจสอบปัญหาความปลอดภัย (credential ที่ hardcode, SQL injection, XSS ฯลฯ)
3. ตรวจสอบปัญหาคุณภาพโค้ด (ฟังก์ชัน >50 บรรทัด, ไฟล์ >800 บรรทัด ฯลฯ)
4. สร้างรายงานพร้อมระดับความรุนแรง

**โหมดตรวจสอบ PR**:
1. รับ metadata และ diff ของ PR
2. ตรวจสอบข้อกำหนดและเอกสารการวางแผนของโปรเจกต์ที่เกี่ยวข้อง
3. อ่านไฟล์ที่เปลี่ยนแปลงอย่างสมบูรณ์
4. ใช้ 7 หมวดหมู่การตรวจสอบ (ความถูกต้อง, ความปลอดภัยประเภท, การปฏิบัติตามรูปแบบ, ความปลอดภัย, ประสิทธิภาพ, ความสมบูรณ์, การบำรุงรักษา)
5. รันคำสั่งตรวจสอบ (ตรวจสอบประเภท, lint, การทดสอบ, การสร้าง)
6. เผยแพร่ความคิดเห็นการตรวจสอบไปยัง GitHub

## /build-fix

**วัตถุประสงค์**: ตรวจจับระบบการสร้างโปรเจกต์และแก้ไขข้อผิดพลาดในการสร้าง/ประเภททีละขั้นตอน

**วิธีใช้**:
```
/build-fix
```

**สถานการณ์การใช้งาน**:
- การสร้างล้มเหลว
- ข้อผิดพลาดประเภท TypeScript/JavaScript
- ข้อผิดพลาดการคอมไพล์
- ปัญหาการพึ่งพา

**ขั้นตอนการทำงาน**:
1. **ตรวจจับระบบการสร้าง** - เลือกคำสั่งการสร้างที่เหมาะสมตามประเภทไฟล์
2. **แยกวิเคราะห์ข้อผิดพลาด** - จัดกลุ่มตามไฟล์และลำดับการพึ่งพา
3. **แก้ไขทีละขั้นตอน** - แก้ไขทีละข้อผิดพลาด
4. **ตรวจสอบ** - รันการสร้างใหม่หลังการแก้ไขแต่ละครั้ง

ระบบการสร้างที่รองรับ:

| ตัวบ่งชี้ | คำสั่งการสร้าง |
|--------|----------|
| `package.json` + `build` script | `npm run build` |
| `tsconfig.json` (TypeScript) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall` หรือ `mypy` |

## /verify

**วัตถุประสงค์**: ตรวจสอบว่าการเปลี่ยนแปลงโค้ดทำงานตามที่คาดไว้ ใช้เพื่อตรวจสอบ PR ยืนยันการแก้ไขหรือทดสอบการเปลี่ยนแปลง

**วิธีใช้**:
```
/verify [quick]           # การตรวจสอบอย่างรวดเร็ว (แนะนำ)
/verify full             # การตรวจสอบอย่างสมบูรณ์
```

**สถานการณ์การใช้งาน**:
- ตรวจสอบการเปลี่ยนแปลง PR
- ยืนยันว่าการแก้ไขได้ผล
- ทดสอบการเปลี่ยนแปลงด้วยตนเอง
- ตรวจสอบในเครื่องก่อน push

**ขั้นตอนการตรวจสอบ**:
1. รันการทดสอบที่เกี่ยวข้อง
2. ตรวจสอบความปลอดภัยประเภท
3. ตรวจสอบว่าการสร้างสำเร็จ
4. ตรวจสอบ linting

## /quality-gate

**วัตถุประสงค์**: รัน ECC quality pipeline สำหรับไฟล์หรือขอบเขตโปรเจกต์และรายงานขั้นตอนการแก้ไข

**วิธีใช้**:
```
/quality-gate [path|.] [--fix] [--strict]
```

**สถานการณ์การใช้งาน**:
- รันการตรวจสอบคุณภาพตามความต้องการ
- บูรณาการใน CI pipeline
- ตรวจสอบว่าโค้ดเป็นไปตามมาตรฐานโปรเจกต์

**ขั้นตอน pipeline**:
1. ตรวจจับภาษา/เครื่องมือเป้าหมาย
2. รันการตรวจสอบ formatter
3. รัน lint/การตรวจสอบประเภท
4. สร้างรายการการแก้ไขอย่างกระชับ

## /tdd

**วัตถุประสงค์**: ขั้นตอนการทำงานการพัฒนาที่ขับเคลื่อนด้วยการทดสอบ ทำตามวงจร red-green-refactor

**วิธีใช้**:
```
/tdd                      # เริ่มขั้นตอนการทำงาน TDD
```

**สถานการณ์การใช้งาน**:
- สร้างฟีเจอร์ใหม่
- แก้ไขข้อบกพร่อง (เขียนการทดสอบที่ล้มเหลวก่อน)
- เพิ่มการครอบคลุมการทดสอบ
- เรียนรู้วิธีการ TDD

**วงจร TDD**:
```
RED    → เขียนการทดสอบที่ล้มเหลว
GREEN  → เขียนโค้ดน้อยที่สุดเพื่อให้การทดสอบผ่าน
REFACTOR → ปรับปรุงโค้ด ให้การทดสอบยังคงเป็นสีเขียว
REPEAT → การทดสอบเคสถัดไป
```

**แนวปฏิบัติที่ดีที่สุด**:
- **เขียนการทดสอบก่อน** - ก่อนโค้ดใดๆ
- **รันการทดสอบหลังการเปลี่ยนแปลงแต่ละครั้ง** - ยืนยันความคืบหน้า
- **ทดสอบพฤติกรรม ไม่ใช่การ implementation** - โฟกัสที่ interface ไม่ใช่รายละเอียด
- **รวมกรณีขอบเขต** - ค่าว่าง, null, ค่าสูงสุด ฯลฯ

## ความสัมพันธ์การบูรณาการคำสั่ง

```
ความต้องการไม่ชัดเจน → /plan-prd (สร้าง PRD)
     ↓
ความต้องการชัดเจน → /plan (สร้างแผนการดำเนินการ)
     ↓
/tdd (หรือ /feature-dev) ดำเนินการ
     ↓
/build-fix แก้ไขข้อผิดพลาดการสร้าง
     ↓
/code-review ตรวจสอบโค้ด
     ↓
/quality-gate ประตูคุณภาพ
     ↓
/verify ตรวจสอบ
     ↓
/pr สร้าง PR
```
