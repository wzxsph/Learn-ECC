# คำสั่งตรวจสอบภาษา

เอกสารนี้แนะนำคำสั่งเฉพาะทางสำหรับการตรวจสอบโค้ดของแต่ละภาษาโปรแกรมใน ECC

## /python-review

**วัตถุประสงค์**: การตรวจสอบโค้ด Python อย่างครอบคลุม ตรวจสอบการ tuân thủ PEP 8 ความปลอดภัยประเภท และแนวปฏิบัติ Python

**วิธีใช้**:
```
/python-review
```

**สถานการณ์การใช้งาน**:
- หลังเขียนหรือแก้ไขโค้ด Python
- ก่อน commit การเปลี่ยนแปลง Python
- ตรวจสอบ PR ที่มีโค้ด Python
- เรียนรู้แนวปฏิบัติและแนวปฏิบัติที่ดีที่สุดของ Python

**หมวดหมู่การตรวจสอบ**:

### CRITICAL (ต้องแก้ไข)
- ช่องโหว่ SQL/command injection
- การใช้ eval/exec ที่ไม่ปลอดภัย
- การ deserialize pickle ที่ไม่ปลอดภัย
- credential ที่ hardcode
- การโหลด YAML ที่ไม่ปลอดภัย
- การใช้ bare except ที่ซ่อนข้อผิดพลาด

### HIGH (ควรแก้ไข)
- ฟังก์ชันสาธารณะที่ขาด type hints
- mutable default arguments
- การ swallow exception แบบเงียบ
- การจัดการทรัพยากรที่ไม่ใช้ context manager
- การใช้ loop แบบ C แทน comprehension
- การใช้ type() แทน isinstance()

### MEDIUM (พิจารณาแก้ไข)
- การละเมิดรูปแบบ PEP 8
- ฟังก์ชันสาธารณะที่ขาด docstrings
- การใช้ Print แทน logging
- การดำเนินการสตริงที่ไม่มีประสิทธิภาพ
- ตัวเลขวิเศษที่ไม่ใช้ named constants
- ไม่ใช้ f-strings

**คำสั่งตรวจสอบอัตโนมัติ**:
```bash
mypy .                              # ตรวจสอบประเภท
ruff check .                        # Linting
black --check .                     # ตรวจสอบการจัดรูปแบบ
isort --check-only .                # การเรียงลำดับ import
bandit -r .                         # การสแกนความปลอดภัย
pip-audit                           # การตรวจสอบการพึ่งพา
pytest --cov=app --cov-report=term-missing  # การครอบคลุมการทดสอบ
```

**การแก้ไขปัญหาทั่วไป**:

```python
# การแก้ไข SQL injection
# ผิด
query = f"SELECT * FROM users WHERE id = {user_id}"

# ถูก - ใช้ parameterized query
query = "SELECT * FROM users WHERE id = %s"
cursor.execute(query, (user_id,))

# การแก้ไข mutable default argument
# ผิด
def process_items(items=[]):
    items.append("new")
    return items

# ถูก
def process_items(items=None):
    if items is None:
        items = []
    items.append("new")
    return items

# การใช้ context manager
# ผิด
f = open("config.json")
data = f.read()
f.close()

# ถูก
with open("config.json") as f:
    data = f.read()
```

## /go-review

**วัตถุประสงค์**: การตรวจสอบโค้ด Go อย่างครอบคลุม ตรวจสอบรูปแบบการใช้งาน ความปลอดภัย concurrency การจัดการข้อผิดพลาด และความปลอดภัย

**วิธีใช้**:
```
/go-review
```

**สถานการณ์การใช้งาน**:
- หลังเขียนหรือแก้ไขโค้ด Go
- ก่อน commit การเปลี่ยนแปลง Go
- ตรวจสอบ PR ที่มีโค้ด Go
- เรียนรู้รูปแบบการใช้งาน Go

**หมวดหมู่การตรวจสอบ**:

### CRITICAL (ต้องแก้ไข)
- ช่องโหว่ SQL/command injection
- race condition ที่ไม่มี synchronization
- goroutine leaks
- credential ที่ hardcode
- การใช้ pointer ที่ไม่ปลอดภัย
- การละเว้นข้อผิดพลาดในเส้นทางสำคัญ

### HIGH (ควรแก้ไข)
- ขาด error context wrapper
- panic แทน error return
- context ไม่แพร่กระจาย
- deadlock จาก unbuffered channel
- interface not implemented error
- ขาด mutex protection

### MEDIUM (พิจารณาแก้ไข)
- รูปแบบโค้ดที่ไม่ใช่รูปแบบการใช้งาน
- ฟังก์ชันส่งออกที่ขาด godoc comments
- การต่อสตริงที่ไม่มีประสิทธิภาพ
- slice ที่ไม่ได้ preallocate
- ไม่ใช้ table-driven tests

**คำสั่งตรวจสอบอัตโนมัติ**:
```bash
go vet ./...                         # การวิเคราะห์แบบคงที่
staticcheck ./...                    # การตรวจสอบขั้นสูง (ถ้าติดตั้ง)
golangci-lint run                   # Linting
go build -race ./...                # การตรวจจับ race
govulncheck ./...                   # ช่องโหว่ความปลอดภัย
```

## /kotlin-review

**วัตถุประสงค์**: การตรวจสอบโค้ด Kotlin อย่างครอบคลุม ตรวจสอบรูปแบบการใช้งาน ความปลอดภัยประเภท null safety coroutine และความปลอดภัย

**วิธีใช้**:
```
/kotlin-review
```

**สถานการณ์การใช้งาน**:
- หลังเขียนหรือแก้ไขโค้ด Kotlin
- ก่อน commit การเปลี่ยนแปลง Kotlin
- ตรวจสอบ PR ของโปรเจกต์ Android/KMP
- เรียนรู้รูปแบบการใช้งาน Kotlin

**หมวดหมู่การตรวจสอบ**:

### CRITICAL (ต้องแก้ไข)
- ช่องโหว่ SQL/command injection
- การใช้ force unwrap `!!` โดยไม่มีเหตุผลเพียงพอ
- การละเมิด null safety ของ platform type
- การใช้ GlobalScope (การละเมิด structured concurrency)
- credential ที่ hardcode
- deserialization ที่ไม่ปลอดภัย

### HIGH (ควรแก้ไข)
- สถานะที่ mutable (เมื่อใช้ immutable ได้)
- การเรียกใช้ blocking ใน coroutine context
- ขาดการตรวจสอบ cancellation ใน loop ยาว
- sealed class `when` ที่ไม่ครบถ้วน
- ฟังก์ชันใหญ่ (>50 บรรทัด)
- การซ้อนลึก (>4 ชั้น)

### MEDIUM (พิจารณาแก้ไข)
- Kotlin ที่ไม่ใช่รูปแบบ (รูปแบบสไตล์ Java)
- ขาด trailing comma
- การใช้ scope functions ผิดหรือซ้อนกัน
- ขาด Sequence ใน chain ของ collection ใหญ่
- ประเภทที่ชัดเจนซ้ำ

**คำสั่งตรวจสอบอัตโนมัติ**:
```bash
./gradlew build                      # การตรวจสอบการสร้าง
./gradlew detekt                     # การวิเคราะห์แบบคงที่
./gradlew ktlintCheck                # การตรวจสอบรูปแบบ
./gradlew test                       # การทดสอบ
```

## /rust-review

**วัตถุประสงค์**: การตรวจสอบโค้ด Rust อย่างครอบคลุม ตรวจสอบ ownership lifetimes การจัดการข้อผิดพลาด การใช้ unsafe และรูปแบบการใช้งาน

**วิธีใช้**:
```
/rust-review
```

**สถานการณ์การใช้งาน**:
- หลังเขียนหรือแก้ไขโค้ด Rust
- ก่อน commit การเปลี่ยนแปลง Rust
- ตรวจสอบ PR ของโปรเจกต์ Rust
- เรียนรู้รูปแบบการใช้งาน Rust

**หมวดหมู่การตรวจสอบ**:

### CRITICAL (ต้องแก้ไข)
- `unwrap()`/`expect()` ที่ไม่ได้ตรวจสอบใน production code path
- `unsafe` ที่ไม่มี `// SAFETY:` comment
- SQL injection จาก string interpolation
- command injection จาก input ที่ไม่ได้ตรวจสอบ
- credential ที่ hardcode
- use-after-free ของ raw pointers

### HIGH (ควรแก้ไข)
- `.clone()` ที่ไม่จำเป็นเพื่อให้ borrow checker พอใจ
- `String` parameter ควรใช้ `&str` หรือ `impl AsRef<str>`
- blocking ใน async context (`std::thread::sleep`)
- shared types ที่ขาด `Send`/`Sync` constraints
- wildcard `_ =>` matching บน enum ที่สำคัญ
- ฟังก์ชันใหญ่ (>50 บรรทัด)

### MEDIUM (พิจารณาแก้ไข)
- การจัดสรรที่ไม่จำเป็นใน hot path
- ขาด `with_capacity` เมื่อทราบขนาด
- การระงับ clippy warnings โดยไม่มีเหตุผลเพียงพอ
- ขาด `///` documentation บน public API
- พิจารณาใช้ `#[must_use]` สำหรับ return types ที่อาจถูกละเว้นค่า

**คำสั่งตรวจสอบอัตโนมัติ**:
```bash
cargo check                           # การตรวจสอบการสร้าง
cargo clippy -- -D warnings            # Lints
cargo fmt --check                      # การตรวจสอบรูปแบบ
cargo test                            # การทดสอบ
cargo audit                           # การตรวจสอบความปลอดภัย (ถ้ามี)
```

## /cpp-review

**วัตถุประสงค์**: การตรวจสอบโค้ด C++ อย่างครอบคลุม ตรวจสอบความปลอดภัยหน่วยความจำ รูปแบบการใช้งาน C++ สมัยใหม่ concurrency และความปลอดภัย

**วิธีใช้**:
```
/cpp-review
```

**สถานการณ์การใช้งาน**:
- หลังเขียนหรือแก้ไขโค้ด C++
- ก่อน commit การเปลี่ยนแปลง C++
- ตรวจสอบ PR ของโปรเจกต์ C++
- ตรวจสอบปัญหาความปลอดภัยหน่วยความจำ

**หมวดหมู่การตรวจสอบ**:

### CRITICAL (ต้องแก้ไข)
- bare `new`/`delete` ที่ไม่มี RAII
- buffer overflow และ use-after-free
- data race ที่ไม่มี synchronization
- command injection ผ่าน `system()`
- การอ่านตัวแปรที่ยังไม่ได้ initialize
- null pointer dereference

### HIGH (ควรแก้ไข)
- การละเมิด rule of five
- ขาด `std::lock_guard` / `std::scoped_lock`
- การจัดการ lifetime ที่ไม่เหมาะสมของ detached threads
- การใช้ C-style casts แทน `static_cast`/`dynamic_cast`
- ขาด `const` correctness

### MEDIUM (พิจารณาแก้ไข)
- การคัดลอกที่ไม่จำเป็น (ส่งค่าแทน `const&`)
- ขาด `reserve()` เมื่อทราบขนาด
- `using namespace std;` ใน header files
- ขาด `[[nodiscard]]` สำหรับ return values ที่สำคัญ
- template metaprogramming ที่ซับซ้อนเกินไป

**คำสั่งตรวจสอบอัตโนมัติ**:
```bash
clang-tidy --checks='*,-llvmlibc-*' src/*.cpp -- -std=c++17
cppcheck --enable=all --suppress=missingIncludeSystem src/
cmake --build build -- -Wall -Wextra -Wpedantic
```

## /flutter-review

**วิธีใช้**:
```
/flutter-review
```

**สถานการณ์การใช้งาน**:
- ก่อนส่ง PR ที่มีการเปลี่ยนแปลง Flutter/Dart (หลังจาก build และทดสอบผ่าน)

| Const propagation, component composition | HIGH |

| สตริงที่ hardcode (l10n) | MEDIUM |

## /fastapi-review

**วัตถุประสงค์**: การตรวจสอบแอป FastAPI ตรวจสอบสถาปัตยกรรม ความถูกต้องของ async การ injection ของ dependencies Pydantic schemas ความปลอดภัย ประสิทธิภาพ และความสามารถทดสอบ

**วิธีใช้**:
```
/fastapi-review [ไฟล์หรือไดเรกทอรี]
```

**พื้นที่ตรวจสอบ**:
- App factory, ขอบเขต routes, middleware และ exception handlers
- Pydantic request และ response schemas แยกกัน
- Database sessions, authentication, pagination และ dependency injection ของ settings
- รูปแบบ async database และ external HTTP
- CORS, authentication, rate limiting, logging และการจัดการ keys
- OpenAPI metadata และ response models ที่ documented
- การตั้งค่า test client และ dependency overrides
