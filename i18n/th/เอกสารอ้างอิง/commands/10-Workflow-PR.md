# คำสั่ง Workflow PR

## ภาพรวม

คำสั่ง workflow PR ใช้สำหรับสร้าง จัดการ และตรวจสอบ GitHub Pull Requests

## รายการคำสั่ง

### /pr

**วัตถุประสงค์**: สร้าง GitHub PR จาก branch ปัจจุบัน พร้อม commits ที่ยังไม่ได้ push

**คำอธิบาย**: ค้นพบเทมเพลต วิเคราะห์การเปลี่ยนแปลง push branch และสร้าง PR วิเคราะห์ประวัติ commit ทั้งหมด ใช้ `git diff [base-branch]...HEAD` เพื่อดูการเปลี่ยนแปลงทั้งหมด

**ข้อมูลนำเข้า**: `$ARGUMENTS` - แบบเลือก สามารถรวมชื่อ base branch และ/หรือ flags (เช่น `--draft`)

**กฎการแยกวิเคราะห์**:
- แยก flags ที่รู้จัก (เช่น `--draft`)
- ข้อความที่เหลือที่ไม่ใช่ flag ใช้เป็นชื่อ base branch
- ค่าเริ่มต้นถ้าไม่ระบุคือ `main`

**ขั้นตอนการทำงาน**:

| ขั้นตอน | คำอธิบาย |
|---|---|
| **ขั้นตอนที่ 1: VALIDATE** | ตรวจสอบเงื่อนไขเบื้องต้น (ไม่อยู่บน base branch, ไม่มีการเปลี่ยนแปลงที่ยังไม่ได้ commit, มี commits ที่ยังไม่ได้ push, ไม่มี PR อยู่) |
| **ขั้นตอนที่ 2: DISCOVER** | ค้นหา PR template, วิเคราะห์ commits, วิเคราะห์ไฟล์ที่เปลี่ยน, ตรวจสอบ planning artifacts ที่เกี่ยวข้อง |
| **ขั้นตอนที่ 3: PUSH** | Push branch (rebase ก่อนถ้าจำเป็น) |
| **ขั้นตอนที่ 4: CREATE** | สร้าง PR โดยใช้ template หรือรูปแบบเริ่มต้น |
| **ขั้นตอนที่ 5: VERIFY** | ตรวจสอบว่าสร้าง PR สำเร็จ |
| **ขั้นตอนที่ 6: OUTPUT** | รายงาน PR URL และขั้นตอนถัดไป |

**การตรวจสอบการสร้าง PR**:

| การตรวจสอบ | เงื่อนไข | การทำงานเมื่อล้มเหลว |
|---|---|---|
| ไม่อยู่บน base branch | branch ปัจจุบัน ≠ base | หยุด: "สลับไป branch ฟีเจอร์ก่อน" |
| พื้นที่ทำงานสะอาด | ไม่มีการเปลี่ยนแปลงที่ยังไม่ได้ commit | เตือน: "commit หรือ stash ก่อน" |
| มี commits ที่ยังไม่ได้ push | `git log origin/<base>..HEAD` ไม่ว่าง | หยุด: "ไม่มี commits ที่ยังไม่ได้ push" |
| ไม่มี PR อยู่ | `gh pr list` ว่าง | หยุด: "PR มีอยู่แล้ว" |

**ลำดับการค้นหา template**:
1. ไดเรกทอรี `.github/PULL_REQUEST_TEMPLATE/` (ถ้ามี)
2. `.github/PULL_REQUEST_TEMPLATE.md`
3. `.github/pull_request_template.md`
4. `docs/pull_request_template.md`

**รูปแบบ PR เริ่มต้น** (เมื่อไม่มี template):
```markdown
## Summary
<คำอธิบาย 1-2 ประโยค>

## Changes
<รายการแบบ bullet จัดกลุ่มตามพื้นที่>

## Files Changed
<ตารางไฟล์ที่เปลี่ยน>

## Testing
<วิธีทดสอบการเปลี่ยนแปลง หรือ "Needs testing">

## Related Issues
<ปัญหาที่เชื่อมโยง หรือ "None">
```

**การจัดการกรณีขอบ**:
- **ไม่มี `gh` CLI**: หยุดและแนะนำการติดตั้ง
- **ไม่ได้ authenticate**: แนะนำรัน `gh auth login`
- **branch แยก**: ใช้ `git fetch && git rebase` แล้ว push
- **PR ใหญ่ (>20 ไฟล์)**: เตือนและแนะนำแยกส่วน

**การบูรณาการกับคำสั่งอื่น**:
- ใช้ `/code-review <number>` เพื่อตรวจสอบ PR
- ใช้ `gh pr merge <number>` เพื่อรวม PR ที่พร้อม
- ใช้ `/plan-prd` หรือ `/plan` เพื่อสร้าง planning artifacts ที่เกี่ยวข้อง

### /review-pr

**วัตถุประสงค์**: ตรวจสอบ GitHub PR หรือการเปลี่ยนแปลงที่ยังไม่ได้ commit ในเครื่อง

**คำอธิบาย**: ตรวจสอบ Pull Request บน GitHub หรือการเปลี่ยนแปลงที่ยังไม่ได้ commit ในเครื่อง โหมดตรวจสอบ PR ปรับจาก PRPs-agentic-eng

**ข้อมูลนำเข้า**: `$ARGUMENTS` - สามารถเป็นหมายเลข PR, URL ของ PR หรือว่างเปล่า (ตรวจสอบในเครื่อง)

---

#### การเลือกโหมด

| ข้อมูลนำเข้า | การทำงาน |
|---|---|
| หมายเลข PR (เช่น `42`) | ใช้เป็นหมายเลข PR |
| URL (เช่น `github.com/.../pull/42`) | แยกหมายเลข PR |
| ชื่อ branch | ค้นหา PR ผ่าน `gh pr list --head <branch>` |
| ว่างเปล่า | ใช้โหมดตรวจสอบในเครื่อง |

---

#### โหมดตรวจสอบในเครื่อง

**ขั้นตอนการทำงาน**:
1. **GATHER** - รวบรวมการเปลี่ยนแปลง: `git diff --name-only HEAD`
2. **REVIEW** - ตรวจสอบแต่ละไฟล์ (ปัญหาความปลอดภัย, ปัญหาคุณภาพ, แนวปฏิบัติที่ดีที่สุด)
3. **REPORT** - สร้างรายงานพร้อมระดับความรุนแรงและคำแนะนำการแก้ไข

**รายการตรวจสอบ**:

| หมวดหมู่ | รายการตรวจสอบ |
|---|---|
| **ปัญหาความปลอดภัย (CRITICAL)** | credential ที่ hardcode, SQL injection, XSS, ขาดการตรวจสอบ input, การพึ่งพาที่ไม่ปลอดภัย, path traversal |
| **คุณภาพโค้ด (HIGH)** | ฟังก์ชัน >50 บรรทัด, ไฟล์ >800 บรรทัด, การซ้อน >4 ชั้น, ขาดการจัดการข้อผิดพลาด, console.log, TODO/FIXME |
| **แนวปฏิบัติที่ดี (MEDIUM)** | รูปแบบการเปลี่ยนแปลง, การใช้ emoji, ขาดการทดสอบสำหรับโค้ดใหม่, ปัญหาความสามารถเข้าถึง |

**การตัดสินใจ**:

| เงื่อนไข | การตัดสินใจ |
|---|---|
| มีปัญหา CRITICAL หรือ HIGH | **BLOCK** - ป้องกันการ commit |
| มีเฉพาะปัญหา MEDIUM/LOW | ผ่านแต่พร้อมความคิดเห็น |
| ไม่มีปัญหาและผ่านการตรวจสอบ | **APPROVE** |

---

#### โหมดตรวจสอบ PR

**ขั้นตอนการทำงาน**:
1. **FETCH** - รับ metadata และ diff ของ PR
2. **CONTEXT** - สร้างบริบทสำหรับตรวจสอบ (กฎโปรเจกต์, planning artifacts, ไฟล์ที่เปลี่ยน)
3. **REVIEW** - อ่านไฟล์ทั้งหมด ใช้ 7 หมวดหมู่การตรวจสอบ
4. **VALIDATE** - รันคำสั่งตรวจสอบของประเภทโปรเจกต์
5. **DECIDE** - สร้างคำแนะนำตามการค้นพบ
6. **REPORT** - สร้าง artifact การตรวจสอบ
7. **PUBLISH** - เผยแพร่การตรวจสอบไปยัง GitHub
8. **OUTPUT** - รายงานให้ผู้ใช้

**7 หมวดหมู่การตรวจสอบ**:

| หมวดหมู่ | รายการตรวจสอบ |
|---|---|
| **ความถูกต้อง** | ข้อผิดพลาดทางตรรกะ, กรณีขอบเขต, การจัดการ null, race conditions |
| **ความปลอดภัยประเภท** | ประเภทไม่ตรงกัน, การแปลงประเภทที่ไม่ปลอดภัย, การใช้ `any`, ขาด generics |
| **การปฏิบัติตามรูปแบบ** | การ tuân thủ ข้อตกลงโปรเจกต์ (การตั้งชื่อ, โครงสร้างไฟล์, การจัดการข้อผิดพลาด, import) |
| **ความปลอดภัย** | injection, ช่องว่างการยืนยันตัวตน, เปิดเผย secret, SSRF, path traversal, XSS |
| **ประสิทธิภาพ** | N+1 queries, ขาดดัชนี, loop ไม่มีขอบเขต, memory leaks, payload ใหญ่ |
| **ความสมบูรณ์** | ขาดการทดสอบ, ขาดการจัดการข้อผิดพลาด, การย้ายที่ไม่สมบูรณ์, ขาดเอกสาร |
| **การบำรุงรักษา** | dead code, ตัวเลขวิเศษ, การซ้อนลึก, การตั้งชื่อไม่ชัดเจน, ขาดประเภท |

**การตรวจจับคำสั่งตรวจสอบ**:

| ประเภทโปรเจกต์ | ไฟล์ตรวจจับ | คำสั่งตรวจสอบ |
|---|---|---|
| Node.js/TypeScript | `package.json` | `npm run typecheck`, `npm run lint`, `npm test`, `npm run build` |
| Rust | `Cargo.toml` | `cargo clippy`, `cargo test`, `cargo build` |
| Go | `go.mod` | `go vet ./...`, `go test ./...`, `go build ./...` |
| Python | `pyproject.toml` | `pytest` |

**ระดับการตัดสินใจ**:

| ระดับ | ความหมาย | การทำงาน |
|---|---|---|
| CRITICAL | ช่องโหว่ความปลอดภัยหรือความเสี่ยงสูญเสียข้อมูล | ต้องแก้ไขก่อนรวม |
| HIGH | ข้อผิดพลาดหรือข้อผิดพลาดทางตรรกะที่อาจทำให้เกิดปัญหา | ควรแก้ไขก่อนรวม |
| MEDIUM | ขาดคุณภาพโค้ดหรือแนวปฏิบัติที่ดี | แนะนำให้แก้ไข |
| LOW | ปัญหารูปแบบหรือคำแนะนำเล็กน้อย | เลือกได้ |

**เผยแพร่การตรวจสอบไปยัง GitHub**:
```bash
# APPROVE
gh pr review <NUMBER> --approve --body "<summary>"

# REQUEST_CHANGES
gh pr review <NUMBER> --request-changes --body "<summary with required fixes>"

# COMMENT only (draft PR หรือข้อมูล)
gh pr review <NUMBER> --comment --body "<summary>"
```

### /multi-workflow (อยู่ระหว่างพัฒนา)

> (อยู่ระหว่างพัฒนา) รายละเอียดคำสั่งรอการเพิ่มเติม

**วัตถุประสงค์**: การพัฒนาทำงานร่วมกันหลายโมเดล

**คำอธิบาย**: ประสานงานโมเดล AI หลายตัวสำหรับการพัฒนาร่วมกัน

---

### /multi-backend (อยู่ระหว่างพัฒนา)

> (อยู่ระหว่างพัฒนา) รายละเอียดคำสั่งรอการเพิ่มเติม

**วัตถุประสงค์**: การพัฒนา backend หลายโมเดล

**คำอธิบาย**: การพัฒนาร่วมกันหลายโมเดลโดยเน้น backend

---

### /multi-frontend (อยู่ระหว่างพัฒนา)

> (อยู่ระหว่างพัฒนา) รายละเอียดคำสั่งรอการเพิ่มเติม

**วัตถุประสงค์**: การพัฒนา frontend หลายโมเดล

**คำอธิบาย**: การพัฒนาร่วมกันหลายโมเดลโดยเน้น frontend

---

### /multi-execute (อยู่ระหว่างพัฒนา)

> (อยู่ระหว่างพัฒนา) รายละเอียดคำสั่งรอการเพิ่มเติม

**วัตถุประสงค์**: การรันพร้อมกันหลายโมเดล

**คำอธิบาย**: ประสานงานโมเดลหลายตัวเพื่อรันงานขนาน

## คำสั่งที่เกี่ยวข้อง

- `/pr` - สร้าง PR
- `/review-pr` - ตรวจสอบ PR
- `/multi-workflow` - การทำงานร่วมกันหลายโมเดล
