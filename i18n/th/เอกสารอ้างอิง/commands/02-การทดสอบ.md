# คำสั่งที่เกี่ยวข้องกับการทดสอบ

เอกสารนี้แนะนำคำสั่งเฉพาะทางสำหรับการทดสอบใน ECC ซึ่งรองรับหลายภาษาโปรแกรมและเฟรมเวิร์กการทดสอบ

---

## /go-test

**วัตถุประสงค์**: ขั้นตอนการทำงาน TDD สำหรับภาษา Go ใช้ table-driven tests เขียน测试ก่อนแล้วค่อยเขียนโค้ด และตรวจสอบการครอบคลุม 80%+

**วิธีใช้**:
```
/go-test [คำอธิบายฟังก์ชัน]
```

**สถานการณ์การใช้งาน**:
- สร้างฟังก์ชัน Go ใหม่
- เพิ่มการครอบคลุมการทดสอบสำหรับโค้ด Go ที่มีอยู่
- แก้ไขข้อบกพร่อง (เขียนการทดสอบที่ล้มเหลวก่อน)
- เรียนรู้วิธีการ TDD ของ Go

**ตัวอย่างขั้นตอนการทำงาน**:
```go
// ขั้นตอนที่ 1: กำหนด interface
func ValidateEmail(email string) error {
    panic("not implemented")
}

// ขั้นตอนที่ 2: เขียน table-driven test (RED)
tests := []struct {
    name    string
    email   string
    wantErr bool
}{
    {"simple email", "user@example.com", false},
    {"empty string", "", true},
    {"no at sign", "userexample.com", true},
}

// ขั้นตอนที่ 3: รันการทดสอบ - ตรวจสอบ FAIL

// ขั้นตอนที่ 4: เขียนโค้ดขั้นต่ำน้อยที่สุด (GREEN)
func ValidateEmail(email string) error {
    if email == "" {
        return ErrEmailEmpty
    }
    if !emailRegex.MatchString(email) {
        return ErrEmailInvalid
    }
    return nil
}

// ขั้นตอนที่ 5: ตรวจสอบ PASS

// ขั้นตอนที่ 6: ตรวจสอบการครอบคลุม
go test -cover ./...
// coverage: 100.0%
```

**คำสั่งสำหรับการครอบคลุม**:
```bash
go test -cover ./...                    # การครอบคลุมพื้นฐาน
go test -coverprofile=coverage.out      # สร้างไฟล์การครอบคลุม
go tool cover -html=coverage.out        # ดูในเบราว์เซอร์
go test -race -cover ./...              # ตรวจจับ race condition
```

## /kotlin-test

**วัตถุประสงค์**: ขั้นตอนการทำงาน TDD สำหรับภาษา Kotlin ใช้เฟรมเวิร์ก Kotest และ MockK เขียนการทดสอบก่อนแล้วค่อยเขียนโค้ด และตรวจสอบการครอบคลุม 80%+ (Kover)

**วิธีใช้**:
```
/kotlin-test [คำอธิบายฟังก์ชัน]
```

**สถานการณ์การใช้งาน**:
- สร้างฟังก์ชันหรือคลาส Kotlin ใหม่
- เพิ่มการครอบคลุมการทดสอบสำหรับโปรเจกต์ Android/KMP
- แก้ไขข้อบกพร่อง (เขียนการทดสอบที่ล้มเหลวก่อน)
- เรียนรู้วิธีการ TDD ของ Kotlin

**รูปแบบการทดสอบ Kotest**:
```kotlin
// StringSpec (ง่ายที่สุด)
class CalculatorTest : StringSpec({
    "add two positive numbers" {
        Calculator.add(2, 3) shouldBe 5
    }
})

// FunSpec (การทดสอบหน่วยมาตรฐาน)
class RegistrationValidatorTest : FunSpec({
    test("valid registration returns Valid") {
        val request = RegistrationRequest(
            name = "Alice",
            email = "alice@example.com",
            password = "SecureP@ss1"
        )
        val result = validateRegistration(request)
        result.shouldBeInstanceOf<ValidationResult.Valid>()
    }
})

// BehaviorSpec (รูปแบบ BDD)
class OrderServiceTest : BehaviorSpec({
    Given("a valid order") {
        When("placed") {
            Then("should be confirmed") { /* ... */ }
        }
    }
})
```

**การทดสอบ Coroutine**:
```kotlin
test("concurrent fetch completes") {
    runTest {
        val result = service.fetchAll()
        result.shouldNotBeEmpty()
    }
}
```

**คำสั่งสำหรับการครอบคลุม**:
```bash
./gradlew koverHtmlReport     # รายงาน HTML
./gradlew koverVerify         # ตรวจสอบเกณฑ์การครอบคลุม
./gradlew koverXmlReport      # รายงาน XML สำหรับ CI
./gradlew test                # รันการทดสอบ
```

## /rust-test

**วัตถุประสงค์**: ขั้นตอนการทำงาน TDD สำหรับภาษา Rust ใช้ `#[test]` rstest, proptest และ mockall เขียนการทดสอบก่อนแล้วค่อยเขียนโค้ด และตรวจสอบการครอบคลุม 80%+ (cargo-llvm-cov)

**วิธีใช้**:
```
/rust-test [คำอธิบายฟังก์ชัน]
```

**สถานการณ์การใช้งาน**:
- สร้างฟังก์ชัน, method หรือ trait ของ Rust ใหม่
- เพิ่มการครอบคลุมการทดสอบสำหรับโปรเจกต์ Rust
- แก้ไขข้อบกพร่อง (เขียนการทดสอบที่ล้มเหลวก่อน)
- เรียนรู้วิธีการ TDD ของ Rust

**รูปแบบการทดสอบ**:
```rust
// การทดสอบหน่วย
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn adds_two_numbers() {
        assert_eq!(add(2, 3), 5);
    }

    #[test]
    fn handles_error() -> Result<(), Box<dyn std::error::Error>> {
        let result = parse_config(r#"port = 8080"#)?;
        assert_eq!(result.port, 8080);
        Ok(())
    }
}

// การทดสอบแบบ parameterize (rstest)
#[rstest]
#[case("hello", 5)]
#[case("", 0)]
#[case("rust", 4)]
fn test_string_length(#[case] input: &str, #[case] expected: usize) {
    assert_eq!(input.len(), expected);
}

// การทดสอบแบบ async
#[tokio::test]
async fn fetches_data_successfully() {
    let client = TestClient::new().await;
    let result = client.get("/data").await;
    assert!(result.is_ok());
}

// การทดสอบแบบ property (proptest)
proptest! {
    #[test]
    fn encode_decode_roundtrip(input in ".*") {
        let encoded = encode(&input);
        let decoded = decode(&encoded).unwrap();
        assert_eq!(input, decoded);
    }
}
```

**คำสั่งสำหรับการครอบคลุม**:
```bash
cargo llvm-cov                    # สรุปการครอบคลุม
cargo llvm-cov --html             # รายงาน HTML
cargo llvm-cov --fail-under-lines 80  # ล้มเหลวถ้าต่ำกว่าเกณฑ์
cargo test                        # รันการทดสอบ
```

## /cpp-test

**วัตถุประสงค์**: ขั้นตอนการทำงาน TDD สำหรับภาษา C++ ใช้ GoogleTest/GoogleMock และ CMake/CTest เขียนการทดสอบก่อนแล้วค่อยเขียนโค้ด และตรวจสอบการครอบคลุม (gcov/lcov)

**วิธีใช้**:
```
/cpp-test [คำอธิบายฟังก์ชัน]
```

**สถานการณ์การใช้งาน**:
- สร้างคลาสหรือฟังก์ชัน C++ ใหม่
- เพิ่มการครอบคลุมการทดสอบ
- แก้ไขข้อบกพร่อง (เขียนการทดสอบที่ล้มเหลวก่อน)
- เรียนรู้วิธีการ TDD ของ C++

**รูปแบบ GoogleTest**:
```cpp
// การทดสอบพื้นฐาน
TEST(SuiteName, TestName) {
    EXPECT_EQ(add(2, 3), 5);
    EXPECT_NE(result, nullptr);
    EXPECT_TRUE(is_valid);
    EXPECT_THROW(func(), std::invalid_argument);
}

// Fixture
class DatabaseTest : public ::testing::Test {
protected:
    void SetUp() override { db_ = create_test_db(); }
    void TearDown() override { db_.reset(); }
    std::unique_ptr<Database> db_;
};

TEST_F(DatabaseTest, InsertsRecord) {
    db_->insert("key", "value");
    EXPECT_EQ(db_->get("key"), "value");
}

// การทดสอบแบบ parameterized
class PrimeTest : public ::testing::TestWithParam<std::pair<int, bool>> {};

TEST_P(PrimeTest, ChecksPrimality) {
    auto [input, expected] = GetParam();
    EXPECT_EQ(is_prime(input), expected);
}

INSTANTIATE_TEST_SUITE_P(Primes, PrimeTest,
    ::testing::Values(
        std::make_pair(2, true),
        std::make_pair(4, false),
        std::make_pair(7, true)
    ));
```

**คำสั่งสำหรับการครอบคลุม**:
```bash
cmake -DCMAKE_CXX_FLAGS="--coverage" -B build
cmake --build build
ctest --test-dir build
lcov --capture --directory build --output-file coverage.info
genhtml coverage.info --output-directory coverage_html
```

## /flutter-test

**วัตถุประสงค์**: การทดสอบ Flutter/Dart รายงานการทดสอบที่ล้มเหลวและแก้ไขปัญหาการทดสอบทีละขั้นตอน ครอบคลุมการทดสอบหน่วย คอมโพเนนต์ golden และการทดสอบแบบบูรณาการ

**วิธีใช้**:
```
/flutter-test                # รันการทดสอบทั้งหมด
/flutter-test --coverage     # พร้อมการครอบคลุม
/flutter-test test/file.dart # รันไฟล์การทดสอบเฉพาะ
```

**สถานการณ์การใช้งาน**:
- ตรวจสอบว่าฟังก์ชันใหม่ไม่ทำให้สิ่งที่มีอยู่เสียหาย
- ตรวจสอบการครอบคลุมการทดสอบของโค้ดใหม่
- แก้ไขไฟล์การทดสอบเฉพาะ
- ก่อนส่ง PR

**การทดสอบที่ล้มเหลวทั่วไปและการแก้ไข**:

| ประเภทการล้มเหลว | การแก้ไขทั่วไป |
|----------|----------|
| `Expected: <X> Actual: <Y>` | อัปเดต assertion หรือแก้ไข implementation |
| `Widget not found` | แก้ไข finder selector หรืออัปเดตชื่อ widget |
| `Golden file not found` | รัน `flutter test --update-goldens` เพื่อสร้าง |
| `MissingPluginException` | Mock platform channel ในการตั้งค่าการทดสอบ |
| `pumpAndSettle timed out` | ใช้ `pump(Duration)` แทน |

## /test-coverage

**วัตถุประสงค์**: วิเคราะห์การครอบคลุมของการทดสอบ ระบุช่องว่าง และสร้างการทดสอบที่ขาดหายเพื่อให้ถึงเกณฑ์เป้าหมาย (ค่าเริ่มต้น 80%)

**วิธีใช้**:
```
/test-coverage
```

**ขั้นตอนการทำงาน**:
1. **ตรวจจับเฟรมเวิร์กการทดสอบ** - เลือกคำสั่งการครอบคลุมตามประเภทโปรเจกต์
2. **วิเคราะห์รายงานการครอบคลุม** - แสดงรายการไฟล์ที่ต่ำกว่า 80%
3. **สร้างการทดสอบที่ขาดหาย** - สร้างการทดสอบตามลำดับความสำคัญ
4. **ตรวจสอบ** - รันการครอบคลุมอีกครั้งเพื่อยืนยันการปรับปรุง

**การตรวจจับเฟรมเวิร์กการทดสอบ**:

| ตัวบ่งชี้ | คำสั่งการครอบคลุม |
|--------|----------|
| `jest.config.*` | `npx jest --coverage` |
| `vitest.config.*` | `npx vitest run --coverage` |
| `pytest.ini` | `pytest --cov=src --cov-report=json` |
| `Cargo.toml` | `cargo llvm-cov --json` |
| `go.mod` | `go test -coverprofile=coverage.out` |

## /python-testing

**วัตถุประสงค์**: ขั้นตอนการทำงานการทดสอบ Python ใช้ pytest รองรับการวิเคราะห์การครอบคลุม

**วิธีใช้**:
```
/python-testing [คำอธิบายฟังก์ชัน]
```

**สถานการณ์การใช้งาน**:
- สร้างฟังก์ชัน Python ใหม่
- เพิ่มการครอบคลุมการทดสอบ
- แก้ไขข้อบกพร่อง
- การทดสอบโปรเจกต์ FastAPI/Django/Flask

**คุณสมบัติ Pytest**:
```python
# การทดสอบแบบ parameterize
@pytest.mark.parametrize("input,expected", [
    ("hello", 5),
    ("", 0),
    ("world", 5),
])
def test_string_length(input, expected):
    assert len(input) == expected

# Fixtures
@pytest.fixture
def client():
    with TestClient(app) as c:
        yield c

# การทดสอบแบบ async
@pytest.mark.asyncio
async def test_async_fetch():
    result = await fetch_data()
    assert result is not None
```
