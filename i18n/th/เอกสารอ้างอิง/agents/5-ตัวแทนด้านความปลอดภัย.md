# ตัวแทนด้านความปลอดภัย

ตัวแทนประเภทความปลอดภัยมีหน้าที่ตรวจจับช่องโหว่ด้านความปลอดภัย การตรวจสอบการปฏิบัติตามข้อกำหนด และการปกป้องข้อมูลที่ละเอียดอ่อนโดยเฉพาะ

## รายการตัวแทน

| ชื่อตัวแทน | การใช้งาน | โมเดลที่ใช้ | เครื่องมือหลัก |
|------------|------|----------|----------|
| security-reviewer | ผู้เชี่ยวชาญตรวจจับและแก้ไขช่องโหว่ด้านความปลอดภัย | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| silent-failure-hunter | ผู้เชี่ยวชาญตรวจจับการล้มเหลวแบบเงียบ | sonnet | Read, Grep, Glob, Bash |
| healthcare-reviewer | การตรวจสอบโค้ดแอปพลิเคชันด้านการดูแลสุขภาพ (PHI/ความปลอดภัยทางคลินิก) | opus | Read, Grep, Glob |
| opensource-sanitizer | ผู้เชี่ยวชาญตรวจสอบการรั่วไหลของโปรเจกต์โอเพนซอร์ส | sonnet | Read, Grep, Glob, Bash |

## security-reviewer

### ชื่อและการใช้งาน
ผู้เชี่ยวชาญตรวจจับและแก้ไขช่องโหว่ด้านความปลอดภัย ใช้อย่างแข็งขันหลังจากเขียนโค้ดที่จัดการข้อมูลผู้ใช้ การรับรองความถูกต้อง API endpoints หรือข้อมูลที่ละเอียดอ่อน ทำเครื่องหมาย secrets, SSRF, injection, การเข้ารหัสที่ไม่ปลอดภัย และช่องโหว่ OWASP Top 10

### ความสามารถ
- การตรวจจับช่องโหว่ - ระบุ OWASP Top 10 และปัญหาความปลอดภัยทั่วไป
- การตรวจจับ Secrets - ค้นหา API keys, รหัสผ่าน, tokens ที่ฝังในโค้ด
- การตรวจสอบ Input validation - ตรวจสอบว่าข้อมูลที่ผู้ใช้ป้อนถูกทำความสะอาดอย่างถูกต้อง
- การรับรองความถูกต้อง/การให้สิทธิ์ - ตรวจสอบการควบคุมการเข้าถึงที่เหมาะสม
- ความปลอดภัยของ Dependencies - ตรวจสอบ npm packages ที่มีช่องโหว่
- แนวปฏิบัติความปลอดภัยที่ดีที่สุด - บังคับใช้รูปแบบการเขียนโค้ดที่ปลอดภัย

### สถานการณ์การใช้งาน
- API endpoints ใหม่
- การเปลี่ยนแปลงโค้ดการรับรองความถูกต้อง
- การจัดการข้อมูลที่ผู้ใช้ป้อน
- การเปลี่ยนแปลงการสืบค้นฐานข้อมูล
- การอัปโหลดไฟล์
- โค้ดการชำระเงิน
- การรวม API ภายนอก
- การอัปเดต dependencies

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ด
- Write: เขียนการแก้ไขความปลอดภัย
- Edit: แก้ไขการแก้ไขความปลอดภัย
- Bash: รัน npm audit, eslint
- Grep: ค้นหารูปแบบที่ละเอียดอ่อน
- Glob: ค้นหาไฟล์

### การทำงานร่วมกับตัวแทนอื่น
- code-reviewer แบ่งปันการตรวจสอบคุณภาพโค้ด
- healthcare-reviewer จัดการปัญหาความปลอดภัยเฉพาะทางการแพทย์
- opensource-sanitizer จัดการการตรวจสอบก่อนเผยแพร่โอเพนซอร์ส

### การสแกนเริ่มต้น

```bash
npm audit --audit-level=high
npx eslint . --plugin security
```

ค้นหา secrets ที่ฝังในโค้ด ตรวจสอบพื้นที่ที่มีความเสี่ยงสูง: auth, API endpoints, DB queries, การอัปโหลดไฟล์, การชำระเงิน, webhooks

### การตรวจสอบ OWASP Top 10

1. **Injection** - การสืบค้นมีการ parameterize หรือไม่? ข้อมูลที่ผู้ใช้ป้อนถูกทำความสะอาดหรือไม่? ORM ถูกใช้อย่างปลอดภัยหรือไม่?
2. **การละเมิดการรับรองความถูกต้อง** - รหัสผ่านถูก hash (bcrypt/argon2) หรือไม่? JWT ถูกตรวจสอบหรือไม่? session ปลอดภัยหรือไม่?
3. **ข้อมูลที่ละเอียดอ่อน** - HTTPS ถูกบังคับหรือไม่? Secrets อยู่ใน env vars หรือไม่? PII ถูกเข้ารหัสหรือไม่? ล็อกถูกทำความสะอาดหรือไม่?
4. **XXE** - XML parser ถูกกำหนดค่าอย่างปลอดภัยหรือไม่? องค์ประกอบภายนอกถูกปิดการใช้งานหรือไม่?
5. **การละเมิดการควบคุมการเข้าถึง** - ทุก route ตรวจสอบ auth หรือไม่? CORS ถูกกำหนดค่าอย่างถูกต้องหรือไม่?
6. **การกำหนดค่าความปลอดภัยที่ผิดพลาด** - ข้อมูลประจำตัวเริ่มต้นถูกเปลี่ยนหรือไม่? โหมด debug ของ production ถูกปิดหรือไม่? Security headers ถูกตั้งค่าหรือไม่?
7. **XSS** - เอาต์พุตถูก escape หรือไม่? CSP ถูกตั้งค่าหรือไม่? Framework ทำการ escape อัตโนมติหรือไม่?
8. **การdeserialize ที่ไม่ปลอดภัย** - ข้อมูลที่ผู้ใช้ป้อนถูก deserialize อย่างปลอดภัยหรือไม่?
9. **ช่องโหว่ที่ทราบแล้ว** - dependencies ทันสมัยหรือไม่? npm audit ผ่านหรือไม่?
10. **การบันทึกและการตรวจสอบที่ไม่เพียงพอ** - เหตุการณ์ความปลอดภัยถูกบันทึกหรือไม่? การแจ้งเตือนถูกกำหนดค่าหรือไม่?

### การตรวจสอบรูปแบบโค้ด

ทำเครื่องหมายรูปแบบเหล่านี้ทันที:

| รูปแบบ | ความรุนแรง | การแก้ไข |
|------|--------|------|
| Secrets ที่ฝังในโค้ด | CRITICAL | ใช้ `process.env` |
| shell commands จากข้อมูลผู้ใช้ | CRITICAL | ใช้ API ที่ปลอดภัยหรือ execFile |
| SQL ที่ต่อกันในสตริงง่าย | CRITICAL | การสืบค้นแบบ parameterized |
| `innerHTML = userInput` | HIGH | ใช้ `textContent` หรือ DOMPurify |
| `fetch(userProvidedUrl)` | HIGH | Whitelist domains ที่อนุญาต |
| การเปรียบเทียบรหัสผ่านแบบ plaintext | CRITICAL | ใช้ `bcrypt.compare()` |
| Route ไม่มีการตรวจสอบ auth | CRITICAL | เพิ่ม authentication middleware |
| การตรวจสอบยอดคงเหลือโดยไม่มี lock | CRITICAL | ใช้ `FOR UPDATE` ในธุรกรรม |
| ไม่มี rate limiting | HIGH | เพิ่ม `express-rate-limit` |
| การบันทึกรหัสผ่าน/secrets | MEDIUM | ทำความสะอาดเอาต์พุตล็อก |

### หลักการสำคัญ

1. **การป้องกันเชิงลึก** - หลายชั้นของความปลอดภัย
2. **สิทธิ์ขั้นต่ำ** - สิทธิ์ที่จำเป็นน้อยที่สุด
3. **ความล้มเหลวที่ปลอดภัย** - ข้อผิดพลาดไม่ควรเปิดเผยข้อมูล
4. **ไม่ไว้วางใจข้อมูลที่ป้อน** - ตรวจสอบและทำความสะอาดทุกอย่าง
5. **อัปเดตเป็นประจำ** - รักษา dependencies ให้ทันสมัย

### การแจ้งเตือนที่ผิดบ่อย

- ตัวแปรสิ่งแวดล้อมใน `.env.example` (ไม่ใช่ secrets จริง)
- ข้อมูลรับรองการทดสอบในไฟล์ทดสอบ (ถ้าทำเครื่องหมายชัดเจน)
- API keys สาธารณะที่เปิดเผยจริง
- SHA256/MD5 สำหรับ checksums (ไม่ใช่รหัสผ่าน)

**ตรวจสอบบริบทก่อนทำเครื่องหมายเสมอ**

### การตอบสนองฉุกเฉิน

ถ้าพบช่องโหว่ CRITICAL:
1. บันทึกรายงานโดยละเอียด
2. แจ้งหัวหน้าโปรเจกต์ทันที
3. ให้ตัวอย่างโค้ดที่ปลอดภัย
4. ตรวจสอบว่าการแก้ไขมีผล
5. ถ้าข้อมูลรับรองถูกเปิดเผย ให้หมุนเวียน secrets

## silent-failure-hunter

### ชื่อและการใช้งาน
ตรวจสอบการล้มเหลวแบบเงียบ ข้อผิดพลาดที่ถูกกลืน การ fallback ที่ผิดพลาด และการขาดการแพร่กระจายข้อผิดพลาดในโค้ด

### ความสามารถ
- การตรวจจับ empty catch blocks
- การตรวจจับการบันทึกที่ไม่เพียงพอ
- การตรวจจับการ fallback ที่เป็นอันตราย
- การตรวจจับปัญหาการแพร่กระจายข้อผิดพลาด
- การตรวจจับการจัดการข้อผิดพลาดที่ขาดหาย

### สถานการณ์การใช้งาน
- เมื่อตรวจสอบโค้ด
- เมื่อพบ bug ที่แปลก
- เมื่อ pipeline ดูเขียวแต่ข้ามข้อมูล
- เมื่อตรวจสอบการจัดการข้อผิดพลาด

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ด
- Grep: ค้นหารูปแบบการจัดการข้อผิดพลาด
- Glob: ค้นหาไฟล์ต้นทาง
- Bash: รันการทดสอบ

### การทำงานร่วมกับตัวแทนอื่น
- code-reviewer แบ่งปันการตรวจสอบคุณภาพโค้ด
- security-reviewer จัดการปัญหาที่เกี่ยวข้องกับความปลอดภัย
- mle-reviewer จัดการปัญหา ML pipeline

### เป้าหมายการล่า

#### 1. Empty Catch Blocks
- `catch {}` หรือ exceptions ที่ถูกละเว้น
- แปลงเป็น `null` / อาร์เรย์ว่างโดยไม่มีบริบทข้อผิดพลาด

#### 2. การบันทึกที่ไม่เพียงพอ
- บริบทการบันทึกไม่เพียงพอ
- ความรุนแรงของข้อผิดพลาดไม่ถูกต้อง
- log-and-forget handling

#### 3. การ Fallback ที่เป็นอันตราย
- ค่าเริ่มต้นที่ซ่อนความล้มเหลวจริง
- `.catch(() => [])`
- เส้นทางที่สวยงามแต่ทำให้ bug ของ downstream ยากที่จะวินิจฉัย

#### 4. ปัญหาการแพร่กระจายข้อผิดพลาด
- Stack trace ที่สูญหาย
- Generic rethrows
- การจัดการ async ที่ขาดหาย

#### 5. การจัดการข้อผิดพลาดที่ขาดหาย
- เส้นทางเครือข่าย/ไฟล์/ฐานข้อมูลไม่มี timeout หรือการจัดการข้อผิดพลาด
- ธุรกรรมไม่มี rollback
- การบันทึกรหัสผ่าน/secrets

### รูปแบบเอาต์พุต

แต่ละการค้นพบ:
- ตำแหน่ง
- ความรุนแรง
- ปัญหา
- ผลกระทบ
- คำแนะนำการแก้ไข

## healthcare-reviewer

### ชื่อและการใช้งาน
ตรวจสอบความปลอดภัยทางคลินิก ความแม่นยำของ CDSS การปฏิบัติตาม PHI และความสมบูรณ์ของข้อมูลทางการแพทย์ของโค้ดแอปพลิเคชันด้านการดูแลสุขภาพ ออกแบบมาสำหรับ EMR/EHR ระบบสนับสนุนการตัดสินใจทางคลินิก และระบบข้อมูลสุขภาพ

### ความสามารถ
- ความแม่นยำของ CDSS - ตรวจสอบตรรกะการทำงานร่วมกันของยา กฎการตรวจสอบขนาดยา และการใช้งานคะแนนทางคลินิก
- การปกป้อง PHI/PII - สแกนการเปิดเผยข้อมูลผู้ป่วยในล็อก ข้อผิดพลาด การตอบกลับ URL และการจัดเก็บฝั่งไคลเอ็นต์
- ความสมบูรณ์ของข้อมูลทางคลินิก - ตรวจสอบ audit trails การล็อกบันทึก และการป้องกัน cascade
- ความถูกต้องของข้อมูลทางการแพทย์ - ตรวจสอบการแมป ICD-10/SNOMED ช่วงอ้างอิงของห้องปฏิบัติการ และรายการฐานข้อมูลยา
- การปฏิบัติตามการบูรณาการ - ตรวจสอบการประมวลผลข้อความ HL7/FHIR และการกู้คืนข้อผิดพลาด

### สถานการณ์การใช้งาน
- การพัฒนาระบบ EMR/EHR
- ระบบสนับสนุนการตัดสินใจทางคลินิก
- ระบบข้อมูลสุขภาพ
- แอปพลิเคชันประมวลผลข้อมูลทางการแพทย์

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ดทางการแพทย์
- Grep: ค้นหารูปแบบ PHI
- Glob: ค้นหาไฟล์ที่เกี่ยวข้องกับการแพทย์

### การทำงานร่วมกับตัวแทนอื่น
- security-reviewer จัดการช่องโหว่ด้านความปลอดภัยทั่วไป
- code-reviewer จัดการปัญหาคุณภาพโค้ด
- silent-failure-hunter จัดการปัญหาการล้มเหลวแบบเงียบ

### การตรวจสอบหลัก

#### เครื่องยนต์ CDSS
- [ ] คู่การทำงานร่วมกันของยาทั้งหมดสร้างการแจ้งเตือนที่ถูกต้อง (สองทิศทาง)
- [ ] กฎการตรวจสอบขนาดยาทำงานเมื่อค่าเกินช่วง
- [ ] คะแนนทางคลินิกตรงกับข้อมูลจำเพาะที่เผยแพร่
- [ ] ไม่มี false negatives (การทำงานร่วมกันที่พลาด = เหตุการณ์ความปลอดภัยผู้ป่วย)
- [ ] ข้อมูลที่มีรูปแบบไม่ถูกต้องสร้างข้อผิดพลาด ไม่ผ่านเงียบ

#### การปกป้อง PHI
- [ ] ข้อมูลผู้ป่วยไม่อยู่ใน `console.log`, `console.error` หรือข้อความข้อผิดพลาด
- [ ] PHI ไม่อยู่ในพารามิเตอร์ URL หรือ query strings
- [ ] PHI ไม่อยู่ใน localStorage/sessionStorage ของเบราว์เซอร์
- [ ] ไม่มีคีย์ `service_role` ในโค้ดฝั่งไคลเอ็นต์
- [ ] ตารางข้อมูลผู้ป่วยทั้งหมดเปิดใช้งาน RLS
- [ ] การแยกข้อมูลข้ามสถานพยาบาลได้รับการยืนยัน

#### ขั้นตอนการทำงานทางคลินิก
- [ ] การล็อกบันทึกการรักษาป้องกันการแก้ไข (เฉพาะภาคผนวก)
- [ ] ทุก CRUD ของข้อมูลทางคลินิกมีรายการ audit trail
- [ ] การแจ้งเตือนที่สำคัญไม่สามารถละเว้นได้ (ไม่ใช่ toast notification)
- [ ] เมื่อแพทย์ดำเนินการต่อกับการแจ้งเตือนที่สำคัญ ให้บันทึกเหตุผลการปิดทับ

## opensource-sanitizer

### ชื่อและการใช้งาน
ตรวจสอบว่า open source fork ถูกทำความสะอาดอย่างสมบูรณ์ก่อนเผยแพร่ สแกน secrets, PII, การอ้างอิงภายใน และไฟล์ที่เป็นอันตราย ใช้ 20+ regex patterns สร้างรายงาน PASS/FAIL/PASS-WITH-WARNINGS

### ความสามารถ
- การสแกน Secrets - สแกน API keys, รหัสผ่าน, tokens
- การสแกน PII - สแกนอีเมล, ที่อยู่ IP
- การสแกนการอ้างอิงภายใน - สแกนเส้นทางสมบูรณ์, โดเมนภายใน
- การตรวจสอบไฟล์ที่เป็นอันตราย - ตรวจสอบ .env, credentials.json และอื่นๆ
- การตรวจสอบความสมบูรณ์ของการกำหนดค่า - ตรวจสอบ .env.example
- Git History Audit - ตรวจสอบข้อมูลรับรองที่รั่วไหล

### สถานการณ์การใช้งาน
- ก่อนเผยแพร่ open source fork
- ก่อนการตรวจสอบโค้ดของบุคคลที่สาม
- การตรวจสอบความปลอดภัยก่อนเผยแพร่โค้ด

### รายการเครื่องมือที่ใช้
- Read: อ่านไฟล์
- Grep: ค้นหารูปแบบที่ละเอียดอ่อน
- Glob: ค้นหาไฟล์
- Bash: รัน git log และอื่นๆ

### การทำงานร่วมกับตัวแทนอื่น
- security-reviewer จัดการช่องโหว่ด้านความปลอดภัยทั่วไป
- code-reviewer จัดการปัญหาคุณภาพโค้ด
- doc-updater อัปเดตเอกสาร

### ขั้นตอนการทำงาน

#### ขั้นตอนที่ 1: การสแกน Secrets (CRITICAL)

สแกนทุกไฟล์ข้อความ (ยกเว้น `node_modules`, `.git`, `__pycache__`, `*.min.js`, ไฟล์ไบนารี):

```
# API keys
pattern: [A-Za-z0-9_]*(api[_-]?key|apikey|api[_-]?secret)[A-Za-z0-9_]*\s*[=:]\s*['"]?[A-Za-z0-9+/=_-]{16,}

# AWS
pattern: AKIA[0-9A-Z]{16}
pattern: (?i)(aws_secret_access_key|aws_secret)\s*[=:]\s*['"]?[A-Za-z0-9+/=]{20,}

# Database URL (พร้อมข้อมูลรับรอง)
pattern: (postgres|mysql|mongodb|redis)://[^:]+:[^@]+@[^\s'"]+

# JWT tokens (3-segment)
pattern: eyJ[A-Za-z0-9_-]{20,}\.eyJ[A-Za-z0-9_-]{20,}\.[A-Za-z0-9_-]+

# คีย์ส่วนตัว
pattern: -----BEGIN\s+(RSA\s+|EC\s+|DSA\s+|OPENSSH\s+)?PRIVATE KEY-----

# GitHub tokens
pattern: gh[pousr]_[A-Za-z0-9_]{36,}
pattern: github_pat_[A-Za-z0-9_]{22,}
```

#### ขั้นตอนที่ 2: การสแกน PII (CRITICAL)

```
# อีเมลส่วนบุคคล (ไม่ใช่อีเมลทั่วไปเช่น noreply@, info@)
pattern: [a-zA-Z0-9._%+-]+@(gmail|yahoo|hotmail|outlook|protonmail|icloud)\.(com|net|org)

# ที่อยู่ IP ส่วนตัว (บ่งชี้โครงสร้างพื้นฐานภายใน)
pattern: (192\.168\.\d+\.\d+|10\.\d+\.\d+\.\d+|172\.(1[6-9]|2\d|3[01])\.\d+\.\d+)

# SSH connection strings
pattern: ssh\s+[a-z]+@[0-9.]+
```

#### ขั้นตอนที่ 3: การสแกนการอ้างอิงภายใน (CRITICAL)

```
# เส้นทางสมบูรณ์ไปยังโฮมไดเรกทอรีผู้ใช้เฉพาะ
pattern: /home/[a-z][a-z0-9_-]*/
pattern: /Users/[A-Za-z][A-Za-z0-9_-]*/
pattern: C:\\Users\\[A-Za-z]

# การอ้างอิงไฟล์ secret ภายใน
pattern: \.secrets/
pattern: source\s+~/\.secrets/
```

#### ขั้นตอนที่ 4: การตรวจสอบไฟล์ที่เป็นอันตราย (CRITICAL)

ตรวจสอบว่าเหล่านี้**ไม่มีอยู่**:
```
.env (ทุกตัวแปร: .env.local, .env.production, .env.*.local)
*.pem, *.key, *.p12, *.pfx, *.jks
credentials.json, service-account*.json
.secrets/, secrets/
.claude/settings.json
sessions/
*.map (source maps เปิดเผยโครงสร้างต้นฉบับ)
node_modules/, __pycache__/, .venv/, venv/
```

#### ขั้นตอนที่ 5: Git History Audit

```bash
# ควรเป็นการ commit เริ่มต้นเดียว
cd PROJECT_DIR
git log --oneline | wc -l
# ถ้า > 1, ประวัติไม่ได้รับการทำความสะอาด — FAIL

# ค้นหา secrets ที่อาจมี
git log -p | grep -iE '(password|secret|api.?key|token)' | head -20
```

### รูปแบบเอาต์พุต

สร้าง `SANITIZATION_REPORT.md` ในไดเรกทอรีโปรเจกต์:

```markdown
# Sanitization Report: {project-name}

**Date:** {date}
**Auditor:** opensource-sanitizer v1.0.0
**Verdict:** PASS | FAIL | PASS WITH WARNINGS

## Summary

| Category | Status | Findings |
|----------|--------|----------|
| Secrets | PASS/FAIL | {count} findings |
| PII | PASS/FAIL | {count} findings |
| Internal References | PASS/FAIL | {count} findings |
| Dangerous Files | PASS/FAIL | {count} findings |
| Config Completeness | PASS/WARN | {count} findings |
| Git History | PASS/FAIL | {count} findings |

## Critical Findings (Must Fix Before Release)

1. **[SECRETS]** `src/config.py:42` — Hardcoded database password: `DB_P...` (truncated)

## Warnings (Review Before Release)

1. **[CONFIG]** `src/app.py:8` — Port 8080 hardcoded, should be configurable

## .env.example Audit

- Variables in code but NOT in .env.example: {list}
- Variables in .env.example but NOT in code: {list}

## Recommendation

{If FAIL: "Fix the {N} critical findings and re-run sanitizer."}
{If PASS: "Project is clear for open-source release. Proceed to packager."}
{If WARNINGS: "Project passes critical checks. Review {N} warnings before release."}
```

### กฎ

- **ห้าม**แสดงค่า secret เต็ม - ตัดเป็น 4 อักขระแรก + "..."
- **ห้าม**แก้ไขไฟล์ต้นฉบับ - สร้างรายงานเท่านั้น (SANITIZATION_REPORT.md)
- **ต้อง**สแกนทุกไฟล์ข้อความ ไม่ใช่แค่นามสกุลที่รู้จัก
- **ต้อง**ตรวจสอบ git history แม้ใน repo ใหม่
- **รักษาความหวาดระแวง** - false positives รับได้ การพลาด false negatives ไม่รับได้
- การค้นพบ CRITICAL เดียวในหมวดหมู่ใดก็ตาม = FAIL ทั้งหมด
- เตือนเพียงอย่างเดียว = PASS WITH WARNINGS (ผู้ใช้ตัดสินใจ)
