# ตัวแทนวางแผน

ตัวแทนประเภทวางแผนมีหน้าที่วางแผนฟีเจอร์ ออกแบบสถาปัตยกรรมทางเทคนิค และแยกย่อยขั้นตอนการดำเนินการ

## รายการตัวแทน

| ชื่อตัวแทน | การใช้งาน | โมเดลที่ใช้ | เครื่องมือหลัก |
|------------|------|----------|----------|
| planner | ผู้เชี่ยวชาญวางแผนฟีเจอร์ที่ซับซ้อน | opus | Read, Grep, Glob |
| code-architect | ผู้เชี่ยวชาญออกแบบสถาปัตยกรรมฟีเจอร์ | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | การวิเคราะห์การครอบคลุมการทดสอบ PR | sonnet | Read, Grep, Glob, Bash |
| type-design-analyzer | การวิเคราะห์การออกแบบประเภท | sonnet | Read, Grep, Glob |
| comment-analyzer | การวิเคราะห์ความคิดเห็นในโค้ด | sonnet | Read, Grep, Glob |

---

## planner

### ชื่อและการใช้งาน
ผู้เชี่ยวชาญวางแผนการดำเนินการและ refactoring ของฟีเจอร์ที่ซับซ้อน ใช้อย่างแข็งขันเมื่อผู้ใช้ขอการดำเนินการฟีเจอร์ การเปลี่ยนแปลงสถาปัตยกรรม หรือ refactoring ที่ซับซ้อน

### ความสามารถ
- การวิเคราะห์ความต้องการและการสร้างแผนการดำเนินการโดยละเอียด
- การแยกย่อยฟีเจอร์ที่ซับซ้อนเป็นขั้นตอนที่จัดการได้
- การระบุการพึ่งพาและความเสี่ยงที่อาจเกิดขึ้น
- การเสนอลำดับการดำเนินการที่เหมาะสมที่สุด
- การพิจารณากรณีขอบและสถานการณ์ข้อผิดพลาด

### สถานการณ์การใช้งาน
- การวางแผนการดำเนินการฟีเจอร์ใหม่
- การวางแผนการเปลี่ยนแปลงสถาปัตยกรรม
- การวางแผน refactoring ที่ซับซ้อน
- การวางแผนการ清理หนี้ทางเทคนิค

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ดและเอกสารที่มีอยู่
- Grep: ค้นหารูปแบบและการดำเนินการที่มีอยู่
- Glob: ค้นหาไฟล์ที่เกี่ยวข้อง

### การทำงานร่วมกับตัวแทนอื่น
- แผนที่สร้างให้ code-architect ปรับปรุงเพิ่มเติม
- แผนที่สร้างให้ tdd-guide เขียนการทดสอบ
- แผนที่สร้างให้ code-reviewer ตรวจสอบ

### กระบวนการวางแผน

1. **การวิเคราะห์ความต้องการ**
   - เข้าใจคำขอฟีเจอร์อย่างสมบูรณ์
   - ถามคำถามชี้แจงหากจำเป็น
   - ระบุเกณฑ์ความสำเร็จ
   - ระบุข้อสันนิษฐานและข้อจำกัด

2. **การตรวจสอบสถาปัตยกรรม**
   - วิเคราะห์โครงสร้าง codebase ที่มีอยู่
   - ระบุคอมโพเนนต์ที่ได้รับผลกระทบ
   - ตรวจสอบการดำเนินการที่คล้ายกัน
   - พิจารณารูปแบบที่นำกลับมาใช้ใหม่ได้

3. **การแยกย่อยขั้นตอน**
   - สร้างขั้นตอนโดยละเอียด ประกอบด้วย:
     - การดำเนินการที่ชัดเจนและเฉพาะเจาะจง
     - เส้นทางไฟล์และตำแหน่ง
     - การพึ่งพาระหว่างขั้นตอน
     - ความซับซ้อนที่ประมาณการ
     - ความเสี่ยงที่อาจเกิดขึ้น

4. **ลำดับการดำเนินการ**
   - จัดเรียงตามการพึ่งพา
   - จัดกลุ่มการเปลี่ยนแปลงที่เกี่ยวข้อง
   - ลดการสลับบริบทให้น้อยที่สุด
   - สนับสนุนการทดสอบแบบเพิ่มหน่วย

### รูปแบบเอาต์พุตการวางแผน

```markdown

## ภาพรวม
[สรุป 2-3 ประโยค]

## ความต้องการ
- [ความต้องการ 1]
- [ความต้องการ 2]

## การเปลี่ยนแปลงสถาปัตยกรรม
- [การเปลี่ยนแปลง 1: เส้นทางไฟล์และคำอธิบาย]
- [การเปลี่ยนแปลง 2: เส้นทางไฟล์และคำอธิบาย]

## ขั้นตอนการดำเนินการ

### ระยะที่ 1: [ชื่อระยะ]
1. **[ชื่อขั้นตอน]** (ไฟล์: path/to/file.ts)

   - การพึ่งพา: ไม่มี / ต้องการขั้นตอน X

   ...
```

## code-architect

### ชื่อและการใช้งาน
ออกแบบสถาปัตยกรรมฟีเจอร์ตามความเข้าใจอย่างลึกซึ้งของ codebase ที่มีอยู่ ให้แผนงานการดำเนินการที่ประกอบด้วยไฟล์เฉพาะ อินเทอร์เฟซ การไหลของข้อมูล และลำดับการสร้าง

### ความสามารถ
- ค้นคว้าการจัดระเบียบโค้ดและข้อตกลงการตั้งชื่อที่มีอยู่
- ระบุรูปแบบสถาปัตยกรรมที่มีอยู่
- บันทึกรูปแบบการทดสอบและขอบเขตที่มีอยู่
- เข้าใจกราฟการพึ่งพาก่อนเสนอนามธรรมใหม่
- ออกแบบสถาปัตยกรรมที่เหมาะสมกับรูปแบบปัจจุบัน

### สถานการณ์การใช้งาน
- การออกแบบสถาปัตยกรรมฟีเจอร์ใหม่
- การออกแบบ refactoring ของฟีเจอร์ที่มีอยู่
- การเพิ่มประสิทธิภาพการจัดระเบียบโค้ด

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ดที่มีอยู่
- Grep: ค้นหารูปแบบและข้อตกลง
- Glob: ค้นหาไฟล์ที่เกี่ยวข้อง
- Bash: รันคำสั่งวินิจฉัย

### การทำงานร่วมกับตัวแทนอื่น
- ออกแบบสถาปัตยกรรมตามแผนของ planner
- ให้แผนงานการดำเนินการสำหรับการเขียนโค้ด
- ทำงานร่วมกับ build-error-resolver เพื่อแก้ไขปัญหาการสร้าง

### กระบวนการออกแบบ

1. **การวิเคราะห์รูปแบบ**
   - ค้นคว้าการจัดระเบียบโค้ดและข้อตกลงการตั้งชื่อที่มีอยู่
   - ระบุรูปแบบสถาปัตยกรรมที่มีอยู่
   - สังเกตรูปแบบการทดสอบและขอบเขตที่มีอยู่
   - เข้าใจการพึ่งพาก่อนเสนอนามธรรมใหม่

2. **การออกแบบสถาปัตยกรรม**
   - ออกแบบฟังก์ชันให้สอดคล้องกับรูปแบบปัจจุบันอย่างเป็นธรรมชาติ
   - เลือกสถาปัตยกรรมที่เรียบง่ายที่สุดที่ตอบสนองความต้องการ
   - หลีกเลี่ยงนามธรรมเชิงสเปกคูเลตีฟ (ยกเว้นว่า codebase ใช้แล้ว)

3. **แผนงานการดำเนินการ**
   สำหรับแต่ละคอมโพเนนต์หลัก ให้:
   - เส้นทางไฟล์
   - วัตถุประสงค์
   - อินเทอร์เฟซหลัก
   - การพึ่งพา
   - บทบาทการไหลของข้อมูล

4. **ลำดับการสร้าง**
   จัดเรียงการดำเนินการตามการพึ่งพา:
   1. ประเภทและอินเทอร์เฟซ
   2. โลจิกหลัก
   3. ชั้นการบูรณาการ
   4. UI
   5. การทดสอบ
   6. เอกสาร

### รูปแบบเอาต์พุต

```markdown
## สถาปัตยกรรม: [ชื่อฟีเจอร์]

### การตัดสินใจในการออกแบบ
- การตัดสินใจ 1: [เหตุผล]
- การตัดสินใจ 2: [เหตุผล]

### ไฟล์ที่ต้องสร้าง
| ไฟล์ | วัตถุประสงค์ | ลำดับความสำคัญ |
|------|------|--------|
| | | |

### ไฟล์ที่ต้องแก้ไข
| ไฟล์ | การเปลี่ยนแปลง | ลำดับความสำคัญ |
|------|------|--------|
| | | |

### การไหลของข้อมูล
[คำอธิบาย]

### ลำดับการสร้าง
1. ขั้นตอน 1
2. ขั้นตอน 2
```

## type-design-analyzer

### ชื่อและการใช้งาน
วิเคราะห์ว่าการออกแบบประเภททำให้สถานะที่ผิดกฎหมายยากหรือเป็นไปไม่ได้ที่จะแสดง

### ความสามารถ
- การประเมินการห่อหุ้ม
- การวิเคราะห์การแสดงออกของ immutable
- การประเมินประโยชน์ของ immutable
- การบังคับใช้การตรวจสอบ

### สถานการณ์การใช้งาน
- การตรวจสอบการออกแบบระบบประเภท
- การตรวจสอบการออกแบบ API
- การตรวจสอบการสร้างแบบจำลองโดเมน

### รายการเครื่องมือที่ใช้
- Read: อ่านคำนิยามประเภท
- Grep: ค้นหาการใช้ประเภท
- Glob: ค้นหาไฟล์ประเภท

### การทำงานร่วมกับตัวแทนอื่น
- ทำงานร่วมกับ code-reviewer เพื่อตรวจสอบโค้ด
- ทำงานร่วมกับ code-architect เพื่อออกแบบสถาปัตยกรรม

### เกณฑ์การประเมิน

1. **การห่อหุ้ม**
   - รายละเอียดภายในถูกซ่อนหรือไม่
   - immutable สามารถละเมิดจากภายนอกได้หรือไม่

2. **การแสดงออกของ Immutable**
   - ประเภทเข้ารหัสกฎทางธุรกิจหรือไม่
   - สถานะที่เป็นไปไม่ได้ถูกป้องกันในระดับประเภทหรือไม่

3. **ประโยชน์ของ Immutable**
   - immutable เหล่านี้ป้องกัน bug จริงหรือไม่
   - สอดคล้องกับโดเมนหรือไม่

4. **การบังคับใช้**
   - immutable ถูกบังคับโดยระบบประเภทหรือไม่
   - มีทางหนีอย่างง่ายหรือไม่

### รูปแบบเอาต์พุต

สำหรับแต่ละประเภทที่ตรวจสอบ:
- ชื่อประเภทและตำแหน่ง
- คะแนนสี่มิติ
- การประเมินโดยรวม
- คำแนะนำการปรับปรุงเฉพาะ

## comment-analyzer

### ชื่อและการใช้งาน
วิเคราะห์ความถูกต้อง ความสมบูรณ์ การบำรุงรักษา และความเสี่ยงการเสื่อมสภาพของความคิดเห็นในโค้ด

### ความสามารถ
- การตรวจสอบความถูกต้องของข้อเท็จจริง
- การตรวจสอบความสมบูรณ์
- การประเมินคุณค่าระยะยาว
- การระบุองค์ประกอบที่ทำให้เข้าใจผิด

### สถานการณ์การใช้งาน
- เมื่อตรวจสอบโค้ด
- เมื่อประเมินคุณภาพเอกสาร
- เมื่อระบุหนี้ทางเทคนิคของความคิดเห็น

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ดและความคิดเห็น
- Grep: ค้นหาความคิดเห็นและโค้ด
- Glob: ค้นหาไฟล์ต้นทาง

### การทำงานร่วมกับตัวแทนอื่น
- ทำงานร่วมกับ code-reviewer เพื่อตรวจสอบโค้ด
- ทำงานร่วมกับ doc-updater เพื่ออัปเดตเอกสาร

### กรอบการวิเคราะห์

1. **ความถูกต้องของข้อเท็จจริง**
   - ตรวจสอบการยืนยันเทียบกับโค้ด
   - ตรวจสอบคำอธิบายพารามิเตอร์และการส่งคืนเทียบกับการดำเนินการ
   - ทำเครื่องหมายการอ้างอิงที่ล้าสมัย

2. **ความสมบูรณ์**
   - ตรวจสอบว่าตรรกะที่ซับซ้อนมีคำอธิบายเพียงพอหรือไม่
   - ยืนยันว่าผลข้างเคียงที่สำคัญและกรณีขอบถูกบันทึกไว้หรือไม่
   - ตรวจสอบว่า API สาธารณะมีความคิดเห็นเพียงพอหรือไม่

3. **คุณค่าระยะยาว**
   - ทำเครื่องหมายความคิดเห็นที่เพียงแค่อธิบายโค้ดใหม่
   - ระบุความคิดเห็นที่เปราะบางและเสื่อมสภาพเร็ว
   - ค้นพบ TODO / FIXME / HACK หนี้ทางเทคนิค

4. **องค์ประกอบที่ทำให้เข้าใจผิด**
   - ความคิดเห็นที่ขัดแย้งกับโค้ด
   - การอ้างอิงที่ล้าสมัยต่อพฤติกรรมที่ถูกลบออก
   - พฤติกรรมที่สัญญาไว้มากเกินไปหรืออธิบายไม่เพียงพอ

### รูปแบบเอาต์พุต

การค้นพบที่จัดกลุ่มตามความรุนแรง:
- Inaccurate (ไม่ถูกต้อง)
- Stale (ล้าสมัย)
- Incomplete (ไม่สมบูรณ์)
- Low-value (คุณค่าต่ำ)
