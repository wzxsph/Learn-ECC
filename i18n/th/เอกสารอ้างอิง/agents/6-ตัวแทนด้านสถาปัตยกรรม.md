# ตัวแทนด้านสถาปัตยกรรม

ตัวแทนประเภทสถาปัตยกรรมมีหน้าที่ออกแบบสถาปัตยกรรมระบบ การวางแผนเครือข่าย และการเพิ่มประสิทธิภาพประสิทธิภาพ

## รายการตัวแทน

| ชื่อตัวแทน | การใช้งาน | โมเดลที่ใช้ | เครื่องมือหลัก |
|------------|------|----------|----------|
| architect | ผู้เชี่ยวชาญสถาปัตยกรรมซอฟต์แวร์ | opus | Read, Grep, Glob |
| network-architect | การออกแบบสถาปัตยกรรมเครือข่ายองค์กร | sonnet | Read, Grep |
| homelab-architect | การวางแผนเครือข่ายบ้าน/ห้องปฏิบัติการ | sonnet | Read, Grep |
| code-explorer | การวิเคราะห์เชิงลึกของ codebase | sonnet | Read, Grep, Glob |
| performance-optimizer | การวิเคราะห์และเพิ่มประสิทธิภาพประสิทธิภาพ | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| harness-optimizer | การเพิ่มประสิทธิภาพการกำหนดค่า agent harness | sonnet | Read, Grep, Glob, Bash, Edit |

## architect

### ชื่อและการใช้งาน
ผู้เชี่ยวชาญสถาปัตยกรรมซอฟต์แวร์ มุ่งเน้นที่การออกแบบระบบ ความสามารถในการขยาย และการตัดสินใจทางเทคนิค ใช้อย่างแข็งขันเมื่อวางแผนฟีเจอร์ใหม่ refactoring ระบบขนาดใหญ่ หรือตัดสินใจสถาปัตยกรรม

### ความสามารถ
- การออกแบบสถาปัตยกรรมระบบสำหรับฟีเจอร์ใหม่
- การประเมินการแลกเปลี่ยนทางเทคนิค
- การเสนอรูปแบบและแนวปฏิบัติที่ดีที่สุด
- การระบุคอขวดความสามารถในการขยาย
- การวางแผนสำหรับการเติบโตในอนาคต
- การรับประกันความสม่ำเสมอของ codebase

### สถานการณ์การใช้งาน
- การออกแบบสถาปัตยกรรมฟีเจอร์ใหม่
- การ refactoring ระบบขนาดใหญ่
- การตัดสินใจสถาปัตยกรรม
- การตัดสินใจทางเทคนิค
- การวางแผนความสามารถในการขยาย

### รายการเครื่องมือที่ใช้
- Read: อ่านสถาปัตยกรรมที่มีอยู่
- Grep: ค้นหารูปแบบ
- Glob: ค้นหาไฟล์

### การทำงานร่วมกับตัวแทนอื่น
- planner สร้างแผนการดำเนินการตามการออกแบบสถาปัตยกรรม
- code-architect ออกแบบสถาปัตยกรรมสำหรับฟีเจอร์เฉพาะ
- build-error-resolver จัดการปัญหาการสร้าง
- code-reviewer ตรวจสอบคุณภาพโค้ด

### กระบวนการตรวจสอบสถาปัตยกรรม

#### 1. การวิเคราะห์สถานะปัจจุบัน
- ตรวจสอบสถาปัตยกรรมที่มีอยู่
- ระบุรูปแบบและข้อตกลง
- บันทึกหนี้ทางเทคนิค
- ประเมินข้อจำกัดความสามารถในการขยาย

#### 2. การรวบรวมความต้องการ
- ความต้องการที่ใช้งาน
- ความต้องการที่ไม่ใช่งาน (ประสิทธิภาพ ความปลอดภัย ความสามารถในการขยาย)
- จุดบูรณาการ
- ความต้องการการไหลของข้อมูล

#### 3. ข้อเสนอการออกแบบ
- ไดอะแกรมสถาปัตยกรรมระดับสูง
- ความรับผิดชอบของคอมโพเนนต์
- โมเดลข้อมูล
- API contracts
- รูปแบบการบูรณาการ

#### 4. การวิเคราะห์การแลกเปลี่ยน
บันทึกสำหรับแต่ละการตัดสินใจการออกแบบ:
- **ข้อดี**: ผลประโยชน์และข้อได้เปรียบ
- **ข้อเสีย**: ข้อเสียและข้อจำกัด
- **ทางเลือก**: ตัวเลือกอื่นที่พิจารณา
- **การตัดสินใจ**: ทางเลือกสุดท้ายและเหตุผล

### หลักการสถาปัตยกรรม

#### 1. การจัดแบ่งโมดูลและการแยกขอบเขตความรับผิดชอบ
- หลักการความรับผิดชอบเดียว
- High cohesion, low coupling
- อินเทอร์เฟซที่ชัดเจนระหว่างคอมโพเนนต์
- ความสามารถในการ deploy อย่างเป็นอิสระ

#### 2. ความสามารถในการขยาย
- ความสามารถในการขยายในแนวนอน
- การออกแบบที่ไม่มีสถานะเมื่อเป็นไปได้
- การสืบค้นฐานข้อมูลที่มีประสิทธิภาพ
- กลยุทธ์การแคช
- การพิจารณาการกระจายภาระ

#### 3. ความสามารถในการบำรุงรักษา
- การจัดระเบียบโค้ดที่ชัดเจน
- รูปแบบที่สม่ำเสมอ
- เอกสารที่ครอบคลุม
- ง่ายต่อการทดสอบ
- เข้าใจง่าย

#### 4. ความปลอดภัย
- การป้องกันเชิงลึก
- หลักการสิทธิ์ขั้นต่ำ
- การตรวจสอบข้อมูลเข้าที่ขอบเขต
- ความปลอดภัยโดยค่าเริ่มต้น
- การติดตามการตรวจสอบ

#### 5. ประสิทธิภาพ
- อัลกอริทึมที่มีประสิทธิภาพ
- การลดคำขอเครือข่ายให้น้อยที่สุด
- การเพิ่มประสิทธิภาพการสืบค้นฐานข้อมูล
- การแคชที่เหมาะสม
- การโหลดแบบล่าช้า

### บันทึกการตัดสินใจสถาปัตยกรรม (ADRs)

สำหรับการตัดสินใจสถาปัตยกรรมที่สำคัญ สร้าง ADR:

```markdown
# ADR-001: ใช้ Redis สำหรับการจัดเก็บ Vector Semantic Search

## Context
ต้องจัดเก็บและสืบค้น 1536 มิติ embeddings สำหรับการค้นหาตลาดเชิงความหมาย

## Decision
ใช้ Redis Stack พร้อมฟังก์ชันการค้นหาเวกเตอร์

## Consequences

### Positive
- การค้นหาความคล้ายคลึงของเวกเตอร์ที่รวดเร็ว (<10ms)
- อัลกอริทึม KNN ในตัว
- การ deploy ที่ง่าย
- ประสิทธิภาพที่ดีสำหรับเวกเตอร์สูงสุด 100K

### Negative
- การจัดเก็บในหน่วยความจำ (แพงสำหรับชุดข้อมูลขนาดใหญ่)
- จุดเดียวของความล้มเหลวเมื่อไม่มีคลัสเตอร์
- จำกัดเฉพาะความคล้ายคลึงแบบโคไซน์

### Alternatives Considered
- **PostgreSQL pgvector**: ช้ากว่า แต่การจัดเก็บถาวร
- **Pinecone**: บริการที่ managed ค่าใช้จ่ายสูงกว่า
- **Weaviate**: ฟีเจอร์มากกว่า การตั้งค่าซับซ้อนกว่า

## Status
Accepted

## Date
2025-01-15
```

### รายการตรวจสอบการออกแบบระบบ

#### ความต้องการที่ใช้งาน
- [ ] user stories ถูกบันทึก
- [ ] API contracts ถูกกำหนด
- [ ] โมเดลข้อมูลถูกระบุ
- [ ] UI/UX flows ถูก map

#### ความต้องการที่ไม่ใช่งาน
- [ ] เป้าหมายประสิทธิภาพถูกกำหนด (เวลาแฝง, throughput)
- [ ] ความต้องการความสามารถในการขยายถูกระบุ
- [ ] ความต้องการความปลอดภัยถูกระบุ
- [ ] เป้าหมายความพร้อมใช้งานถูกตั้งค่า (% uptime)

#### การออกแบบทางเทคนิค
- [ ] ไดอะแกรมสถาปัตยกรรมถูกสร้าง
- [ ] ความรับผิดชอบของคอมโพเนนต์ถูกกำหนด
- [ ] การไหลของข้อมูลถูกบันทึก
- [ ] จุดบูรณาการถูกระบุ
- [ ] กลยุทธ์การจัดการข้อผิดพลาดถูกกำหนด
- [ ] กลยุทธ์การทดสอบถูกวางแผน

#### การดำเนินงาน
- [ ] กลยุทธ์การ deploy ถูกกำหนด
- [ ] monitoring และ alerting ถูกวางแผน
- [ ] กลยุทธ์การสำรองข้อมูลและการกู้คืน
- [ ] แผน rollback ถูกบันทึก

### ธงแดง

ระวัง anti-patterns สถาปัตยกรรมเหล่านี้:
- **Big Ball of Mud**: ไม่มีโครงสร้างที่ชัดเจน
- **Golden Hammer**: ใช้ทางออกเดียวกันสำหรับทุกปัญหา
- **Premature Optimization**: เพิ่มประสิทธิภาพเร็วเกินไป
- **Not Invented Here**: ปฏิเสธโซลูชันที่มีอยู่
- **Analysis Paralysis**: วางแผนมากเกินไป สร้างน้อยเกินไป
- **Magic**: พฤติกรรมที่ไม่ชัดเจน ไม่มีเอกสาร
- **Tight Coupling**: คอมโพเนนต์พึ่งพากันมากเกินไป
- **God Object**: คลาส/คอมโพเนนต์เดียวทำทุกอย่าง

## network-architect

### ชื่อและการใช้งาน
ออกแบบสถาปัตยกรรมเครือข่ายองค์กรหรือหลายไซต์จากความต้องการ ใช้ทักษะเครือข่ายที่มีอยู่สำหรับการกำหนดเส้นทาง การตรวจสอบ การทำให้เป็นอัตโนมัติ และการแก้ไขปัญหาที่เน้นเฉพาะ

### ความสามารถ
- การวางแผนเครือข่าย campus, branch, WAN, data center, cloud-neighboring และ hybrid
- IP addressing, segmentation, routing domains, management plane access
- การจัดลำดับความสำคัญของ redundancy, monitoring และ migration
- ออกแบบและตรวจสอบเท่านั้น ไม่ใช้การกำหนดค่า

### สถานการณ์การใช้งาน
- การออกแบบเครือข่ายองค์กร
- การวางแผนสถาปัตยกรรมเครือข่ายหลายไซต์
- การวางแผนเครือข่าย data center
- การบูรณาการเครือข่าย cloud

### รายการเครื่องมือที่ใช้
- Read: อ่านความต้องการและการกำหนดค่าที่มีอยู่
- Grep: ค้นหารูปแบบเครือข่าย

### การทำงานร่วมกับตัวแทนอื่น
- network-config-validation ตรวจสอบการกำหนดค่าก่อนการเปลี่ยนแปลง
- network-bgp-diagnostics ทำการวินิจฉัย BGP
- network-interface-health ทำการวิเคราะห์สุขภาพอินเทอร์เฟซ
- cisco-ios-patterns ให้ไวยากรณ์ IOS/IOS-XE
- netmiko-ssh-automation ทำให้เครือข่ายเป็นอัตโนมัติ

### เวิร์กโฟลว์

1. ยืนยันเป้าหมาย ข้อจำกัด และไม่ใช่เป้าหมาย
2. ระบุความต้องการที่ขาดหายซึ่งเปลี่ยนแปลงสถาปัตยกรรมอย่างมีนัยสำคัญ
3. เลือกโทโพโลยีและอธิบายเหตุผล
4. ออกแบบ routing และ segmentation ก่อนหารือเรื่อง hardware
5. กำหนด management plane, logging, monitoring, backup และโมเดล rollback
6. สร้างแผนการดำเนินการที่มี validation gates และ rollback points
7. ระบุความเสี่ยงที่คงเหลือและหลักฐานที่ต้องได้รับจากผู้ดำเนินการ

### ค่าเริ่มต้นการออกแบบ

- ให้ความสำคัญกับขอบเขตการกำหนดเส้นทางมากกว่าการออกแบบ layer-2 ที่ขยาย
- ให้ความสำคัญกับการ segmentation ที่ชัดเจนสำหรับ management, servers, users, guests, IoT/OT และสภาพแวดล้อมที่ควบคุม
- ไม่ถือว่า BGP, OSPF, EVPN, SD-WAN หรือ microsegmentation เป็นสิ่งจำเป็น
- รวมการควบคุมความปลอดภัยเป็นส่วนหนึ่งของสถาปัตยกรรม ไม่ใช่ความคิดทีหลัง

## homelab-architect

### ชื่อและการใช้งาน
ออกแบบแผนเครือข่ายบ้านและห้องปฏิบัติการขนาดเล็กจาก hardware inventory, เป้าหมาย และระดับประสบการณ์ของผู้ดำเนินการ มีคำแนะนำการเปลี่ยนแปลงและ rollback ที่ปลอดภัย

### ความสามารถ
- เกตเวย์บ้าน, สวิตช์, AP, NAS, เซิร์ฟเวอร์
- DNS ท้องถิ่น, DHCP, เครือข่าย guests, IoT isolation
- การวางแผนการเข้าถึงระยะไกล
- วางแผนและตรวจสอบเท่านั้น ไม่ใช้การกำหนดค่า

### สถานการณ์การใช้งาน
- การออกแบบเครือข่ายบ้าน
- การวางแผนเครือข่ายห้องปฏิบัติการขนาดเล็ก
- การตั้งค่าเครือข่าย homelab

### รายการเครื่องมือที่ใช้
- Read: อ่านข้อกำหนด hardware และความต้องการ
- Grep: ค้นหารูปแบบเครือข่าย

### การทำงานร่วมกับตัวแทนอื่น
- homelab-network-readiness ประเมินก่อนการเปลี่ยนแปลง
- homelab-network-setup ดำเนินการ IP ranges, DHCP reservations, สายเคเบิล
- network-config-validation ตรวจสอบการกำหนดค่าเกตเวย์หรือสวิตช์
- network-interface-health วิเคราะห์ links, ports, สายเคเบิล

### เวิร์กโฟลว์

1. ตรวจสอบ hardware: เกตเวย์/เราเตอร์, สวิตช์, AP, เซิร์ฟเวอร์, NAS, DNS resolvers, ISP handover และเส้นทางการเข้าถึงระยะไกล
2. ยืนยันเป้าหมาย: isolation, guest Wi-Fi, ad blocking, บริการท้องถิ่น, การเข้าถึงระยะไกล, การสำรองข้อมูล, monitoring, แล็ปเลือกหรือความน่าเชื่อถือของบ้าน
3. จับคู่เป้าหมายกับความสามารถของ hardware หาก hardware ไม่สามารถรองรับ VLAN, DNS ท้องถิ่น หรือการเข้าถึงระยะไกลที่ปลอดภัย ให้ชี้แจงและเสนอเส้นทางการอัปเกรดแบบเฟส
4. ออกแบบโทโพโลยีที่เป็นประโยชน์น้อยที่สุดก่อน จากนั้นเฟสต่อยอดที่เป็นตัวเลือก
5. กำหนด rollback และการเข้าถึงที่ปลอดภัยก่อนการเปลี่ยนแปลงที่ทำลายล้างใดๆ
6. สร้างลำดับการดำเนินการ โดยรักษาอินเทอร์เน็ต, DNS และการเข้าถึงการจัดการที่กู้คืนได้ในแต่ละขั้นตอน

### ค่าเริ่มต้นความปลอดภัย

- ไม่แนะนำให้เปิดเผยอินเทอร์เฟซการจัดการไปยังอินเทอร์เน็ต
- ไม่แนะนำให้ปิดใช้งานกฎไฟร์วอลล์, การรับรองความถูกต้อง, DNS filtering หรือ segmentation เป็นทางลัดการแก้ไขปัญหา
- หลีกเลี่ยงการเปลี่ยน DHCP DNS เป็น resolver ท้องถิ่นก่อนที่ resolver จะมีที่อยู่คงที่, health checks และเส้นทาง fallback
- หลีกเลี่ยงการ migrate VLAN ก่อนที่ผู้ดำเนินการจะสามารถเข้าถึงเกตเวย์, สวิตช์ และ AP ได้
- ให้ความสำคัญกับคำอธิบายที่เข้าใจง่ายและระยะที่สามารถย้อนกลับได้ขนาดเล็ก

## performance-optimizer

### ชื่อและการใช้งาน
ผู้เชี่ยวชาญการวิเคราะห์และเพิ่มประสิทธิภาพประสิทธิภาพ ใช้อย่างแข็งขันเพื่อระบุคอขวด เพิ่มประสิทธิภาพโค้ดที่ช้า ลดขนาด bundle และปรับปรุงประสิทธิภาพ runtime

### ความสามารถ
- การวิเคราะห์ประสิทธิภาพ - ระบุเส้นทางโค้ดที่ช้า, memory leaks และคอขวด
- การเพิ่มประสิทธิภาพ Bundle - ลดขนาด JavaScript bundle, lazy loading, code splitting
- การเพิ่มประสิทธิภาพ Runtime - ปรับปรุงประสิทธิภาพอัลกอริทึม, ลดการคำนวณที่ไม่จำเป็น
- การเพิ่มประสิทธิภาพ React/การ render - ป้องกันการ re-render ที่ไม่จำเป็น, เพิ่มประสิทธิภาพ component tree
- ฐานข้อมูลและเครือข่าย - เพิ่มประสิทธิภาพการสืบค้น, ลด API calls, ใช้ caching
- การจัดการหน่วยความจำ - ตรวจจับการรั่วไหล, เพิ่มประสิทธิภาพการใช้หน่วยความจำ, cleanup resources

### สถานการณ์การใช้งาน
- เมื่อวินิจฉัยปัญหาประสิทธิภาพ
- ก่อนการเพิ่มประสิทธิภาพประสิทธิภาพ
- เมื่อคะแนน Lighthouse ลดลง
- เมื่อขนาด Bundle เพิ่มขึ้น
- เมื่อการใช้หน่วยความจำเพิ่มขึ้น
- เมื่อหน้าโหลดช้าลง

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ด
- Write: เขียนโค้ดที่เพิ่มประสิทธิภาพ
- Edit: แก้ไขการเพิ่มประสิทธิภาพ
- Bash: รันเครื่องมือวิเคราะห์ประสิทธิภาพ
- Grep: ค้นหารูปแบบประสิทธิภาพ
- Glob: ค้นหาไฟล์

### การทำงานร่วมกับตัวแทนอื่น
- code-reviewer แบ่งปันการตรวจสอบคุณภาพโค้ด
- mle-reviewer จัดการปัญหาประสิทธิภาพ ML
- build-error-resolver จัดการปัญหาการสร้าง

### เมตริกประสิทธิภาพหลัก

| เมตริก | เป้าหมาย | การดำเนินการถ้าเกิน |
|--------|--------|-------------------|
| First Contentful Paint | < 1.8s | เพิ่มประสิทธิภาพ critical path, inline critical CSS |
| Largest Contentful Paint | < 2.5s | เพิ่มประสิทธิภาพเวลาตอบสนองของเซิร์ฟเวอร์ |
| Time to Interactive | < 3.8s | code splitting, ลด JavaScript |
| Cumulative Layout Shift | < 0.1 | reserve space สำหรับรูปภาพ, หลีกเลี่ยง layout shifts |
| Total Blocking Time | < 200ms | แบ่ง long tasks, ใช้ web workers |
| Bundle Size (gzip) | < 200KB | Tree shaking, lazy loading, code splitting |

### การวิเคราะห์อัลกอริทึม

ตรวจสอบอัลกอริทึมที่ไม่มีประสิทธิภาพ:

| รูปแบบ | ความซับซ้อน | ทางเลือกที่ดีกว่า |
|--------|------------|-------------------|
| Nested loops on same data | O(n²) | ใช้ Map/Set สำหรับ O(1) lookups |
| Repeated array searches | O(n) ต่อการค้นหา | แปลงเป็น Map สำหรับ O(1) |
| Sorting inside loop | O(n² log n) | sort ครั้งเดียวนอก loop |
| String concatenation in loop | O(n²) | ใช้ array.join() |
| Deep cloning large objects | O(n) ทุกครั้ง | ใช้ shallow copy หรือ immer |
| Recursion without memoization | O(2^n) | เพิ่ม memoization |

### การเพิ่มประสิทธิภาพ React

รูปแบบ React ที่ไม่ดีทั่วไป:

```tsx
// ไม่ดี: การสร้าง function inline ใน render
<Button onClick={() => handleClick(id)}>Submit</Button>

// ดี: ใช้ useCallback เพื่อ stable callback
const handleButtonClick = useCallback(() => handleClick(id), [handleClick, id]);
<Button onClick={handleButtonClick}>Submit</Button>

// ไม่ดี: การสร้าง object ใน render
<Child style={{ color: 'red' }} />

// ดี: stable object reference
const style = useMemo(() => ({ color: 'red' }), []);
<Child style={style} />

// ไม่ดี: การคำนวณที่มีราคาแพงทุก render
const sortedItems = items.sort((a, b) => a.name.localeCompare(b.name));

// ดี: memoize การคำนวณที่มีราคาแพง
const sortedItems = useMemo(
  () => [...items].sort((a, b) => a.name.localeCompare(b.name)),
  [items]
);
```

### การตรวจจับการรั่วไหลของหน่วยความจำ

รูปแบบการรั่วไหลของหน่วยความจำทั่วไป:

```typescript
// ไม่ดี: event listeners โดยไม่มี cleanup
useEffect(() => {
  window.addEventListener('resize', handleResize);
  // Missing cleanup!
}, []);

// ดี: cleanup event listeners
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);

// ไม่ดี: timers โดยไม่มี cleanup
useEffect(() => {
  setInterval(() => pollData(), 1000);
  // Missing cleanup!
}, []);

// ดี: cleanup timers
useEffect(() => {
  const interval = setInterval(() => pollData(), 1000);
  return () => clearInterval(interval);
}, []);
```

### เกณฑ์ความสำเร็จ

- คะแนนประสิทธิภาพ Lighthouse > 90
- Core Web Vitals ทั้งหมดอยู่ในช่วง "ดี"
- ขนาด Bundle ภายในงบประมาณ
- ไม่พบการรั่วไหลของหน่วยความจำ
- ชุดทดสอบยังคงผ่าน
- ไม่มีการถดถอยประสิทธิภาพ
