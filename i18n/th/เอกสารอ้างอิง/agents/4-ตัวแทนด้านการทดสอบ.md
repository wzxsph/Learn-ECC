# ตัวแทนด้านการทดสอบ

ตัวแทนประเภทการทดสอบมีหน้าที่พัฒนาที่ขับเคลื่อนด้วยการทดสอบ การทดสอบ end-to-end การวิเคราะห์การครอบคลุมการทดสอบ และการประกันคุณภาพ

## รายการตัวแทน

| ชื่อตัวแทน | การใช้งาน | โมเดลที่ใช้ | เครื่องมือหลัก |
|------------|------|----------|----------|
| tdd-guide | ผู้เชี่ยวชาญ TDD ที่ขับเคลื่อนด้วยการทดสอบ | sonnet | Read, Write, Edit, Bash, Grep |
| e2e-runner | ผู้เชี่ยวชาญการทดสอบ end-to-end | sonnet | Read, Write, Edit, Bash, Grep, Glob |
| mle-reviewer | การตรวจสอบโค้ด ML วิศวกรรม (การฝึก/การอนุมาน/การตรวจสอบ) | sonnet | Read, Grep, Glob, Bash |
| pr-test-analyzer | การวิเคราะห์การครอบคลุมการทดสอบ PR | sonnet | Read, Grep, Glob, Bash |

## tdd-guide

### ชื่อและการใช้งาน
ผู้เชี่ยวชาญ TDD ที่ขับเคลื่อนด้วยการทดสอบ บังคับใช้วิธีการเขียนการทดสอบก่อน ใช้อย่างแข็งขันสำหรับการเขียนฟีเจอร์ใหม่ การแก้ไข bug หรือ refactoring โค้ด รับประกันการครอบคลุมการทดสอบ 80%+

### ความสามารถ
- การบังคับใช้วิธีการทดสอบก่อน
- การแนะนำรอบ Red-Green-Refactor
- การรับประกันการครอบคลุมการทดสอบ 80%+
- การเขียนชุดการทดสอบที่ครอบคลุม (หน่วย การบูรณาการ E2E)
- การตรวจจับกรณีขอบก่อนการดำเนินการ

### สถานการณ์การใช้งาน
- เมื่อเขียนฟีเจอร์ใหม่
- เมื่อแก้ไข bug
- เมื่อ refactoring โค้ด
- เมื่อต้องการเวิร์กโฟลว์ TDD

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ดที่มีอยู่
- Write: เขียนการทดสอบ
- Edit: แก้ไขการทดสอบและการดำเนินการ
- Bash: รันคำสั่งทดสอบ
- Grep: ค้นหารูปแบบการทดสอบ

### การทำงานร่วมกับตัวแทนอื่น
- code-reviewer ตรวจสอบโค้ดหลังรอบ TDD
- build-error-resolver แก้ไขข้อผิดพลาดการสร้างการทดสอบ
- e2e-runner ดำเนินการทดสอบ E2E ของเส้นทางผู้ใช้หลัก

### เวิร์กโฟลว์ TDD

#### 1. เขียนการทดสอบก่อน (RED)
เขียนการทดสอบที่อธิบายพฤติกรรมที่คาดหวัง

#### 2. รันการทดสอบ - ยืนยันว่ามัน FAIL
```bash
npm test
```

#### 3. เขียนการดำเนินการขั้นต่ำ (GREEN)
เขียนโค้ดเพียงพอให้การทดสอบผ่าน

#### 4. รันการทดสอบ - ยืนยันว่ามัน PASS

#### 5. Refactor (IMPROVE)
ลบการซ้ำซ้อน ปรับปรุงชื่อ เพิ่มประสิทธิภาพ - การทดสอบต้องยังคงเป็นสีเขียว

#### 6. ยืนยันการครอบคลุม
```bash
npm run test:coverage
# ความต้องการ: การครอบคลุม 80%+ branches, functions, lines, statements
```

### กรณีขอบที่ต้องทดสอบ

1. **Input Null/Undefined**
2. **อาร์เรย์/สตริงว่าง**
3. **ส่งประเภทที่ไม่ถูกต้อง**
4. **ค่าขอบ** (น้อยที่สุด/มากที่สุด)
5. **เส้นทางข้อผิดพลาด** (ความล้มเหลวของเครือข่าย ข้อผิดพลาดฐานข้อมูล)
6. **เงื่อนไขการแข่งขัน** (การดำเนินการพร้อมกัน)
7. **ข้อมูลขนาดใหญ่** (ประสิทธิภาพ 10k+ รายการ)
8. **อักขระพิเศษ** (Unicode, emoji, อักขระ SQL)

### รูปแบบต้องหลีกเลี่ยง

- ทดสอบรายละเอียดการดำเนินการ (สถานะภายใน) แทนพฤติกรรม
- การทดสอบที่พึ่งพากัน (สถานะที่ใช้ร่วมกัน)
- การยืนยันน้อยเกินไป (ผ่านแต่ไม่ตรวจสอบอะไรเลย)
- External dependencies ไม่ได้ mock (Supabase, Redis, OpenAI ฯลฯ)

### รายการตรวจสอบคุณภาพ

- [ ] ฟังก์ชันสาธารณะทั้งหมดมีการทดสอบหน่วย
- [ ] API endpoints ทั้งหมดมีการทดสอบการบูรณาการ
- [ ] เส้นทางผู้ใช้หลักมีการทดสอบ E2E
- [ ] กรณีขอบครอบคลุม (null, empty, invalid)
- [ ] เส้นทางข้อผิดพลาดถูกทดสอบ (ไม่ใช่แค่ happy path)
- [ ] External dependencies ถูก mock
- [ ] การทดสอบเป็นอิสระต่อกัน (ไม่มีสถานะที่ใช้ร่วมกัน)
- [ ] การยืนยันเฉพาะเจาะจงและมีความหมาย
- [ ] การครอบคลุม 80%+

### ภาคผนวก v1.8 Eval-Driven TDD

รวม eval-driven development เข้ากับกระบวนการ TDD:

1. กำหนด capability + regression evals ก่อนการดำเนินการ
2. รัน baseline และจับ failure signature
3. ดำเนินการเปลี่ยนแปลงขั้นต่ำที่ผ่าน
4. รันการทดสอบและ evals ใหม่ รายงาน pass@1 และ pass@3
5. ปล่อย - เส้นทางหลักควรมี pass^3 stability ก่อนการรวม

## e2e-runner

### ชื่อและการใช้งาน
ผู้เชี่ยวชาญการทดสอบ end-to-end ใช้ Vercel Agent Browser (ต้องการ) และ Playwright สำรอง ใช้อย่างแข็งขันสำหรับการสร้าง บำรุงรักษา และรันการทดสอบ E2E

### ความสามารถ
- การสร้างเส้นทางการทดสอบ - เขียนการทดสอบสำหรับ user flows
- การบำรุงรักษาการทดสอบ - ทำให้การทดสอบตรงกับการเปลี่ยนแปลง UI
- การจัดการการทดสอบที่ไม่เสถียร - ระบุและแยกการทดสอบที่ไม่เสถียร
- การรายงานการทดสอบ - สร้างรายงาน HTML และ JUnit XML
- การรวม CI/CD - ทำให้การทดสอบทำงานในไปป์ไลน์อย่างน่าเชื่อถือ
- การจัดการ screenshot/video/trace

### สถานการณ์การใช้งาน
- เมื่อเส้นทางผู้ใช้หลักต้องการการตรวจสอบ
- เมื่อต้องการการตรวจสอบ end-to-end
- เมื่อรันการทดสอบ E2E ในไปป์ไลน์ CI/CD
- เมื่อค้นพบปัญหาการบูรณาการ

### รายการเครื่องมือที่ใช้
- Read: อ่านเนื้อหาหน้า
- Write: เขียนการทดสอบ
- Edit: แก้ไขการทดสอบ
- Bash: รันคำสั่ง Playwright/Agent Browser
- Grep: ค้นหาการทดสอบ
- Glob: ค้นหาไฟล์การทดสอบ

### การทำงานร่วมกับตัวแทนอื่น
- tdd-guide เขียนการทดสอบหน่วย/การบูรณาการ
- code-reviewer ตรวจสอบคุณภาพการทดสอบ
- mle-reviewer ตรวจสอบการทดสอบที่เกี่ยวข้องกับ ML

### เครื่องมือหลัก: Agent Browser

**ใช้ Agent Browser แทน Playwright ดิบ** - semantic selectors, AI optimization, auto-wait, built on Playwright

```bash
# การตั้งค่า
npm install -g agent-browser && agent-browser install

# เวิร์กโฟลว์หลัก
agent-browser open https://example.com
agent-browser snapshot -i          # รับ elements พร้อม refs [ref=e1]
agent-browser click @e1            # คลิกตาม ref
agent-browser fill @e2 "text"     # เติม input ตาม ref
agent-browser wait visible @e5     # รอให้ element มองเห็นได้
agent-browser screenshot result.png
```

### สำรอง: Playwright

เมื่อ Agent Browser ไม่พร้อมใช้งาน ใช้ Playwright โดยตรง

```bash
npx playwright test                        # รันการทดสอบ E2E ทั้งหมด
npx playwright test tests/auth.spec.ts     # รันไฟล์เฉพาะ
npx playwright test --headed               # เบราว์เซอร์มองเห็นได้
npx playwright test --debug                # แก้ไขข้อบกพร่องด้วย inspector
npx playwright test --trace on             # รันพร้อม trace
npx playwright show-report                 # ดูรายงาน HTML
```

### เวิร์กโฟลว์

#### 1. วางแผน
- ระบุเส้นทางผู้ใช้หลัก (รับรองความถูกต้อง, ฟีเจอร์หลัก, การชำระเงิน, CRUD)
- กำหนดสถานการณ์: happy path, กรณีขอบ, กรณีข้อผิดพลาด
- จัดลำดับความเสี่ยง: HIGH (การเงิน, รับรองความถูกต้อง), MEDIUM (ค้นหา, นำทาง), LOW (UI optimization)

#### 2. สร้าง
- ใช้รูปแบบ Page Object Model (POM)
- ให้ความสำคัญกับตัวเลือก `data-testid`
- เพิ่มการยืนยันในขั้นตอนสำคัญ
- จับ screenshot ในช่วงเวลาสำคัญ
- ใช้การรอที่เหมาะสม (ไม่เคยใช้ `waitForTimeout`)

#### 3. ดำเนินการ
- รันภายใน 3-5 ครั้งเพื่อตรวจสอบความไม่เสถียร
- ใช้ `test.fixme()` หรือ `test.skip()` เพื่อแยกการทดสอบที่ไม่เสถียร
- อัปโหลด artifact ไปยัง CI

### หลักการสำคัญ

- **ใช้ semantic selectors**: `[data-testid="..."]` > CSS selectors > XPath
- **รอเงื่อนไขไม่ใช่เวลา**: `waitForResponse()` > `waitForTimeout()`
- **Auto-wait ในตัว**: `page.locator().click()` รออัตโนมัติ; `page.click()` ไม่รอ
- **แยกการทดสอบ**: การทดสอบแต่ละครั้งควรเป็นอิสระ; ไม่มีสถานะที่ใช้ร่วมกัน
- **ล้มเหลวเร็ว**: ใช้ `expect()` ยืนยันในแต่ละขั้นตอนสำคัญ
- **Retry พร้อม trace**: กำหนดค่า `trace: 'on-first-retry'` สำหรับการแก้ไขข้อบกพร่อง

### การจัดการการทดสอบที่ไม่เสถียร

```typescript
// แยก
test('flaky: market search', async ({ page }) => {
  test.fixme(true, 'Flaky - Issue #123')
})

// ระบุความไม่เสถียร
// npx playwright test --repeat-each=10
```

สาเหตุทั่วไป: เงื่อนไขการแข่งขัน (ใช้ auto-wait selectors), การจับเวลาเครือข่าย (รอ response), การจับเวลาแอนิเมชัน (รอ `networkidle`)

### เกณฑ์ความสำเร็จ

- เส้นทางหลักทั้งหมดผ่าน (100%)
- อัตราผ่านโดยรวม > 95%
- อัตราไม่เสถียร < 5%
- ระยะเวลาการทดสอบ < 10 นาที
- Artifact อัปโหลดและเข้าถึงได้

## mle-reviewer

### ชื่อและการใช้งาน
ผู้เชี่ยวชาญการตรวจสอบ ML engineering ของ production มุ่งเน้นที่ data contracts, feature pipelines, การทำซ้ำการฝึก, การประเมินออฟไลน์/ออนไลน์, model serving, monitoring และ rollback

### ความสามารถ
- การตรวจสอบคุณภาพของนิยามปัญหาและการตัดสินใจ
- การตรวจสอบเมตริก ค่าขีดจำกัด และการวิเคราะห์ข้อผิดพลาด
- การตรวจสอบ data contracts และการรั่วไหล
- การตรวจสอบการทำซ้ำการฝึก
- การตรวจสอบกระบวนการประเมินและการเลื่อนขั้น
- การตรวจสอบความปลอดภัยของการ serving และการ deploy
- การวางแผน monitoring และการตอบสนองต่อเหตุการณ์

### สถานการณ์การใช้งาน
- เมื่อมีการเปลี่ยนแปลงโค้ด ML, MLOps, การฝึกโมเดล
- เมื่อมีการเปลี่ยนแปลงโค้ด inference, feature store, การประเมิน
- เมื่อตัดสินใจเลื่อนขั้นโมเดล

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ด ML และการกำหนดค่า
- Grep: ค้นหารูปแบบ
- Glob: ค้นหาไฟล์
- Bash: รัน pytest, ruff, mypy

### การทำงานร่วมกับตัวแทนอื่น
- python-reviewer จัดการสไตล์ Python, ประเภท, การจัดการข้อผิดพลาด
- pytorch-build-resolver จัดการ tensor/CUDA/การฝึกที่ล้มเหลว
- database-reviewer จัดการตาราง features, การจัดเก็บ labels
- security-reviewer จัดการ secrets, PII, ความปลอดภัยของ pickle
- performance-optimizer จัดการเวลาแฝง, หน่วยความจำ, การใช้ GPU
- build-error-resolver จัดการ CI, dependencies, native extensions ที่ล้มเหลว
- pr-test-analyzer วิเคราะห์การครอบคลุมการทดสอบ
- silent-failure-hunter ค้นพบการล้มเหลวแบบเงียบของ pipeline
- e2e-runner ดำเนินการทดสอบ E2E ของ product flows
- a11y-architect ตรวจสอบ accessibility ของ prediction
- doc-updater อัปเดตเอกสาร

### พื้นที่ตรวจสอบหลัก

#### นิยามปัญหาและคุณภาพการตัดสินใจ
- การเปลี่ยนแปลงเริ่มต้นจากการตัดสินใจของผู้ใช้หรือระบบ ไม่ใช่ความชอบสถาปัตยกรรมโมเดล
- ผู้มีส่วนได้ส่วนเสียและต้นทุนความล้มเหลวชัดเจน
- การเลือกเมตริกเป็นไปตาม error budget

#### เมตริก ค่าขีดจำกัด และการวิเคราะห์ข้อผิดพลาด
- เปรียบเทียบ baseline และพฤติกรรม production ปัจจุบัน
- ค่าขีดจำกัดและการกำหนดค่าเป็นการตัดสินใจของผลิตภัณฑ์
- ตรวจสอบโดยตรง false positives และ false negatives

#### Data Contracts และการรั่วไหล
- รายละเอียด entity, primary keys, timestamps ชัดเจน
- การแบ่งส่วนเป็นไปตามเวลา, กลุ่มผู้ใช้/entity
- การเชื่อมต่อ features ถูกต้องตามจุดเวลา
- แอตทริบิวต์ที่ละเอียดอ่อนถูก exclude หรือจัดการอย่างเหมาะสม

#### การทำซ้ำการฝึก
- การฝึกสามารถรันจากโค้ด การกำหนดค่า เวอร์ชันข้อมูล seed
- Hyperparameters, preprocessing, dependencies versions ถูกบันทึก
- Randomness และ GPU non-determinism ถูกจัดการอย่างตั้งใจ

#### การประเมินและการเลื่อนขั้น
- เมตริกเปรียบเทียบกับ baseline และโมเดล production ปัจจุบัน
- ประตูการเลื่อนขั้นถูกประกาศก่อนการเลือก
- Slice metrics ครอบคลุมคิวที่สำคัญ

#### การ serving และการ deploy
- การแปลง preprocessing ระหว่างการฝึกและ serving ถูกแชร์หรือทดสอบเทียบเท่า
- Input schema ปฏิเสธ features ที่ล้าสมัย หายไป ไม่ถูกต้อง
- เส้นทาง inference มี timeout, resource limits, fallback logic

#### Monitoring และการตอบสนองต่อเหตุการณ์
- Monitoring ครอบคลุม service health, feature drift, prediction drift
- ล็อกมี identifiers เพียงพอ
- rollback ตั้งชื่อ artifact, configuration, data dependencies ก่อนหน้า

### blockers ทั่วไป

- การแบ่ง train/test แบบสุ่มบนข้อมูลที่เกี่ยวข้องกับเวลาหรือผู้ใช้
- การสร้าง feature ใช้ฟิลด์ที่ไม่พร้อมใช้งาน ณ เวลาทำนาย
- เมตริกออฟไลน์ปรับปรุงแต่ slices หลัก regression
- Preprocessing การฝึกถูก copy แบบ manual ไปยังโค้ด serving
- เวอร์ชันโมเดลไม่อยู่ในล็อกการทำนาย

### เกณฑ์การอนุมัติ

- **APPROVE**: ไม่มีความเสี่ยง MLE ระดับ critical/high, การทดสอบที่เกี่ยวข้องหรือ evaluation gates ผ่าน
- **APPROVE WITH WARNINGS**: มีเพียงปัญหาระดับ medium, มีการดำเนินการตามมาเฉพาะ
- **BLOCK**: การรั่วไหลที่อาจเกิดขึ้นใดๆ, การเลื่อนขั้นที่ไม่สามารถทำซ้ำได้, พฤติกรรม serving ที่ไม่ปลอดภัย, การ缺าการ rollback ของ production deployment, การเปิดเผยข้อมูลที่ละเอียดอ่อน, หรือช่องว่างการประเมินที่สำคัญ

## pr-test-analyzer

### ชื่อและการใช้งาน
ตรวจสอบคุณภาพและความสมบูรณ์ของการครอบคลุมการทดสอบ Pull Request เน้นพฤติกรรมและการป้องกัน bug จริง

### ความสามารถ
- การระบุการแมปการเปลี่ยนแปลงโค้ด
- การตรวจสอบการครอบคลุมพฤติกรรม
- การประเมินคุณภาพการทดสอบ
- การค้นพบช่องว่างการครอบคลุม

### สถานการณ์การใช้งาน
- เมื่อตรวจสอบ PR
- เมื่อประเมินการครอบคลุมการทดสอบ
- เมื่อยืนยันความเพียงพอของการทดสอบ

### รายการเครื่องมือที่ใช้
- Read: อ่านโค้ดการเปลี่ยนแปลงและการทดสอบ
- Grep: ค้นหาการทดสอบ
- Glob: ค้นหาไฟล์การทดสอบ
- Bash: รันคำสั่งทดสอบ

### การทำงานร่วมกับตัวแทนอื่น
- tdd-guide ปรับปรุงการทดสอบ
- e2e-runner ดำเนินการทดสอบ end-to-end coverage
- code-reviewer ตรวจสอบคุณภาพโค้ด

### กระบวนการวิเคราะห์

#### 1. ระบุโค้ดที่เปลี่ยนแปลง
- แมปฟังก์ชัน คลาส และโมดูลที่เปลี่ยน
- ค้นหาการทดสอบที่เกี่ยวข้อง
- ระบุเส้นทางโค้ดใหม่ที่ยังไม่ได้ทดสอบ

#### 2. การครอบคลุมพฤติกรรม
- ตรวจสอบว่าฟังก์ชันแต่ละรายการมีการทดสอบหรือไม่
- ยืนยันกรณีขอบและเส้นทางข้อผิดพลาด
- ตรวจสอบว่าการบูรณาการที่สำคัญถูกครอบคลุม

#### 3. คุณภาพการทดสอบ
- ให้ความสำคัญกับการยืนยันที่มีความหมายมากกว่าการตรวจสอบที่ไม่โยน
- ทำเครื่องหมายรูปแบบที่ไม่เสถียร
- ตรวจสอบการแยกและความชัดเจนของชื่อการทดสอบ

#### 4. ช่องว่างการครอบคลุม
จัดกลุ่มตามการจัดอันดับผลกระทบ:
- critical (สำคัญ)
- important (สำคัญ)
- nice-to-have (มีได้ดี)

3. คำแนะนำการปรับปรุง
4. การสังเกตเชิงบวก
