# การประสานงานตัวแทน

## ตัวแทนที่ใช้ได้

อยู่ที่ `~/.claude/agents/`:

| ตัวแทน | วัตถุประสงค์ | เวลาที่ใช้ |
|-------|------|----------|
| planner | การวางแผนการดำเนินการ | ฟีเจอร์ที่ซับซ้อน, การปรับโครงสร้าง |
| architect | การออกแบบระบบ | การตัดสินใจสถาปัตยกรรม |
| tdd-guide | การพัฒนาที่ขับเคลื่อนด้วยการทดสอบ | ฟีเจอร์ใหม่, การแก้ไข bug |
| code-reviewer | การตรวจสอบโค้ด | หลังเขียนโค้ด |
| security-reviewer | การวิเคราะห์ความปลอดภัย | ก่อน commit |
| build-error-resolver | แก้ไขข้อผิดพลาดการสร้าง | เมื่อการสร้างล้มเหลว |
| e2e-runner | การทดสอบ end-to-end | user flows ที่สำคัญ |
| refactor-cleaner | ลบ dead code | การบำรุงรักษาโค้ด |
| doc-updater | อัปเดตเอกสาร | เมื่ออัปเดตเอกสาร |
| rust-reviewer | การตรวจสอบโค้ด Rust | โปรเจกต์ Rust |
| harmonyos-app-resolver | การพัฒนาแอป HarmonyOS | โปรเจกต์ HarmonyOS/ArkTS |

## การใช้ตัวแทนทันที

โดยไม่ต้องรอคำขอจากผู้ใช้:
1. คำขอฟีเจอร์ที่ซับซ้อน → ใช้ตัวแทน **planner**
2. เพิ่งเขียน/แก้ไขโค้ด → ใช้ตัวแทน **code-reviewer**
3. การแก้ไข bug หรือฟีเจอร์ใหม่ → ใช้ตัวแทน **tdd-guide**
4. การตัดสินใจสถาปัตยกรรม → ใช้ตัวแทน **architect**

## การรันงานขนาน

สำหรับ operations ที่เป็นอิสระต่อกัน ต้องใช้การรัน Task แบบขนานเสมอ:

```markdown
# ดี: รันขนาน
เริ่มตัวแทน 3 ตัวขนาน:
1. ตัวแทน 1: การวิเคราะห์ความปลอดภัยของโมดูลการตรวจสอบสิทธิ์
2. ตัวแทน 2: การตรวจสอบประสิทธิภาพของระบบ cache
3. ตัวแทน 3: การตรวจสอบประเภทของ utilities

# ไม่ดี: รันตามลำดับโดยไม่จำเป็น
ทำตัวแทน 1 ก่อน แล้วจึงตัวแทน 2 แล้วตัวแทน 3
```

## การวิเคราะห์หลายมุมมอง

สำหรับปัญหาที่ซับซ้อน ใช้ sub-agents แบบแบ่งบทบาท:
- ผู้ตรวจสอบข้อเท็จจริง
- วิศวกรอาวุโส
- ผู้เชี่ยวชาญด้านความปลอดภัย
- ผู้ตรวจสอบความสอดคล้อง
- ผู้ตรวจสอบความซ้ำซ้อน
