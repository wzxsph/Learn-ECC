# กฎรูปแบบการเขียนโค้ด

## ภาพรวมของกฎ

ECC Rules กฎรูปแบบการเขียนโค้ดกำหนดหลักการและแนวปฏิบัติหลักสำหรับการจัดระเบียบโค้ด กฎเหล่านี้ทำให้โค้ดคงความสามารถอ่าน ความสามารถในการบำรุงรักษา และความสอดคล้อง ครอบคลุมทุกอย่างตั้งแต่ความไม่เปลี่ยนแปลงไปจนถึงรูปแบบการตั้งชื่อ

## ข้อกำหนดหลัก

### ความไม่เปลี่ยนแปลง (หลักการสำคัญ)

**ต้อง**: สร้าง object ใหม่เสมอ ไม่แก้ไข object ที่มีอยู่

```typescript
// ตัวอย่างผิด - แก้ไข object เดิม
function modify(user, name) {
  user.name = name;  // เปลี่ยน object เดิม
  return user;
}

// ตัวอย่างถูก - ส่งคืน object ใหม่
function update(user, name) {
  return { ...user, name };  // สร้าง object ใหม่
}
```

**เหตุผล**: ข้อมูลที่ไม่เปลี่ยนแปลงป้องกันผลข้างเคียงที่ซ่อนเร้น ทำให้การ debug ง่ายขึ้น และสนับสนุน concurrency ที่ปลอดภัย

### หลักการออกแบบหลัก

| หลักการ | คำอธิบาย | การปฏิบัติ |
|------|------|------|
| **KISS** | Keep It Simple | �เลือกวิธีแก้ปัญหาที่เรียบง่ายที่สุดที่ใช้งานได้จริง หลีกเลี่ยงการ optimize ก่อนเวลาอันควร ยึดความชัดเจนมากกว่าความชาญฉลาด |
| **DRY** | Don't Repeat Yourself | แยก logic ที่ซ้ำกันออกเป็นฟังก์ชันหรือ utility classes ที่ใช้ร่วมกัน หลีกเลี่ยงการคัดลอก-paste ที่ทำให้เกิดการ drift ของ implementation |
| **YAGNI** | You Aren't Gonna Need It | ไม่สร้าง features หรือ abstractions ที่อาจต้องการในอนาคต เริ่มอย่างเรียบง่าย แล้ว refactor เมื่อจำเป็นจริงๆ |

### การจัดระเบียบไฟล์

| ข้อกำหนด | คำอธิบาย |
|------|------|
| ไฟล์เล็กจำนวนมาก > ไฟล์ใหญ่จำนวนน้อย | high cohesion, low coupling |
| จำนวนบรรทัดทั่วไป | 200-400 บรรทัด, สูงสุด 800 บรรทัด |
| แยก module ใหญ่ | แยก utility functions ออกจาก module ใหญ่ |
| การจัดระเบียบ | ตาม feature/domain ไม่ใช่ตามประเภท |

### การจัดการข้อผิดพลาด

| ข้อกำหนด | คำอธิบาย |
|------|------|
| จัดการอย่างชัดเจน | ทุกระดับต้องจัดการข้อผิดพลาดอย่างชัดเจน |
| โค้ด UI | ให้ข้อความข้อผิดพลาดที่เป็นมิตรกับผู้ใช้ |
| ฝั่งเซิร์ฟเวอร์ | บันทึกบริบทข้อผิดพลาดโดยละเอียด |
| สิ่งต้องห้าม | ไม่ดูกลืนข้อผิดพลาดโดยเงียบ |

### การตรวจสอบข้อมูลนำเข้า

| ข้อกำหนด | คำอธิบาย |
|------|------|
| การตรวจสอบขอบเขต | ตรวจสอบข้อมูลนำเข้าทั้งหมดที่ขอบเขตระบบ |
| Schema validation | ใช้การตรวจสอบแบบ Schema ที่ใช้ได้ |
| Fail fast | Fail อย่างรวดเร็วพร้อมข้อความข้อผิดพลาดที่ชัดเจน |
| ข้อมูลภายนอก | ไม่เคยไว้วางใจข้อมูลภายนอก (API responses, user input, file content) |

## รายละเอียดการนำไปใช้

### รูปแบบการตั้งชื่อ

| ประเภท | รูปแบบ | ตัวอย่าง |
|------|------|------|
| ตัวแปรและฟังก์ชัน | camelCase + ชื่อที่สื่อความหมาย | `calculateTotal`, `userName` |
| บูลีน | คำนำหน้า is/has/should/can | `isActive`, `hasPermission` |
| interface, types, components | PascalCase | `UserProfile`, `ButtonComponent` |
| ค่าคงที่ | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT`, `API_TIMEOUT` |
| custom Hooks | camelCase + คำนำหน้า use | `useUserData`, `useFetch` |

### กลิ่นโค้ด

#### การซ้อนลึกเกินไป

ควร refactor ด้วย early return เมื่อซ้อนเกิน 4 ระดับ:

```typescript
// หลีกเลี่ยง
function processOrder(order) {
  if (order) {
    if (order.items) {
      if (order.items.length > 0) {
        // ลอจิกการประมวลผล
      }
    }
  }
}

// แนะนำ
function processOrder(order) {
  if (!order || !order.items || order.items.length === 0) {
    return;
  }
  // ลอจิกการประมวลผล
}
```

#### ตัวเลขวิเศษ

ใช้ค่าคงที่ที่ตั้งชื่อแทนตัวเลขวิเศษ:

```typescript
// หลีกเลี่ยง
setTimeout(callback, 300000);

// แนะนำ
const SESSION_TIMEOUT_MS = 5 * 60 * 1000; // 5 นาที
setTimeout(callback, SESSION_TIMEOUT_MS);
```

#### ฟังก์ชันยาว

แยกฟังก์ชันใหญ่ออกเป็นฟังก์ชันย่อยที่มีหน้าที่ชัดเจน:

| ความยาวฟังก์ชัน | คำแนะนำ |
|----------|------|
| <50 บรรทัด | เหมาะสม |
| 50-100 บรรทัด | รับได้ พิจารณาแยก |
| >100 บรรทัด | ต้องแยก |

## การจัดการการละเมิด

### รายการตรวจสอบคุณภาพโค้ด

ต้องตรวจสอบก่อนทำเครื่องหมายงานเสร็จ:

- [ ] โค้ดอ่านง่ายและตั้งชื่อดี
- [ ] ฟังก์ชันสั้น (<50 บรรทัด)
- [ ] ไฟล์มีจุดประสงค์ชัดเจน (<800 บรรทัด)
- [ ] ไม่ซ้อนลึกเกิน (>4 ระดับ)
- [ ] มีการจัดการข้อผิดพลาดที่เหมาะสม
- [ ] ไม่มีค่าที่ hardcode (ใช้ค่าคงที่หรือ config)
- [ ] ใช้รูปแบบที่ไม่เปลี่ยนแปลง (ไม่มี mutation)

### การจัดระดับความรุนแรง

| ระดับ | ความหมาย | การทำงาน |
|------|------|------|
| CRITICAL | ช่องโหว่ความปลอดภัยหรือความเสี่ยงสูญเสียข้อมูล | **ป้องกัน** - ต้องแก้ไขก่อนรวม |
| HIGH | Bug หรือปัญหาคุณภาพที่สำคัญ | **เตือน** - ควรแก้ไขก่อนรวม |
| MEDIUM | ปัญหาความสามารถในการบำรุงรักษา | **แนะนำ** - พิจารณาแก้ไข |
| LOW | ปัญหารูปแบบหรือคำแนะนำเล็กน้อย | **หมายเหตุ** - เลือกได้ |

## กฎที่เกี่ยวข้อง

| กฎที่เกี่ยวข้อง | คำอธิบาย |
|----------|------|
| กฎการทดสอบ | ข้อกำหนดความครอบคลุมของการทดสอบ |
| กฎความปลอดภัย | รายการตรวจสอบความปลอดภัย |
| กฎการตรวจสอบโค้ด | มาตรฐานการตรวจสอบโดยละเอียด |
| กฎขั้นตอนการทำงานการพัฒนา | TDD และขั้นตอนการตรวจสอบโค้ด |
