# กฎการทดสอบ

## ภาพรวมของกฎ

ECC Rules กฎการทดสอบกำหนดมาตรฐานการทดสอบที่บังคับเพื่อให้มั่นใจว่าโค้ดมีคุณภาพ กฎนี้ครอบคลุมการพัฒนาที่ขับเคลื่อนด้วยการทดสอบ (TDD), ข้อกำหนดความครอบคลุมขั้นต่ำ และมาตรฐานโครงสร้างการทดสอบ ทำให้มั่นใจว่าโค้ดทั้งหมดได้รับการตรวจสอบอย่างเพียงพอ

## ข้อกำหนดหลัก

### ความครอบคลุมการทดสอบขั้นต่ำ: 80%

ต้องมีสามประเภทการทดสอบต่อไปนี้:

| ประเภทการทดสอบ | คำอธิบาย | ขอบเขต |
|----------|------|----------|
| การทดสอบหน่วย | ฟังก์ชัน, utilities, components แต่ละรายการ | Individual units |
| การทดสอบการบูรณาการ | API endpoints, การดำเนินการฐานข้อมูล | Integration points |
| การทดสอบ end-to-end | user flows ที่สำคัญ | Critical user flows |

### การพัฒนาที่ขับเคลื่อนด้วยการทดสอบ (TDD)

**ขั้นตอนการทำงานบังคับ**:

```
1. เขียนการทดสอบ (RED)   - เขียนการทดสอบที่จะล้มเหลวก่อน
2. รันการทดสอบ          - ตรวจสอบว่าการทดสอบล้มเหลว
3. เขียน implementation ขั้นต่ำ (GREEN) - เขียนโค้ดน้อยที่สุดเพื่อให้การทดสอบผ่าน
4. รันการทดสอบ          - ตรวจสอบว่าการทดสอบผ่าน
5. Refactor (IMPROVE)    - ปรับปรุงโครงสร้างโค้ด
6. ตรวจสอบความครอบคลุม        - ตรวจสอบให้ถึง 80%+
```

## รายละเอียดการนำไปใช้

### รูปแบบ AAA (Arrange-Act-Assert)

การทดสอบทั้งหมดต้อง tuân thủ โครงสร้าง AAA:

```typescript
test('คำนวณ cosine similarity ถูกต้อง', () => {
  // Arrange - เตรียมข้อมูลทดสอบ
  const vector1 = [1, 0, 0];
  const vector2 = [0, 1, 0];

  // Act - ดำเนินการที่ถูกทดสอบ
  const similarity = calculateCosineSimilarity(vector1, vector2);

  // Assert - ตรวจสอบผลลัพธ์
  expect(similarity).toBe(0);
});
```

### มาตรฐานการตั้งชื่อการทดสอบ

ใช้ชื่อที่สื่อความหมายซึ่งอธิบายพฤติกรรมที่ถูกทดสอบ:

```typescript
// แนะนำ - อธิบายพฤติกรรมที่คาดหวัง
test('เมื่อไม่มีตลาดที่ตรงกับคำค้นหาจะส่งคืน array ว่าง', () => {});
test('เมื่อ API key หายจะโยนข้อผิดพลาด', () => {});
test('เมื่อ Redis ไม่พร้อมจะ fallback ไปยังการค้นหาตัวอักษรย่อย', () => {});

// หลีกเลี่ยง - การตั้งชื่อที่คลุมเครือ
test('test1', () => {});
test('edge case', () => {});
```

### การแก้ไขปัญหาการทดสอบล้มเหลว

เมื่อการทดสอบล้มเหลว:

| ขั้นตอน | การทำงาน |
|------|------|
| 1 | ใช้ตัวแทน **tdd-guide** |
| 2 | ตรวจสอบการแยกการทดสอบ |
| 3 | ยืนยันความถูกต้องของ Mock |
| 4 | แก้ไข implementation ไม่ใช่การทดสอบ (เว้นแต่การทดสอบเองผิด) |

### การสนับสนุนตัวแทน

| ตัวแทน | วัตถุประสงค์ | เวลาที่ใช้ |
|-------|------|----------|
| **tdd-guide** | การชี้นำการพัฒนาที่ขับเคลื่อนด้วยการทดสอบ | บังคับเมื่อพัฒนาฟีเจอร์ใหม่และแก้ไขข้อผิดพลาด |
| **code-reviewer** | การตรวจสอบโค้ด | หลังเขียนโค้ดเสร็จ |

## การจัดการการละเมิด

### ความครอบคลุมไม่เพียงพอ

- โค้ดที่มีความครอบคลุมต่ำกว่า 80% ต้องไม่ถูกรวม
- CI/CD pipeline ควรป้องกันการส่งโค้ดที่ความครอบคลุมต่ำ
- ใช้ `c8` หรือเครื่องมือที่คล้ายกันเพื่อสร้างรายงานความครอบคลุม

### การจัดการการทดสอบล้มเหลว

```
1. วิเคราะห์สาเหตุของการล้มเหลว
2. วินิจฉัยว่าเป็นปัญหาของ implementation หรือการทดสอบ
3. ถ้าเป็นปัญหาของ implementation - แก้ไขโค้ด implementation
4. ถ้าเป็นปัญหาของการทดสอบ - แก้ไขโค้ดการทดสอบ
5. ตรวจสอบว่าการทดสอบทั้งหมดผ่านก่อนส่ง
```

### ปัญหาทั่วไป

| ปัญหา | สาเหตุ | วิธีแก้ไข |
|------|------|----------|
| การทดสอบ flaky | การดำเนินการ async, race conditions | เพิ่มเวลารอ, ใช้ mock |
| การแยกการทดสอบล้มเหลว | การ share state ที่ปนเปื้อน | รีเซ็ตสถานะก่อนการทดสอบแต่ละครั้ง |
| Mock ไม่ถูกต้อง | ค่าที่คาดหวังไม่ตรงกับค่าจริง | ยืนยันว่าการตั้งค่า mock ถูกต้อง |

## กฎที่เกี่ยวข้อง

| กฎที่เกี่ยวข้อง | คำอธิบาย |
|----------|------|
| กฎรูปแบบการเขียนโค้ด | ความสามารถในการอ่านและความสามารถในการบำรุงรักษาโค้ด |
| กฎความปลอดภัย | ข้อกำหนดการทดสอบความปลอดภัย |
| กฎการตรวจสอบโค้ด | การตรวจสอบความครอบคลุมการทดสอบ |
| กฎขั้นตอนการทำงานการพัฒนา | การบูรณาการขั้นตอน TDD |
