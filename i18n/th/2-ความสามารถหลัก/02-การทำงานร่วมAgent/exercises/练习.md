# 练习：Agent 协作

## 练习 1: 使用代码审查类 Agent

### 目标
学习调用代码审查 Agent，并理解其反馈机制。

### 场景
假设你编写了以下 Python 代码（不需要实际执行，只是在脑海中）：

```python
def get_user_data(user_id, include_password=False):
    user = db.query("SELECT * FROM users WHERE id = " + user_id)
    if include_password:
        return user
    return {"name": user.name, "email": user.email}
```

### 步骤

1. **分析这段代码**
   思考可能的问题：
   - SQL 注入漏洞
   - 密码泄露风险
   - 错误处理缺失

2. **选择合适的 Agent**
   对于这段代码，应该使用：
   - `security-reviewer` - 检查安全漏洞
   - `python-reviewer` - 检查 Python 最佳实践

3. **模拟 Agent 反馈**
   记录你认为 Agent 会给出的反馈：

   ```markdown
   ## 代码审查反馈

   ### 安全问题 (CRITICAL)
   - SQL 注入：直接拼接 user_id 到 SQL 语句
   - 密码泄露：include_password 选项可能暴露密码

   ### 质量问题 (HIGH)
   - 缺少错误处理
   - 缺少输入验证

   ### 改进建议
   - 使用参数化查询
   - 不要返回密码字段
   - 添加异常处理
   ```

4. **制定修复计划**
   使用 `/plan` 的思路，制定修复计划：

   ```markdown
   # 修复计划

   1. 修复 SQL 注入：使用参数化查询
   2. 移除密码返回：永远不要返回密码
   3. 添加输入验证：验证 user_id 格式
   4. 添加错误处理：处理用户不存在情况
   ```

### 验收标准
- [ ] 能识别代码中的安全问题
- [ ] 理解审查 Agent 的反馈结构
- [ ] 能制定修复计划

## 练习 2: 使用构建修复类 Agent

### 目标
学习如何使用构建 Agent 诊断和修复构建错误。

### 场景
假设 Go 项目有以下构建错误：

```
# github.com/myproject/pkg
./main.go:15: undefined: " fmt"
./main.go:23: undefined: calculateTotal
./main.go:31: too many arguments to function call
```

### 步骤

1. **分析错误类型**
   将错误分类：

   | 错误 | 类型 | 严重程度 |
   |------|------|----------|
   | undefined: " fmt" | 导入错误 | HIGH |
   | undefined: calculateTotal | 函数未定义 | HIGH |
   | too many arguments | 参数错误 | MEDIUM |

2. **设计修复流程**
   使用 Agent 协作：

   ```markdown
   # 构建修复流程

   1. /go-build - 运行构建获取完整错误
   2. /build-fix - 调用构建修复 Agent
   3. 分析错误优先级：高优先级先修
   4. 逐个修复并验证
   5. /verify - 确认构建成功
   ```

3. **为每个错误设计修复方案**

   **错误 1: undefined: " fmt"**
   ```go
   // 问题：多余的引号
   import "fmt"  // 正确写法
   
   // 修复：移除引号
   ```

   **错误 2: undefined: calculateTotal**
   ```go
   // 问题：函数未定义或命名错误
   // 修复：检查函数定义，可能需要添加或重命名
   ```

   **错误 3: too many arguments**
   ```go
   // 问题：函数调用参数数量不匹配
   // 修复：检查函数签名，修正调用参数
   ```

4. **记录修复经验**
   创建错误模式总结：

   ```markdown
   # Go 构建错误模式

   ## 常见错误
   1. undefined: 导入/函数问题
   2. too many/few arguments: 参数不匹配
   3. undefined: " xxx": 导入语法错误

   ## 修复策略
   - 先看错误数量，再看错误位置
   - 高优先级错误可能连锁导致低优先级错误
   - 修复后重新构建验证
   ```

### 验收标准
- [ ] 能分析构建错误类型和优先级
- [ ] 设计了完整的修复流程
- [ ] 记录了常见错误模式

## 练习 3: Agent 组合工作流

### 目标
设计一个复杂任务，使用多个 Agent 协作完成。

### 场景
开发一个新的用户认证模块，包含以下功能：
- 用户注册（含密码强度验证）
- 用户登录（含暴力破解防护）
- JWT token 生成和验证

### 步骤

1. **任务分解**
   将任务分解为子任务：

   ```markdown
   # 认证模块开发任务

   ## 子任务
   1. 数据库设计 (Schema)
   2. 用户注册 API
   3. 用户登录 API
   4. JWT Token 处理
   5. 安全机制（密码强度、暴力破解防护）
   ```

2. **Agent 分配**
   为每个子任务分配 Agent：

   | 子任务 | Agent | 理由 |
   |--------|-------|------|
   | 数据库设计 | architect | 需要架构视角 |
   | 注册 API | tdd-guide + security-reviewer | TDD + 安全 |
   | 登录 API | tdd-guide + security-reviewer | TDD + 安全 |
   | JWT 处理 | security-reviewer | 安全敏感 |
   | 安全机制 | security-reviewer (优先) | 安全优先 |

3. **设计工作流**

   ```
   [开始]
      ↓
   /plan (规划整体)
      ↓
   ┌─────────────────────────────────────┐
   │  并行执行:                            │
   │  - architect: 数据库设计               │
   │  - tdd-guide: 设计测试策略            │
   └─────────────────────────────────────┘
      ↓
   [实施阶段]
      ↓
   ┌─────────────────────────────────────┐
   │  串行 + 并行混合:                      │
   │  1. tdd-guide: 编写注册测试            │
   │  2. 实现注册功能                       │
   │  3. security-reviewer: 安全审查        │
   │  ──────────────────────────────────   │
   │  1. tdd-guide: 编写登录测试            │
   │  2. 实现登录功能                       │
   │  3. security-reviewer: 安全审查        │
   └─────────────────────────────────────┘
      ↓
   [整合阶段]
      ↓
   ┌─────────────────────────────────────┐
   │  - architect: JWT 架构设计            │
   │  - security-reviewer: 整体安全审查     │
   │  - code-reviewer: 代码质量审查        │
   │  - /verify: 整体验证                  │
   └─────────────────────────────────────┘
      ↓
   [完成]
   ```

4. **编写工作流文档**

   ```markdown
   # 用户认证模块开发工作流

   ## 阶段 1: 规划
   - 使用 /plan 制定整体计划
   - 使用 architect 设计数据库和 JWT 架构

   ## 阶段 2: TDD 开发
   - 使用 tdd-guide 指导测试编写
   - 先写测试，再实现功能
   - 每个子功能单独验证

   ## 阶段 3: 安全审查
   - 使用 security-reviewer 审查每个 API
   - 特别关注：密码处理、Token 安全、注入防护

   ## 阶段 4: 整合验证
   - 使用 code-reviewer 整体审查
   - 使用 /verify 验证功能完整性

   ## Agent 协作总结
   - architect: 架构设计 (1次)
   - tdd-guide: 测试指导 (2次)
   - security-reviewer: 安全审查 (3次)
   - code-reviewer: 质量审查 (1次)
   ```

### 验收标准
- [ ] 任务分解合理，包含 5+ 个子任务
- [ ] Agent 分配策略明确
- [ ] 工作流设计清晰
- [ ] 文档化工作流便于复用
