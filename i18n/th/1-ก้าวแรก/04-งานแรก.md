# 04 - ภารกิจแรก

## วัตถุประสงค์การเรียนรู้

- เข้าใจกระบวนการพัฒนาที่สมบูรณ์
- ผูกขาดกระบวนการครบ /ecc:plan → /ecc:code-review → /ecc:build-fix → commit
- สามารถพัฒนาฟังก์ชันง่ายๆ ได้ด้วยตัวเอง

---

## กระบวนการพัฒนาที่สมบูรณ์

ECC แนะนำให้ใช้กระบวนการพัฒนาดังนี้:

```
ขั้นตอนที่ 1: /ecc:plan        → วางแผนภารกิจ
ขั้นตอนที่ 2: /ecc:code-review → Review code
ขั้นตอนที่ 3: /ecc:build-fix   → แก้ไข build
ขั้นตอนที่ 4: Commit code     → Git commit
```

---

## ขั้นตอนที่ 1: ใช้ /ecc:plan วางแผน

### ข้อมูลนำเข้า

สมมติว่าเราต้อง implement ฟังก์ชันเครื่องคิดเลขง่ายๆ:

```
/ecc:plan Implement เครื่องคิดเลขที่รองรับบวก ลบ คูณ หาร 4 รูปแบบ
```

### ผลลัพธ์ที่คาดหวัง

คำสั่ง /ecc:plan จะ:

1. **ทวนความต้องการ** - "Implement เครื่องคิดเลขที่รองรับบวก ลบ คูณ หาร 4 รูปแบบพื้นฐาน..."
2. **ประเมินความเสี่ยง** - "ต้องระวัง: กรณีหารด้วยศูนย์..."
3. **แผนทีละขั้นตอน** -
   - Phase 1: สร้างฟังก์ชันหลักเครื่องคิดเลข
   - Phase 2: Implement บวก ลบ คูณ หาร 4 รูปแบบ
   - Phase 3: เพิ่มการจัดการข้อผิดพลาด
   - Phase 4: เขียน unit test
4. **รอการยืนยัน** - "กรุณายืนยันว่าสามารถเริ่มต้นได้"

### หลังยืนยัน

หลังจากผู้ใช้ยืนยัน ECC จะรอผลลัพธ์จากผู้ใช้เพื่อเริ่มต้นแต่ละ phase

### เคล็ดลับ

- ใช้โหมดไฟล์ PRD เพื่อให้ได้แผนที่ละเอียดกว่า
- ฟังก์ชันที่ซับซ้อนควรเขียน PRD ก่อนแล้วค่อยวางแผน

---

## ขั้นตอนที่ 2: ใช้ /ecc:code-review review

### ข้อมูลนำเข้า

หลังจากเขียน code เสร็จ รัน:

```
/ecc:code-review
```

### ผลลัพธ์ที่คาดหวัง

/ecc:code-review จะ:

1. รวบรวมไฟล์ที่เปลี่ยนแปลง
2. ดำเนินการ review 7 หมวด:
   - ความถูกต้อง
   - ความปลอดภัยของชนิดข้อมูล
   - การปฏิบัติตาม pattern
   - ความปลอดภัย
   - ประสิทธิภาพ
   - ความสมบูรณ์
   - ความสามารถในการบำรุงรักษา
3. สร้างรายงาน review ระดับ CRITICAL/HIGH/MEDIUM/LOW

### ตัวอย่างรายงาน review

```
## รายงาน Code Review

### CRITICAL
- [ ] Hardcoded credentials: บรรทัดที่ 23 ใน src/auth.js

### HIGH
- [ ] Function ยาวเกินไป: บรรทัด 45-78 ใน src/calculator.js (34 บรรทัด)
- [ ] ขาดการจัดการข้อผิดพลาด: function หารใน src/calculator.js

### MEDIUM
- [ ] การตั้งชื่อไม่มาตรฐาน: ตัวแปร `tmp` ควรเปลี่ยนเป็น `temporary`

### ข้อเสนอแนะ
1. ลบ hardcoded API Key
2. Refactor function หารใน calculator.js เพิ่มการตรวจสอบหารด้วยศูนย์
3. แบ่ง function ที่ยาวเกินออกเป็นส่วนเล็กๆ
```

### เคล็ดลับ

- ต้องรัน /ecc:code-review ก่อน commit เสมอ
- ปัญหา CRITICAL และ HIGH ต้องแก้ไข

---

## ขั้นตอนที่ 3: ใช้ /ecc:build-fix แก้ไข

### ข้อมูลนำเข้า

ถ้า build ล้มเหลว รัน:

```
/ecc:build-fix
```

### ผลลัพธ์ที่คาดหวัง

/ecc:build-fix จะ:

1. **ตรวจจับระบบ build** - "ตรวจพบ TypeScript project"
2. **รัน build** - "รัน tsc --noEmit..."
3. **แยกวิเคราะห์ข้อผิดพลาด** - จัดกลุ่มตามไฟล์
4. **แก้ไขทีละขั้นตอน** - แก้ไขทีละข้อผิดพลาด
5. **ตรวจสอบ** - build ใหม่หลังแต่ละการแก้ไข

### ตัวอย่างผลลัพธ์

```
[build-fix] ตรวจพบ TypeScript project
[build-fix] รัน tsc --noEmit...
[build-fix] ตรวจพบ 3 ข้อผิดพลาด:

ข้อผิดพลาดที่ 1: src/calculator.ts:15 - ชนิด 'string' ไม่สามารถกำหนดให้ชนิด 'number'
ข้อผิดพลาดที่ 2: src/calculator.ts:28 - คุณสมบัติ 'result' ไม่มีอยู่
ข้อผิดพลาดที่ 3: src/calculator.ts:42 - พารามิเตอร์ 'b' ไม่ได้ใช้งาน

[build-fix] เริ่มแก้ไขข้อผิดพลาดที่ 1...
[build-fix] แก้ไขเสร็จ กำลังตรวจสอบ...
[build-fix] Build สำเร็จ!
```

### เคล็ดลับ

- การแก้ไข build เป็นแบบ incremental แก้ไขทีละข้อผิดพลาด
- สามารถกด Ctrl+C ที่ any เวลาเพื่อหยุด

---

## ขั้นตอนที่ 4: Commit code

### ขั้นตอนการ commit

1. ตรวจสอบว่าทดสอบทั้งหมด pass
2. ตรวจสอบว่า build สำเร็จ
3. ตรวจสอบว่าปัญหา review ถูกแก้ไขแล้ว

### กฎการ commit Git

ECC ใช้รูปแบบ conventional commits:

```
<type>: <description>

[optional body]
```

**ประเภท Type**:
- `feat` - ฟังก์ชันใหม่
- `fix` - แก้ไข bug
- `refactor` - refactor
- `docs` - เอกสาร
- `test` - ทดสอบ
- `chore` - บำรุงรักษาประจำวัน
- `perf` - เพิ่มประสิทธิภาพ
- `ci` - ตั้งค่า CI

### ตัวอย่าง commit

```
feat: Implement บวก ลบ คูณ หาร 4 รูปแบบพื้นฐานสำหรับเครื่องคิดเลข

- เพิ่ม function บวก
- เพิ่ม function ลบ
- เพิ่ม function คูณ
- เพิ่ม function หาร (มีการตรวจสอบหารด้วยศูนย์)

Fixes: Issue #123
```

### เคล็ดลับ

- ใช้ `/ecc:verify quick` เพื่อตรวจสอบอย่างรวดเร็วก่อน commit
- ฟังก์ชันใหญ่ควร commit หลายครั้ง ฟังก์ชันเล็ก commit ครั้งเดียว

---

## ตัวอย่างที่สมบูรณ์

### สถานการณ์

Implement utility function จัดการ string ง่ายๆ: `truncate(str, maxLength)` ตัด string ให้สั้นลงตามความยาวที่กำหนด

### ขั้นตอนที่ 1: วางแผน

```
/ecc:plan Implement function truncate ตัด string ให้สั้นลงตามความยาวที่กำหนด
```

ผลลัพธ์:
```
## ทวนความต้องการ
Implement function truncate(str, maxLength), ส่วนที่เกิน maxLength แทนที่ด้วย ...

## ประเมินความเสี่ยง
- การจัดการเมื่อ maxLength น้อยกว่า 3
- การจัดการเมื่อ str ว่างหรือเป็น null
- การจัดการตัวอักษร Unicode

## แผนทีละขั้นตอน
1. สร้าง src/string-utils.ts
2. Implement function truncate
3. เพิ่ม test กรณีขอบเขต
4. เพิ่ม JSDoc documentation

กรุณายืนยันว่าสามารถเริ่มต้นได้
```

### ขั้นตอนที่ 2: Implement

(หลังผู้ใช้ยืนยัน ECC ช่วย implement code)

### ขั้นตอนที่ 3: Review

```
/ecc:code-review
```

ผลลัพธ์:
```
## รายงาน Code Review
สถานะ: APPROVED

### ผลการ review
- ความถูกต้อง: Pass
- ความปลอดภัยของชนิดข้อมูล: Pass
- การปฏิบัติตาม pattern: Pass
- ความปลอดภัย: Pass
- ประสิทธิภาพ: Pass
- ความสมบูรณ์: แนะนำเพิ่ม test edge cases มากขึ้น
- ความสามารถในการบำรุงรักษา: Pass

### ข้อเสนอแนะ
- เพิ่ม test string ว่าง
- เพิ่ม test null
```

### ขั้นตอนที่ 4: ตรวจสอบ build

```
/ecc:build-fix
```

```
[build-fix] ตรวจพบ TypeScript project
[build-fix] รัน tsc --noEmit...
[build-fix] Build สำเร็จ!
```

### ขั้นตอนที่ 5: Commit

```
feat: Implement function truncate ตัด string

- รองรับความยาวที่กำหนด
- ส่วนที่เกินแทนที่ด้วย ...
- จัดการ string ว่างและค่า null
```

---

## สรุป

1. **กระบวนการที่สมบูรณ์**: /ecc:plan → Implement → /ecc:code-review → /ecc:build-fix → Commit
2. **/ecc:plan** ใช้เพื่อวางแผน ตรวจสอบทิศทางที่ถูกต้อง
3. **/ecc:code-review** ใช้เพื่อ review ค้นหาปัญหา
4. **/ecc:build-fix** ใช้เพื่อแก้ไขข้อผิดพลาด build
5. **Commit** ใช้รูปแบบ conventional commits
