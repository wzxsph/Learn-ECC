## 练习 2: 编写自定义 Rule

### 目标
学习如何编写自定义 Rule 来扩展 Claude Code 的行为。

### 场景
为一个假设的项目创建一个代码审查规则。

### 步骤

1. **定义规则需求**
   你的项目需要以下规则：
   - 函数长度检查（不超过 50 行）
   - 文件长度检查（不超过 800 行）
   - 命名规范检查

2. **设计规则结构**

   ```markdown
   # 自定义规则：代码质量规则

   ## 规则名称
   code-quality

   ## 触发条件
   当用户提交代码时触发

   ## 检查项
   1. 函数长度 <= 50 行
   2. 文件长度 <= 800 行
   3. 变量命名使用 camelCase
   4. 常量命名使用 UPPER_SNAKE_CASE

   ## 违规处理
   - CRITICAL: 阻止提交
   - HIGH: 警告但允许提交
   - MEDIUM: 建议但允许提交
   ```

3. **实现规则**
   按照 Claude Code Rules 编写流程实现规则。

### 验收标准
- [ ] 理解了 Rules 的编写方法
- [ ] 创建了完整的规则文档
- [ ] 在实际项目中测试了规则

## 练习 3: Rules 组合工作流

### 目标
设计一个复杂任务，使用多个 Rules 协作完成。

### 场景
开发一个新的用户认证模块，需要以下 Rules：
- 安全规则：密码加密检查
- 代码风格规则：命名规范
- 测试规则：覆盖率要求

### 步骤

1. **任务分解**
   将任务分解为子任务，并为每个子任务分配 Rule。

2. **Rule 分配**
   | 子任务 | Rule | 理由 |
   |--------|------|------|
   | 密码处理 | 安全规则 | 安全敏感 |
   | 命名规范 | 代码风格规则 | 一致性 |
   | 测试覆盖 | 测试规则 | 质量保证 |

3. **设计工作流**
   ```
   [开始]
      ↓
   /plan (规划整体)
      ↓
   [编写代码]
      ↓
   ┌─────────────────────────────────────┐
   │  Rules 检查:                         │
   │  - 安全规则: 密码加密                 │
   │  - 代码风格: 命名规范                 │
   │  - 测试规则: 覆盖率 >= 80%           │
   └─────────────────────────────────────┘
      ↓
   [完成]
   ```

### 验收标准
- [ ] 理解了 Rules 的组合逻辑
- [ ] 设计了完整的工作流
- [ ] 文档化了工作流便于复用
