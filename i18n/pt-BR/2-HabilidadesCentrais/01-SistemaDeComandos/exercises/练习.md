## 练习 2: 组合使用多个命令

### 目标
学习如何将多个命令组合使用，创建连贯的开发工作流。

### 场景
修复一个假设的构建错误，并进行代码审查。

### 步骤

1. **模拟场景**
   假设你正在开发一个 Go 项目，遇到构建错误：

   ```
   # 场景设置（不需要实际执行）
   假设有以下构建错误需要修复：
   - import cycle detected
   - undefined: someFunction
   ```

2. **命令组合练习**
   按顺序思考并记录以下命令的组合：

   a. 首先使用 `/plan` 规划修复步骤
   b. 使用 `/go-build` 尝试构建
   c. 使用 `/build-fix` 修复错误
   d. 使用 `/code-review` 进行审查
   e. 使用 `/verify` 验证修复

3. **记录工作流**
   创建你的修复工作流文档：

   ```markdown
   ## 构建修复工作流

   1. /plan - 分析问题，制定修复计划
   2. /build-fix - 执行修复
   3. /code-review - 检查修复质量
   4. /verify - 验证功能正常
   ```

## 练习 3: 创建自定义工作流

### 目标
基于项目需求，创建自动化的工作流命令组合。

### 场景
为一个假设的 Flutter 项目创建开发工作流。

### 步骤

1. **定义工作流目标**
   你的 Flutter 项目需要以下工作流：

   - 日常开发：编码 → 测试 → 构建
   - 代码审查：自检 → 审查 → 修复
   - 发布准备：测试 → 构建 → 验证

2. **设计命令组合**

   **日常开发工作流**:
   ```
   /flutter-build → /flutter-test → /verify
   ```

   **代码审查工作流**:
   ```
   /flutter-review → /build-fix → /code-review
   ```

3. **创建工作流文档**
   在项目中创建 `CLAUDE_WORKFLOWS.md`：

   ```markdown
   # Claude Code 工作流

   ## Flutter 开发工作流

   ### 日常开发
   1. 使用 `/plan` 规划任务
   2. 使用 `/flutter-build` 构建
   3. 使用 `/flutter-test` 测试
   4. 使用 `/verify` 验证

   ### 代码审查
   1. 使用 `/flutter-review` 自检
   2. 使用 `/build-fix` 修复问题
   3. 使用 `/code-review` 请求审查

   ### 发布准备
   1. 运行完整测试套件
   2. 执行构建验证
   3. 使用 `/verify` 最终确认
   ```

4. **实践验证**
   如果有实际项目，尝试执行这个工作流。

### 验收标准
- [ ] 设计了 3 个完整的工作流
- [ ] 创建了工作流文档
- [ ] 在实际项目中测试了至少 1 个工作流

### 练习 3 工作流模板

```markdown
# [项目名] Claude Code 工作流

## 开发流程
- 日常: /plan → [编码] → /verify
- 修复: /build-fix → /code-review → /verify
- 发布: /test → /build → /verify

## 使用说明
1. 使用 `/plan` 开始任何任务
2. 编码完成后运行 `/verify`
3. 重要变更使用 `/code-review`
```
