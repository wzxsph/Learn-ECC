# 上下文预算

分析 Claude Code 会话中每个加载组件的 token 开销，并提供可操作的优化建议以释放上下文空间。

## 何时使用

- 会话运行缓慢或输出质量下降
- 最近添加了很多技能、智能体或 MCP 服务器
- 想知道实际还有多少上下文余量
- 计划添加更多组件，需要知道是否有空间
- 运行 `/context-budget` 命令（此技能为其提供支持）

## 工作原理

### 阶段 1：清单盘点

扫描所有组件目录并估算 token 消耗：

**智能体**（`agents/*.md`）
- 统计每个文件的行数和 token 数（字数 × 1.3）
- 提取 `description` frontmatter 长度
- 标记：文件 >200 行（重型）、描述 >30 词（膨胀 frontmatter）

**技能**（`skills/*/SKILL.md`）
- 统计每个 SKILL.md 的 token 数
- 标记：文件 >400 行
- 检查 `.agents/skills/` 中是否有重复副本 — 跳过相同副本以避免重复计数

**规则**（`rules/**/*.md`）
- 统计每个文件的 token 数
- 标记：文件 >100 行
- 检测同一语言模块中规则文件之间的内容重叠

**MCP 服务器**（`.mcp.json` 或活跃的 MCP 配置）
- 统计配置的服务器数量和工具总数
- 估算模式开销：每个工具约 500 token
- 标记：工具 >20 个的服务器、包装简单 CLI 命令的服务器（`gh`、`git`、`npm`、`supabase`、`vercel`）

**CLAUDE.md**（项目级 + 用户级）
- 统计 CLAUDE.md 链中每个文件的 token 数
- 标记：合并后总计 >300 行

### 阶段 2：分类

将每个组件排序到对应的桶中：

| 桶 | 标准 | 操作 |
|--------|----------|--------|
| **始终需要** | 在 CLAUDE.md 中被引用、支持活跃命令、或匹配当前项目类型 | 保留 |
| **有时需要** | 领域特定（如语言模式），在 CLAUDE.md 中未被引用 | 考虑按需激活 |
| **很少需要** | 没有命令引用、内容重叠、或没有明显项目匹配 | 移除或延迟加载 |

### 阶段 3：检测问题

识别以下问题模式：

- **膨胀的智能体描述** — frontmatter 中 description >30 词会在每次 Task 工具调用时加载
- **重型智能体** — 文件 >200 行会在每次 spawn 时膨胀 Task 工具上下文
- **冗余组件** — 重复智能体逻辑的技能、重复 CLAUDE.md 的规则
- **MCP 过度订阅** — >10 个服务器，或包装免费 CLI 工具的服务器
- **CLAUDE.md 膨胀** — 冗长的解释、过时的章节、应该是规则的指令

### 阶段 4：报告

生成上下文预算报告：

```
上下文预算报告
═══════════════════════════════════════

估算总开销：约 XX,XXX tokens
上下文模型：Claude Sonnet（200K 窗口）
有效可用上下文：约 XXX,XXX tokens（XX%）

组件分解：
┌─────────────────┬────────┬───────────┐
│ 组件            │ 数量    │ Tokens    │
├─────────────────┼────────┼───────────┤
│ 智能体          │ N      │ ~X,XXX    │
│ 技能            │ N      │ ~X,XXX    │
│ 规则            │ N      │ ~X,XXX    │
│ MCP 工具        │ N      │ ~XX,XXX   │
│ CLAUDE.md       │ N      │ ~X,XXX    │
└─────────────────┴────────┴───────────┘

发现问题（个）：
[按 token 节省量排序]

前 3 项优化：
1. [操作] → 节省约 X,XXX tokens
2. [操作] → 节省约 X,XXX tokens
3. [操作] → 节省约 X,XXX tokens

潜在节省：约 XX,XXX tokens（当前开销的 XX%）
```

在详细模式下，额外输出按文件逐一的 token 计数、 heaviest 文件的逐行分解、重叠组件之间的重复行对比，以及每个工具模式大小估算的 MCP 工具列表。

## 示例

**基础审计**
```
用户：/context-budget
技能：扫描设置 → 16 个智能体（12,400 tokens）、28 个技能（6,200）、87 个 MCP 工具（43,500）、2 个 CLAUDE.md（1,200）
      标记：3 个重型智能体、14 个 MCP 服务器（3 个可被 CLI 替代）
      最高节省：移除 3 个 MCP 服务器 → -27,500 tokens（减少 47% 开销）
```

**详细模式**
```
用户：/context-budget --verbose
技能：完整报告 + 逐文件分解，显示 planner.md（213 行，1,840 tokens）、
      每个工具大小的 MCP 工具列表、重复规则行的并排对比
```

**扩展前检查**
```
用户：我想再添加 5 个 MCP 服务器，还有空间吗？
技能：当前开销 33% → 添加 5 个服务器（约 50 个工具）会增加约 25,000 tokens → 推高到 45% 开销
      建议：先移除 2 个可被 CLI 替代的服务器以保持在 40% 以下
```

## 最佳实践

- **Token 估算**：散文用 `字数 × 1.3`，代码密集文件用 `字符数 / 4`
- **MCP 是最大的杠杆**：每个工具模式约 500 tokens；一个 30 工具的服务器比所有技能加起来还大
- **智能体描述始终加载**：即使智能体从未被调用，其描述字段也存在于每个 Task 工具上下文中
- **详细模式用于调试**：当你需要精确定位导致开销的文件时使用，而非常规审计
- **变更后审计**：在添加任何智能体、技能或 MCP 服务器后运行，以尽早发现增长
