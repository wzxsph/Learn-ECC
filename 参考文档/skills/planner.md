# 规划技能

为复杂功能和架构决策创建详细的项目计划。

## 何时激活

- 用户请求实现计划或路线图
- 需要分解的多步骤项目
- 架构决策需要记录
- PRD（产品需求文档）创建
- 系统设计文档
- 任务分解和时间线规划

## 工作流程

### 步骤 1：理解需求

与用户澄清以下内容：
1. **目标** - 我们要构建什么？
2. **约束** - 时间表、预算、技术限制？
3. **质量标准** - 覆盖率、可维护性要求？
4. **风险** - 已知的未知因素？

### 步骤 2：研究

1. 检查现有的代码库和模式
2. 搜索类似的实现以获取灵感
3. 查阅相关的库/框架文档
4. 识别技术债务和依赖

### 步骤 3：创建计划

生成的文档应包含：

#### 概述
- 目标陈述
- 背景/动机
- 假设

#### 系统架构
- 组件图
- 数据流
- API 契约

#### 实施计划

按优先级排序的任务列表：

```
## 阶段 1：基础（1-2 周）
- [ ] 任务 1.1
- [ ] 任务 1.2
- [ ] 任务 1.3

## 阶段 2：核心功能（2-3 周）
- [ ] 任务 2.1
- [ ] 任务 2.2
- [ ] 任务 2.3

## 阶段 3：完善（1 周）
- [ ] 任务 3.1
```

#### 风险和缓解

```
| 风险 | 影响 | 可能性 | 缓解策略 |
|------|------|--------|----------|
| API 变更 | 高 | 中 | 版本控制和平滑迁移 |
| 性能问题 | 中 | 低 | 早期性能测试 |
```

#### 验收标准

- [ ] 功能标准 1
- [ ] 功能标准 2
- [ ] 非功能标准（性能、安全等）

## 计划格式

### PRD 格式

```markdown
# 产品需求文档：[功能名称]

## 概述
[功能是什么以及为什么重要]

## 用户故事
- 身为 [用户类型]，当 [场景] 时，我想要 [功能]
- ...

## 功能需求
1. [需求 1]
2. [需求 2]

## 非功能需求
- 性能：[要求]
- 安全性：[要求]
- 可用性：[要求]

## 排除范围
- [明确排除的内容]

## 依赖
- [外部依赖]
- [内部依赖]
```

### 技术设计格式

```markdown
# 技术设计文档：[变更名称]

## 概述
[高层描述]

## 背景
[为什么需要这个变更]

## 详细设计
### 组件 A
[设计]

### 组件 B
[设计]

## 数据模型
[ schema 变更]

## API 设计
[端点定义]

## 实施计划
[阶段和里程碑]

## 测试策略
[测试方法]

## 回滚计划
[如何回滚]
```

## 输出位置

将计划文档保存到：
- `docs/planning/` 目录
- 使用描述性名称：`feature-name-prd.md` 或 `feature-name-design.md`

## 质量检查

完成计划后：

- [ ] 所有假设已记录
- [ ] 所有依赖已识别
- [ ] 风险已评估
- [ ] 时间线是现实的
- [ ] 验收标准是具体的
- [ ] 文档格式一致
