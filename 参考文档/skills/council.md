# 委员会

为模糊决策召集四位顾问：
- 上下文中的Claude声音
- 怀疑者子智能体
- 务实者子智能体
- 批评者子智能体

这是用于**模糊下的决策**，不是代码审查、实施规划或架构设计。

## 何时使用

在以下情况使用委员会：
- 一个决策有多条可信路径且没有明显赢家
- 需要明确展示权衡
- 用户要求第二意见、分歧或多个视角
- 存在会话锚定风险
- go/no-go决策需要对抗性挑战

示例：
- monorepo vs polyrepo
- 现在发布 vs 等待完善
- 功能开关 vs 全面推出
- 简化范围 vs 保持战略广度

## 何时不使用

| 不要用委员会 | 应该用 |
| --- | --- |
| 验证输出是否正确 | `santa-method` |
| 将功能分解为实施步骤 | `planner` |
| 设计系统架构 | `architect` |
| 审查代码的bug或安全问题 | `code-reviewer`或`santa-method` |
| 直接的事实问题 | 直接回答 |
| 明显的执行任务 | 直接完成任务 |

## 角色

| 声音 | 视角 |
| --- | --- |
| 架构师 | 正确性、可维护性、长期影响 |
| 怀疑者 | 前提挑战、简化、假设打破 |
| 务实者 | 发布速度、用户影响、运营现实 |
| 批评者 | 边界情况、下行风险、失败模式 |

三个外部声音应作为全新子智能体启动，**仅获取问题和相关上下文**，而非完整持续会话的对话历史。这是反锚定机制。

## 工作流

### 1. 提取真正的问题

将决策简化为一个明确的提示：
- 我们在决定什么？
- 哪些约束是重要的？
- 什么算成功？

如果问题模糊，在召集委员会前先问一个澄清问题。

### 2. 仅收集必要的上下文

如果决策特定于代码库：
- 收集相关文件、片段、问题文本或指标
- 保持简洁
- 只包含做决策所需的上下文

如果决策是战略性的/一般的：
- 除非实质性地改变答案，否则跳过代码库片段

### 3. 首先形成架构师立场

在阅读其他声音之前，先写下：
- 你的初始立场
- 支持它的三个最强理由
- 你首选路径的主要风险

首先这样做，这样综合结果就不会简单地镜像外部声音。

### 4. 并行启动三个独立声音

每个子智能体获取：
- 决策问题
- 紧凑的上下文（如需要）
- 严格角色
- 无不必要的对话历史

提示格式：

```text
你是四人决策委员会中的[角色]。

问题：
[决策问题]

上下文：
[仅相关片段或约束]

回复：
1. 立场 — 1-2句话
2. 推理 — 3个简洁要点
3. 风险 — 你推荐中最大的风险
4. 意外 — 其他声音可能错过的一件事

直接。不要对冲。控制在300字以内。
```

角色强调：
- 怀疑者：挑战框架、质疑假设、提出最简单可信的替代方案
- 务实者：为速度、简单性和现实世界执行优化
- 批评者：展示下行风险、边界情况和计划可能失败的原因

### 5. 使用护栏进行综合

你既是参与者也是综合者，因此使用这些规则：
- 不要在没有解释的情况下驳回外部观点
- 如果外部声音改变了你的建议，明确说出来
- 始终包含最强烈的反对意见，即使你拒绝它
- 如果两个声音与你的初始立场一致，将其视为真实信号
- 在 verdict 之前保持原始立场可见

### 6. 呈现简洁的裁决

使用以下输出格式：

```markdown
## 委员会：[简短决策标题]

**架构师：** [1-2句话立场]
[一行原因]

**怀疑者：** [1-2句话立场]
[一行原因]

**务实者：** [1-2句话立场]
[一行原因]

**批评者：** [1-2句话立场]
[一行原因]

### 裁决
- **共识：** [它们一致的地方]
- **最强烈的异议：** [最重要的分歧]
- **前提检查：** [怀疑者是否挑战了问题本身？]
- **建议：** [综合后的路径]
```

保持手机屏幕上可扫描。

## 持久化规则

**不要**从此技能向`~/.claude/notes`或其他隐藏路径写临时笔记。

如果委员会实质性地改变了建议：
- 使用`knowledge-ops`将经验教训存储到正确的持久位置
- 或如果结果属于会话内存，则使用`/save-session`
- 或如果决策改变了活跃执行事实，则直接更新相关GitHub/Linear问题

只有在改变真实事物时才持久化决策。

## 多轮后续

默认是一轮。

如果用户想要另一轮：
- 保持新问题聚焦
- 仅在必要时包含之前的裁决
- 尽可能保持怀疑者独立，以保留反锚定价值

## 反模式

- 将委员会用于代码审查
- 在任务只是实施工作时使用委员会
- 向子智能体提供完整对话记录
- 在最终裁决中隐藏分歧
- 将每个决策都作为笔记持久化

## 相关技能

- `santa-method` — 对抗性验证
- `knowledge-ops` — 正确持久化持久决策增量
- `search-first` — 需要时在委员会前收集外部参考材料
- `architecture-decision-records` — 当决策成为长期系统策略时将其正式化

## 示例

问题：

```text
我们应该现在发布ECC 2.0作为alpha，还是等到控制平面UI更完善后再发布？
```

可能的委员会结构：
- 架构师推动结构完整性和避免混乱的表面
- 怀疑者质疑UI是否真的是限制因素
- 务实者问现在可以发布什么而不损害信任
- 批评者关注支持负担、预期债务和推出混淆

价值不在于 unanimity（一致）。价值在于在做选择前使分歧清晰可见。
